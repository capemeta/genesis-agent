# Genesis Agent — 产品分发架构设计

> 版本：v1.0  
> 适用范围：整体 Monorepo 结构、三产品边界划分、文件系统设计、Product Profile 体系

---

## 一、核心判断：三个产品，一个内核

Genesis Agent 本质上是**一个共享内核 + 三条产品线**。三个产品面向不同的用户、运行环境、安全边界、持久化形态和 UI 形态，如果放在同一个 binary 中打包，会带来：

- **冗余代码**：CLI 二进制包含企业版 Docker 沙箱 SDK；企业版包含 macOS Seatbelt 代码
- **依赖污染**：桌面版 GUI 框架进入 CLI；企业版 PostgreSQL / OIDC / RBAC 适配进入个人工具
- **配置混乱**：三套权限策略、工具集、沙箱机制、持久化策略无法独立收敛
- **发布困难**：无法独立版本、独立发布、独立开源

**结论：必须拆分产品入口和产品装配，但当前阶段不拆多仓库、不拆多 Go module。**
采用一个 GitHub 仓库、一个 Go module、多个 `cmd/<product>` 入口，通过 `products/` 层隔离产品特有代码。独立打包靠 Go import graph 保证：某个产品入口不 import 的实现，不会进入该产品二进制。

## 二、三产品定位与边界

### 2.1 genesis-cli（命令行 Agent）

**定位**：个人本地使用的 Claude Code-like CLI Agent

| 维度 | 说明 |
|---|---|
| **用户** | 开发者、个人效率用户 |
| **运行环境** | 本地宿主机（macOS/Linux/Windows） |
| **UI** | 终端 TUI |
| **文件系统** | 宿主机文件系统（WorkDir 权限限制） |
| **沙箱** | 轻量 OS 原生沙箱（macOS Seatbelt / Linux bwrap+Landlock / Windows JobObject） |
| **工具集** | 文件读写、bash 执行、搜索、网络请求 |
| **多 Agent** | 单 Agent 为主，支持基础 sub-agent（task delegation） |
| **分发** | 单二进制，跨平台（macOS/Linux/Windows arm64/amd64） |
| **无需** | Docker、数据库、RBAC、多租户、企业 SSO |

### 2.2 genesis-desktop（桌面工作台 Agent）

**定位**：面向个人/团队的桌面 GUI Agent，未来演化为 OpenClaw-like 产品，多 Agent 编排中枢

| 维度 | 说明 |
|---|---|
| **用户** | 个人高级用户、小团队、Agent 开发者 |
| **运行环境** | 本地桌面（macOS/Windows/Linux） |
| **UI** | 桌面 GUI（Wails = Go + Web 前端嵌入） |
| **文件系统** | 宿主机文件系统（GUI 文件选择器授权 + WorkDir） |
| **沙箱** | 同 CLI 轻量沙箱（宿主机直接执行，选择性隔离） |
| **工具集** | 全套（文件/执行/搜索/网络/MCP/桌面原生集成） |
| **多 Agent** | **核心差异化**：丰富的多 Agent 编排支持（规划/委派/并行/反思/人在回路/长期运行） |
| **分发** | 桌面安装包（DMG/EXE/AppImage） |
| **特有** | 多 Agent 可视化画布、任务看板、会话历史、本地记忆管理 |

### 2.3 genesis-enterprise（企业版 Web 平台）

**定位**：企业级 AI Agent 管理与执行平台，OpenClaw-like 企业产品底座

| 维度 | 说明 |
|---|---|
| **用户** | 企业、团队、平台管理员 |
| **运行环境** | 服务器/云（Linux） |
| **UI** | Web 前端（React + Ant Design Pro） |
| **文件系统** | **只通过 genesis-sandbox API 操作**（不直接访问宿主机 FS） |
| **沙箱** | Docker/gVisor（genesis-sandbox 独立服务） |
| **工具集** | 企业工具集（受治理、RBAC 过滤后的工具子集） |
| **多 Agent** | 支持多 Agent 协作，服务端编排 |
| **分发** | Docker 镜像 / K8s Helm Chart |
| **特有** | 多租户、RBAC、审计日志、工具治理、使用量统计、SSO、Agent 配置管理 |

## 三、文件系统设计反思

### 3.1 原设计（错误）

前期设计将 `LocalFileSystemBackend` 和 `SandboxFileSystemBackend` 都放在 `internal/capabilities/filesystem/backend/` 下：

```text
internal/capabilities/filesystem/
  backend/
    interface.go              ← OK
    local/local.go            ← 错误：这是 CLI/Desktop 特有的
    sandbox/sandbox.go        ← 错误：这是 Enterprise 特有的
```

**问题**：
- 企业版二进制打包时会包含 `os.ReadFile` 等本地 FS 调用（完全不需要）
- CLI 二进制会包含 genesis-sandbox HTTP client（不应存在）
- OS 级沙箱代码（Seatbelt/bwrap）和 Docker 沙箱客户端混在同一个包里

### 3.2 正确设计：接口在 core，实现在 products

**接口（产品无关）放在 `internal/capabilities/`**：
- `FileSystemBackend` interface
- `SandboxRunner` interface
- 权限检查器 interface

**实现（产品特有）放在 `products/<product>/`**：
- CLI/Desktop: `LocalFileSystemBackend` + `LightweightSandboxRunner`（Seatbelt/bwrap）
- Enterprise: `SandboxFileSystemBackend` + `SandboxServiceRunner`（genesis-sandbox client）

### 3.3 CLI 和 Desktop 的 local backend 共享

CLI 和 Desktop 共用同一套本地 FS 后端实现，通过 `shared/local/` 共享：

```text
products/
  local/                    ← CLI 和 Desktop 共享的本地执行能力
    fs_backend/             ← LocalFileSystemBackend
    sandbox/                ← 轻量沙箱（Seatbelt/bwrap/JobObject）
    workdir/                ← WorkDir 管理
```

## 四、Product Profile 体系设计

### 4.1 为什么需要 Product Profile

三个产品和任意自定义 Agent 都需要**配置不同的提示词、工具、技能、权限组合**。这个配置体系需要是可编程、可热更新、可继承的 Profile 系统。

```
系统内置 Profile（代码 hardcode）
  └── ProductProfile（products/<product>/profile/）
       └── PlatformProfile（企业管理员配置，只有 Enterprise 有）
            └── AgentProfile（数据库/配置文件中的 Agent 配置）
                 └── SessionProfile（运行时会话级临时覆盖）
```

规则：**高优先级覆盖低优先级**（同 Claude Code 的配置分层逻辑）。

### 4.2 Profile 结构

```go
// internal/capabilities/profile/model/profile.go
type Profile struct {
    ID          string
    Name        string
    Description string

    Scope       CapabilityScope   // 能力适用范围：端、租户、项目、Agent、用户、角色、运行环境
    Tools       ToolSet           // 工具集配置
    Skills      SkillSet          // 技能集配置
    Permissions PermissionPolicy  // 权限策略（是否允许执行，是否需要确认/审批）
    Prompts     PromptConfig      // 提示词配置
    Memory      MemoryConfig      // 记忆配置
    Sandbox     SandboxPolicy     // 沙箱策略
    LLM         LLMConfig         // LLM 配置
    MultiAgent  MultiAgentConfig  // 多 Agent 配置
}

type CapabilityScope struct {
    Channels     []ChannelType        // cli / desktop / enterprise / api / webhook
    TenantIDs    []string             // 为空表示不按租户限制，实际查询仍必须带 tenant_id
    ProjectIDs   []string
    AgentIDs     []string
    UserIDs      []string
    RoleIDs      []string
    Environments []RuntimeEnvironment // local / desktop / server / sandbox
}

type ToolSet struct {
    Enabled  []string                    // 启用的工具（支持通配符 "filesystem.*"）
    Disabled []string                    // 显式禁用（优先级高于 Enabled）
    Config   map[string]json.RawMessage  // 工具特定配置
}

type PermissionPolicy struct {
    Mode         PermissionMode  // default/acceptEdits/dontAsk/yolo/bypass
    AllowedPaths []PathRule
    BlockedPaths []PathRule
    AllowedCmds  []CommandRule
    BlockedCmds  []CommandRule
}

type MultiAgentConfig struct {
    SupportedPatterns []MultiAgentPattern  // 支持的编排模式
    MaxConcurrent     int
    MaxDepth          int                  // 子 Agent 嵌套深度上限
}
```

### 4.3 三个产品的默认 Profile 对比

| 能力 | CLI Profile | Desktop Profile | Enterprise Profile |
|---|---|---|---|
| **工具集** | filesystem.* + execution.bash + search.* + network.fetch | 全套（含桌面集成工具） | 受治理工具集（沙箱内可用） |
| **沙箱** | LightweightSandbox（可选） | LightweightSandbox（可选） | SandboxService（强制） |
| **权限模式** | default（WorkDir 限制） | acceptEdits（WorkDir 内自动允许） | dontAsk（企业策略） |
| **多 Agent** | BasicDelegation | Full（所有模式） | ManagedDelegation（受治理） |
| **记忆** | 本地文件 | 本地文件 + 向量索引 | 数据库持久化 |
| **提示词** | CLI 系统提示词 | Desktop 系统提示词 | 企业系统提示词 + 租户自定义 |


### 4.4 持久化 Profile 与产品装配

CLI、Desktop、Enterprise 对数据库的需求不同，但上层业务不应该知道自己使用的是内存、文件、SQLite 还是 PostgreSQL。持久化边界按接口隔离：

```text
internal/capabilities/*/contract      # Store / Repository 接口
internal/capabilities/*/adapter       # memory / file / sql 等可复用实现
internal/platform/db                  # DB / Tx / Dialect / Migration 基础设施

shared/local/persistence       # SQLite、本地路径、轻量迁移（CLI/Desktop 可复用）
products/enterprise/persistence  # PostgreSQL、连接池、企业迁移、租户约束
```

装配规则：

| 产品 | 默认持久化 | 说明 |
|---|---|---|
| CLI | memory / file，可选 SQLite | 个人本地工具优先轻量，允许完全无 DB 运行 |
| Desktop | SQLite | 本地会话、任务看板、多 Agent 图状态、记忆索引优先落本地库 |
| Enterprise | PostgreSQL | 多租户、RBAC、审计、Usage、任务恢复等必须使用服务端数据库 |

约束：`runtime`、`app`、`domain` 不 import 任何具体 DB driver；产品入口通过 `products/<product>/bootstrap` 选择具体 Store 实现。CLI 不 import PostgreSQL，Enterprise 不 import SQLite 本地路径管理，二进制即可独立裁剪。

## 六、依赖方向与产品隔离约束

```
cmd/<product>/main.go
  └── products/<product>/bootstrap/
       ├── shared/local/          ← 仅 CLI/Desktop 可引入
       │     └── internal/capabilities/ ← backend interface 实现
       ├── products/enterprise/     ← 仅 Enterprise 可引入
       │     └── internal/capabilities/ ← sandbox interface 实现
       ├── internal/runtime/
       ├── internal/capabilities/        ← 接口 + 工具（产品无关）
       ├── internal/platform/
       └── products/<product>/internal/<interfaces|command|tui>/
```

### 硬性隔离约束

1. `internal/capabilities/` **不能** import `products/` 任何包
2. `internal/runtime/` **不能** import `products/` 任何包
3. `products/cli/` **不能** import `products/enterprise/`
4. `products/enterprise/` **不能** import `shared/local/`（企业版绝对不用本地 FS）
5. `shared/local/` **可以**被 `products/cli/` 和 `products/desktop/` 共同 import
6. `internal/domain/` 不 import 任何外部框架
7. `internal/bootstrap` 只能保留产品无关的共享 builder，不能同时 import CLI/Desktop/Enterprise 的具体实现
8. 产品入口必须只 import 自己的 `products/<product>/bootstrap`，由该 bootstrap 选择具体持久化、Trace、Auth、Sandbox 适配

## 七、文件系统操作数据流

### CLI/Desktop 路径
```
ReadFileTool.call(input)
  → PermissionEngine.Check("read_file", path)  # 路径权限检查
  → FileSystemBackend.Read(ctx, path, opts)     # 接口调用
  → LocalFileSystemBackend.Read()               # shared/local/ 实现
  → os.ReadFile(path)                           # 读本地文件
```

### Enterprise 路径
```
ReadFileTool.call(input)
  → PermissionEngine.Check("read_file", path)   # RBAC 过滤 + 路径检查
  → FileSystemBackend.Read(ctx, path, opts)      # 接口调用
  → SandboxFileSystemBackend.Read()             # products/enterprise/ 实现
  → genesis-sandbox HTTP: GET /v1/sandboxes/{id}/files/download
```

## 八、多 Agent 编排边界

Desktop 和 Enterprise 都需要多 Agent，但两者的运行环境、持久化、Trace、治理策略并不相同。因此多 Agent 不能简单地“全部放 Desktop”，也不能把所有实现硬做成一套不可替换的公共服务。

正确边界是：**核心协议与状态机通用，产品级存储、Trace、投影和治理适配由 products 注入。**

### 8.1 通用核心放在 internal/runtime/multiagent

`internal/runtime/multiagent` 只承载产品无关的编排内核：

```text
internal/runtime/multiagent/
  contract/              # Orchestrator / Store / Tracer / Scheduler 等端口接口
  model/                 # AgentNode / Edge / Task / Delegation / Handoff / GraphSnapshot
  event/                 # agent.started / agent.completed / handoff.created / graph.updated
  orchestrator/          # 通用编排状态机，不绑定 DB / UI / HTTP
  patterns/              # plan-execute / reflection / parallel / handoff / supervisor
```

通用核心只依赖端口接口，不直接依赖 SQLite、PostgreSQL、HTTP Handler、Wails、前端画布或企业 RBAC。这样 Desktop 和 Enterprise 可以共享编排语义，但不会共享不该共享的基础设施实现。

### 8.2 产品适配放在 products

| 产品 | 适配重点 | 落点 |
|---|---|---|
| CLI | 基础 sub-agent / task delegation，通常内存或文件状态即可 | `products/cli/bootstrap` |
| Desktop | 可视化画布、任务看板、本地 SQLite 投影、桌面通知 | `products/desktop/multiagent` |
| Enterprise | PostgreSQL 状态持久化、分布式调度、租户隔离、RBAC、审计、OpenTelemetry Trace | `products/enterprise/multiagent` |

Desktop 的核心差异化不是“拥有唯一的多 Agent 编排实现”，而是“围绕通用多 Agent 事件提供最强的本地可视化与交互体验”。Enterprise 的核心差异化是“围绕同一套通用编排语义提供服务端治理、审计、恢复和多租户隔离”。

### 8.3 支持的编排模式

| 模式 | 通用核心 | 产品差异 |
|---|---|---|
| Plan-Execute | Planner 拆解任务，Executor 执行任务 | Desktop 展示计划树；Enterprise 持久化任务和审计 |
| Reflection | Executor + Critic 迭代改进 | Desktop 展示反思链路；Enterprise 记录 Trace 与质量评估 |
| Parallel Fan-out | 并行启动子 Agent，汇聚结果 | Desktop 展示并行分支；Enterprise 处理队列、限流和重试 |
| Handoff | Agent 间移交上下文和控制权 | Desktop 展示交接边；Enterprise 按 RBAC 校验接收方 |
| Human-in-the-Loop | 任意节点等待人工确认/输入 | Desktop 本地弹窗/通知；Enterprise 审批流和 SSE/Webhook |
| Supervisor | 监控子 Agent 状态，失败时重新分配 | Desktop 本地恢复；Enterprise 分布式恢复和告警 |

### 8.4 多 Agent 可视化画布

Desktop 的画布只消费通用多 Agent 事件或 GraphSnapshot，不反向成为编排内核的依赖：

```text
internal/runtime/multiagent/event
  → products/desktop/multiagent/projection
  → web/desktop 画布 / 任务看板
```

这样 Enterprise 可以复用同一批事件生成服务端 Trace、审计日志和管理控制台视图，而不需要 import Desktop 代码。

## 九、编码落地纪律

本节用于约束后续编码，避免目录正确但依赖重新耦合。

### 9.1 Policy / Permission / RBAC 分层

当前文档中会出现多个 policy/permission 目录，它们语义不同，不能互相替代：

| 层级 | 位置 | 职责 |
|---|---|---|
| 工具权限 | `internal/capabilities/tool/permission` | 单次工具调用的 Allow / Ask / Deny、运行时授权、人工确认 |
| 命令策略 | `internal/capabilities/execution/policy` | 命令前缀、参数规则、执行风险、shell 级限制 |
| 能力治理 | `internal/capabilities/policy` | CapabilityScope、RBAC / ABAC 通用模型、跨 Tool/MCP/Skill/Sandbox 的可见性与治理 |
| 企业 RBAC 存储 | `products/enterprise/rbac/postgres` | 企业版角色、权限、租户策略的 PostgreSQL 适配 |

规则：
- `CapabilityScope` 解决“当前上下文是否可见/可用”。
- `PermissionPolicy` 解决“这次执行是否允许，以及是否需要确认/审批”。
- RBAC / ABAC 解决“当前身份对资源和动作是否有权限”。
- RateLimit / Quota 解决“允许用多少”。

四者可以组合，但不能合并成一个巨大 Policy 对象。

### 9.2 Adapter 必须由产品 bootstrap 注入

`internal/capabilities/*/adapter` 可以放可复用适配器，但通用 service 不能为了方便直接 import 所有 adapter。例如：

```text
允许：
products/enterprise/bootstrap
  → sandbox/adapter/http.NewClient(...)
  → 注入 capabilities/sandbox/service

禁止：
internal/capabilities/sandbox/service
  → import adapter/http + adapter/grpc
  → 根据配置自己选择实现
```

原因：后者会让 CLI 即使不使用 sandbox，也可能因为共享 service 的 import graph 把 HTTP/gRPC sandbox client 打进二进制。

### 9.3 SQLite 是 CLI 的可选能力，不是默认依赖

CLI 默认应优先支持 memory / file 持久化，保持单二进制轻量。SQLite 对 CLI 是可选增强能力，落地时可采用以下方式之一：

```text
cmd/genesis-cli              # 默认，不 import sqlite driver
cmd/genesis-cli-sqlite       # 可选入口，import local/persistence/sqlite
// 或 build tag: sqlite
```

Desktop 默认可以使用 SQLite，因为它天然需要本地会话、任务看板、多 Agent 图状态和本地记忆索引。Enterprise 默认使用 PostgreSQL。

### 9.4 MultiAgent patterns 只依赖端口

`internal/runtime/multiagent/patterns` 可以承载通用编排模式，但只能依赖端口接口：

```text
允许依赖：
- Store / GraphStore
- Tracer / EventSink
- Scheduler / QueuePort
- InterventionPort
- AgentInvoker / ToolGateway

禁止依赖：
- SQLite / PostgreSQL driver
- HTTP Handler / SSE Handler
- Wails / Desktop UI
- Enterprise RBAC adapter
- OpenTelemetry SDK 具体实现
```

Desktop 和 Enterprise 的差异通过 `products/<product>/multiagent` 注入端口实现，不在 patterns 中分支判断产品类型。

### 9.5 产品入口是唯一 Composition Root

每个产品的 `cmd/<product>/main.go` 必须只 import 自己的产品 bootstrap：

```go
// CLI
import "genesis-agent/products/cli/bootstrap"

// Enterprise
import "genesis-agent/products/enterprise/bootstrap"
```

`internal/bootstrap` 只能提供产品无关的共享构建函数，例如创建 logger、HTTP client、配置解析辅助、基础 registry builder；不能 import 产品实现、DB driver、Wails、OIDC、PostgreSQL、SQLite 或 sandbox 具体 adapter。

## 十、构建脚本设计

```powershell
# 分产品构建：每个产品入口只 import 自己的 bootstrap
New-Item -ItemType Directory -Force bin | Out-Null
go build -o bin/genesis-cli.exe ./cmd/genesis-cli
go build -o bin/genesis-enterprise.exe ./cmd/genesis-enterprise

# Desktop 由 Wails 打包，仍应由 cmd/genesis-desktop 或 Wails 入口只 import desktop bootstrap
wails build -o genesis-desktop.exe

# 分产品测试
go test ./cmd/genesis-cli/... ./products/cli/... ./shared/local/...
go test ./cmd/genesis-enterprise/... ./products/enterprise/...

# 验证产品依赖隔离：检查 import graph，而不是只 grep 源码目录
$cliDeps = go list -deps ./cmd/genesis-cli
$enterpriseDeps = go list -deps ./cmd/genesis-enterprise

$cliForbidden = @(
  'genesis-agent/products/enterprise',
  'github.com/jackc/pgx',
  'github.com/lib/pq',
  'github.com/wailsapp/wails'
)
foreach ($pattern in $cliForbidden) {
  if ($cliDeps | Select-String -SimpleMatch $pattern -Quiet) {
    throw "genesis-cli 引入了禁止依赖: $pattern"
  }
}

$enterpriseForbidden = @(
  'genesis-agent/shared/local',
  'genesis-agent/products/desktop',
  'github.com/wailsapp/wails'
)
foreach ($pattern in $enterpriseForbidden) {
  if ($enterpriseDeps | Select-String -SimpleMatch $pattern -Quiet) {
    throw "genesis-enterprise 引入了禁止依赖: $pattern"
  }
}
```

后续可将上述检查固化为 `scripts/check_product_isolation.ps1` 或 Go 小工具，并在 CI 中对每个产品入口执行。

### 10.1 安装包配置分发契约

CLI 的 Windows 安装包应采用外置配置目录，安装后的基础布局如下：

```text
Genesis Agent/
  genesis-cli.exe
  configs/
    config.yaml
    llm.yaml
    mcp.yaml
    hooks.yaml
```

`configs/` 与可执行文件同级，属于**安装级基线配置**：它可随安装包升级，也可由运维在不重新构建二进制的情况下调整。它不是项目配置，不能继续与源码仓库中的“项目共享 `configs/`”混为一个概念；项目共享配置应由工作区配置来源承载，项目私有值继续放其本地覆盖文件。

二进制可以使用 `go:embed` 持有一份只读默认模板，作用仅限于外置配置缺失时的保底或安装器首次物化。真正运行时的优先级必须让**外置的安装目录 `configs/` 覆盖二进制内置默认值**。外置文件存在但格式非法、权限不允许或校验失败时必须报错，不能静默回退到内置值，否则会把部署错误伪装成正常运行。

目标优先级从低到高应为：

```text
二进制内置默认值
  < 安装目录 configs/
  < 用户级 ~/.genesis-agent/<product>/config.yaml
  < 工作区项目共享配置
  < 工作区项目本地覆盖
  < AGENT_ 显式运行时覆盖
```

当前实现尚未完成“未传 `--config` 时按 `os.Executable()` 自动定位同级 `configs/`”及通用工作区配置发现；默认 `configs` 仍相对于进程工作目录。发布前必须补齐该解析器和对应测试，不能仅依赖启动脚本先 `cd` 到安装目录的偶然行为。

## 十一、关键决策汇总

| 决策点 | 结论 | 理由 |
|---|---|---|
| 是否拆分为独立仓库？ | **不拆，Monorepo + products 层隔离** | 共享代码维护成本低；隔离通过 import 约束实现 |
| CLI 和 Desktop 的 local backend 是否共享？ | **共享（shared/local/）** | 本地 FS 和轻量沙箱实现完全相同 |
| 文件系统 backend 实现放在哪里？ | **products/ 各产品目录** | 不放 internal/，确保产品隔离和二进制无冗余 |
| 持久化实现如何隔离？ | **Store 契约在 internal，SQLite/PostgreSQL 装配在 products** | CLI 可无 DB，Desktop 可 SQLite，Enterprise 使用 PostgreSQL |
| 多 Agent 核心放在哪里？ | **核心放 internal/runtime/multiagent，产品适配放 products** | Desktop 和 Enterprise 共享编排语义，但存储、Trace、治理、可视化不同 |
| Product Profile 如何实现？ | **Profile 继承链 + CapabilityScope + Gateway 二次校验** | Profile 管默认值，可见性和执行权限不能混为一谈 |
| Adapter 如何装配？ | **只能由产品 bootstrap 选择并注入** | 避免 internal service import 所有 adapter 导致二进制污染 |
| CLI 是否默认引入 SQLite？ | **不默认，SQLite 是可选增强** | 保持 CLI 轻量，Desktop 默认可用 SQLite |
| 企业版是否访问宿主机 FS？ | **绝对不允许（安全红线）** | 全部走 genesis-sandbox |
| Windows 轻量沙箱怎么办？ | **Job Object（弱）+ 推荐 WSL2** | Windows 内核沙箱能力弱 |
