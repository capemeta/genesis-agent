# 网络搜索与网络获取工具设计

## 一、目标与边界

本文设计 Genesis Agent 的两个网络研究工具：

- `web_search`：面向公开互联网或受治理搜索源的检索工具，返回结构化搜索结果与来源元数据。
- `web_fetch`：面向公开网页内容获取、正文抽取、Markdown 化和可引用片段提取的只读工具。

两者不要与现有 `http_request` 混为一谈：

| 工具 | 定位 | 典型输入 | 典型输出 | 是否推荐带认证 |
|---|---|---|---|---|
| `http_request` | 通用 HTTP/API 调用 | method、url/base_url、headers、body、connection_ref | 原始 HTTP 状态、Header、Body | 是，优先通过 `connection_ref` |
| `web_search` | 研究检索 | query、domain filters、recency、limit | 搜索结果列表、来源、摘要、时间 | 否，搜索 provider 凭证由平台配置 |
| `web_fetch` | 网页获取与内容抽取 | url、prompt/format、limits | Markdown/文本、标题、来源、引用片段 | 默认否，仅允许公开网页 |

核心目标：

1. 统一能力域实现，避免 CLI / Desktop / Enterprise 三套重复工具。
2. 产品侧只负责选择 provider、治理策略、默认启停、凭证注入和用户交互方式。
3. 从第一版就把 SSRF、防重定向绕过、大小限制、超时、审计、缓存、来源引用做成一等能力。
4. 为后续 Hosted Search、MCP Search、企业搜索网关、浏览器渲染抓取、PDF/图片内容抽取留下接口。

## 三、总体架构

### 3.1 搜索 Provider 落地策略

Provider 不建议按“所有搜索 API 平铺”组织，而应先按部署/地域/合规域分组，再在组内做优先级选择。原因是 CLI、Desktop、Enterprise 的默认可用 provider 不一定相同：开源私有化优先考虑 self-hosted，国际公网优先考虑成熟商业 API，中国区企业客户需要能走本地云厂商、备案、审计和数据出境边界。

推荐 provider family 与 adapter id：

```text
web_search providers
├── self_hosted
│   └── searxng
├── international
│   ├── brave
│   ├── tavily
│   ├── exa
│   └── serpapi
└── china
    └── baidu_ai_search
```

这里的“流行度排行”不是互联网搜索引擎用户量排行，而是从 AI Agent/开发者集成角度综合判断：文档成熟度、Agent 适配度、商业可用性、可自部署性、生态出现频率、合规风险、是否仍对新用户开放。排序用于 Genesis 的接入优先级，不代表搜索质量的绝对排名。

| 分组 | 优先级 | Adapter ID | Provider | 推荐定位 | Genesis 落地建议 |
|---|---|---|---|---|---|
| self_hosted | P0 | `searxng` | SearXNG | 企业/私有化元搜索 | 第一批实现。作为 Enterprise/self-hosted 默认候选，要求产品 bootstrap 提供 `base_url`，不默认依赖公共实例。 |
| international | P0 | `brave` | Brave Search API | 通用公网搜索 API | 第一批实现。适合 CLI/Desktop/云部署的默认商业 provider。 |
| international | P1 | `tavily` | Tavily | Agent 搜索/搜索+抽取 | 第一批或第二批实现。适合研究型 Agent；注意 `include_raw_content` 和 advanced search 的成本/延迟。 |
| international | P1 | `exa` | Exa | 语义搜索/研究搜索 | 第二批实现。适合深度研究、论文/公司/人物搜索，不建议作为普通搜索默认 provider。 |
| international | P2 | `serpapi` | SerpAPI | SERP 聚合 | 可选 adapter。只在用户明确配置 key 后启用，Enterprise 默认需法务/合规确认。 |
| china | P1 | `baidu_ai_search` | 百度 AI Search / 百度智能云搜索能力 | 中国区搜索与 AI 搜索候选 | 中国区第一版唯一实现 adapter。百度搜索生态强，但开放 API 形态需按当期智能云产品核实；不作为未配置默认。 |
| legacy | P2 | `google_cse` | Google Custom Search JSON API | 存量 Google Programmable Search | 不作为新项目默认接入。仅为存量客户保留 adapter 设计位置。 |
| legacy | P3 | `bing_web` | 传统 Bing Web Search API | 传统 Bing Web Search | 不实现传统 Bing Web Search API。未来如接 Microsoft，走 `azure_bing_grounding` hosted adapter。 |
| fallback | P3 | `duckduckgo_html` | DuckDuckGo HTML | 无 key fallback | 仅用于本地开发/CLI best-effort fallback，不能作为 Enterprise provider。 |

组内选择建议：

1. Self-hosted 优先：`searxng`。适合企业、私有化、内网代理和可审计部署，也是 Enterprise 最稳的默认候选。
2. International 通用优先：`brave`。它是最适合第一版通用公网搜索的商业 API adapter。
3. Agent 搜索增强：`tavily`。适合需要 answer/raw content/domain filter 的研究型 Agent。
4. 深度语义研究：`exa`。适合 research/company/paper/news 等高阶搜索，不作为普通搜索默认。
5. 聚合 SERP：`serpapi`。覆盖面广，但合规/稳定性/成本边界更复杂，默认不启用。
6. China 组：第一版只实现 `baidu_ai_search` 一个 adapter，降低国内 provider 适配面；实现前必须核实百度智能云当期 API 文档、开通方式、计费、返回结构、数据出境和内容合规要求。腾讯、阿里、火山先不建目录，后续有明确客户/产品需求再补。
7. Legacy：`google_cse`、`bing_web` 不进入新项目主路径。Google Custom Search JSON API 文档已标注不向新客户开放并将在 2027-01-01 停用；传统 Bing Web Search API 已进入 previous versions 路径。
8. Fallback：`duckduckgo_html` 只能是 local/dev fallback，metadata 必须标记 `BestEffort=true`、`Unofficial=true`。

Provider 引入边界：

- `internal/capabilities/web/contract` 定义统一请求/响应、Provider metadata、Credential schema、Health check 和能力声明。
- 每个 provider adapter 只负责把 Genesis contract 映射到外部 API，不做权限判断、不读产品配置、不直接写审计。
- provider API key 从产品 bootstrap 注入的 credential resolver 获取，adapter 不直接读取环境变量或配置文件。
- allow/deny domain、safe search、region、locale、max result、raw content 等策略由 `SearchService` 合并产品配置与工具输入后下发。
- provider 返回的原始响应只在 debug trace 中按策略采样，默认不写普通日志。
- 所有外部 provider 都必须支持 `ValidateCredentials(ctx)` 或 `HealthCheck(ctx)`，用于管理端配置验证。
- 中国区 provider 必须额外记录 `region=china`、`data_residency`、`content_compliance`、`egress_policy` 等 metadata，Enterprise 侧才能做租户策略和审计。

### 3.2 Web Fetch 后端选型

`web_fetch` 不建议一开始绑定 Crawl4AI、Firecrawl 或 Playwright。它们都可以私有化，但定位不同，应作为可插拔 fetch/crawl backend，而不是核心依赖。

| Backend | 能否私有化 | 定位 | Genesis 建议 |
|---|---|---|---|
| `http` | 是，内置 Go 实现 | 普通网页 GET、正文抽取、Markdown 化 | 第一版默认实现，必须把 SSRF、DNS 私网校验、redirect、大小限制、审计做好。 |
| `jina` | 可以，Jina Reader 有开源/自托管形态，也可走 SaaS | URL -> LLM-friendly Markdown/Text，适合 Reader 服务 | 作为 Phase 2 可选 adapter；不作为默认依赖。 |
| `crawl4ai` | 可以，开源 Python 包，支持 CLI、Docker/FastAPI server、Playwright 浏览器能力 | LLM-friendly crawler/scraper，适合私有化、复杂抽取、深度 crawl | 作为 Phase 2/3 可选自托管 adapter；更适合 `web_crawl` 或高级 `web_fetch(render_mode=crawl4ai)`。 |
| `firecrawl` | 可以，官方支持 self-host，也可走 SaaS | search/scrape/crawl/batch/structured extract，能力完整但组件较重 | 作为 Enterprise/云可选 adapter；适合复杂抓取和批量 crawl，不进入默认构建路径。 |
| `playwright` | 可以，官方 Docker 镜像/远程 server，可自建浏览器 worker | JS 渲染、登录态、点击、等待 selector、截图 | 只放 `adapter/fetch/browser/playwright`，必须运行在 sandbox/browser worker，不嵌入主进程。 |

选择规则：

1. 默认永远先走 `http` fetcher，失败或用户明确要求渲染/复杂抽取时再升级 backend。
2. `crawl4ai` 和 `firecrawl` 都不应放进 `internal/capabilities/web` 的核心 service 依赖，只通过 adapter 接口接入。
3. `crawl4ai` 是 Python 生态，Genesis 不直接 import Python 包；推荐以独立服务方式接入，如 `CRAWL4AI_BASE_URL` + token。
4. `firecrawl` 比 `crawl4ai` 更像完整服务平台，适合 Enterprise 部署，但需要把 LLM key、队列、Playwright service、代理和资源限制纳入运维边界。
5. `playwright` 只解决“浏览器渲染/交互”，不负责搜索、策略、缓存和审计；这些仍由 Genesis `FetchService` 管。
6. 所有外部 fetch backend 都必须复用统一 `WebPolicy`、URL 安全校验、审计、Usage 和租户限流，不能让 adapter 自行绕过策略。

约束：

- `internal/capabilities/web` 不 import `products/*`、`shared/local/*`。
- `runtime` 不直接依赖 web provider、HTTP client、浏览器或搜索服务。
- 产品差异只通过 bootstrap 注入：provider、policy、cache、approval requester、是否允许 live web、是否允许 browser render。
- `http_request` 继续保留在 `builtin`，但后续可迁出到 `internal/capabilities/http/tool/http_request`；本次不强制搬迁，避免扩大改动面。

## 四、核心接口设计

### 4.1 领域模型

建议模型保持结构化，工具最后统一 JSON 输出：

```go
type SearchRequest struct {
    TenantID        string
    Query           string
    Limit           int
    RecencyDays     int
    AllowedDomains  []string
    BlockedDomains  []string
    Locale          string
    Region          string
    SafeSearch      string
    Mode            SearchMode // disabled/cached/indexed/live
    ContextSize     string
}

type SearchResult struct {
    Query     string
    Provider  string
    Results   []SearchHit
    Cached    bool
    Duration  time.Duration
}

type SearchHit struct {
    Title       string
    URL         string
    Snippet     string
    Source      string
    PublishedAt *time.Time
    Rank        int
}

type FetchRequest struct {
    TenantID        string
    URL             string
    Prompt          string
    Format          FetchFormat // markdown/text/html/metadata
    MaxBytes        int64
    MaxChars        int
    Timeout         time.Duration
    AllowedDomains  []string
    BlockedDomains  []string
    FollowRedirects bool
    RenderMode      RenderMode // http/browser/sandbox
}

type FetchResult struct {
    URL          string
    FinalURL     string
    StatusCode   int
    ContentType  string
    Title        string
    Description  string
    Markdown     string
    Text         string
    Citations    []Citation
    Bytes        int
    Cached       bool
    Duration     time.Duration
}
```

### 4.2 端口接口

```go
type Searcher interface {
    Search(ctx context.Context, req SearchRequest) (SearchResult, error)
}

type Fetcher interface {
    Fetch(ctx context.Context, req FetchRequest) (FetchedDocument, error)
}

type Extractor interface {
    Extract(ctx context.Context, doc FetchedDocument, req FetchRequest) (ExtractedDocument, error)
}

type Summarizer interface {
    Summarize(ctx context.Context, input SummarizeRequest) (SummarizeResult, error)
}

type WebPolicy interface {
    AuthorizeSearch(ctx context.Context, req SearchRequest) (PolicyDecision, error)
    AuthorizeFetch(ctx context.Context, req FetchRequest) (PolicyDecision, error)
}
```

设计取舍：

- `web_fetch` 不应直接依赖具体 LLM adapter；如果支持 `prompt` 后处理，通过 `Summarizer` port 注入。
- 第一版可以不启用 `Summarizer`，直接返回 Markdown + metadata，让 Agent 自己读结果并总结。
- `Searcher` provider 可以多实现，`SearchService` 根据配置选择 primary/fallback，而不是在 tool 内写分支。
- `Fetcher` 和 `Extractor` 分离，后续 browser render、PDF、正文抽取算法可以独立替换。

## 五、工具 Schema

### 5.1 `web_search`

输入：

```json
{
  "query": "string",
  "limit": 5,
  "recency_days": 30,
  "allowed_domains": ["example.com"],
  "blocked_domains": ["spam.example"],
  "locale": "zh-CN",
  "region": "CN",
  "safe_search": "moderate"
}
```

约束：

- `query` 必填，允许短 query，不做过强长度校验；空字符串拒绝。
- `allowed_domains` 与 `blocked_domains` 可以共存，但 deny 优先；与 Kode 不同，Genesis 可以更灵活，但必须在文档说明。
- `limit` 默认 5，最大 20。
- 输出必须提醒模型使用 Markdown 链接引用来源。

输出：

```json
{
  "query": "...",
  "provider": "duckduckgo",
  "duration_ms": 1234,
  "cached": false,
  "results": [
    {
      "rank": 1,
      "title": "...",
      "url": "https://...",
      "snippet": "...",
      "source": "example.com",
      "published_at": "2026-07-03T00:00:00Z"
    }
  ]
}
```

### 5.2 `web_fetch`

输入：

```json
{
  "url": "https://example.com/page",
  "prompt": "提取安装步骤",
  "format": "markdown",
  "max_bytes": 4194304,
  "max_chars": 20000,
  "allowed_domains": ["example.com"],
  "blocked_domains": [],
  "follow_redirects": false,
  "render_mode": "http"
}
```

约束：

- 仅允许 `http` / `https`。
- 禁止 URL username/password。
- 禁止 localhost、loopback、link-local、private IPv4、ULA/private IPv6、metadata IP、裸 IP 可按策略默认拒绝。
- DNS 解析后也要校验目标 IP，避免域名指向内网。
- 每次重定向后重新执行 URL/IP/domain 策略校验。
- 默认不自动跨 host 跟随重定向；返回 `redirect_required` 结果，让模型或用户显式再 fetch。
- 默认最大响应体沿用 `http_client.max_response_body_bytes`，工具可配置更小上限。
- HTML 转 Markdown 后再按 `max_chars` 截断，返回 `truncated=true`。

输出：

```json
{
  "url": "https://example.com/page",
  "final_url": "https://example.com/page",
  "status_code": 200,
  "content_type": "text/html; charset=utf-8",
  "title": "...",
  "description": "...",
  "markdown": "...",
  "citations": [
    {
      "url": "https://example.com/page",
      "title": "...",
      "start_offset": 0,
      "end_offset": 280
    }
  ],
  "bytes": 12345,
  "truncated": false,
  "cached": false,
  "duration_ms": 820
}
```

## 六、产品装配策略

### 6.1 统一部分

放在 `internal/capabilities/web`：

- 输入 schema 与工具实现。
- Search/Fetch service、provider/fetcher/extractor/cache/policy 端口。
- URL 安全校验、domain match、内容大小限制、缓存 key、结构化结果。
- 工具 traits：`ReadOnly=true`、`ConcurrencySafe=true`、`NeedsPermission=true`。
- 单元测试：schema、domain filter、SSRF、redirect、extractor、cache、service fallback。

放在 `internal/bootstrap`：

- 增加 `BuildOptions.Web`，允许产品注入 `web.Options`。
- 只在 options 声明启用或配置存在时注册 `web_search`/`web_fetch`。
- 不在 shared builder 里写 CLI/Desktop/Enterprise 语义。

建议装配契约：

```go
type WebBuildOptions struct {
    Enabled        bool
    Searcher       webcontract.Searcher
    Fetcher        webcontract.Fetcher
    Extractor      webcontract.Extractor
    Summarizer     webcontract.Summarizer
    Policy         webcontract.WebPolicy
    Cache          webcontract.Cache
    RegisterSearch bool
    RegisterFetch  bool
}
```

规则：

- `Enabled=false` 时 shared builder 不注册任何 web 工具。
- `RegisterSearch=true` 但 `Searcher=nil` 时注册失败并返回清晰错误，不提供静默空实现。
- `RegisterFetch=true` 时必须提供 `Fetcher` 与 `Extractor`；`Summarizer` 可选。
- 产品 profile 仍是最终可见性开关：即使工具已注册，也必须在 `Profile.Tools.Enabled` 中出现才会暴露给模型。
- Enterprise 不应依赖“注册但 profile 禁用”作为唯一安全边界，还必须在 `WebPolicy` 或 Gateway Authorizer 中做租户/域名/角色校验。

### 6.2 CLI

CLI 推荐策略：

- 默认启用 `web_search` 与 `web_fetch`，但网络访问需要审批。
- provider 默认优先：`duckduckgo` 或配置化 `brave/bing`；没有 provider 时 `web_search` 返回清晰错误。
- `web_fetch` 默认 `http` fetcher，不启用 browser render。
- 权限粒度借鉴 Kode：
  - `web_fetch(domain:example.com)`
  - `web_fetch(domain:*.example.com)`
  - `web_search(query:golang release)`
  - `web_search`
  - deny 优先于 allow。
- CLI approval requester 复用终端交互，quiet 模式默认 deny 或仅允许显式 allowlist。

需要改动：

- `products\cli\bootstrap\container.go`：构建 web service/tool，追加到 AdditionalTools。
- `products\cli\internal\profile\default_profile.go`：加入 `web_search`、`web_fetch`。
- 后续 CLI config 增加 web provider、domain allowlist、默认 mode。

### 6.3 Desktop

Desktop 推荐策略：

- 复用同一套 `web_search`/`web_fetch` 工具和 service。
- 可新增 browser render adapter，利用桌面内置浏览器或 Wails/native 能力处理需要 JS 渲染的页面。
- 用户审批使用桌面弹窗或权限中心，而不是终端 requester。
- 本地缓存可使用用户目录，支持按工作区隔离。

需要改动：

- `products\desktop\bootstrap`：真正实现产品 Composition Root。
- `products\desktop\internal`：提供 Desktop approval requester、browser render fetcher、用户缓存路径解析。

### 6.4 Enterprise

Enterprise 推荐策略：

- 默认不直接开放任意 live web。
- `web_search` 优先接企业搜索网关、合规搜索 API、受控 MCP search 或代理服务。
- `web_fetch` 走企业 egress proxy / sandbox fetcher，强制租户策略、域名 allowlist、审计和限流。
- 所有请求必须带 `tenant_id`，策略按租户/角色/Agent/项目评估。
- 返回结果中不记录完整正文到普通日志；审计只记录 URL、domain、status、bytes、hash、request_id。

需要改动：

- `products\enterprise\bootstrap\container.go`：不再只调用 shared builder 默认能力，增加企业 web provider/policy 注入。
- `products\enterprise\internal\profile\default_profile.go`：按治理策略启用 `web_search`/`web_fetch`。
- 后续 Enterprise HTTP 管理 API：维护 provider、domain policy、租户限流、审计查询。

## 七、安全与治理设计

### 7.1 SSRF 与网络边界

`web_fetch` 必须实现以下防护：

- scheme allowlist：仅 `http` / `https`。
- 禁止 URL credentials：`https://user:pass@example.com`。
- host 规范化：小写、去尾点、IDN punycode 处理。
- DNS 解析后拦截：
  - loopback：`127.0.0.0/8`、`::1`
  - private：`10/8`、`172.16/12`、`192.168/16`、IPv6 ULA
  - link-local：`169.254/16`、`fe80::/10`
  - carrier-grade NAT：`100.64/10`
  - metadata：`169.254.169.254` 等云厂商元数据地址
  - multicast、unspecified、reserved
- redirect 每跳重新校验，限制最大跳数。
- Enterprise 可要求所有 fetch 走代理或 sandbox，禁止服务端直连公网。

### 7.2 权限模型

建议新增 Web 专用 authorizer，挂到 Gateway `Authorizer` 或 Web service policy：

| 能力 | 权限 key | 说明 |
|---|---|---|
| `web_search` | `web_search` | 允许任意搜索。 |
| `web_search` | `web_search(query:<normalized-query>)` | 允许指定 query。 |
| `web_search` | `web_search(domain:<domain>)` | 允许结果限定在指定域。 |
| `web_fetch` | `web_fetch(domain:<domain>)` | 允许抓取指定域。 |
| `web_fetch` | `web_fetch(domain:*.example.com)` | 允许子域通配。 |
| `web_fetch` | `web_fetch(url:<normalized-url>)` | 允许指定 URL，一次性授权。 |

规则：

- deny 优先。
- project/user/tenant/role/agent 多级策略后续纳入统一能力适用范围配置。
- CLI/Desktop 可 ask 用户；Enterprise 默认不 ask，走预配置策略。

### 7.3 审计、Usage、Trace

审计字段：

- tool name、tenant_id、agent_id、session_id、request_id。
- query hash / query preview，避免完整敏感 query 直接进日志。
- URL、domain、final_url、status_code、bytes、duration_ms、provider、cached。
- policy decision、deny reason、redirect reason、truncated。

Usage 字段：

- provider、result_count、response_bytes、cache_hit、duration_ms、success。

Trace span：

- `web.search`
- `web.fetch`
- `web.extract`
- `web.summarize`

## 八、缓存与内容处理

缓存分层：

| 缓存 | 作用 | 默认 |
|---|---|---|
| 会话内 memory cache | 避免同一轮重复 fetch/search | 默认开启 |
| 本地 file cache | CLI/Desktop 可选，减少重复抓取 | 默认关闭或短 TTL |
| 企业共享 cache | Enterprise 可接 Redis/DB，带租户隔离 | 后续 |

内容处理：

- HTML 使用正文抽取 + Markdown 转换，保留标题、canonical URL、description。
- `text/plain`、`text/markdown` 直接读取并按大小截断。
- JSON 默认不作为 `web_fetch` 主场景；如果用户要 API，提示使用 `http_request`。
- PDF、图片、视频先返回 unsupported content type，后续由 media/document extractor 扩展。
- 版权合规：工具只返回必要片段和摘要，避免长篇逐字复制；完整引用由 Agent 响应阶段控制。

## 九、实施计划

### Phase 1：最小可用版本

- 新增 `internal/capabilities/web/contract` 与模型。

- 新增 `tool/web_search`、`tool/web_fetch`。

- Enterprise 第一版默认不启用任意公网 live web；只有配置了企业 search gateway / egress proxy / domain policy 后，才在 profile 中显式开放 `web_search` 或 `web_fetch`。
- 单元测试覆盖 schema、URL 校验、私网拦截、redirect、domain filter、fetch size limit、cache hit。

### Phase 2：产品治理完善

- Gateway Authorizer 接入 WebPolicy，统一 domain/query 权限。
- Enterprise bootstrap 注入企业 egress proxy/search gateway。
- 增加 config：
  - `web.mode: disabled|cached|indexed|live`
  - `web.search.provider`
  - `web.fetch.max_bytes`
  - `web.policy.allowed_domains`
  - `web.policy.blocked_domains`
  - `web.search.user_location`
  - `web.search.context_size`
- 增加审计/usage 专用 metadata。
- CLI/Desktop 增加授权缓存与配置命令。

### Phase 3：高级能力

- Hosted Search adapter：对接 OpenAI/Anthropic/企业模型托管搜索能力。
- Jina/Crawl4AI/Firecrawl fetch adapter：以外部服务方式接入，支持私有化部署或 SaaS endpoint。
- MCP Search adapter：将 MCP search/fetch provider 纳入同一 `Searcher`/`Fetcher`。
- Browser render fetcher：Desktop 或 sandbox 内通过 Playwright 渲染 JS 页面。
- PDF/文档/图片 extractor。
- 搜索 open/find/page cache：借鉴 Codex `web.run`，可支持 `open(ref_id)`、`find(ref_id, pattern)`。

## 十一、最佳实践结论

推荐方案是“统一能力域 + 产品注入策略”：

- `web_search` 和 `web_fetch` 都属于 `internal/capabilities/web`，不是 CLI/Desktop/Enterprise 各自实现。
- provider、policy、cache、approval、browser render、enterprise gateway 通过产品 bootstrap 注入。
- `http_request` 保持底层 API 工具定位，不承载网页研究语义。
- `web_fetch` 第一版返回 Markdown/metadata/citation，由 Agent 总结；需要 prompt 后处理时通过 `Summarizer` port 注入，避免工具直接绑定 LLM。
- Enterprise 的默认 posture 是受控搜索/受控抓取，不默认开放任意公网 live web。

## 十二、review-fix-rereview 审查记录

- 需要明确 `web_fetch` 与 `http_request` 的职责边界，避免后续实现把网页抓取做成万能 HTTP 的别名。

修复结果：

- 已加入目标边界、参考源码清单、统一架构、产品装配、安全治理、实施计划和测试策略。
