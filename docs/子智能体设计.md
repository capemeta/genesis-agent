## 一、设计目标与硬原则

### 1.1 要解决的问题

Genesis 需要「主 Agent 按需启子 Agent」能力，覆盖：

1. **Agent-as-Tool**：主循环通过固定网关工具委派专家子任务（探索、规划、审查、领域专家）。
2. **用户自定义 SubAgent**：类似 Claude Code / Kode 的 Markdown Agent 模板（prompt、工具白名单、模型、权限）。
3. **产品统一与差异**：CLI / Desktop / Enterprise 共享编排语义与协议，但存储、治理、可视化、审批各自演进。
4. **与 Skill 边界清晰**：Skill 是知识/流程包；SubAgent 是可独立跑完一轮（或多轮）的执行单元。二者可组合，不可混名。

### 1.2 硬原则（开发纪律）

| ID | 原则 |
| --- | --- |
| P1 | **开发阶段不兼容半吊子实现**：现有预留、占位、错误方向的代码，开发时该删就删、该重写就重写；不为「已经写过」保留兼容层。 |
| P2 | **代码保持干净与最佳实践**：优先清晰边界、可测试契约、可替换端口；禁止为赶工留下双轨 API、死字段、假实现。 |
| P3 | **LLM function schema 只出现 Tool 名**：禁止把 `explore`、`office-reviewer` 等 SubAgent 名注册为 Tool；委派必须经固定网关（见 §4）。 |
| P4 | **子 Agent 权限只收紧不放宽**：子实例可见工具 / Skill / MCP / Sandbox / 审批模式 ⊆ 父上下文 ∩ 定义白名单。 |
| P5 | **内核产品无关**：`internal/runtime/multiagent` 与 `internal/capabilities/subagent` 不 import CLI/Desktop/Enterprise、不绑 SQLite/PostgreSQL/Wails。 |
| P6 | **产品差异只在 bootstrap / products 注入**：Store、Source、审批上抛、UI 投影、租户 RBAC 由产品装配。 |
| P7 | **不做半吊子子 Run**：`Skill(context=fork)` 必须走统一 `Task` / Controller 路径，不另开旁路。 |
| P8 | **事件、审计、Usage、日志分离且委派可见**：Progress 给 UX；`agent.log` 给排障；Audit/Usage/Trace 给治理；transcript 给 resume。启动子智能体时 **Progress 与 agent 日志都必须有**；日志顶层不得漏 `run_id`、`session_id`。 |

### 1.3 非目标（本设计明确不做）

- 不把 Kode 的 Ink UI / Claude 路径硬编码（`.claude/agents` 可作为可选兼容 Source，不是内核依赖）。
- 不照搬 Codex 的 v1+v2 双栈；Genesis **只设计一套**委派工具面与事件模型。
- 不在 Phase 1 实现完整 @Agent 协作空间 / Swarm / Group Chat（模式文档已有；本设计只预留扩展点）。
- 不把 SubAgent 名塞进 Tool Registry 冒充「专家工具」。
- 不默认给每个子 Agent 独立宿主机 cwd；工作区语义仍走现有 PathContract / Sandbox（可后续加 per-agent workspace 扩展）。

## 二、概念模型

### 2.1 三层对象

```text
SubAgentDefinition（定义 / 模板）
  = name + description + system_prompt + tools/skills/mcp 策略
    + model + permission_mode + sandbox_profile + metadata
  → 可来自内置 / 用户文件 / 项目 / Plugin / Marketplace / 企业库

SubAgentInstance（运行实例）
  = agent_id + parent_run_id + child_run_id + definition_ref + status + transcript
    + depth + created_from(Task|SkillFork|System)
  → 一次可恢复的子执行句柄
  → `agent_id`：委派句柄（resume / TaskOutput / 审计）
  → `child_run_id`：子 RunEngine 的 run_id（Trace / Usage）
  → Mention **不**直接创建 Instance；只提示父模型调用 `Task`（created_from 仍为 Task）

Delegation（委派动作）
  = 父 Run 通过网关工具发起的 spawn / followup / wait / stop / resume
  → 工具结果回到父 ReAct 观察；进度走 Progress Event
```

| 概念 | 类比 Kode | 类比 Codex | Genesis 落点 |
| --- | --- | --- | --- |
| Definition | `.kode/agents/*.md` | `agents.roles` + role.toml | `SubAgentDefinition` + Source |
| Instance | `agentId` + sidechain jsonl | `ThreadId` + rollout | `SubAgentInstance` + Store |
| Control plane | TaskTool + backgroundTasks | `AgentControl` + ThreadManager | `runtime/multiagent/controller`（Phase 1 名 Controller；patterns/Orchestrator 二期） |
| Gateway tool | `Task` / `TaskOutput` / `TaskStop` | `spawn_agent` / `followup_task` / `wait_agent` | 固定工具族 `Task*`（§4） |
| Catalog | Task prompt 动态列出 agents | spawn 工具 description 列 roles | `Task` description 挂 Catalog |

### 2.2 与 Skill / Tool / Package 的关系

```text
Package / Plugin
  └─ 可投影 CapabilityIndex：skill | tool | mcp | subagent | ...

Skill（知识包）
  ├─ context=inline → 注入主线程（已实现）
  └─ context=fork  → 解析为「按 Skill 元数据 spawn SubAgent」（本设计落地后）

SubAgent（执行单元）
  └─ 独立子 Run（复用 RunEngine / 策略），工具经 ToolGateway，权限只收紧

Tool（原语）
  └─ 唯一委派入口（已锁定）：`Task` / `TaskOutput` / `TaskStop`
```

**禁止**：

- Skill 名 = Tool 名 = SubAgent 名 混用。
- `Skill` 工具内部偷偷 `exec` 半吊子子循环而不走 SubAgent Controller。
- Marketplace 安装了 `subagent` capability 却没有 RuntimeAdapter 时静默「假装可用」。

### 2.3 与「多 Agent 协作空间」的边界

| 能力 | 本文（SubAgent） | `Agent设计模式与多Agent协作空间设计.md` |
| --- | --- | --- |
| 主控委派专家子任务 | ✅ 核心 | Orchestrator + Subagent / Agent-as-Tool |
| 用户自定义 Agent 模板 | ✅ 核心 | 配置层输入 |
| 并行 fan-out / wait | ✅ Phase 1–2 | Parallel 模式 |
| Handoff / Supervisor / Swarm / @Agent 房间 | 预留端口与事件 | 后续专项 |
| Desktop 画布 / Enterprise 分布式调度 | 产品适配 | 产品分发文档 §8 |

**结论**：本文是「可落地的 SubAgent 运行时与协议」；协作空间是其上的编排模式与产品体验，不在此重复展开。

## 三、参考实现结论（深度借鉴）

### 3.1 Kode-CLI：Claude Code 式自定义 SubAgent（优先对齐用户体验）

**核心做法**：Markdown Agent 模板 + `Task(subagent_type, prompt, ...)` Agent-as-Tool；子循环嵌套 `query()`；`agentId` 可 resume；编排工具对子 Agent 硬禁。

**应借鉴**：

1. **定义即数据**：frontmatter + body，多源合并（built-in → plugin → user → project → CLI → policy）。
2. **单一委派网关**：`Task`，动态 catalog 进工具 description。
3. **双层工具过滤**：定义白/黑名单 + 全局 orchestration 硬禁。
4. **权限只收紧**。
5. **可选 forkContext**（默认不共享全历史）。
6. **同步 + 后台 + resume**（`TaskOutput` / `TaskStop`）。
7. **Mention 不硬 spawn**：提示父模型走 `Task`（可与 Genesis `SelectForTurn` / SystemReminder 对齐）。
8. **`SubagentStop` 独立生命周期事件**。

**应避免**：

- CLI `Task` 与 SDK `task_run` 双命名并存。
- 子 Agent 与父共享 cwd 却无明确工作区契约文档。
- Ink/React 渲染逻辑渗入内核。
- Swarm best-effort 静默吞失败。

### 3.2 Codex：控制平面与会话树（优先对齐工程结构）

**核心做法**：子代理 = 独立 Thread；会话级 `AgentControl`；role = config layer；depth/concurrency 硬限制；协作事件一等公民；审批可上抛父会话。

**应借鉴**：

1. **会话级控制平面**（避免 Run ↔ Manager 循环依赖；Weak/接口注入）。
2. **Spawn 预留-提交**（并发槽位：reserve → spawn → commit/release）。
3. **Role/Definition 作为配置层**，不是仅拼接 prompt。
4. **深度 / 并发在工具层硬失败**（直接返回模型可读错误，禁止继续幻想 spawn）。
5. **Fork 历史过滤**（精确决定哪些消息进入子上下文）。
6. **完成通知注入父上下文**（截断错误 + next action 提示）。
7. **产品事件映射**：Core Event → CLI/Desktop/Enterprise 各自投影（对齐 Progress / SSE）。
8. **内部委派与工具委派分离**：Guardian/Review 类系统子线程可不暴露为 LLM 工具，但复用同一 spawn 基础设施。

**应避免**：

- v1/v2 双工具面长期并存。
- 把 review/compact/memory 全部塞进同一个 `SubAgentSource` 大杂烩而不建模。
- 过度绑定某一家 Responses API namespace。

### 3.3 Genesis 综合选型（已选定）

| 维度 | 选型 | 理由 |
| --- | --- | --- |
| 用户自定义格式 | **Markdown + YAML frontmatter**（Kode/Claude 路线） | 生态熟悉、易编辑、易校验；Enterprise 可再投影到 DB |
| 运行实例 | **独立 SubRun / agent_id + child_run_id**（Codex Thread 语义） | 可审计、可 resume、可限流 |
| 委派 API | **固定工具族 `Task` / `TaskOutput` / `TaskStop`**（Kode 体验） | 与 Skill 网关原则一致；禁止别名双轨 |
| 控制平面 | **`runtime/multiagent/controller`**（Codex） | 产品无关、可测；不引入 v1/v2 双栈 |
| 权限 | **只收紧 + Gateway/Approval 全覆盖** | 与现有 Tool/Skill 治理一致 |
| 协作空间 | **二期扩展** | 先把委派做对 |

### 3.4 源码精读：自动委派 vs 用户定义（关键结论）

用户直觉「没有自定义 agents 时，复杂任务也会自动起子智能体」——**方向对，机制常被误解**。Kode-CLI 与 Codex **都没有**「复杂度打分器 / 启发式引擎在 runtime 直接 fork 子进程」；实际是：

```text
内置 Definition 始终在 Catalog
  + System / Tool description / 模式 reminder 提示主模型
  + 主模型自行决定是否调用 Task / spawn_agent
  (± 少数硬编码路径：Skill fork、Plan 退出 swarm、mention reminder)
```

#### 3.4.1 Kode-CLI

| 问题 | 答案 | 证据要点 |
| --- | --- | --- |
| 无 `.kode/agents` 能否委派？ | **能**。`BUILTIN_AGENTS` 始终 merge 进 catalog | `packages/core/src/agent/builtin.ts`、`loader.ts` |
| 内置类型 | `general-purpose`、`Explore`、`Plan`、`statusline-setup`、`capabilities-manager` | 同上（Explore/Plan **大小写敏感**） |
| 「自动」是什么？ | **提示词驱动**：system policy「搜代码优先 Task」「匹配 description 应主动用 Task」；compat 提示对非 needle 探索**强制 Explore**；Plan mode reminder 要求 Phase1 并行 Explore、Phase2 启 Plan | `constants/prompts.ts`、`plan/mode/reminders.ts`、`TaskTool/prompt.ts` |
| 非模型自动的硬编码路径 | ① `Skill`/`SlashCommand` 的 `context:fork` → 直接 `callTaskTool`；② `ExitPlanMode` launch swarm → 后台 spawn Plan；③ `@run-agent-*` → system-reminder **迫使**模型调 Task（仍非代码直接 spawn） | SkillTool / ExitPlanModeTool / mention events |
| 自动/运行时个数限制 | **无全局后台 agent 硬上限**（Map 存储）；Plan mode：Explore 默认并行 3（env 上限 10）、Plan 默认 1（上限 10）；swarm teammates **1–10**；**禁止嵌套 Task**（工具剥离，无 depth 计数器）；Task 入参 `max_turns` schema 有、**运行时忽略** | `backgroundTasks.ts`、`reminders.ts`、`ExitPlanModeTool`、`TaskTool/prompt.ts` |
| 用户 md 限制 | 必填 `name`/`description`；tools 白/黑名单 + **硬禁** orchestration 工具；`permissionMode` **只收紧**；`forkContext` 须字符串 true/false；merge：built-in→plugin→user→project→flag→**policy 最高**；用户可覆盖同名内置 | `validator.ts`、`permissions.ts`、`loader.ts`、`docs/agents-system.md` |

#### 3.4.2 Codex

| 问题 | 答案 | 证据要点 |
| --- | --- | --- |
| 无 `~/.codex/agents` 能否委派？ | **能**。内置 role：`default` / `explorer` / `worker` | `core/src/agent/role.rs` |
| 「自动」是什么？ | **默认不主动**：V1 tool description 写明除非用户/AGENTS.md/skill **明确**要求，否则不要 spawn；「要深入/研究」**不算**授权。V2 + `ReasoningEffort::Ultra` 时注入 developer：**Proactive**——并行能明显提速/提质时可主动用子代理（仍是文案 nudge，**无复杂度引擎**） | `multi_agents_spec.rs`、`multi_agent_mode_instructions.rs`、`session/multi_agents.rs` |
| 硬编码批量 | `spawn_agents_on_csv`（fanout feature）；默认最多 16 worker、上限 64，再被 session 线程上限截断 | `agent_jobs.rs` |
| 个数/深度限制 | V1：`max_threads` 默认 **6**，`max_depth` 默认 **1**（超限工具层拒绝）；V2：`max_concurrent_threads_per_session` 默认 **4**（有效子槽约 3）+ residency LRU；V2 spawn **不强制 depth**；delegate 子线程 **禁用** multi-agent | `config/mod.rs`、`registry.rs`、`residency.rs`、`codex_delegate.rs` |
| 用户 role 限制 | TOML / `agents/*.toml`：必填 description；独立文件必填 `developer_instructions`；nickname 规则；可覆盖同名内置文案；V2 启用时禁止再设 V1 的 `agents.max_threads` | `config/agent_roles.rs` |
| `orchestrator.md` | 模板含并行委派指引，但**代码零引用**（未接线） | `templates/agents/orchestrator.md` |

#### 3.4.3 对 Genesis 的直接借鉴（写死）

1. **内置 + 用户定义是两条互补路径，不是二选一**；用户未放任何 md 时，内置 explore / plan / general-purpose **必须仍可委派**。  
2. **默认「自动」= Catalog 常驻 + System/Task description 引导主模型调用 `Task`**；**禁止**实现「复杂度检测器静默 Spawn」。  
3. **少数硬编码 Spawn** 仅允许：`Skill(context=fork)`、产品明确的 Plan/swarm 出口、（可选）mention→reminder→仍经 `Task`。  
4. **委派姿态可配置**：默认偏 Kode「可主动」（搜代码/匹配 description）；Enterprise 可切 Codex 式 `explicit_request_only`。  
5. **限制分层**：运行时 `max_concurrent` / `max_depth`（工具层硬失败）+ 提示层并行建议；用户 md 走校验 + 只收紧 + 硬禁 orchestration。

细节与 Genesis 规范见 **§6 双路径**、**§6.5–6.6 限制**。

## 四、协议与工具面（唯一委派入口）

### 4.1 命名决策（已锁定，禁止双轨）

对外 LLM 可见工具名**锁定**为：

| 工具名 | 职责 | Phase |
| --- | --- | --- |
| `Task` | 创建或 resume 子 Agent；默认同步等待完成；可 `run_in_background` | 1 |
| `TaskOutput` | 拉取**后台子 Agent**增量/最终输出（按 `agent_id`） | 2 |
| `TaskStop` | 停止后台子 Agent | 2 |

说明：

- 与 `Skill` 对称：`Skill` 加载知识；`Task` 委派执行单元。对齐 Kode/Claude 心智，**不**再提供 `Agent` / `spawn_agent` 等别名。
- Profile / CollisionGuard：若用户安装的 tool 与 `Task*` 撞名，按现有 collision 策略处理；**不得**把 subagent 名当 tool 名。

**与本仓库其他「Task」语义消歧（强制）**：

| 名称 | 含义 | 位置 |
| --- | --- | --- |
| LLM 工具 `Task` / `TaskOutput` / `TaskStop` | **仅** SubAgent 委派与后台子 Agent 生命周期 | `capabilities/subagent/tool/...` |
| 执行域命令会话输出 | shell/sandbox 命令的 stdout 流、session 订阅 | `capabilities/execution`（文档里曾称 TaskOutput，**实现与 LLM 工具名无关**） |
| `TodoWrite` | 计划待办，与 SubAgent 无关 | `tool/adapter/builtin/todo_write.go` |

文档与代码注释必须写全称或带域前缀，避免把「命令 TaskOutput」写成可调用的 LLM 工具。

### 4.2 `Task` 输入契约（逻辑 schema）

```json
{
  "description": "3-5 词摘要，用于进度与审计",
  "prompt": "子任务完整指令",
  "subagent_type": "explore | plan | general-purpose | <custom>",
  "model": "inherit | <profile_or_model>",
  "resume": "<agent_id>",
  "run_in_background": false,
  "fork_context": false,
  "max_turns": 0
}
```

规则：

- `subagent_type` 必须来自当前可见 Catalog（CapabilityScope 过滤后）。
- `resume` 与新建互斥：有 `resume` 时忽略新建字段中与实例冲突的定义，仅追加 `prompt`。
- 子 Agent **默认禁止**再次调用 `Task` / `TaskOutput` / `TaskStop`（防递归，`max_depth=1`）；仅当显式放宽到 `max_depth=2` 时，depth=1 的子 Agent 才获得 `Task`（depth=2 的实例仍禁）；**硬上限 2，不设 `max_depth>2` 的开关**（见 §6.1.3「深度硬上限」）。
- `fork_context=true` 时强制 `model=inherit`（与 Kode 一致），并走「过滤后的父消息快照」。

### 4.3 Catalog 主通道

与 Skill 相同模式：

- `Task` 的 `DescriptionFunc` 动态渲染 `<available_agents>...</available_agents>`。
- System prompt 只保留短规则：禁止把 agent 名当 tool 名；必须 `Task(subagent_type=...)`。
- Catalog 受 CapabilityScope / 租户 / Profile 过滤。

### 4.4 `Skill(context=fork)` 映射

落地后唯一合法路径（Definition 解析**有序、唯一**）：

```text
Skill(skill=X) 且 meta.Context == fork
  → 解析 Definition：
       1. 若 meta.Agent 非空：必须能在可见 Catalog 中 resolve(meta.Agent)，否则失败（不静默回退）
       2. 若 meta.Agent 为空：合成临时 Definition
            name = "skill-fork:" + skill.qualified_name
            system_prompt = skill body
            tools = skill.allowed-tools（再与父可见集求交）
            model = skill.model 或 inherit
  → SubAgentController.Spawn(SpawnRequest{
       From: SkillFork,
       Definition: <上一步>,
       Prompt: args 与任务说明（body 已在 definition/system 中时不再重复全量注入父线程）,
       AllowedTools: parent ∩ definition.tools ∩ skill.allowed-tools,
     })
  → 主线程不注入 skill body
  → ToolResult 返回摘要 + agent_id（同步）或 async_launched（若显式后台；fork 默认同步）
```

当前实现中，Skill 保留自身的解析、依赖检查与审批链，再调用由共享 bootstrap 延迟注入的固定 `Task` 网关；`Task` 负责 Catalog 校验、工具交集、审批与 Controller Spawn。`meta.Agent` 为空时暂以 `general-purpose` 承载 Skill 正文和参数；后续可提升为不进入 Catalog 的临时 Definition，但不得绕过该统一链路。

### 4.5 Mention

| 产品 | 行为 |
| --- | --- |
| CLI | `@agent-<type>` / `@run-agent-<type>` → 注入高优先级 reminder 或等价 Select，**不绕过** `Task` |
| Desktop | 提及选择器 → 同语义 |
| Enterprise | 同语义 + 审计「用户指定委派」 |

禁止：mention 直接在 UI 层 spawn 而不经 Gateway/Approval。

## 六、双路径：内置自动委派 vs 用户定义 Agent

本章把「不配任何 agents md 能否委派」和「用户配置有何限制」说死，对齐 Kode / Codex 源码结论（§3.4）。

### 6.0 一句话

| 路径 | 有无用户文件 | 谁决定 spawn | Genesis 是否支持 |
| --- | --- | --- | --- |
| **A. 内置自动委派** | **不需要** `.genesis/agents` / `.kode/agents` | 主模型受 **System + Task Catalog 提示** 调用 `Task(subagent_type=内置)`；另有少量硬编码（Skill fork 等） | **必须支持（Phase 1）** |
| **B. 用户 / 项目定义** | `.genesis/agents/*.md`（可选兼容 `.kode`/`.claude`） | 同上，但 Catalog 多出自定义 `subagent_type`；用户可 `@agent-xxx` 指定 | Phase 2+ 完整管理；Phase 1 可先只读加载 |

**否定误解：**

- ❌ 「没有用户 md 就不能起子智能体」——错；内置常驻。  
- ❌ 「runtime 会按任务复杂度自动 fork」——错；Kode/Codex 都没有复杂度引擎静默 Spawn。  
- ✅ 「感觉像自动」——因为提示词要求主 Agent **主动**在合适场景调用 `Task`（尤其探索类），模型照做。

### 6.1 路径 A：内置自动委派（用户零配置）

#### 6.1.1 内置 Definition（Phase 1 锁定）

| `name`（= subagent_type） | 用途 | 默认工具策略 | 对齐参考 |
| --- | --- | --- | --- |
| `general-purpose` | 多步研究/执行 | 父可见集 ∩ 禁 orchestration | Kode `general-purpose` |
| `explore` | 只读代码探索 | 只读文件/搜索；禁写/禁 Task* | Kode `Explore`（Genesis **统一小写**） |
| `plan` | 只读规划 | 只读；禁写禁执行禁 Task* | Kode `Plan`（Genesis **统一小写**） |

- 落点：`internal/capabilities/subagent/adapter/embedded`。  
- 产品 bootstrap 可关某类型，但 **CLI 默认 Profile 应开启三者**，保证零配置仍有委派能力。  
- **不**把 Kode 的 `statusline-setup` / `capabilities-manager` 当通用 SubAgent 一期范围（属产品专用；需要时另立定义）。  
- **「只读」的判定来源须复用现有工具能力分类，不新造清单**：explore/plan 的只读白名单应取自现有 read-only / safeMode 工具集合（对齐 Kode `getReadOnlyTools`）与工具 `ToolTraits`；再与父可见集求交。禁止在 embedded 里硬编码一份平行的工具名单。

#### 6.1.2 「自动」如何发生（机制分层）

| 层级 | 机制 | 是否代码直接 Spawn | Genesis 取舍 |
| --- | --- | --- | --- |
| L1 提示 | System 短规则 + `Task` DescriptionFunc 列出内置 `whenToUse` | 否：模型调 `Task` | **采用（主路径）** |
| L2 模式 reminder | 如 Plan 模式要求并行 explore | 否 | 二期与 plan 能力对齐时采用 |
| L3 硬编码 | `Skill(context=fork)`、批准后 swarm、系统子线程 | 是：走 Controller | **fork 必须**；swarm 二期 |
| L4 mention | `@agent-explore` → reminder「必须 Task(...)」 | 否 | 采用（与 Skill mention 同构） |
| L5 复杂度引擎 | 静态分析任务难度自动 Spawn | — | **禁止** |

**Phase 1 提示词要点（借鉴 Kode，写入 prompt builder，勿散落魔法字符串）：**

1. 文件搜索 / 非 needle 的代码库探索：优先 `Task(subagent_type=explore)`，以节省主上下文。  
2. 当任务匹配某 agent 的 `description`（含自定义）：应主动使用 `Task`，无需用户点名。  
3. 已知精确路径 / 单文件 needle：直接 `read_file`/`grep`，**不要**为小事 spawn。  
4. 可并行时：同一条 assistant 消息内多个 `Task`（受 `max_concurrent` 约束）。  
5. Enterprise / 保守 Profile：可改为 Codex 式 `explicit_request_only`——「除非用户、AGENTS.md 或 Skill 明确要求委派，否则不要 spawn；『要深入』不算授权」。

```text
委派姿态 delegation_posture:
  proactive          # 默认 CLI/Desktop：对齐 Kode
  explicit_request_only  # Enterprise 或高敏租户：对齐 Codex V1 / 非 Ultra
```

#### 6.1.3 路径 A 的运行时限制（自动/内置）

| 限制 | Genesis Phase 1 默认 | 借鉴 | 说明 |
| --- | --- | --- | --- |
| `max_depth` | **默认 1，硬上限 2** | Codex V1；Kode 工具剥离 | 默认子 Agent **不可**再 `Task*`；仅显式放宽后允许到 2；超限工具层返回可读错误（见下「深度硬上限」） |
| `max_concurrent` | CLI **3**；Desktop **4**；Enterprise 策略可配，建议 ≤ **6** | Codex V1=6 / V2≈3 子槽；Kode Plan Explore=3 | **工具层硬限**（不像 Kode 后台 Map 无上限） |
| 提示层并行建议 | explore 类建议 ≤ `max_concurrent` | Kode Plan reminder | 写进 Task description，非第二套计数器 |
| 单子 `max_turns` | Definition 可配；**运行时必须生效** | Kode schema 有但忽略→**反例，我们要落实** | 入参 `max_turns` 与定义取更严者 |
| 嵌套 | 禁止 | Kode/Codex 默认 | 编排工具硬禁列表 |
| 全局无限后台 | **禁止** | Kode 无硬限→借鉴 Codex 槽位 | 后台也占 `max_concurrent` |

**深度硬上限（拍板，不得配置突破）**：

- **默认 `max_depth = 1`**：子 Agent 无 `Task*`，不能再委派。
- **仅显式放宽到 2**：企业策略 / 特定 Definition 可显式设 `max_depth = 2`（如「主办理 → 子校验」）；不提供全局默认放宽。
- **硬上限 = 2，任何配置/策略/入参都不能超过**：加载 Definition 时校验 `max_depth ∈ {1,2}`，>2 直接 **拒绝加载并报错**；运行时对第 3 层（depth ≥ 2 的实例再 spawn）**工具层硬失败**，绝不允许无限嵌套。
- **depth = 2 时的附加硬约束**：`max_concurrent` 与预算（§7.6.2）**必须按整棵子树累计**，不是按单层——防止 2 层放大成指数级并发/成本；取消与超时**必须逐层级联**（§7.1.4）。

超限行为：**Controller.Spawn 失败**，`Task` ToolResult 明确错误（如 `agent concurrent limit reached: max=3` 或 `agent depth limit reached: max=2`），并给模型可执行建议（「减少同批并行 Task 数或改串行」/「本层不可再委派，请自行完成」），禁止静默丢弃或假装成功。

**并行实现说明**：并行 fan-out **不由 Controller 另造编排器**，而是复用同轮 sibling tool call 调度（`internal/capabilities/tool/scheduler`）——模型在一条消息里发多个 `Task`，调度器并发执行，各自 `Spawn`+`Wait`；Controller 只做槽位硬限与生命周期（详见 §7.1.4）。因此「提示层建议 ≤ `max_concurrent`」与「工具层硬失败」是同一限额的两面，不是两套计数器。

### 6.2 路径 B：用户 / 项目定义（对齐 `.kode/agents/*.md`）

#### 6.2.1 文件位置

```text
# Genesis 首选（产品可配，内核只认 Source）
<workspace>/.genesis/agents/<name>.md
~/.genesis/agents/<name>.md

# 可选兼容 Source（默认关闭；开启后只读解析，不作为内核依赖）
<workspace>/.kode/agents/*.md
~/.kode/agents/*.md
<workspace>/.claude/agents/*.md
~/.claude/agents/*.md
```

无这些文件时：**路径 A 仍完整可用**；有文件时 Catalog = 内置 ∪ 用户 ∪ …（按优先级覆盖）。

#### 6.2.2 文件格式示例

仓库提供了一份与**当前加载器实际支持字段完全一致**的样例：[`configs/subagents/api-designer.md`](../configs/subagents/api-designer.md)。使用时复制到目标工作区的 `.genesis/agents/api-designer.md`，再执行 `genesis agents validate`。

下方是目标态字段的完整示意，其中标注“后续”的字段会被当前严格校验器拒绝；不要直接把它当作可运行配置。

```md
---
name: api-designer
description: |
  设计清晰的 API 契约与错误模型。在需要接口设计时使用。
tools: ["read_file", "grep", "glob", "write_file"]
disallowed_tools: ["Task", "TaskOutput", "TaskStop"]
skills: []
model: inherit
permission_mode: plan
fork_context: false
sandbox_profile: inherit
max_turns: 40
---

你是 API 设计专家。输出必须包含：资源模型、错误码、兼容性说明与示例。
```

#### 6.2.3 字段语义

| 字段 | 必填 | 当前状态 | 语义 |
| --- | --- | --- |
| `name` | 是 | `[x]` | = `subagent_type`；当前校验字母数字、`-`、`_`。 |

**`permission_mode`「只收紧」需要可判定的全序（对齐 Kode rank，落 Genesis 语义）**：仅凭「≤ 父严格度」无法实现，必须给权限模式定一个严格度序，取 `min(严格度)`——即子实际模式 = 父与定义中**更严**的一个，绝不放宽。建议序（由宽到严，具体枚举以 `统一配置权限与审批治理设计.md` 为准，命名须与其一致）：

```text
bypass  <  accept_edits  <  default(ask)  <  plan  <  read_only
（越靠右越严；子 = max_strict(parent, definition)，不得反向放宽）
```

企业 `SandboxRequire`、危险工具 scope 等**强约束优先于**该序，用户定义不得借 `permission_mode` 绕过。

#### 6.2.4 合并优先级（后者覆盖前者同名）

1. embedded 内置  
2. plugin / marketplace 安装  
3. user（`~/.genesis/agents`）  
4. project（工作区向上查找）  
5. 运行时注入（CLI flag / API，受策略约束）

用户 **可以** 覆盖同名内置（如自定义 `explore`）——与 Kode 一致；企业 policy 可禁止覆盖内置安全基线。

## 七、控制平面与执行流

### 7.1 Controller 职责与工具映射

#### 7.1.0 生命周期粒度（拍板：**根会话级**，消歧）

Controller 的持有粒度**定为「根会话级」（per root session），不是 per-Run**。理由：

- 后台子 Agent（Phase 2 `run_in_background`）可能跨多个父 Turn / Run 存活，`max_concurrent` 必须按会话累计，否则每个 Turn 新建 Controller 会把并发计数清零、限流失效（对齐 Codex `max_concurrent_threads_per_session`）。
- 一个根会话内的父 Run 与其全部子 Run 共享同一 Controller 句柄与同一套槽位/审计视图，`Graph()` 才能给出完整子树。

因此 **`max_concurrent` 语义 = 单根会话内同时存活（running + background）子实例上限**；`max_depth` 是从根 Run 起算的委派链深度。**根父 Run 本身不占子槽**。

「根会话」以抽象 `SessionScope` 标识（内核只认标识，不绑具体存储）：CLI/Desktop 单进程即内存 Session；Enterprise 分布式调度（二期）下同一 `SessionScope` 可能跨节点，由 §7.1.2 的分布式 `SlotLimiter` 后端保证跨节点计数一致——**内核 Controller 代码不变，只换注入的后端**。

> 命名统一：本文 `Channel = agent`、能力类型统一以代码常量 `capmodel.CapabilityTypeSubAgent`（值 `"subagent"`）为准；文档旧写法 `CapabilitySubAgent` / `ChannelAgent` 均指向该常量，实现时不得再造第二个名字。

#### 7.1.1 接口（按 Phase 收敛，禁止一次性大接口）

Phase 1 只落**最小闭环**，其余端口 Phase 2+ 再加，避免接口膨胀与死方法：

```text
SubAgentController（根会话级）

# Phase 1（同步委派最小闭环）
  - Catalog(ctx) → []DefinitionSummary
  - Spawn(ctx, SpawnRequest) → (Instance, error)     # 含 reserve→commit/release
  - Wait(ctx, agent_id, timeout) → (TerminalStatus, Summary, error)
  - Stop(ctx, agent_id) → error                       # 供父 context 取消/超时级联

# Phase 2（后台 / resume / 续问）
  - Resume(ctx, agent_id, prompt) → ...
  - FollowUp(ctx, agent_id, prompt) → ...             # 可由 Task(resume=...) 覆盖

# Phase 3（产品投影 / 持久化）
  - Graph(ctx) → parent/child 边（供 Desktop/Enterprise）
  - 依赖 Store 端口（Phase 1 用内存实现即可，不引持久化）
```

| LLM 工具 | Controller 调用 | 说明 |
| --- | --- | --- |
| `Task`（新建，同步） | `Spawn` + 内部 `Wait` | 阻塞到终态，一次 ToolResult 带回 summary |
| `Task`（`run_in_background`） | `Spawn`（不等待） | 立即返回 `async_launched` + `agent_id` |
| `Task`（带 `resume`） | `Resume`；若未后台则再 `Wait` | 追加 prompt 续跑 |
| `TaskOutput` | 读 Store / 可选短 `Wait` | 按 `agent_id` 取增量或终态；**不**另造第四个工具名 |
| `TaskStop` | `Stop` | 取消运行中实例 |
| （无 LLM 工具） | `FollowUp` / `Graph` | FollowUp 可由 `Task(resume=...)` 覆盖；Graph 供产品投影 |

实现要点（借 Codex）：

- 子 Run 通过 `RunEngine.Start`（或等价内部 API），`RuntimeCapabilityContext.Channel = agent`，带上 `parent_run_id` / `agent_id` / `child_run_id`。
- 子 Run 的 ToolGateway 使用**收窄后的** Info 集与 Approval 链；Enterprise 可将高风险审批 **上抛** 到父会话/人工队列。
- Phase 1 默认配置建议：与 §6.1.3 一致——`max_depth` 默认 1（硬上限 2，仅显式放宽）；`max_concurrent` CLI=3 / Desktop=4 / Enterprise 可配；`max_turns` 运行时生效。

#### 7.1.2 并发槽契约（端口化：语义统一，后端三端独立）

对齐 Codex `SpawnReservation` 的「预留-提交-回滚」。**关键：把限流抽成 `SlotLimiter` 端口，语义（reserve/commit/release + depth/concurrent 判定）三端统一，后端实现三端独立**——否则会把 CLI 的「单进程内存计数」写死进内核，堵死 Enterprise 分布式调度（§8.2）。

```text
// 内核端口：语义统一，禁止内核假设单进程
type SlotLimiter interface {
    Reserve(ctx, key SessionScope, depth int) (SlotToken, error) // 超限 → 模型可读错误
    Commit(token SlotToken, agentID string) error
    Release(token SlotToken) error                               // 幂等，defer 兜底
}

func (c *controller) Spawn(ctx, req) (Instance, error) {
    tok, err := c.limiter.Reserve(ctx, req.SessionScope, req.Depth)
    if err != nil { return nil, err }         // 见 §6.1.3 超限行为
    committed := false
    defer func(){ if !committed { _ = c.limiter.Release(tok) } }() // panic/早退回滚
    inst, err := c.startChildRun(ctx, req)
    if err != nil { return nil, err }
    _ = c.limiter.Commit(tok, inst.AgentID); committed = true
    return inst, nil
}
```

| 后端实现 | 用于 | reserve/commit/release 载体 |
| --- | --- | --- |
| **in-process**（`sync.Mutex` + 计数，同锁内 reserve+校验 避免 TOCTOU） | CLI / Desktop 默认 | 进程内 map/counter |
| **分布式**（共享存储原子操作 / 租约 TTL） | Enterprise 分布式调度（二期） | Store/Redis/DB，实现**同一** `SlotLimiter` 契约 |

- 后台实例在 `commit` 后仍占槽，直到终态（complete/failed/cancelled）由 Controller 显式 `Release`；`Stop` / 超时也必须归还。
- Definition 名 / agent_id / nickname 的占位同样走「预留-回滚」，防止并行 Spawn 撞名。
- **内核只依赖 `SlotLimiter` 端口**，`SessionScope` 抽象「根会话」标识；单机与分布式共用一套 Controller 代码，产品在 bootstrap 注入对应后端。

#### 7.1.3 生命周期、所有权与破环

- **所有权**：Controller 由根会话装配时创建并持有；父/子 Run 只拿 Controller **接口引用**，不反向持有 Session/RunEngine 强引用。
- **破环**：内核禁止 `runtime/multiagent.controller` ⇄ `RunEngine` 相互强引用成环（对齐 Codex 用 `Weak` 破 `ThreadManager↔Session` 环的动机）。Genesis 采用**单向依赖 + 接口注入**：Controller 只依赖 `RunEngine` 端口（`Start`）与 `Store`/`EventSink` 端口；子 Run 需要回到 Controller 时，经由**注入的接口**或 `agent_id` 向 Controller 反查，不在子 Run 内嵌 Controller 具体类型。
- 子 Run 与父 Run **共享 `session_id`**、独立 `run_id(child_run_id)`；关联键注入见 §9。

#### 7.1.4 并行 fan-out 与取消/超时

- **并行 = 复用现有 `scheduler`，Controller 不自建编排引擎（YAGNI）**：同一条 assistant 消息里的多个 `Task` tool call 由现有同轮 sibling 调度器（`internal/capabilities/tool/scheduler`）并发执行，每个 `Task` 内部 `Spawn`+`Wait`。Controller 只负责**槽位限流 + 单实例生命周期**，Phase 1 不做 parallel/handoff 编排（留 `patterns/` 二期）。
- **取消传播**：父 Run 的 `context` 取消（用户中断 / 父超时）**必须**级联取消所有在途子 Run（`Stop` + child ctx cancel），并归还槽位；子终态记为 `cancelled`，照发 Progress + 写 `agent.log`。
- **超时**：子 Run 受两道闸——`max_turns`（轮次，运行时生效）与可选 wall-clock 超时；触发后终态 `failed`（或 `cancelled`），ToolResult 给可执行 next action（见 §7.5）。

### 7.2 同步委派时序

```text
父 ReAct
  → LLM: Task(subagent_type=explore, prompt=...)
  → ToolGateway / Approval
  → Controller.Spawn
       ├─ Progress kind=subagent phase=start（必发）
       └─ agent.log：subagent spawn start（必写，含关联键，见 §9）
  → 子 RunEngine 循环（独立 transcript；子 context 继承 session_id，注入 child_run_id）
  → Progress: subagent progress/complete/error
  → agent.log：subagent complete/error
  → 完成：Summary + artifacts + agent_id
  → 父观察 ToolResult（用户默认不直接看子全文；产品可另开面板）
```

### 7.3 后台委派

- `run_in_background=true` → 立即返回 `async_launched` + `agent_id`。
- 父用 `TaskOutput` / `TaskStop` 管理。
- CLI 可后续做「前景转后台」交互；内核先支持显式 flag。

### 7.4 输入上下文：最小授权快照，而非父历史复制

子智能体的输入在 **Spawn 前**一次性构造，但读取与变换必须分层：`TranscriptSnapshotSource` 负责 flush/materialize 并读取一致的父快照，纯函数 `ContextSnapshotBuilder.Build(snapshot, policy)` 只负责筛选、脱敏、预算裁剪和组装。二者集中在 `runtime/multiagent/contextsnapshot`；Controller 只编排调用，Task、Skill fork 和产品 UI 都不得自行拼接消息。这样既保留可测试的纯变换，也避免不同入口产生不同的泄漏面。

```text
ChildInput =
  ChildSystemContract(definition, effective_tools, effective_policy, inherited_runtime_contract)
  + OptionalParentSnapshot(fork_mode)
  + DelegationEnvelope(task, constraints, artifact_contract)
```

| 模式 | 子 Agent 看到什么 | 适用场景 |
| --- | --- | --- |
| `isolated`（默认） | 定义 system prompt、经过清洗的委派 `prompt`、工作区/沙箱/产物契约 | explore、plan、业务办理；绝大多数委派 |
| `last_n_turns` | `isolated` + 最近 N 个**已完成**父回合的白名单消息 | 子任务确实依赖最近讨论结论 |
| `filtered_history` | `isolated` + 经过白名单过滤的完整父历史 | 仅显式 `fork_context=true`，且须预算允许 |
| `skill_isolated` | Skill 正文作为 definition system prompt，`args` 作为委派输入；不复制父聊天 | `Skill(context=fork)` |

`fork_context` 在 API 层仍兼容为布尔值：`false → isolated`，`true → filtered_history`。内部模型必须使用枚举，未来可安全增加模式，禁止继续扩张布尔语义。模式不命名为 `full_history`，因为无论何种配置都不允许复制未过滤的完整历史。

Phase 1 已接入的 `TranscriptSnapshotSource` 由两部分合并：ReAct 在执行当前 `Task` 前复制 `RunContext.Messages`，持久化 Source 通过 `SessionHistoryStore.Replay`（不支持时回退 `GetRecent`）读取同一 `SessionRef` 的可恢复历史，再按最长重叠合并。当前 `Task` 调用本身因携带 tool call 会被 Builder 丢弃；本轮尚未落盘的消息由活跃快照补齐。它不引入 DB，后续可替换为带 flush/materialize 与分布式一致性保证的 enterprise Source，而不改变 Task/Builder 契约。

**父快照白名单（锁定）**：借鉴 Codex `keep_forked_rollout_item` 的保守策略；Kode 的 fork 会读取更宽的对话记录，但 Genesis 不采用其“原样保留工具块”的行为。

| 处理 | 内容 |
| --- | --- |
| 保留 | `user`；`assistant` 的最终回答；已脱敏的会话摘要 / 压缩摘要 |
| 丢弃 | 全部 function/tool 调用和结果、reasoning/思考、中间 assistant 草稿、进度事件、Skill 注入正文、SubAgent 委派/完成消息、审批记录、usage hint、当前 `Task` 调用本身 |
| 重写 | 对保留消息执行路径脱敏、密钥/credential redaction、每条与总 token 截断；无效引用替换为稳定占位符 |
| 合成 | 边界消息：`以上内容仅为只读背景；父线程工具、权限、审批与未列出的资源在子线程不可用。仅完成下方委派任务。`，再追加 `DelegationEnvelope` |

父级 `system` / `developer` 消息不得原文继承。它们可能包含仅对父 Agent 生效的产品指令、内部连接信息或更高权限能力。需要继承的租户、工作区、沙箱、路径格式和治理约束，必须由可信运行时状态投影为白名单 `InheritedRuntimeContract`，再合入子 `ChildSystemContract`；对话文本不能充当权限或运行契约来源。

构造顺序必须是：**Source 先 flush/materialize 父 transcript → Builder 选取完整回合 → 结构过滤 → 敏感信息清洗 → 语义边界裁剪 → 插入只读边界消息与委派信封 → 最后由 Controller 组装子 system contract**。结构过滤必须先于内容检测；脱敏器异常、无法分类的敏感块或非法消息形态一律 fail closed（丢弃该块或拒绝 Spawn），不得降级为原文透传。

预算裁剪按固定优先级执行：子 system contract 与 `DelegationEnvelope` 是不可截断的保留区；父快照先丢弃最旧的完整回合，再裁剪低优先级摘要，禁止按字节切开消息或 tool call 对。若不可截断部分本身超预算，Spawn 返回分类错误，不能把残缺任务交给子 Agent。审计默认只记录快照清单、来源、过滤原因、内容哈希与计数；不得为审计额外复制一份敏感正文。

Phase 1 的内核实现以保守的 **rune 上限** 作为无模型依赖的安全硬边界（默认委派输入 32,000 runes、结果摘要 4,000 runes）；模型级 token 估算和按模型精确裁剪将在后续通过独立 estimator port 实现，不把 provider tokenizer 引入 Controller。

`DelegationEnvelope` 是结构化数据，不让定义 prompt 解析自由文本：

```json
{
  "task_id": "agent-...",
  "parent_run_id": "run-...",
  "tool_call_id": "call-...",
  "prompt_origin": "model",
  "objective": "来自 Task.prompt 的完整任务",
  "expected_output": "结论、证据、建议与可验证 artifacts",
  "workspace": {"root": ".", "path_format": "workspace-relative"},
  "capabilities": ["read_file", "grep"],
  "constraints": {"max_turns": 8, "max_tokens": 4000, "max_tool_calls": 12},
  "return_contract": "仅返回 TaskResultEnvelope；不要回放完整过程或敏感原文"
}
```

`Task.prompt` 属于父模型生成的不可信委派输入：必须校验长度、编码和危险控制字符，并记录来源与哈希；它不能覆盖 system contract、治理约束和 return contract。产物路径只允许 workspace-relative / sandbox path / resource id。子 Agent 的工具、权限、审批和 credential 均重新按交集求值，父快照绝不能成为扩权凭据。

### 7.5 结果回传：分态 Envelope + Reducer，而非 transcript 透传

目标态中，子 Run 结束后先原子写入 `TerminalCandidate`（最终回答、结构化 final payload、已登记 ArtifactManifest、Usage、终态错误）和独立 transcript；然后由幂等的 `ResultReducer` 一次性生成并保存规范化 `TaskResultRecord`，交付时再由 `ResultProjector` 生成父模型可见的 `TaskResultEnvelope`。当前 Controller 在实例终态时归约并保存 `Instance.Result`；CLI 通过 `.genesis/subagents/<agent_id>.json` 文件 Store 持久化该安全结果，可支持新进程 `TaskOutput` 读取和 `Task(resume=...)` 基于摘要续跑，但尚不提供 transcript / `TaskResultRecord` 的原子持久化；该能力随 DB/企业 Store 阶段补齐。Reducer 是第二个安全边界：它只接收 `TerminalCandidate`，不查询或回传完整工具轨迹。证据与 artifact 的存在性、路径校验由独立 `EvidenceValidator` port 完成，Reducer 不直接读文件系统；当前调用者权限由 Projector 在每次交付时重新校验。同步 `Task`、`TaskOutput`、resume 与 Skill fork 必须复用同一套 Reducer + Projector。

当前内核已为每个子 Run 注入并发安全的 `ArtifactManifest` 登记器；执行能力只能显式登记候选 artifact / finding，不能直接写父结果。默认 `RejectingEvidenceValidator` 与 `DenyingResourceProjector` 会省略全部可选证据和产物，因此 Desktop、Enterprise 在尚未注入各自受控资源后端前不会错误地把路径或文件交付给父模型。CLI 已注入 workspace resource 后端：仅接受工作区内已存在的常规文件、投影 workspace-relative slash 路径、计算 `content_hash`，并在每次 `Task`/`TaskOutput` 交付时重新校验；未登记、越界、绝对路径、hash 变化或证据引用不匹配的 artifact/finding 均被省略。后续产品资源后端只需实现同两个端口，不改变 Controller、Task 或结果协议。

`TaskOutput` / `TaskStop` 的终态结果交付必须经过 `ResultDeliveryStore` 幂等门：键为 `tenant_id + session_id + parent_run_id + agent_id + result_id`，第一次交付返回 `result` 并标记 `result_delivered=true`；后续重复轮询只返回 `retrieval_status=ready`、实例状态与 `duplicate_result=true`，不得再次带回 `result`。这样父 Agent 的 transcript 只消费一次安全摘要，CLI/产品管理面仍可通过受权查询展示状态。当前 CLI 用 `.genesis/subagents/deliveries/*.json` 文件记录交付键；默认内存实现只服务无产品 Store 的测试/本地装配。Enterprise 后续把同一端口换成 DB/Redis 原子 create-once，不改 LLM 工具协议。

启动、轮询和终态不得混用同一结构：

- `TaskLaunchEnvelope`：`status=async_launched`、`agent_id`、`child_run_id`，只表示成功受理。
- `TaskOutputEnvelope`：`retrieval_status=ready|not_ready|timeout`、当前实例状态、可选 `result`；运行中不得伪造终态摘要。
- `TaskResultEnvelope`：仅表示 `completed|partial|failed|cancelled` 终态。

终态 ToolResult（父模型可见）结构化 JSON：

```json
{
  "schema_version": 1,
  "result_id": "result-...",
  "status": "completed | partial | failed | cancelled",
  "agent_id": "...",
  "child_run_id": "run-...",
  "subagent_type": "explore",
  "summary": "经长度、敏感信息和格式过滤后的最终结论",
  "findings": [{"claim": "...", "evidence": ["src/a.go:18"]}],
  "artifacts": [{"resource_id": "res-...", "path": "OUTPUT_DIR/...", "kind": "file", "description": "...", "content_hash": "..."}],
  "error": null,
  "usage": {"input_tokens": 0, "output_tokens": 0, "tool_calls": 0, "budget_exhausted": false},
  "truncated": false,
  "omitted_sections": [],
  "next_action": "可选，供父 Agent 决定继续、重试或询问用户"
}
```

| 字段 | Reducer 规则 |
| --- | --- |
| `summary` | 首选子 Run 最终 assistant 回答；先执行与输入侧一致的敏感信息清洗，再按语义边界截断。Phase 1 使用 4,000 runes 的保守上限；接入 estimator 后改为默认 1,000 tokens。不得回传 reasoning、tool output 或原始 prompt。 |
| `findings` | 仅当 definition 约定了结构化输出且校验通过时保留；证据只能是已验证的相对路径、资源 ID 或短引文。校验失败时降级为空数组，不把原始 JSON 塞给父。 |
| `artifacts` | Reducer 仅接收 ArtifactManifest 中已登记且路径合法的项；Projector 只投影父在**本次读取时**仍有权限的项。稳定 `resource_id` 优先，路径仅作 workspace-relative 展示元数据；不扫描子工作区猜测文件。 |
| `usage` | 聚合数值和预算终态；不含 provider 原始响应、prompt 或 token 明细。 |
| `error` / `next_action` | `error` 使用 `{code, message, retryable}`；失败信息有独立截断预算，不能挤占正常摘要，并附固定可执行建议。 |

**绝不回传**：完整 transcript、工具输入/输出、思考内容、审批记录、credential/connection 标识、未登记 artifact、父快照、内部路径或 provider 原始响应。它们分别只进入子 transcript、audit/usage、产品详情视图，并受原权限控制。

同步 Task 只返回 `TaskResultEnvelope`；后台 Task 首次只返回 `TaskLaunchEnvelope`。`TaskOutput` 可选择 `block` 与短等待，返回 `TaskOutputEnvelope`，不能把后台累积日志直接塞进父上下文。Kode 的 TaskOutput 采用“有限长度、尾部保留、完整输出给文件路径”的思想可借鉴；Genesis 的完整内容应通过受授权的 transcript/artifact 查询 API 提供，而不是暴露宿主文件路径。

`TaskStop` 的跨进程语义必须诚实：当前 CLI 文件 Store 能让新进程读取已完成/失败/取消的终态，或对已完成实例执行幂等 no-op；但不能取消仍在另一个进程中运行的子 Run。遇到 Store 中仍为 `running` 且当前 Controller 无本地 entry 时，必须返回可读错误，等待后续 Enterprise/后台任务管理器通过租约、心跳和远程 cancellation channel 补齐。后台 Manager 的产品无关骨架已经固定为：worker 获取 `LeaseStore` ownership，按 tick 写 `HeartbeatStore` 并续租，从 `CancellationStore` 消费 stop intent，只有持有租约的 worker 执行真实 `Controller.Stop` 并清理 intent，终态或取消后释放租约。当前 shared bootstrap 已默认注入进程内 `MemoryControlStore`，因此 `Task(run_in_background=true)` 不再只是返回 `agent_id`，而会启动后台 runner 持有租约；`TaskStop` 在存在取消通道时只写 stop intent，不直接绕过 worker。

CLI 管理面当前提供三类轻量命令：`agents tasks` 按更新时间倒序列出 `.genesis/subagents` 中保存的实例摘要；`agents task <agent_id>` 展示父可见安全摘要、状态、`result_id` 与已投影 artifact 元数据；`agents cleanup --days N` 只清理早于阈值的终态记录，默认保留 `running`，除非显式 `--include-running`。这些命令只读取 `TaskResultEnvelope` 与控制面元数据，不读取子 transcript、工具输入输出或宿主绝对路径。

模型投影与产品投影必须分离：父模型只看到上述最小 Envelope；CLI/Desktop/Enterprise 详情页可通过受权 API 查询 transcript、artifact 和审计信息，但每次读取重新鉴权，且内部绝对路径、credential 与 reasoning 永不进入产品投影。后台完成事件只携带 `result_id` 与状态；只有显式 `TaskOutput`/resume 消费才把结果加入父模型上下文，并由 `ResultDeliveryStore` 按交付键去重，防止重复通知污染上下文。

产品投影统一使用最小 `ProjectionEvent` 协议：`type/channel/tenant_id/session_id/parent_run_id/agent_id/child_run_id/status/result_id/occurred_at/metadata`。CLI 将其折叠为任务摘要和状态栏；Desktop 将同一事件投到画布节点、任务看板和本地通知；Enterprise 将同一事件接 SSE/Webhook/审计治理。三端可以选择不同存储与展示，但不得扩展父模型可见 Envelope，也不得把子 transcript、工具原始输出或未重新鉴权的 artifact 放入投影事件。当前 Controller 已在 spawn 与终态时发 `ProjectionEvent`，三端 bootstrap 已分别注入 `cli` / `desktop` / `enterprise` channel 的内存 sink；`ProjectionReader` 对内存 sink 提供 tenant/session/agent 筛选和 200 条上限，Enterprise 已通过 `GET /v1/subagents/events` 暴露该安全投影。CLI TUI、Desktop Wails ViewModel 与 Enterprise 鉴权 SSE/审计持久化仍需各产品实现。

Reducer 必须对同一 `TerminalCandidate` 产生相同 `result_id` 和语义等价的规范化 Record，并采用 create-once 语义，避免重试覆盖已交付结果；Projector 可因权限撤销进一步省略 artifact，但不得恢复 Reducer 已过滤的内容。若脱敏、证据校验或 Reducer 失败，终态记录不能丢失：存在安全可用摘要但可选字段被省略时返回 `partial`，无安全摘要时返回 `failed`；两者都只含分类错误且不带 findings/artifacts，并保留内部审计错误供运维查看。交付端按 `result_id` 幂等去重，重复轮询不得重复注入父 transcript。

失败/取消时，`summary` 必须是截断后的错误摘要，`next_action` 固定说明可重新委派、resume 或调整任务；任何部分失败都独立回传，不能让并行批次静默吞错。

### 7.6 生产化不变量（真实业务场景必需）

> Kode/Codex 是「编码 CLI」，其默认（同步、浅委派、无预算、无幂等）不能直接搬到 Genesis 的**通用业务 Runtime**（RAG / 业务办理 / 监控 / 长期执行）。以下为面向真实业务的补充不变量；场景默认见 §8.4 矩阵。

#### 7.6.1 execution_mode 场景化默认（不做全局同步）

- **同步（sync）**：探索 / 规划 / 短研究——父需立即拿结果续跑。
- **异步（async）**：业务办理、长流程、需人工审批、预计耗时 > 阈值——`Task` 立即返回 `async_launched`+`agent_id`，父用 `TaskOutput`/事件跟进。
- 默认由**类别/定义**决定，不是全局一刀切；调用入参 `run_in_background` 可显式覆盖为更保守方向。
- **内核硬规则（三端一致，不是产品权衡）**：**任何委派若会触发「需上抛的人工审批」，一律强制异步**——`Task` 立即返回 `async_launched`+`agent_id`，审批进队列，子 Run 挂起等审批结果，**禁止同步 `Task` 阻塞父 Turn 干等人工审批**（否则父 Turn 长时间挂起、且会被 wall-clock 超时误杀）。判定「是否需上抛审批」由 Approval 链在 Spawn 前给出；命中即忽略 sync 请求、转 async。Enterprise 侧「异步上抛如何接 Approval/SSE 队列」是实现细节（Phase 3），不改变本规则。

#### 7.6.2 预算 / 成本熔断（生产必备）

- 子 Run 放大 token 成本，**必须**有预算闸：`max_tokens` / `max_tool_calls` /（可选）成本上限，作用于**单次委派**与**会话累计**两级。
- 超预算行为：**硬失败并回传可读错误**（`budget exceeded: tokens`），或按 Profile 降级（换更便宜 model / 截断）；禁止静默烧钱。
- Usage 不仅记录还要**参与判定**（对齐 Codex `rollout_budget`，本设计原先只记不熔断）。

#### 7.6.3 幂等与重试（业务办理红线）

- 业务办理类委派 spawn **必须**支持幂等键（`idempotency_key`，默认可由 `parent_run_id + tool_call_id` 派生）：同键重复 spawn 返回**既有实例**而非新建，防止重复下单 / 重复扣款 / 重复工单。
- 重试语义：父重试或 resume 时以幂等键去重；有副作用的子 Agent 应在其 system 契约声明可重入。
- 与取消/超时配合：超时后重试走**新幂等键或显式覆盖**，由业务定义决定，不在内核默认自动重试有副作用的任务。

#### 7.6.4 分场景超时（默认分层）

| 类别 | 默认 wall-clock | 说明 |
| --- | --- | --- |
| explore / plan | 秒~分（如 60–120s） | 交互期望快速 |
| research / RAG fan-out | 分级（如 5–10min） | 多源检索综合 |
| 业务办理 / 长流程 | 分~小时（定义显式声明） | 通常配 `async` |
| system 常驻（监控） | 无固定上限 / 周期心跳 | 见 §10 |

超时终态 `failed`（或 `cancelled`）+ 归还槽位 + 成对 Progress/日志；给 next action。

#### 7.6.5 fan-out 部分失败聚合（禁止静默吞）

- 一批并行 `Task`（同轮多 tool call）中**部分失败**：每个 `Task` 各自回传自身 `status`（`completed`/`failed`），父模型据此决定重试/继续——不做「一失全败」也**不静默丢弃**失败项（避免 Codex swarm best-effort 吞错的坑）。
- 若产品需要聚合视图（Desktop 画布 / Enterprise 报表），由 `EventSink` 投影层汇总，不改单个 `Task` 的独立回传语义。

### 8.1 统一什么

| 统一项 | 说明 |
| --- | --- |
| 定义模型与校验 | 同一 `SubAgentDefinition` |
| 委派工具协议 | `Task*` schema / 错误码 / Catalog XML |
| 实例状态机 | pending / running / completed / failed / cancelled / interrupted |
| 事件名与字段 | `runtime/multiagent/event` + Progress kind 扩展 |
| 权限不变量 | 只收紧、Gateway 必经、Channel=agent；`permission_mode` 严格度序（§6.2.3） |
| 深度/并发语义 | 配置项名字与失败行为；`SlotLimiter` **端口契约**（reserve/commit/release） |
| 上下文/回传契约 | `fork_context` 过滤白名单（§7.4）、`summary` 来源与截断（§7.5） |
| Progress 路由 | 父时间线仅 `kind=subagent`；子 Run 内部走 `EventSink` 子通道（§9.2） |

### 8.2 各自独立什么

| 产品 | 独立点 |
| --- | --- |
| **CLI** | 文件 Source、内存/文件 Store、**in-process `SlotLimiter`**、TUI 状态栏 + 结束后摘要、`genesis agent` / `agents validate` 命令、轻量后台任务；子 `EventSink` 可 no-op |
| **Desktop** | SQLite 投影、in-process `SlotLimiter`、**订阅子 `EventSink`** 渲染画布/transcript 面板、任务看板消费 Graph 事件、本地通知 |
| **Enterprise** | PG 持久化、租户隔离、RBAC 对 Definition/Invoke、审批上抛、SSE/Webhook、OTel、配额与审计检索、**分布式 `SlotLimiter`（二期）+ 分布式调度** |

**三端同源、后端异构的落点**：同一 Controller/Task 工具代码，通过 bootstrap 注入 `Store` / `SlotLimiter` / `EventSink` / `Source` 四类端口的不同后端实现三端差异；内核零 `if product == ...` 分支。

为后续 Enterprise 后台任务管理预留的端口分三层：`LeaseStore` 负责跨进程 ownership 和过期租约，`HeartbeatStore` 负责管理面判断活跃/失联，`CancellationStore` 只写入 stop intent 并由持有租约的 worker 消费。当前代码提供产品无关 Manager 与内存控制 Store，用于验证 ownership、heartbeat tick、stop intent 消费和终态释放；shared bootstrap 已接入进程内默认实现，不接 DB/Redis，不承诺跨进程远程取消。

### 8.2.1 三端差异点（端口后端差异，非内核分支）

| 差异点 | CLI/Desktop | Enterprise | 归属 |
| --- | --- | --- | --- |
| 后台子 Agent 存活时长 | 进程内、随会话结束回收 | 需持久化 + 配额 + 跨会话可查 | Store 端口后端差异 |
| `EventSink` 消费 | CLI 仅聚合状态栏 | Desktop 画布 / Enterprise SSE 全量投影 | EventSink 端口后端差异 |

> **已拍死、不再作为产品权衡**：「委派若触发**需上抛的人工审批**则**强制异步**」是**内核硬规则**（见 §7.6.1），三端一致——不允许同步 `Task` 阻塞父 Turn 干等人工审批。此处只剩 Enterprise 侧的**实现细节**（异步上抛如何接 Approval/SSE 队列），属 Phase 3 落地，不再改变该硬规则。

### 8.3 Profile 建议

| 产品 Profile | SubAgent |
| --- | --- |
| CLI 默认 | 启用内置 explore/plan/general-purpose + 项目/用户 agents；`Task*` 进 Enabled Tools |
| Desktop | 同 CLI + UI 管理自定义 Agent |
| Enterprise | 默认仅管理员/策略下发的 agents 进入 Catalog；终端用户自定义受租户策略显式开启；`SandboxRequire` 对子 Agent 不可降级 |

### 8.4 业务场景 → 默认与不变量映射（决定各字段缺省，非另立机制）

以下把 §7.6 / §10 的不变量落到**场景默认值**；同一套内核 + 定义字段，靠默认与策略适配不同业务，不为某场景新造旁路。

| 场景 | `execution_mode` | `max_depth` | 并发 / 预算 | 幂等 | `timeout` | 审批 | 典型产品 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 代码探索 / 规划 | sync | 1 | 3–5，token 小 | 不需要 | 秒~分 | 无 | CLI / Desktop |
| 深度研究 / RAG fan-out | sync 或 async | 1 | **高并发（可 5–10）+ 预算闸** | 建议（可缓存复用） | 分级 | 无 | 全端 |
| 业务办理 / 合规审查 | **async** | 1（必要时 2） | 低并发 + **严格预算/配额** | **必须幂等键** | 分~时 | **人工上抛（异步）** | Enterprise |
| 监控 / 告警 / 长期常驻 | async 常驻 | 1 | **独立配额池**，不占交互会话 | 幂等（崩溃重启） | 无固定 / 周期心跳 | 策略 | Enterprise（`from=system`） |

**要点**：`max_depth=1` 是安全默认，但业务办理确有「主办理 Agent → 子校验 Agent」两层需求，此时由**企业策略/Definition 显式放宽到 2**；**硬上限 = 2，任何配置都不能突破**（>2 拒绝加载，第 3 层 spawn 工具层硬失败）。放宽到 2 时限流与预算须按子树累计（详见 §6.1.3「深度硬上限」）。

## 九、Progress、日志、审计与 Usage

启动子智能体时，**Progress 与 agent 日志都必须有**；二者职责不同，不得互相替代。审计 / Usage 按治理需要另行写入（见下）。实现须同时改 `docs/运行进度事件与前端展示契约.md` 与遵循 `docs/superpowers/specs/2026-07-09-runtime-log-rotation-design.md`。

### 9.1 硬规则

| ID | 规则 |
| --- | --- |
| O1 | Spawn / Resume / Stop / 终态（completed、failed、cancelled）**必须**发 Progress，且 **必须**写 `agent.log`。缺任一端视为实现缺陷。 |
| O2 | Progress 不是日志；日志不是 Progress。UI 只消费 Progress（及产品 Graph 事件）；排障检索走三类日志文件。 |
| O3 | 凡 SubAgent 相关 `agent` / `audit` / `usage` 日志：**不得漏 `run_id`、`session_id`**。有值必须作为**顶层字段**写出（经 `correl.Enrich` / `correl.AttachLogger`），禁止只埋在自由文本里。 |
| O4 | 父会话的 `session_id` **必须**随 Spawn 继承到子 context；不得因开启子 Run 而清空。 |
| O5 | 关联键语义固定：`run_id` = **父** Run id（排障主检索键）；`child_run_id` / `agent_id` 另字段；子 Run 内部循环写日志时仍带父 `run_id` + `session_id`，并带 `child_run_id`。 |

### 9.2 Progress（UX）

- `kind` **锁定**为 `subagent`（扩展现有 `progress.Kind`；不复用含糊的 `run` + metadata 猜测）。
- `Event.run_id` = **父** run_id（与主线程其他 Progress 同维，便于一条时间线）。
- 必带 metadata / 顶层扩展：`session_id`、`agent_id`、`child_run_id`、`parent_run_id`（可与 `run_id` 同值）、`subagent_type`。
- 必带：`phase`、`summary`（一行可读，如 `启动子智能体 explore`）。
- `component` 建议：`subagent`；`name`：工具名或动作（`Task` / `spawn` / `resume` / `stop`）。

**子 Run 内部 Progress 的父子路由（锁定，避免时间线串味）**：

子 Run 由 `RunEngine.Start` 起，会自发 `KindRun`/`KindLLM`/`KindTool` 等 Progress，其 `run_id = child_run_id`。若直接灌进父 Sink，会让「带 child_run_id 的工具事件」混入父时间线，前端无法归位。故规定：

| 事件来源 | 用哪个 Sink | `run_id` | 是否进父主时间线 |
| --- | --- | --- | --- |
| 父侧委派节拍（`kind=subagent` 的 start/progress/complete/error） | **父 Sink** | **父 run_id** | 是（唯一进入父时间线的子 Agent 事件） |
| 子 Run 内部（`kind=run/llm/tool/...`） | **子 `EventSink`（独立通道）** | child_run_id | 否（默认不上抛） |

子 `EventSink` 是内核端口（§8.1），三端注入不同后端：
- CLI：no-op 或仅落日志；父时间线只出现聚合的 `kind=subagent`。
- Desktop/Enterprise：**订阅子 `EventSink`**（或由 multiagent event 投影）渲染独立 transcript / 画布节点，靠 `agent_id`+`child_run_id` 与父 `kind=subagent` 事件关联，**不**把子内部事件塞回父主流。

最低事件集（Phase 1 同步委派）：

| 时机 | phase | summary 示例 |
| --- | --- | --- |
| Controller 接受 Spawn、子 Run 即将开始 | `start` | `启动子智能体: explore` |
| 子 Run 内关键节拍（可选，防刷屏可抽样） | `progress` | `子智能体执行中: explore` |
| 子 Run 成功结束 | `complete` | `子智能体完成: explore` |
| 失败 / 取消 / 超限 | `error` | `子智能体失败: explore` |

Phase 2 后台：`async_launched` 对应父侧 `start`（或 `progress`）标明后台；`TaskOutput`/`TaskStop` 各发对应 Progress。

CLI 展示：状态栏可显示「启动子智能体: explore」；Run 结束后摘要须能出现 spawn/complete 条目（对齐进度契约 §4）。

Desktop/Enterprise 可另订 Graph 事件：`subagent.spawned`、`edge.created`、`subagent.completed`（由 Progress / multiagent event 投影，不替代 Progress）。

### 9.3 日志（排障，`agent.log`）

对齐运行日志三类通道：SubAgent 运行叙事进 **agent**；审批进 **audit**；计量进 **usage**。

Spawn 成功路径至少打两条 agent 日志（失败路径在失败点打 error，仍带全套关联键）：

1. **spawn start**：即将启动子 Run（已分配 `agent_id` / `child_run_id`）。
2. **spawn end**：子 Run 终态（complete / error / cancelled），含短错误信息（无密钥、无超长 transcript）。

字段契约（**全部为结构化键，缺省即违规**）：

| 字段 | 是否必须 | 说明 |
| --- | --- | --- |
| `run_id` | **必须** | 父 Run id；与 I8 检索主键一致 |
| `session_id` | **必须**（父有则子必有） | 跨 Run 会话辅键；父 context 有值时禁止漏传 |
| `agent_id` | 必须 | 委派句柄 |
| `child_run_id` | 必须（已分配后） | 子 RunEngine 的 run id |
| `parent_run_id` | 建议 | 显式等于父 `run_id`，避免阅读歧义 |
| `subagent_type` | 必须 | 如 `explore` |
| `component` | 必须 | 固定 `subagent` |
| `from` | 建议 | `task` / `skill_fork` / `system` |

实现要点：

- Spawn 入口：`log = correl.AttachLogger(parentCtx, runtimeLogger)`，再 `With("component","subagent","agent_id",...,"child_run_id",...,"subagent_type",...)`。
- 开启子 Run 前：`childCtx` 注入 `session_id`（继承）+ 子侧需要的 id；子循环内继续 `AttachLogger(childCtx, ...)`，**不得**只带 `child_run_id` 而丢掉父 `run_id`/`session_id`。
- 写 audit/usage 时同样 `correl.Enrich`；file Sink 仅作兜底，**不能**当成「调用方可以漏填」的借口。

检索约定：按父 `run_id` 可拉出主 Agent + 全部子智能体叙事；需要单看子执行再用 `child_run_id` 或 `agent_id` 过滤。

### 9.4 审计与 Usage

- 审计决策键：`Task(subagent_type)` / `Task(resume)` / `TaskStop`；资源含 `agent_id`、`parent_run_id`；**同样必须带 `run_id` + `session_id`**。
- Usage：子 Run token / 工具次数归 `child_run_id`（及 `agent_id`），并可汇总到父 `run_id`；事件 metadata 仍带 `session_id`。
- 需要排障文案 + 审批真相时：**agent 一句摘要 + audit 结构化事件**（日志分类设计 §2.2），两边都含关联键。

### 9.5 验收（实现自检）

- 任意一次成功 `Task` 委派：Progress 至少有 `start` + `complete`（或 `error`）；`agent.log` 至少有对应 start/end。
- 对 `agent.log` / `audit.log` / `usage.log` 中该次委派相关行：`run_id`、`session_id` 均可被检索（父会话存在 `session_id` 的前提下）。
- 单元/集成测试覆盖：`correl.AttachLogger` 后子 Spawn 日志字段；故意清空 session 的负例不允许合入。

## 十、安全与治理

1. **CapabilityScope**：SubAgent 定义与 `Task` 工具均受 scope；`ChannelAgent` 用于「被子调用时是否仍可见某危险工具」。
2. **编排工具硬禁列表**（子上下文默认）：`Task`、`TaskOutput`、`TaskStop`，以及未来的协作空间管理工具。
3. **Sandbox**：企业 `SandboxRequire` 对子 Agent 同样生效；禁止子定义把 require 降为 none。
4. **密钥**：子 Agent 不自动获得父未授权的 connection/credential；继承策略显式配置。
5. **提示注入**：子 Agent 的用户可控 prompt 按不可信输入处理；定义 body 来自可信 Source 时仍要防项目目录投毒（校验 + 企业策略）。
6. **租户不变量（Enterprise 硬约束）**：子 Run **必须继承父 `tenant_id` 且不可篡改**；子 Agent 的所有 DB 查询 / 资源访问带该 `tenant_id`（对齐 AGENTS.md 多租户规则）；定义或入参**不得**跨租户指定数据域。跨租户委派视为安全事件。
7. **预算 / 配额治理**：预算闸（§7.6.2）在 Enterprise 与 RBAC/配额联动——按租户、用户、Agent 定义分配子 Run token/次数配额；耗尽即拒绝新 spawn 并审计，禁止无上限烧钱。
8. **system 发起的常驻子 Agent（监控 / 长期执行）**：`created_from=System` 的 agent **不绑单个交互会话**，走独立配额池与生命周期（策略启停、周期心跳、崩溃重启幂等），不占交互会话的 `max_concurrent`；其治理（谁能创建常驻 agent、资源上限）由 Enterprise 策略下发。

## 十二、测试策略

本节描述完整路线的测试目标，不表示所有条目均已在第一阶段完成。第一阶段已通过的测试证据以 §1.4 和 §11 为准；权限策略、业务幂等、生产租户治理等用例随对应后续能力落地。

- 单元：frontmatter 解析、合并优先级、工具求交、权限只收紧、depth/concurrency。
- 契约：`Task` schema、Catalog 渲染、ToolResult JSON。
- 集成：父 ReAct → Task → 子 Run → 摘要回传；fork skill；resume；**Progress start/complete 与 agent.log 成对出现**。
- 关联键：Spawn 路径日志断言 `run_id`、`session_id`、`agent_id`、`child_run_id`；子 context 继承 `session_id`。
- 产品隔离：`go list -deps` 确保 CLI 不拉 PG/Wails，Enterprise 不拉 `shared/local` 主机执行。
- **并发正确性**：`max_concurrent` 满时第 N+1 个 `Spawn` 硬失败；reserve 后 `startChildRun` 失败/ panic 必须 `release` 归还槽位（无泄漏）；并行多个 `Task` 达上限时的行为可断言。
- **取消/超时**：父 `context` 取消 → 在途子 Run 级联 `cancelled` + 槽位归还 + 成对 Progress/日志；`max_turns`/wall-clock 超时终态正确。
- **Progress 路由**：断言父时间线仅出现 `kind=subagent`（run_id=父）；子 Run 内部 `kind=tool/llm` 不进父 Sink。
- **权限只收紧**：父/定义各组合下 `min(严格度)` 取值正确；放宽用例必须失败。
- **预算熔断**：超 `max_tokens`/`max_tool_calls` 的子 Run 被硬失败并回传可读错误；会话累计预算生效。
- **幂等**：同 `idempotency_key` 重复 spawn 返回既有实例，不产生第二个副作用（业务办理用例）。
- **租户不变量**：子 Run 继承父 `tenant_id`；构造跨租户委派的用例必须被拒绝。
- **部分失败**：并行批中单子失败时，其余子 `completed` 正常回传，失败项不被静默丢弃。
- **输入快照安全**：父 `system/developer`、tool 调用/结果、当前 Task 调用不得进入子快照；可信运行约束只能来自 `InheritedRuntimeContract`。
- **输入构造一致性**：Source flush 后的完整回合可见；Builder 纯函数、确定性；脱敏器异常 fail closed；不可截断的 system contract / DelegationEnvelope 超预算时拒绝 Spawn。
- **结果收敛安全**：超长或含密钥的最终回答被脱敏并按语义边界截断；Reducer 对同一 TerminalCandidate 幂等；Reducer/证据校验失败返回安全最小 Envelope。
- **异步协议**：分别覆盖 launch、`TaskOutput` 的 `not_ready/timeout/ready`、终态结果；同一 `result_id` 重复拉取不重复注入父 transcript。
- **Artifact 授权**：未登记、非法路径、权限撤销后的 artifact 不进入结果；模型投影不得暴露宿主绝对路径，产品读取时重新鉴权。
- 负面：子 Agent 调用 `Task` 应失败；扩权 tools 应失败；禁用 capability 后 Catalog 消失；有 session 却漏 `session_id` 的日志视为失败。

## 十五、决策摘要

1. **体验对齐 Kode/Claude**：内置常驻 + Markdown 用户 Agent + 锁定网关 `Task*`；零配置仍可提示词驱动委派。  
2. **工程对齐 Codex**：会话级 Controller、独立实例、深度/并发硬限；可选 `explicit_request_only`；不引入复杂度自动 fork。  
3. **边界对齐 Genesis**：与 Skill 网关同构；内核 / 产品分离；权限只收紧；委派必发 Progress + agent 日志且带 `run_id`/`session_id`。  
4. **干净实现**：不兼容半吊子预留；实现时删除冲突代码；一套工具面做到底。  
5. **扩展路径**：第一阶段单进程安全闭环 → 第二阶段统一治理与运行时可管理性 → 第三阶段产品/分布式深化 → 第四阶段协作空间。

## 十六、已拍板项与残留风险

**目标态已拍板（实现不得再争论出双轨）**：下表同时包含已实现规则和后续阶段必须遵守的设计决策；是否已落地仍以 §1.4 为准。

| 项 | 结论 |
| --- | --- |
| LLM 工具名 | `Task` / `TaskOutput` / `TaskStop` |
| Progress `kind` | `subagent`；Spawn 启停必发 |
| 日志 | Spawn 必写 `agent.log`；**不得漏 `run_id` / `session_id`**（顶层字段） |
| 双路径 | 无用户 md 时内置仍可委派；「自动」= 提示词驱动，禁止复杂度引擎静默 Spawn |
| `delegation_posture` | 默认 `proactive`（CLI/Desktop）；Enterprise 可 `explicit_request_only` |
| `max_depth` / `max_concurrent` | `max_depth` 默认 **1、硬上限 2**（仅显式放宽，>2 拒绝加载、第3层 spawn 硬失败；depth=2 时限流/预算按子树累计）；`max_concurrent` CLI=3·Desktop=4（工具层硬限） |
| 第一阶段子上下文 Skill | 默认禁用；`Skill(context=fork)` 统一转入 Task/Controller，不开放子 Run 内直接 Skill 网关 |
| 第一阶段嵌套 Task | 默认禁用（`max_depth=1`）；Definition 可显式放宽至硬上限 2 |
| Skill fork Definition | `meta.Agent` 优先且必须可解析；否则合成 `skill-fork:<qualified>` |
| `.claude` / `.kode` agents | 可选 Source，默认关闭；内核零依赖 |
| Controller 粒度 | **根会话级**（per root session，抽象 `SessionScope`）；`max_concurrent` 按会话累计，父 Run 不占子槽 |
| 并发槽 | **`SlotLimiter` 端口**（reserve→commit/release，defer 兜底）；CLI/Desktop in-process、Enterprise 分布式后端；后台占槽至终态 |
| 三端同源 | Controller/Task 代码统一；`Store`/`SlotLimiter`/`EventSink`/`Source` 四端口注入实现三端差异；内核零产品分支 |
| 并行 fan-out | 第一阶段复用 `scheduler` 同轮并发多个 `Task`；Controller 不自建编排 |
| 取消/超时 | 父 ctx 取消级联子 Run cancelled + 归还槽位；`max_turns`+可选 wall-clock |
| Progress 路由 | 父时间线仅 `kind=subagent`（run_id=父）；子 Run 内部事件走独立子 Sink，默认不上抛 |
| `summary` 来源 | 子 Run 终态最终 assistant 回答，经 Reducer 脱敏、语义截断；失败给分类错误摘要 + next action |
| `fork_context` 过滤 | Source flush/materialize；Builder 白名单裁剪（保留 user/最终回答/摘要，不原文继承 system/developer，丢工具轨迹/推理）；单一纯函数 |
| 结果交付 | Reducer create-once 生成规范化 Record；Projector 每次交付重新鉴权；`result_id` 幂等去重 |
| `permission_mode` | 定义全序，子 = `min(严格度)`，强约束（沙箱/scope）优先 |
| execution_mode | 场景化默认（探索 sync / 业务办理 async）；Enterprise 需审批上抛的委派强制 async |
| 预算熔断 | 第一阶段已实现 `max_tokens`/`max_tool_calls` 单次与子树累计硬限制；Usage/租户配额联动在后续阶段实现 |
| 幂等 | 业务办理 spawn 带 `idempotency_key`（默认派生自 `parent_run_id+tool_call_id`）；同键返回既有实例 |
| 租户 | 子 Run 继承父 `tenant_id` 不可篡改；跨租户委派视为安全事件 |
| system 常驻 | `from=system` 不绑交互会话、独立配额池、崩溃重启幂等 |
| 部分失败 | 并行批单子失败不牵连其余、不静默吞；聚合视图由 EventSink 投影 |
| 审批上抛 | 委派触发「需上抛人工审批」时**强制异步**（内核硬规则，三端一致，§7.6.1）；同步 `Task` 不得干等人工审批 |

**第一阶段结束时的已知限制**：

1. `permission_mode`、`sandbox_profile`、`skills` 和 Agent 级 `model` 尚未成为可执行 Definition 字段；当前严格解析器会拒绝这些目标态字段。
2. Enterprise 审批上抛、生产认证/RBAC、SSE、共享 Store、跨进程远程取消依赖后续治理与基础设施；第一阶段只承诺单进程内存控制闭环。
3. Marketplace/Plugin Definition 在启动时进入 Catalog，安装、启停后尚不能热更新；动态 RuntimeAdapter 后续实现。
4. CLI 已有管理命令但尚未接 TUI 状态摘要；Desktop 尚无 Wails 界面；Enterprise 只提供绑定服务端租户范围的安全只读投影查询。
5. 业务级 `idempotency_key`、模型白名单/配额/fallback、system 常驻 Agent 和 handoff/swarm/@Agent 协作空间均未实现。
