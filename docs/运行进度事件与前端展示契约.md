# 运行进度事件与前端展示契约

## 1. 结论

Agent 运行过程不能只在 UI 层显示“Agent 思考中”。应在公共 runtime 层定义结构化 **Progress Event**，由 ReAct、Tool、Skill、Sandbox、文件执行等能力发出事件，再由 CLI、Desktop、Enterprise 各自按场景渲染。

公共层只回答“发生了什么”，不决定“怎么展示”。

## 2. 公共层职责

公共层位于 `internal/runtime/progress`，事件字段包括：

- `kind`：`run`、`llm`、`tool`、`skill`、`sandbox`、`file`、`system`、`subagent`。
- `phase`：`start`、`progress`、`complete`、`error`。
- `level`：`debug`、`info`、`warn`、`error`。
- `run_id`、`step_id`、`call_id`：用于关联 trace、audit 和 UI 时间线。
- `component`、`name`：组件和动作名，例如 `genesis-sandbox`、`read_file`、`subagent`/`Task`。
- `summary`：适合一行展示的短文本。
- `detail`：可折叠详情，CLI 可截断，Web/Desktop 可展开。
- `metadata`：结构化扩展字段（SubAgent 必含 `session_id`、`agent_id`、`child_run_id`、`subagent_type` 等，见下）。

设计约束：

- 事件发送必须非阻塞，不能影响主运行流程。
- 事件不是审计替代物，审计仍走 audit sink；事件用于运行可见性和用户体验。
- 事件不泄露敏感内容，参数和结果详情必须截断或按产品策略脱敏。
- Progress Event 不应该被当作普通 assistant 消息实时插入对话历史，否则会造成重复展示和上下文污染。

## 3. 展示分层

| 产品 | 默认展示 | 详情展示 | 持久化 |
| --- | --- | --- | --- |
| CLI TUI | 底部状态栏展示最新事件，例如“调用工具: read_file” | Run 结束后展示最近关键动作摘要 | 默认不持久化，仅在当前会话 UI 中展示 |
| CLI 非交互/CI | 按 `--progress=auto|off|text|jsonl` 输出事件；默认普通模式输出 text，`--json`/`--quiet` 自动关闭 | JSONL 或压缩摘要，输出到 stderr，避免污染 stdout | 可由 CI 日志保存 |
| Desktop | 时间线/步骤面板，支持折叠参数和结果 | 详情抽屉、错误定位、artifact 链接 | 可按会话保存 |
| Enterprise Web/API | SSE/WebSocket 事件流，前端渲染时间线 | 结合 trace/audit/usage 做治理视图 | 按租户策略持久化 |

## 4. CLI 展示规则

CLI TUI 使用两层信息：

1. **实时状态栏**：只展示最新事件，避免刷屏。
2. **运行过程摘要**：Run 完成或失败后，把最近关键动作一次性写入历史，例如：

```text
运行过程:
- 启动 Agent 运行
- 第 1 轮调用 LLM
- 调用工具: read_file
- 执行工具: read_file
- 工具执行完成: read_file
- 启动子智能体: explore
- 子智能体完成: explore
- sandbox job 已完成
```

说明：非流式下 `tool_input` 的 `phase=complete` 仅闭合协议块，`display=false`，不写入运行过程，避免与带参数的「调用工具」重复。

流式 LLM 中间思考（工具前 `Content` / CoT）使用 `assistant_draft` 或 `thinking` 且 `display=true`，写入当前 Turn 的 Agent 气泡，供用户查看推理过程；多轮之间空行分隔。确认无工具调用后另发 `final_answer`（保留已展示的思考，再接「最终回答」分段）。进度通道必须可靠投递，避免丢事件导致段落叠加；TUI 对累计型 stream delta 做合并去重。

不应把工具进度事件插入对话历史。状态栏只展示最新事件；运行过程摘要在 Run 结束后写入。

## 5. 事件来源规范

推荐事件来源：

- ReAct Loop：run start/complete/error、LLM start/complete/error、final answer。
- Tool Registry / Tool Scheduler：tool start/complete/error。
- Skill Loader：Skill、read_skill_resource、search_skill_resources。
- Sandbox Adapter：lease、renew、stage input、submit job、poll job、download artifact、release。
- File Tools：read/write/edit/search 的目标摘要，路径展示按权限和产品策略脱敏。
- SubAgent Controller：`kind=subagent` 的 start / progress / complete / error（委派启停必发；细节见 `docs/子智能体设计.md` §9）。

### 5.1 SubAgent Progress

| 约定 | 要求 |
| --- | --- |
| `kind` | 固定 `subagent` |
| `run_id` | **父** Run id（与主时间线对齐） |
| metadata | 至少：`session_id`、`agent_id`、`child_run_id`、`subagent_type`；建议含 `parent_run_id` |
| 启停 | Spawn 接受时 `phase=start`；终态 `complete` 或 `error`；不可只打日志不发 Progress |
| 展示文案 | summary 一行，如「启动子智能体: explore」 |

Progress **不替代** `agent.log`。子智能体相关排障日志须带顶层 `run_id` 与 `session_id`（见运行日志分类设计关联键契约）。

## 6. 非目标

- 不把 Progress Event 当成模型上下文。
- 不把 Progress Event 替代 trace/audit/usage。
- 不在公共层写 CLI、React、Desktop 组件逻辑。
- 不默认展示完整工具参数或完整工具结果。
