# Genesis Agent 目录结构重构建议

> 目的：在不推翻当前架构原则的前提下，解决目录按技术横切分层导致的分散感，收敛为更适合 `genesis-agent` 长期演进的目录结构与命名方案。

---

## 1. 结论

迁移前目录：

```text
internal/
  adapters/
  app/
  bootstrap/
  contracts/
  domain/
  engine/
  infra/
  transport/
```

这套结构本身不是错的，优点是边界清晰、依赖方向明确、适合项目早期建立架构纪律；本次迁移就是在此基础上做收口与正名。

但随着能力变多，它会逐渐暴露三个问题：

1. **能力实现被按技术层切散**  
   例如一个 LLM、Memory 或 Tool 能力，经常同时散落在 `contracts`、`adapters`、`app`、`transport` 等多个目录中。
2. **目录命名过泛，语义分辨率不够**  
   `app`、`infra`、`transport`、`contracts` 都是常见名词，但当能力增多时，容易让人纠结“这个文件到底该放哪”。
3. **新增能力时开发路径不顺手**  
   做一个完整功能时，经常要跨多个顶层目录来回跳转，认知成本偏高。

因此，建议做一次 **收口 + 正名 + 渐进迁移**，而不是推翻重来。

---

## 2. 目标原则

新的目录结构建议同时满足以下目标：

1. **保留分层纪律**：核心运行时、基础设施、接入层、能力实现仍然要有边界。
2. **按能力域聚合**：LLM、Tool、Memory、Sandbox Client 等能力尽量在同一片目录中聚拢。
3. **让目录名更贴近语义**：少用过泛名词，多用更接近实际职责的命名。
4. **支持渐进迁移**：允许旧目录逐步收拢，避免一次性大迁移破坏开发节奏。
5. **兼容未来拆分**：后续若某项能力独立成模块、服务或 SDK，目录结构应天然支持。

---

## 3. 推荐目标结构

推荐将 `internal/` 逐步收敛为六个一级域：

```text
internal/
  bootstrap/      # 依赖装配根
  app/            # 少量跨域应用编排（逐步收缩）
  platform/       # 通用基础设施与跨域底座
  runtime/        # Agent Runtime 内核
  capabilities/   # 各能力域
  interfaces/     # 外部接入面
```

推荐完整结构：

```text
internal/
  bootstrap/
  app/

  platform/
    cache/
    config/
    db/
    errors/
    httpclient/
    idgen/
    logger/
    observability/

  runtime/
    action/
    context/
    session/
    stop/
    strategy/

  capabilities/
    llm/
      contract/
      adapter/
      service/
      model/

    tool/
      contract/
      adapter/
      service/
      model/

    memory/
      contract/
      adapter/
      service/
      model/

    sandbox/
      contract/
      adapter/
      service/

    trace/
      contract/
      adapter/

    profile/
      contract/
      adapter/
      service/
      model/

    usage/
      contract/
      adapter/
      service/
      model/

  interfaces/
    http/
      dto/
      handler/
      router/
    cli/
      tui/
```

说明：

- `bootstrap` 保留为系统装配根，不归属某个能力域。
- `app` 只保留少量跨能力域应用编排，不再承接所有域内 service。
- `platform` 放全局可复用底座，不绑定某个能力域。
- `runtime` 是 Agent 运行时内核，不与具体能力实现耦合。
- `capabilities` 是能力聚合层，是本次重构的重点。
- `interfaces` 是系统外部入口，比 `transport` 更贴切当前职责。

## 4. 命名优化建议

### 4.1 `transport` -> `interfaces`

原因：

- `transport` 更像传输协议层。
- 当前目录实际承载的是 HTTP、CLI、TUI、SSE、未来 Webhook/Desktop 等系统外部入口。
- 用 `interfaces` 更符合“接入面”的真实职责。

推荐：

```text
internal/interfaces/http
internal/interfaces/cli
```

### 4.2 `infra` -> `platform`

原因：

- `infra` 往往让人联想到运维基础设施。
- 当前项目里的 `config`、`idgen`、未来的 `httpclient`、`errors`、`observability`，更适合作为“平台底座”。
- `platform` 的语义包容性更强，也更贴近工程实践。

推荐：

```text
internal/platform/config
internal/platform/idgen
internal/platform/httpclient
internal/platform/errors
```

### 4.3 `engine` 上收为 `runtime`

原因：

- 项目的核心不是单一执行引擎，而是一个 Agent Runtime。
- 当前的 `engine/runtime` 命名组合读起来容易绕。

推荐：

```text
internal/runtime/
  action/
  context/
  session/
  stop/
  strategy/
```

若需要总入口文件，可采用：

```text
internal/runtime/engine.go
```

### 4.4 不再继续扩大全局 `contracts/`

原因：

- `contracts/llm`、`contracts/tool`、`contracts/memory` 会随能力数线性膨胀。
- 这样会让“能力的接口”和“能力的实现”分散在太远的位置。

推荐：

```text
internal/capabilities/llm/contract/
internal/capabilities/tool/contract/
internal/capabilities/memory/contract/
```

### 4.5 不再继续扩大全局 `adapters/`

原因：

- 全局 `adapters` 最后容易演化成一个按技术实现堆砌的大目录。
- 实际上，大多数 adapter 都从属于某个能力域。

推荐：

```text
internal/capabilities/llm/adapter/
internal/capabilities/tool/adapter/
internal/capabilities/memory/adapter/
```

### 4.6 `app` 更建议域内改名为 `service`

原因：

- `app` 太泛，长期看会越来越抽象。
- 在能力域内部，`service` 更贴近“用例编排与应用服务”的职责。

推荐：

```text
internal/capabilities/llm/service/
internal/capabilities/tool/service/
```

### 4.7 `bootstrap` 保留顶层，不建议下沉

原因：

- `bootstrap` 的职责是系统依赖装配、容器初始化、运行时与接口层的组装。
- 它天然不属于某个能力域，也不属于平台基础设施的具体子模块。
- 将其继续保留在顶层，能让系统入口更清楚。

推荐：

```text
internal/bootstrap/
  container.go
  runtime.go
  interfaces.go
  capabilities.go
```

说明：

- 当前即使只有一个 `container.go` 也没问题。
- 后续规模变大时再细分，不建议为了统一风格而硬迁移。

### 4.8 `app` 保留，但要收职责

原因：

- `app` 作为顶层目录并非完全错误，但它最容易因为名字过泛而持续膨胀。
- 如果所有能力自己的应用服务都继续往 `internal/app` 放，它会逐渐变成一个“大杂烩”。

建议：

- `internal/app` 只保留少量跨能力域 orchestration。
- 各能力自己的应用服务优先进入对应的 `internal/capabilities/*/service`。
- 后续若跨域编排逐渐增多，可视情况将 `app` 再重命名为 `orchestration`，但当前无需立刻调整。

推荐：

```text
internal/app/
  run_orchestrator.go
  agent_orchestrator.go
```

约束：

- `app` 不再承接 LLM、Tool、Memory 等单一能力域内部 service。
- `app` 中的逻辑应以“跨域编排”为主，而非具体实现细节。

## 5. 旧目录到新目录映射建议

### 5.1 可以直接迁移的目录

| 当前目录 | 推荐目录 | 说明 |
|---|---|---|
| `internal/infra/config` | `internal/platform/config` | 明确是平台配置底座 |
| `internal/infra/idgen` | `internal/platform/idgen` | 全局通用能力 |
| `internal/transport/http` | `internal/interfaces/http` | 外部 HTTP 接入面 |
| `internal/transport/cli` | `internal/interfaces/cli` | 外部 CLI 接入面 |
| `internal/engine/*` | `internal/runtime/*` | 运行时内核归拢 |
| `internal/bootstrap` | `internal/bootstrap` | 顶层保留，作为系统装配根 |
| `internal/app` | `internal/app` | 顶层保留，但仅限少量跨域编排 |

### 5.2 建议逐步迁移的目录

| 当前目录 | 推荐目录 | 说明 |
|---|---|---|
| `internal/contracts/llm` | `internal/capabilities/llm/contract` | 与能力实现聚拢 |
| `internal/adapters/llm` | `internal/capabilities/llm/adapter` | 域内化 adapter |
| `internal/contracts/tool` | `internal/capabilities/tool/contract` | 与 registry / builtin 聚合 |
| `internal/adapters/tool` | `internal/capabilities/tool/adapter` | 后续接 gateway / sandbox 更自然 |
| `internal/contracts/memory` | `internal/capabilities/memory/contract` | 为短期/长期记忆收口 |
| `internal/adapters/memory` | `internal/capabilities/memory/adapter` | file/db/redis/vector 更好展开 |
| `internal/contracts/trace` | `internal/capabilities/trace/contract` 或 `internal/platform/observability/trace` | 二选一，取决于是否保留 trace 为独立能力 |
| `internal/adapters/trace` | 同上 | 建议和 trace contract 同域 |

### 5.3 暂时不要急着迁的目录

| 当前目录 | 原因 |
|---|---|
| `internal/domain` | 当前规模还小，先别为了“统一风格”强行拆散 |

## 6. 针对当前项目的专项建议

### 6.1 LLM

推荐目标：

```text
internal/capabilities/llm/
  contract/
  adapter/
    eino/
  service/
  model/
```

原因：

- LLM Provider、Model Pool、Router、调用策略、本地配置会持续增长。
- 继续拆散在全局 `contracts` 与 `adapters` 下，后面会越来越难维护。

### 6.2 Tool

推荐目标：

```text
internal/capabilities/tool/
  contract/
  adapter/
    builtin/
    registry/
  service/
  model/
```

原因：

- 后面还会接 Tool Gateway、权限、审计、沙箱执行。
- Tool 是高频扩展点，最需要聚拢。

### 6.3 Memory

推荐目标：

```text
internal/capabilities/memory/
  contract/
  adapter/
    inmemory/
    file/
    db/
  service/
  model/
```

原因：

- 记忆很快会从简单接口演进到短期记忆、长期记忆、用户画像、记忆管理 API。
- 提前按能力域收口，能明显降低后续复杂度。

### 6.4 HTTP 请求工具

推荐落点：

```text
internal/platform/httpclient/
```

原因：

- 它是平台通用基础设施，不属于某个特定能力域。
- 会被 LLM Adapter、Sandbox Client、Webhook、未来外部 API 调用共同复用。
- 放在 `capabilities` 下会显得过窄，放在 `platform` 下更合理。
- 这里应承载的是“项目级 HTTP 基础设施能力”，而不是某个具体上游协议的业务语义。

核心建议：

- **基于 `net/http` 自建一层项目级 HTTP Client**，不要直接把 `resty` 作为核心抽象。
- `resty` 不是不能用，但更适合快速接普通 REST API；当前项目后续会遇到 JSON、SSE、Webhook、Sandbox、MCP Gateway、长连接流、统一 Trace/审计/错误治理等场景，长期看还是自有抽象更稳。
- 如果未来确实要借用第三方库能力，也应藏在 `platform/httpclient` 内部，不能把第三方调用风格扩散到上层能力代码里。

推荐职责边界：

```text
platform/httpclient        # 通用 HTTP 能力：请求发送、超时、重试、认证、日志、trace、SSE、错误归一
capabilities/*/adapter/http # 具体上游协议适配：Sandbox / LLM Provider / Webhook / External API
interfaces/http           # 服务端对外暴露的 HTTP 入口，不复用 client 侧协议模型
```

也就是说：

- `platform/httpclient` 负责“怎么稳定地发 HTTP 请求”。
- `capabilities/sandbox/adapter/http` 负责“怎么把 Sandbox 领域请求映射成 HTTP”。
- `capabilities/llm/adapter/...` 负责“怎么把模型调用映射到不同 Provider 协议”。
- `interfaces/http` 负责“主平台如何对外提供 HTTP API”。

建议子结构：

```text
internal/platform/httpclient/
  client.go        # Client 接口与入口实现
  request.go       # Request / Response / StreamRequest 等通用模型
  errors.go        # 统一错误模型与错误分类
  transport.go     # http.Transport / RoundTripper 装配
  middleware.go    # middleware / decorator 链
  retry.go         # RetryPolicy、退避与 Retry-After 处理
  auth.go          # bearer / api-key / basic / custom header
  json.go          # JSON encode / decode
  sse.go           # SSE 事件流读取与解码
  logging.go       # 请求日志、脱敏、截断
  trace.go         # trace/span 注入
```

第一版建议能力：

- `Do(ctx, req)`
- `StreamSSE(ctx, req)`
- `BaseURL + Path + Query` 拼装
- Header / Auth
- JSON 编解码
- Timeout
- Retry
- 统一错误模型
- Logging / Trace
- 可配置 `http.Transport`
- `Context` 全链路贯穿

错误模型建议统一，而不是直接向上抛 `net/http` 原始错误。至少包含：

- `timeout`
- `canceled`
- `network`
- `unauthorized`
- `forbidden`
- `not_found`
- `rate_limited`
- `upstream`
- `invalid_response`
- `decode`
- `sse`

并建议附带：

- `status_code`
- `retryable`
- `operation`
- `url`
- `request_id`
- `upstream_code`
- `raw_body`（可截断）

超时与重试建议：

- 普通 JSON 请求可以有默认总超时，如 30s 或 60s。
- SSE 不应复用同一套短总超时逻辑，更适合依赖 `context.Context` 控制取消。
- 默认只对幂等请求或显式带 `Idempotency-Key` 的请求重试。
- 默认可重试场景建议限定为：网络瞬时错误、`429`、`502/503/504`。
- 退避策略建议使用 `exponential backoff + jitter`，并尊重 `Retry-After`。

当前目录结构下，更推荐的接口落点是：

```text
internal/platform/httpclient/client.go
internal/platform/httpclient/request.go
internal/platform/httpclient/errors.go
```

而不是旧结构中的：

```text
internal/contracts/httpclient/
internal/adapters/httpclient/
```

因为现在项目已经从“全局横切 contracts/adapters”收敛为“platform + capabilities + interfaces”骨架，HTTP Client 作为平台底座，直接放 `platform/httpclient` 更符合当前架构方向。
### 6.5 Sandbox Client

注意：这里说的是主平台侧对接独立 `genesis-sandbox` 的客户端，而不是沙箱服务实现本体。

推荐落点：

```text
internal/capabilities/sandbox/
  contract/
  adapter/
    http/
    grpc/
  service/
```

原因：

- 它在主平台中属于一个能力域，而不是全局平台底座。
- 后面会涉及租约、作业、日志流、文件、策略映射等领域行为。

## 7. 渐进式迁移顺序

建议分三步走，避免一次性大改：

### 第一步：先改名字最明确的

目标：先把最“确定”的部分正名，提升整体语义。

推荐顺序：

1. `transport` -> `interfaces`
2. `infra` -> `platform`
3. `engine/*` -> `runtime/*`

这一步收益高、认知改善明显、风险相对可控。

### 第二步：新能力按新结构落，旧能力先不强迁

规则：

- 新增的 HTTP Client 直接放 `internal/platform/httpclient`
- 新增的 Sandbox Client 直接放 `internal/capabilities/sandbox`
- 新增的 Profile / Usage / Session 相关能力优先按 `capabilities/*` 落地

这样可以先建立新结构，不必立刻大搬迁。

### 第三步：把 LLM / Tool / Memory 逐步收口

顺序建议：

1. Tool
2. Memory
3. LLM

原因：

- Tool 和 Memory 最容易继续发散，优先级最高。
- LLM 当前虽然重要，但已有结构相对清晰，可以稍后再做域内化。

---

## 8. 迁移时的约束

1. **不要为统一而统一**  
   不是所有目录都要立刻迁完，避免“结构洁癖式重构”。
2. **新结构优先接新能力**  
   先用增量方式建立新目录，再逐步回收旧目录。
3. **避免同时引入太多新抽象**  
   本次重构目标是收拢，不是增加层数。
4. **尽量用更少但更清晰的顶层目录**  
   顶层目录越多，越容易再次分散。
5. **文档与 AGENTS 要同步**  
   一旦目录方案定稿，应及时同步 `AGENTS.md` 与 `docs/agent loop设计.md` 的描述。

---

## 9. 最终建议

综合当前项目阶段、能力增长趋势和长期演进需求，推荐的最佳实践是：

```text
internal/
  bootstrap/
  app/
  platform/
  runtime/
  capabilities/
  interfaces/
```

这是本项目当前最值得采用的目录收敛方向。

相较于现在的纯横向技术分层，这个方案的优势是：

- 更聚合，减少能力实现分散
- 命名更贴近真实职责
- 更适合 LLM / Tool / Memory / Sandbox 这类持续扩展的能力域
- 更适合未来引入统一 HTTP Client、Usage、Profile、Session、治理策略
- 明确保留 `bootstrap` 的系统装配职责，同时收缩 `app` 的语义边界
- 更方便独立服务、SDK、客户端和平台内核并行演进

一句话总结：

> 不是继续增加抽象层，而是把目录从“技术横切”收敛成“系统装配 + 跨域编排 + 平台底座 + 运行时内核 + 能力域 + 外部接口”。
