# Sandbox Runtime 设计方案（Go 实现版 - 全量终极版）

> 适用范围：AI平台的低代码 Code 节点、Agent 工具调用、MCP/Skills 执行、文件处理、数据分析、后续类 Claude Code / OpenClaw 的长期自主执行能力。  
> 技术路线：主平台调用独立的 Sandbox Service；Sandbox Service 使用 Go 实现；当前阶段只做稳定 Docker 版本：Windows/macOS 开发环境使用 Docker (`runc`)，Linux 生产优先 Docker + gVisor (`runsc`)，通过 Driver 接口抽象实现能力探测、策略降级与平滑切换。containerd / K8s / RuntimeClass / Kata / Firecracker / egress sidecar / Credential Vault 等云原生能力暂不实现，仅保留远期扩展点，等第三阶段或真实云原生需求出现后再评估。

> 说明：Sandbox Service 实现现已迁移至独立仓库 `D:\workspace\go\genesis-sandbox`。文档中若出现旧的 monorepo 示例，请以独立仓库结构与 HTTP/gRPC 对接边界为准。

---

## 1. 核心结论

沙箱不要做成主后端里的一个执行函数，也不要只用简单新进程来跑代码。推荐设计为：

```text
Genesis Agent 主平台
  ├─ Workflow / Agent Loop / Tool Router / Policy Engine
  └─ 调用 Sandbox Service Client (Adapters)
          │
          ▼ HTTP/gRPC (网络隔离)
Sandbox Service（Go 独立服务）
  ├─ 沙箱池管理 (Pool Manager)
  ├─ 统一 Driver 抽象 (RuntimeDriver)
  ├─ 容器生命周期管理
  ├─ 代码执行 & 工具执行
  ├─ 文件挂载与回收 (File Staging)
  ├─ 网络策略控制（Phase 1 仅做 Docker 基础网络策略，预留 Tool-Proxy / egress 扩展点）
  ├─ CPU / 内存 / 磁盘 / 超时限制
  ├─ 日志与审计（Phase 1 记录审计字段，Falco/eBPF 远期云原生阶段再评估）
  └─ Docker / gVisor Runtime Driver（containerd / K8s / Kata / Firecracker / WASM 仅预留）
```

最终建议与落地路线：

```text
项目形态：独立仓库 genesis-sandbox，运行进程名 sandbox-service，通过 HTTP/gRPC 与主平台解耦部署
实现语言：Go
第一阶段 (MVP & 生产级路线)：
  - Windows & macOS 开发环境：Docker Engine (runc)
  - Linux 生产环境：Docker Engine + gVisor (runsc)
  - 核心驱动：统一 RuntimeDriver 接口，通过 Runtime Capability Probe 探测 Docker daemon 支持的 runtime，再按 Policy 决定使用 runsc 或是否允许降级 runc。
第二阶段增强：Snapshot / Restore 快照恢复机制探索，仍以 Docker 路线为主
远期云原生增强（暂不实现）：containerd / K8s / RuntimeClass / Kata / Firecracker / Tool-Proxy / egress sidecar / Credential Vault / Falco，在第三阶段或真实云原生需求出现后再评估
轻量函数沙箱：WASM (wazero) 可作为后续补充
```

## 2. 另起项目还是放本项目？

建议直接保持独立仓库运行；主平台通过 HTTP/gRPC/SDK 对接。当前已采用该形态。

当前推荐边界：

```text
genesis-agent/
  cmd/
    agent/                 # Agent Runtime CLI / 主入口之一
    api/                   # 主平台 HTTP/API 入口
  internal/
    contracts/             # 主平台内部契约
    adapters/              # 主平台适配器；后续放 sandbox HTTP/gRPC/SDK client
    governance/            # 治理层 (Policy Engine / Quota)
  configs/
    config.yaml            # 主平台配置
  web/
  docs/
    沙箱环境设计.md

D:\workspace\go\genesis-sandbox\
  cmd/
    sandbox-service/       # 沙箱基础设施服务进程入口
  internal/
    app/                   # lease/job 生命周期编排
    contracts/             # sandbox 内部服务与 driver 契约
    domain/                # sandbox 纯领域模型
    config/                # sandbox.yaml 独立配置
    adapters/docker/       # DockerDriver (支持 runc/runsc 探测)
    transport/http/        # HTTP API
  configs/
    sandbox.yaml           # 沙箱服务配置
  api/
  docs/
```

关键不是仓库是否独立，而是运行时必须独立：

```text
主平台进程 != 沙箱执行进程
主平台服务 != 沙箱服务
业务数据库权限 != 沙箱权限
```

### 为什么不建议放在主项目进程里？

| 问题 | 说明 |
|---|---|
| 安全风险 | 用户代码、AI 生成代码、第三方工具可能读取文件、环境变量、内网地址 |
| 稳定性风险 | 死循环、内存爆炸、CPU 打满会拖垮主平台 |
| 依赖冲突 | Python/Node/系统命令依赖会污染主平台运行环境 |
| 扩展困难 | 后续上 Docker、gVisor、Kata、K8s 会很难 |
| 审计困难 | 执行日志、资源统计、工具审计需要独立治理 |

结论：

```text
代码仓库可以在一个 monorepo；
沙箱运行时必须独立服务化。
```

## 3. 新进程、Docker、gVisor、Kata、Firecracker 怎么选？

可以分成几个安全等级。

| 等级 | 技术 | 安全性 | 启动速度 | 适合阶段与落地策略 |
|---|---|---:|---:|---|
| L0 | 主进程内执行 | 极低 | 最快 | 禁用 |
| L1 | 子进程执行 | 低 | 很快 | 只适合可信内部工具 (Trusted Runtime) |
| **L2** | **Docker (`runc`)** | **中等** | **较快 (100ms~1s)** | **MVP 阶段 Windows 开发环境首选** |
| **L3** | **Docker + gVisor (`runsc`)** | **高** | **中等 (200ms~1s)** | **MVP 阶段 Linux 生产环境首选** |
| L4 | Kata Containers | 很高 | 中等偏慢 (1~3s) | 政企强隔离（暂不实现，目前作为备选） |
| L5 | Firecracker MicroVM / E2B | 极高 | 慢(传统)/快(Snapshot) | 大规模云端沙箱（暂不实现，目前作为备选） |
| L6 | WASM/WASI | 高但能力受限 | 很快 (<5ms) | 轻量纯计算函数/规则/插件（暂不实现，目前作为备选） |

推荐路线：

```text
MVP & 生产级：Windows/macOS (开发测试 runc) / Linux (优先 Docker+gVisor，按能力探测启用 runsc)
企业版/二期：Snapshot/Restore 快照恢复，继续保持 Docker 稳定路线
远期云原生/三期：containerd / K8s / RuntimeClass / Kata / Firecracker / egress sidecar / Falco（目前暂不实现）
```

## 4. 行业常见沙箱技术

### 4.1 Docker / containerd

最常见的一阶段方案。

适合：

```text
低代码 Code 节点
Agent 工具执行
Python/JS/Go 脚本执行
Excel/PDF/图片处理
私有化部署
```

优点：

```text
成熟
生态好
部署简单
镜像管理方便
支持 CPU / 内存 / 网络 / 文件限制
```

缺点：

```text
共享宿主机内核
安全性不是最高
需要配合 seccomp / AppArmor / capabilities / read-only filesystem
```

第一版建议直接用 Docker Engine SDK；containerd 暂不实现，仅作为远期云原生适配器候选，用于未来 K8s/CRI/RuntimeClass、规模化调度和更细粒度 runtime 管理。

---

### 4.2 gVisor

gVisor 是比普通容器更安全的容器运行时（用户态 Linux 内核实现 `runsc`），适合多租户和不可信代码执行。

适合：

```text
企业私有化
多租户沙箱
AI 生成代码执行
第三方工具执行
Linux 生产环境
```

优点：

```text
比普通 Docker 更安全 (拦截非法系统调用)
可以和 Docker / containerd / Kubernetes 集成
配置极简单 (`docker run --runtime=runsc`)
比 Firecracker 更容易落地与运维
```

缺点：

```text
兼容性不如原生 Docker (少数极端系统调用可能有坑)
部分系统调用开销略高于原生 runc
```

**MVP 生产级强力推荐：Linux 环境直接采用 Docker + gVisor。**

---

### 4.3 Kata Containers

Kata 是“容器体验 + 虚拟机隔离”。

适合：

```text
政企客户
强隔离要求
多租户高安全场景
不信任用户代码
```

优点：

```text
每个 workload 有独立内核
安全边界更接近虚拟机
```

缺点：

```text
部署复杂
资源开销更大
启动速度慢于普通容器
```

---

### 4.4 Firecracker MicroVM

适合构建类似云端代码执行平台、AI Agent 执行云、E2B 类沙箱服务。

适合：

```text
大规模云端沙箱
强隔离 AI Agent 执行环境
高风险不可信代码
```

优点：

```text
隔离强
适合多租户
安全边界清晰
```

缺点：

```text
自研成本高
网络、文件、镜像、生命周期管理复杂
MVP 阶段偏重
```

**目前明确暂不实现，作为后续三期大云端方案备选。**

---

### 4.5 WASM / WASI

适合轻量函数沙箱，不适合替代完整 Docker 环境。

适合：

```text
规则函数
表达式执行
轻量工具插件
安全脚本
MCP 工具风险隔离
```

优点：

```text
启动快
资源可控
边界清晰
```

缺点：

```text
生态受限
文件、网络、系统命令能力受限
不适合完整 Python/Node 生态
```

Go 里可以考虑：

```text
wazero
wasmtime-go
```

---

### 4.6 nsjail / bubblewrap / firejail

适合 Linux 单机轻量进程隔离。

不是你的第一优先级。

## 5. Windows / macOS / Linux 平台实现建议

### 5.1 Linux

生产环境首选 Linux。

推荐部署：

```text
sandbox-service 运行在 Linux 宿主机/容器中
容器运行时驱动：Docker Engine (`runtime=runsc` 即 gVisor)
企业版/三期支持 K8s + RuntimeClass
```

---

### 5.2 Windows

Windows 不建议作为生产沙箱宿主，但必须支持开发测试。

推荐开发配置：

```text
Windows 开发者机器:
  - Docker Desktop / WSL2 Linux containers，驱动通常使用 runc。
  - 不通过 runtime.GOOS 判断安全等级，因为 Sandbox Service 可能运行在 WSL2/Linux VM 内。
  - 由 Runtime Capability Probe 查询 Docker daemon 是否支持 runsc。
  - 不支持 runsc 时按 Policy 判断是否允许降级 runc。
  - 高风险任务、第三方 Skills、未知 MCP 工具默认不允许静默降级。
```

Windows containers 可作为未来 Windows 专用工具的独立 profile，但不作为默认代码执行沙箱。

---

### 5.3 macOS

macOS 适合作为团队开发/测试环境，但**不适合作为云端生产宿主**。

1. **为什么不适合生产？**  
   macOS 基于 Darwin 内核，原生缺少 Linux 的 cgroups 和 namespaces 功能。在 macOS 上运行 Docker 无论如何都必须依赖一层轻量级 Linux 虚拟机（Linux VM）。因此云端生产宿主依然首选 Linux（内核原生 + gVisor `runsc`）。
2. **macOS 开发者生态工具链支持**：
   - **OrbStack（官方强烈推荐）**：启动仅需 2 秒，CPU 和内存开销极低，极大提升 Mac 机器上的沙箱开发与调试体验。
   - **Docker Desktop / Colima / Lima**：常规兼容支持。
3. **Apple Silicon (ARM64) 镜像多架构构建规范 (Multi-Arch)**：
   - 团队中有大量开发者使用 M1/M2/M3/M4 等 Apple Silicon 芯片（arm64 架构），而 Linux 服务器普遍是 amd64 (x86_64)。
   - **规范**：沙箱基础镜像（如 `genesis/sandbox-python:3.11`）统一使用 `docker buildx` 打包为包含 `linux/amd64` 和 `linux/arm64` 的多架构镜像。
   - **效果**：Mac 开发者本地启动时直接运行原生 arm64 容器，无 Rosetta/QEMU 转译，性能几乎无损耗。

## 6. 为什么用 Go 实现 Sandbox Service？

Go 更适合写沙箱服务本身。

原因：

```text
并发能力强
适合长期运行的 runtime manager
适合容器生命周期管理
适合 gRPC / HTTP 服务
适合对接 Docker / containerd / Kubernetes
部署简单，单 binary
比脚本语言更适合执行层基础设施
```

Go 服务适合承担：沙箱池管理、容器生命周期管理、任务调度、超时强杀、日志流式返回、资源限制、健康检查、清理任务、工具执行策略。

## 7. Go 技术栈建议与 Driver 抽象接口

### 7.1 服务框架

推荐技术栈：`gRPC + HTTP(Gin/chi) + Redis + PostgreSQL`。`gRPC` 作为内部高性能协议，`HTTP` 作为调试、跨语言 SDK、前端/运维接入协议，两者共享同一套 application service。

### 7.2 容器运行时 SDK

第一版：`Docker Engine SDK for Go`（当前代码使用 Moby split modules：`github.com/moby/moby/client` + `github.com/moby/moby/api`）。Docker SDK 更适合 MVP 与私有化部署，开发者安装成本低、调试直观、镜像和网络管理简单。`containerd` 更接近生产底层运行时，优势是少一层 Docker 管理封装、更贴近 K8s/CRI 与 RuntimeClass、镜像 snapshotter 能力更强、生命周期控制更细；缺点是 API 更底层、调试体验不如 Docker、网络/卷/日志等周边能力需要平台自己补齐，第一版实现成本更高。

选型建议：

```text
Phase 1: Docker Engine SDK
  - 开发和私有化部署简单
  - 易于支持 Docker Desktop / WSL2 / Linux
  - 方便验证池化、租约、文件 staging、资源限制

远期云原生阶段: containerd adapter（暂不实现）
  - 面向 K8s / CRI / RuntimeClass
  - 更适合大规模生产调度和节点级 runtime 统一治理
  - 便于接 gVisor/Kata 运行时、镜像 snapshotter 与更细粒度容器生命周期控制
  - 仅在 Sandbox Service 已稳定且出现真实 K8s/云原生集成需求时引入
```

---

### 7.3 底层 RuntimeDriver 接口抽象与 Runtime Capability Probe（关键设计）

不要用 `runtime.GOOS` 判断是否启用 `runsc`。Sandbox Service 可能运行在 Windows/macOS 的 Docker Desktop / WSL2 / Linux VM 内，此时 Go 看到的是 `linux`，但 Docker daemon 未必安装了 gVisor。正确方式是：**先探测 Runtime 能力，再由 Policy 决定是否允许降级**。

```text
RuntimeEngine 选择顺序：
  1. SandboxSpec 显式指定 runtime_engine
  2. RuntimeCapabilityProbe 查询 Docker daemon 支持的 runtime：runc / runsc / kata
  3. Policy Engine 根据 risk_level、tenant_plan、runtime_profile 决定目标 runtime
  4. 支持 runsc 且策略要求高隔离 → 使用 runsc
  5. 不支持 runsc → 判断 allow_runtime_downgrade
  6. 高风险任务不允许降级，直接返回 RUNTIME_UNAVAILABLE
```

```go
package runtime

import "context"

type SandboxSpec struct {
    SandboxID       string            `json:"sandbox_id"`
    Image           string            `json:"image"`
    RuntimeEngine   string            `json:"runtime_engine"` // "runc" / "runsc" / "kata"，可由 Policy 填充
    RuntimeProfile  string            `json:"runtime_profile"` // code-python-basic / tool-basic / browser-playwright
    RiskLevel       string            `json:"risk_level"`      // low / medium / high
    AllowDowngrade  bool              `json:"allow_downgrade"`
    CPULimit        float64           `json:"cpu_limit"`
    MemoryMB        int64             `json:"memory_mb"`
    NetworkPolicy   string            `json:"network_policy"`
    EnvVars         map[string]string `json:"env_vars"`
    Mounts          []MountConfig     `json:"mounts"`
}

type JobSpec struct {
    JobID          string   `json:"job_id"`
    LeaseID        string   `json:"lease_id"`
    IdempotencyKey string   `json:"idempotency_key"`
    Command        []string `json:"command"`
    ScriptContent  string   `json:"script_content"`
    TimeoutSeconds int      `json:"timeout_seconds"`
}

type JobResult struct {
    ExitCode     int      `json:"exit_code"`
    Stdout       string   `json:"stdout"`
    Stderr       string   `json:"stderr"`
    Artifacts    []string `json:"artifacts"`
    DurationMs   int64    `json:"duration_ms"`
    ErrorCode    string   `json:"error_code,omitempty"`
    ErrorMessage string   `json:"error_message,omitempty"`
}

type RuntimeCapabilities struct {
    Runtimes           []string `json:"runtimes"`             // docker info 返回的 runtimes
    SupportsRunsc      bool     `json:"supports_runsc"`
    SupportsKata       bool     `json:"supports_kata"`
    SupportsCheckpoint bool     `json:"supports_checkpoint"`
}

type RuntimeCapabilityProbe interface {
    Detect(ctx context.Context) (*RuntimeCapabilities, error)
}

// RuntimeDriver 为底层沙箱驱动统一契约。
type RuntimeDriver interface {
    CreateSandbox(ctx context.Context, spec SandboxSpec) error
    ExecJob(ctx context.Context, sandboxID string, job JobSpec) (*JobResult, error)
    CancelJob(ctx context.Context, sandboxID string, jobID string) error
    DestroySandbox(ctx context.Context, sandboxID string) error
    GetSandboxStatus(ctx context.Context, sandboxID string) (string, error)
}
```

#### Runtime 选择示例

```go
func SelectRuntime(spec SandboxSpec, caps RuntimeCapabilities) (string, error) {
    if spec.RuntimeEngine != "" {
        if contains(caps.Runtimes, spec.RuntimeEngine) {
            return spec.RuntimeEngine, nil
        }
        if !spec.AllowDowngrade || spec.RiskLevel == "high" {
            return "", ErrRuntimeUnavailable
        }
    }

    if spec.RiskLevel == "high" {
        if caps.SupportsRunsc {
            return "runsc", nil
        }
        return "", ErrRuntimeUnavailable
    }

    if caps.SupportsRunsc {
        return "runsc", nil
    }
    return "runc", nil
}
```

这一抽象使业务逻辑与底层 Runtime 解耦，同时避免把“开发机可运行”误认为“生产高隔离可运行”。

## 8. OpenSandbox 可借鉴的边界与本项目取舍

OpenSandbox 的成熟经验可以借鉴，但不能照搬。Genesis 当前只做稳定 Docker 版，因此只吸收对 Docker 路线有帮助的架构边界：

```text
Lifecycle 层：管理沙箱资源、租约、状态、池、文件 staging、endpoint 元数据
Execution 层：执行命令/代码、收集日志、处理取消、上传下载文件
Network/Policy 层：裁决 network_policy、SecretRef、Tool 权限；Phase 1 只落基础 Docker 网络策略
Client 层：Go SDK / Python SDK / HTTP client / MCP Server / 未来 CLI
```

取舍原则：

- 借鉴 OpenSandbox 的 `Lifecycle Server + execd + SDK/CLI/MCP + specs/OSEP` 产品边界。
- Phase 1 不实现 OpenSandbox 风格的 egress sidecar、Credential Vault、K8s CRD、透明 MITM、完整多语言 SDK。
- Phase 1 可先用 Docker exec + file staging 实现执行能力，但接口要预留 `genesis-execd`，避免未来长期 Agent、文件代理、会话、metrics 时重做协议。
- 所有高级能力必须通过 `RuntimeCapabilityProbe` / `Capabilities API` 显式探测，不允许静默假设或自动降级。
- 重大沙箱能力建议后续采用 `docs/adr/` 或 `docs/oseps/` 形式写设计提案，再进入实现。

推荐目标边界：

```text
Sandbox Service
  ├─ Lifecycle API：Create / Lease / Release / Drain / Destroy / Renew / Endpoint
  ├─ Job API：Exec / Cancel / Status / StreamLogs
  ├─ File API：Upload / Download / Artifacts
  ├─ Pool API：Profile pool config / queue / reconcile
  ├─ Diagnostics API：Logs / Events / Metrics
  └─ Capabilities API：runtime / network / filesystem / exec features

Phase 1 Execution
  └─ Docker exec + 文件 staging

Future genesis-execd（Phase 1B/2，可选）
  ├─ command / session / files / metrics
  ├─ endpoint-proxy 支撑浏览器/桌面/长期 Agent
  └─ isolated session / diff / commit（远期）
```

## 9. Sandbox Service 模块设计

推荐目录：

```text
sandbox/
  cmd/
    sandbox-service/
      main.go
  internal/
    app/               # Service 编排 lease / exec / release / destroy
    contracts/         # Service / RuntimeDriver 接口和内部 DTO
    domain/            # 纯领域模型
    config/            # sandbox.yaml 独立配置
    adapters/
      docker/          # DockerDriver (支持 runc 和 runsc)
      containerd/      # 远期云原生候选，当前不实现
      wasm/            # 远期轻量函数候选，当前不实现
    transport/
      http/            # HTTP router/server/handler
  pkg/                 # 后续 Go SDK 或公开 types，当前暂不暴露
  deploy/              # 后续独立部署清单
```

## 10. 工具调用和代码执行是否共用沙箱？

建议分为四类 Runtime：`Code Sandbox`、`Tool Sandbox`、`Trusted Runtime`、`Browser Sandbox`。

### 10.1 Code Sandbox
用于：低代码 Code 节点、用户自定义代码、AI 生成代码、数据分析代码。  
特点：风险最高、默认断网、输入只读、输出临时可写、强制超时、执行后清理。

### 10.2 Tool Sandbox
用于：PDF 解析、Excel 处理、图片处理、Markdown 转换、HTML 清洗、HTTP 请求工具、爬虫工具、第三方 MCP 工具。  
特点：风险中等、按工具授权、部分工具允许走代理联网。

### 10.3 Trusted Runtime
用于平台内置可信工具（知识库检索、数据库查询、模型调用）。无需进入沙箱容器，直接走 Tool Gateway / Policy Engine 鉴权执行。

### 10.4 Browser Sandbox
用于 Playwright/Headless Chrome 自动化。资源开销大，独立池化管理。

### 10.5 Skills 脚本的执行映射策略
Skills（技能包）通常包含一组自动化脚本（如 `skills/<skill_name>/scripts/eval.py`）。脚本必须在沙箱或受控 Trusted Runtime 中执行，不能直接在主平台进程执行。执行 Skills 脚本时不应简单固化，必须依据**信任来源**与**安全声明**动态映射到上述 Runtime：
1. **平台内置官方 Skills (Built-in Skills)**：属于高信度资产，若无深度隔离要求可直接映射到 **`Trusted Runtime`** 或 **`Tool Sandbox`**。
2. **用户/工作区自定义 Skills (User-defined Skills)**：由用户或企业管理员上传。映射到 **`Tool Sandbox`** 或 **`Code Sandbox`**，默认 `network=none`，限制工作区读写权限，通过预热池快速调用。
3. **第三方/开放集市 Skills (Third-party Skills)**：属于不可信资产，映射到专用 **`skill-*` Runtime Profile**，底层可使用 Code Sandbox 的高隔离能力（Linux 生产优先 gVisor `runsc`）。针对其自带依赖脚本的安装与联网需求，采取 **“注册期静态构建 + 运行期受控授权”** 机制（详见第 31.3 节），网络根据声明通过 Policy Engine 动态裁决；当前 Docker 稳定版只落 `none` / 基础 allowlist / 策略字段，`tool-proxy` 仅作为远期扩展点，绝不在对话运行期卡顿安装。

## 11. 沙箱生命周期设计

沙箱生命周期必须区分“容器实例状态”和“租约状态”。Agent / Tool 不能直接持有容器，只能通过 `lease_id` 使用沙箱，便于取消、幂等、回收和审计。

```text
created
  -> starting
  -> ready
  -> leased
  -> busy
  -> idle
  -> draining
  -> destroyed

异常分支：
  starting / ready / busy -> failed -> draining -> destroyed
```

状态语义：

| 状态 | 说明 |
|---|---|
| created | 已创建元数据，尚未启动底层容器 |
| starting | 正在拉镜像、创建容器、初始化工作区 |
| ready | 容器已启动，可进入池 |
| leased | 已分配给某个 job/run/tool，尚未开始执行 |
| busy | 正在执行 job |
| idle | 执行完成，等待复用或回收 |
| draining | 不再接收新 job，准备销毁或重建 |
| destroyed | 容器和临时资源已销毁 |
| failed | 启动或执行失败，需要清理 |

沙箱元数据示例：

```json
{
  "sandbox_id": "sbx_xxx",
  "lease_id": "lease_xxx",
  "tenant_id": "t1",
  "user_id": "u1",
  "workspace_id": "w1",
  "run_id": "run_xxx",
  "runtime_profile": "code-python-basic",
  "runtime_engine": "runsc",
  "type": "code",
  "status": "busy",
  "created_at": "2026-06-29T00:00:00Z",
  "last_used_at": "2026-06-29T00:10:00Z",
  "ttl_seconds": 1800,
  "max_cpu": "1",
  "max_memory": "512Mi",
  "network_policy": "none"
}
```

必须支持的控制动作：

```text
LeaseSandbox(profile, policy) -> lease_id
ReleaseLease(lease_id)
CancelJob(job_id)
DrainSandbox(sandbox_id)
DestroySandbox(sandbox_id)
Heartbeat(sandbox_id)
```

## 12. 三种启动策略

### 12.1 Job 级冷启动
每次新建容器。隔离最好，但启动较慢 (1~2s)。适合高风险代码及未知第三方 MCP 工具。

### 12.2 预热容器池
提前启动空闲沙箱，响应快 (50~100ms)。适合高频普通工具调用与数据转换。

### 12.3 会话级沙箱
用户会话绑定沙箱。多轮连续执行速度最快，可保留临时文件，适合 Code Interpreter 和 Agent 编码。

## 13. Docker 速度问题与 Snapshot/Restore 恢复机制

### 13.1 传统优化策略
预拉镜像、预热容器池、依赖打入镜像（禁止运行时 `pip install`）、使用 tmpfs 内存盘做 `/tmp`。

### 13.2 Snapshot / Restore 快照恢复机制（二期探索）

Snapshot / Restore 可作为第二阶段优化方向，但不作为 Phase 1 SLA 承诺。CRIU、gVisor checkpoint、Firecracker snapshot 都会受到内核、runtime、文件系统、网络连接和进程状态限制，落地复杂度明显高于预热池。

Phase 1 优先级：

```text
1. 预拉镜像
2. 依赖预构建进镜像
3. 预热容器池
4. tmpfs / workspace staging 优化
```

Phase 2 目标：

```text
探索 Snapshot / Restore，把冷启动耗时从秒级降到亚秒级；
具体收益以压测为准，不在第一版承诺固定 100ms。
```

## 14. 沙箱池设计与回收完整性验证（关键改进）

沙箱池按 `runtime_profile + isolation_strategy` 分类，不能只按语言分类。不同 profile 的 Release 行为不同：有的回到 idle，有的进入 draining 并销毁重建。

沙箱池分类：`code-python-basic`、`code-python-data`、`code-node-basic`、`tool-basic`、`skill-python-basic`、`skill-node-basic`、`skill-build-python`、`skill-build-node`、`browser-playwright` 等。

### 14.1 池配置与容量策略

每类沙箱都应配置固定的预热数量、启动数量、最大数量和排队策略。第一版建议按 profile 配置，不在代码中写死默认值。

| 配置项 | 含义 | 建议默认 |
|---|---|---|
| `min_idle` | 空闲保底数量 | code/tool 为 1~2，browser 为 0~1 |
| `initial_idle` | 服务启动后预热数量 | 不超过 `min_idle`，可延迟预热 |
| `max_idle` | 最大空闲数量 | 防止预热池占满机器 |
| `max_total` | 运行中 + 空闲总数上限 | 受租户 quota 和节点资源共同限制 |
| `max_queue_depth` | 等待队列长度 | 超出直接返回 `QUEUE_FULL` |
| `max_wait_seconds` | 最大等待时间 | 超时返回 `SANDBOX_BUSY` 或降级策略 |
| `lease_timeout_seconds` | 租约未执行过期时间 | 释放沙箱，避免泄漏 |
| `exec_timeout_seconds` | 单次执行超时 | soft kill 后 hard kill |
| `start_timeout_seconds` | 容器启动超时 | 标记 runtime/pool 异常 |
| `idle_timeout_seconds` | 空闲回收时间 | 控制成本 |
| `max_lifetime_seconds` | 容器最大存活时间 | 防止长期污染 |
| `max_exec_count` | 最大复用次数 | 高风险为 1 |
| `dirty_score_threshold` | 污染评分阈值 | 超过销毁重建 |
| `cpu_limit` / `memory_limit` / `pids_limit` / `disk_limit` | 资源上限 | 按 profile 配置 |
| `stdout_limit_bytes` / `stderr_limit_bytes` | 日志截断上限 | 防止刷屏和撑爆存储 |
| `artifact_limit_bytes` | 产物大小上限 | 防止输出炸盘 |
| `network_policy` | 网络策略 | 默认 `none` |
| `dependency_policy` | 依赖安装策略 | 默认禁止运行期安装 |
| `fallback_policy` | Docker 不可用时策略 | 默认禁用本地执行 |

推荐排队规则：同一租户按 `priority + created_at` 公平调度；超过租户配额直接排队或拒绝；系统总资源不足时先保护高优先级和已租约任务，低优先级返回可重试错误。

池参数必须做成配置，不能写死在代码里。配置维度至少支持：全局默认值、租户覆盖、workspace 覆盖、runtime_profile 覆盖；最终由 Policy Engine 合并生成执行策略。

```yaml
runtime: code-python-basic
image: genesis/sandbox-python:3.11
isolation_strategy: ephemeral_destroy
min_idle: 2
max_idle: 10
max_total: 50
ttl_seconds: 1800
idle_timeout_seconds: 300
exec_timeout_seconds: 30
memory_limit: 512Mi
cpu_limit: 1
network: none
filesystem: ephemeral
queue:
  max_wait_seconds: 10
  max_queue_depth: 200
  reject_when_full: true
  priority_levels: [high, normal, low]
error_policy:
  lease_timeout: reject
  exec_timeout: hard_kill
  start_failure: retry_then_mark_unhealthy
  max_start_retries: 1
  unhealthy_threshold: 3
cleanup_policy:
  release_action: destroy
  max_exec_count: 1
  dirty_score_threshold: 1
dependency_policy:
  allow_runtime_install: false
  allowed_package_sources: []
  allow_postinstall_scripts: false
```


第一阶段同步执行的推荐超时分层：

| profile / 层级 | default timeout | max timeout | 说明 |
|---|---:|---:|---|
| `code-python-basic` / `code-python-isolated` | 45s | 180s | 普通代码片段、轻量数据处理，避免无限占用容器 |
| `tool-basic` | 30s | 120s | 工具调用通常应更快失败，方便上层重试或降级 |
| `skill-python-basic` | 120s | 600s | Skills 可能包含多步骤脚本，允许更长但仍有硬上限 |
| `build` | 10m | 10m | 仅用于注册期/构建期依赖处理，不用于普通运行期 |
| HTTP `write_timeout` | 11m | - | 第一版 `/v1/jobs` 同步返回，必须大于最长执行上限；后续异步/SSE 后再缩短 |
| LLM `fast/default/reasoning/coder/local` | 45s/90s/180s/240s/300s | - | 按模型用途区分，避免 coding/reasoning 被 60s 过早截断 |

关键原则：租约超时 (`lease_timeout`) 不等于执行超时 (`exec_timeout`)；租约控制沙箱占用泄漏，执行超时控制单个 job。执行超时必须由 runtime profile 决定，并允许调用方请求更短时间，但不能超过 profile 的 `max_timeout`。
### 14.2 按 profile 决定复用策略

| profile | 风险 | 复用策略 | Release 后行为 |
|---|---:|---|---|
| high_risk_code | 高 | 不复用 | `draining -> destroy -> recreate` |
| third_party_skill | 高 | 不复用 | `draining -> destroy -> recreate` |
| user_code_interpreter | 中高 | session/workspace 级复用 | `idle`，受 TTL / max_exec_count / dirty_score 限制 |
| trusted_tool | 低 | 可复用 | 清理 job 工作目录后回到 `idle` |
| file_convert_tool | 中 | 可复用或短 TTL | 清理后回到 `idle`，超过 dirty_score 销毁 |
| browser_playwright | 中高 | 会话级复用 | 强 TTL，关闭后销毁 |

### 14.3 预热池“回收后清理”的完整性验证机制

仅 `rm -rf /workspace/*` 无法清理残留子进程、内存映射、后台任务或被修改的系统环境。

推荐双策略：

1. **Ephemeral 销毁重建策略（高风险默认）**：沙箱借出执行完一个 Job 后进入 `draining`，后台异步销毁并补齐 `min_idle`。这是真正干净的默认安全策略。
2. **受控复用策略（低风险/会话型）**：每个 Job 使用独立 `/workspace/jobs/{job_id}/`；执行后强杀非 PID 1 子进程、清理工作目录、校验进程表、检查磁盘增量；超过 `max_exec_count`、`max_lifetime` 或 `dirty_score` 后销毁。

受控复用必须满足：

```text
no background process
no unexpected listening port
workspace usage <= disk_limit
stdout/stderr already collected and truncated
no secret file left in workspace
```

## 15. 工具执行是否可以复用沙箱？

可复用工具：文本处理、JSON 转换、Excel 解析、PDF 提取、图片转换。每次必须独立工作目录 `/workspace/jobs/{job_id}/`，完成后彻底清理。  
不可复用工具：第三方上传 MCP 工具、未知代码包、爬虫、浏览器自动化。建议独立沙箱。

## 16. 网络权限设计与 Tool-Proxy 代理实现（Docker 优先，Tool-Proxy 远期）

网络策略分为：`none`（断网）、`internet`（公网）、`allowlist`（白名单）、`internal-api`（内部 API）、`tool-proxy`（代理中转）。默认策略应为 `none`，由 Policy Engine 按工具声明、租户策略、风险等级裁决是否放开。当前阶段只做稳定 Docker 版本：实现 `network=none`、租户/项目级 Docker internal bridge、基础 allowlist 策略字段和审计字段。Tool-Proxy、egress sidecar、Credential Vault、透明 MITM、K8s NetworkPolicy 不进入当前主流程，仅作为远期云原生扩展点。

### 16.1 Tool-Proxy 推荐拓扑（远期）

`--internal tenant bridge` 会阻断容器直接出网。如果需要经过 Tool-Proxy 出网，Proxy 必须同时连接租户 internal network 和受控 egress network。

```text
[Sandbox Container]
      │  only tenant internal network
      ▼
[tenant-net-{tenant_id} --internal]
      │
      ▼
[Tool-Proxy / Egress Gateway]
      │  dual-homed: tenant internal + egress network
      │  1. 域名/IP allowlist 校验
      │  2. 阻断云元数据地址、内网 CIDR、Docker bridge gateway、localhost 回环
      │  3. 全量网络请求日志审计
      ▼
[egress-net]
      ▼
[Internet / Allowed API]
```

### 16.2 Tool-Proxy 实现路径（远期）

Tool-Proxy 是受控出网网关，主要负责 allowlist、请求审计、阻断内网/云元数据地址和统一代理外部 API。它不影响当前 Docker 主流程；当前只预留 `network_policy=tool-proxy` 与策略字段，真正的 Envoy/Squid 代理治理等有明确受控出网需求后再实现。

远期可采用 Envoy 或 Squid 作为 HTTP/HTTPS 代理中转组件。若未来引入 K8s/RuntimeClass，再评估 egress sidecar（DNS/nftables/Credential Vault）作为云原生实现。沙箱容器注入 `HTTP_PROXY` / `HTTPS_PROXY` 只能作为兼容手段，不能作为安全边界，因为恶意代码可以绕过环境变量。

安全边界必须由网络层兜底：

```text
- network=none：完全断网
- tool-proxy：容器只加入 tenant internal network，无法直接访问 egress network
- iptables/nftables/CNI：阻断非代理出站
- DNS：只允许受控 DNS 或由 proxy 解析
```

第一版若暂不做透明代理，应明确限制：

```text
tool-proxy 仅治理 HTTP/HTTPS；
非 HTTP 协议默认禁止；
高风险任务默认 network=none。
```

### 16.3 单机 Docker 下的租户网络隔离（当前重点）

Docker 默认的 default `bridge` 网络会导致不同容器间 IP 互通。在 MVP/单机阶段：

- 断网容器：强制指定 `--network none`。
- 联网容器：Sandbox Service 动态为各租户创建独立的孤立 Bridge 网络（如 `tenant-net-{tenant_id}`），沙箱容器只加入该 internal network。
- Tool-Proxy：当前不实现，只预留策略值；远期作为唯一受控出网网关，连接 `tenant-net-{tenant_id}` 和 `egress-net`。
- 禁止沙箱访问：`169.254.169.254`、RFC1918 内网 CIDR、IPv6 本地地址、宿主网关、Docker bridge gateway、`localhost` 映射服务。

## 17. 文件系统设计

```text
/sandbox
  /workspace
    /input   (只读)
    /output  (可写)

允许的 MIME 类型 / 扩展名 allowlist

```

## 18. 环境变量和密钥设计

禁止向沙箱传递 `DATABASE_URL`、`MINIO_SECRET`、`OPENAI_API_KEY` 等长效密钥。仅提供短期动态 `GENESIS_JOB_TOKEN` 与白名单文件访问权限。当前 Docker 版优先使用 SecretRef / Tool Gateway 受控调用 / 临时最小权限 token，执行后销毁。OpenSandbox 风格的透明 Credential Vault 依赖 egress sidecar 和透明代理，放到远期云原生阶段评估，不进入当前 Docker 主流程。

## 19. 资源限制设计

默认限制：CPU 0.5~1 core、内存 256MB~512MB、执行时间 10s~60s、PID 限制 32 或 64（防止 fork 炸弹）、stdout/stderr 限制 1MB~5MB。

## 20. 等待、排队、超时策略

等待、排队、超时策略必须由 profile 配置驱动，不能写死在代码里。推荐默认：获取空闲沙箱 <=2s 直接执行；2s~10s 排队；>10s 返回 `SANDBOX_BUSY`。队列满返回 `QUEUE_FULL`，租户配额不足返回 `QUOTA_EXCEEDED`。支持高优先级队列，但同一租户内要按 `priority + created_at` 公平调度，避免低优先级任务永久饥饿。

## 21. 多租户隔离架构设计（Docker 稳定版优先，云原生远期）

当前 Docker 稳定版只做“弱多租户隔离”：tenant_id 贯穿元数据、配额、目录、审计和 Docker internal bridge。强多租户隔离（K8s namespace、ResourceQuota、RuntimeClass、Kata/Firecracker、egress sidecar）属于远期云原生阶段，当前不实现。

```yaml
tenant_plan:
  basic:
    max_running_jobs: 2
    max_sandboxes: 2
    max_daily_executions: 1000
  enterprise:
    max_running_jobs: 50
    max_sandboxes: 100
    max_daily_executions: 100000
```

1. **计算与配额隔离**：租户维度 Quota 管控，禁止跨租户复用会话级沙箱。
2. **网络隔离**：使用单机 Docker 租户/项目专属 internal Bridge 子网隔离 (`tenant-net-{tenant_id}` / `project-net-{project_id}`)；当前默认断网或基础 allowlist，不强依赖 Tool-Proxy。
3. **存储隔离**：挂载路径严格带有 `/workspace/tenants/{tenant_id}/jobs/{job_id}/` 前缀。

## 22. API 与客户端设计（gRPC + HTTP + SDK + MCP）

Sandbox Service API 必须同时支持 gRPC 和 HTTP。gRPC 用于 Agent Runtime / Tool Gateway / Worker 等内部服务调用，HTTP 用于调试、运维、前端控制台、跨语言接入和 Webhook 式集成。两种协议共享同一套业务服务与 DTO 映射，不能各写一套逻辑。

客户端与入口策略：

```text
后端 API 服务：HTTP 为公开入口；内网服务间可使用 gRPC
Go SDK：默认 HTTP，可选 gRPC，面向主平台、Tool Gateway、Worker、CLI
Python SDK：默认 HTTP，可选 gRPC，面向数据分析、脚本工具、Python 生态 Agent
Java SDK：默认 HTTP，可选 gRPC，面向企业后端、Spring 服务、内部业务系统
HTTP API：稳定开放协议，供前端控制台、运维脚本、第三方系统和非官方语言调用
MCP Server：Agent/模型工具生态入口，把沙箱能力包装成 MCP tools/resources/prompts
```

协议选择原则：

| 场景 | 推荐协议 | 原因 |
|---|---|---|
| 对外开放、第三方集成、前端控制台、运维调试 | HTTP/OpenAPI | 易调试、易穿透网关、跨语言成本低、生态成熟 |
| 内网高频服务间调用、Worker、Tool Gateway、长日志流 | gRPC | 强类型、低开销、双向/服务端流更自然 |
| Go/Python/Java SDK 默认配置 | HTTP | 本地开发、私有化部署、桌面端和跨网络环境更稳定 |
| Go/Python/Java SDK 内网增强模式 | gRPC | 明确部署在内网且追求性能/流式体验时启用 |
| MCP Server | MCP + 内部 HTTP/gRPC client | MCP 面向 Agent，不直接替代平台协议 |

第一版建议以 `proto/openapi` 为协议源，生成或手写薄 SDK：

- Go SDK、Python SDK、Java SDK 默认使用 HTTP/OpenAPI；配置 `protocol=grpc` 或提供 `grpc_addr` 时可切换 gRPC。
- SDK 暴露领域方法：lease、exec、logs、files、artifacts、capabilities、policy/effective_policy 等，不只是裸 HTTP endpoint 包装。
- SDK 应支持 `protocol=auto|http|grpc`、`base_url`、`grpc_addr`、`request_timeout`、`retry_policy`、`tenant_id`、`workspace_id`、`default_runtime_profile`。
- gRPC 和 HTTP 只能是 transport，必须共享同一个 application service、PolicyResolver、审计和错误码体系，不能各写一套业务逻辑。
- MCP 不直接暴露所有底层 API，只暴露适合 Agent 使用的安全工具，例如 `sandbox_exec_code`、`sandbox_run_skill`、`sandbox_get_artifact`、`sandbox_stream_logs`。

API 必须显式支持租约、幂等、取消、文件传输、日志流、诊断和能力探测，不能只提供“执行一次代码”的简单接口。参考 OpenSandbox 的资源拆分，但 Genesis 以 gRPC + HTTP 双协议为准。

MCP 不能替代 HTTP：

- HTTP 是平台稳定开放协议，适合 UI、SDK、运维、第三方系统、非 Agent 客户端。
- MCP 是面向 Agent/LLM 的工具协议，适合模型发现、调用和组合工具。
- MCP Server 应作为 Sandbox Service 的适配层，通过 Go/Python/HTTP/gRPC client 调用 Sandbox Service，而不是绕过 Policy Engine 或直接操作 Docker。
- 即使有 MCP，也必须保留 HTTP；否则前端控制台、普通服务集成、Webhook、调试和非 MCP 生态语言都会变差。
关键请求字段：

```text
tenant_id, user_id, workspace_id
run_id, step_id, tool_name
runtime_profile, risk_level
network_policy, filesystem_policy
lease_id
idempotency_key
exec_timeout_seconds
resource_limits
secret_refs, credential_policy
metadata, labels
```

幂等规则：

```text
同一 tenant_id + idempotency_key 重复提交，返回同一个 job 结果或当前状态；
避免 Agent 重试、网络重试造成重复执行副作用。
```

取消规则：

```text
CancelJob 先发送 soft kill；
超时后 hard kill；
被取消的 sandbox 根据 profile 决定复用或 draining 销毁。
```


### 22.1 客户端覆盖与服务端策略裁决

客户端可以表达执行意图，但不能直接决定最终执行策略。第一阶段采用 **Client Intent -> PolicyResolver -> EffectivePolicy -> RuntimeDriver Spec** 的链路：

```text
HTTP/gRPC/Go SDK/Python SDK/MCP
  -> PolicyRequest（客户端意图）
  -> PolicyResolver（profile + tenant/workspace/agent/skill + risk + capability）
  -> EffectivePolicy（服务端裁决结果，可审计、可返回给客户端）
  -> SandboxSpec / DriverExecRequest（内部运行时规格）
```

客户端允许请求的字段：

| 字段 | 是否允许客户端请求 | 第一阶段裁决规则 |
|---|---|---|
| `runtime_profile` | 允许 | 必须是服务端已配置 profile；未知 profile 回退默认或拒绝，后续按租户可见性过滤 |
| `exec_timeout_seconds` | 允许 | 可请求更短；请求更长时不能超过 profile `max_timeout` |
| `resource_limits.memory_mb/cpus/pids` | 允许 | 只能小于或等于 profile 上限；不能借请求扩大资源 |
| `network_policy.mode` | 有限允许 | 当前仅允许 `none`；allowlist/internet/tool-proxy 后续由 Policy Engine/egress 实现后开放 |
| `filesystem_policy.read_only_rootfs` | 允许加强 | 可请求只读；不能关闭 profile 要求的只读 |
| `filesystem_policy.allowed_host_paths` | 禁止 | 第一阶段拒绝所有 host bind mount |
| `env` / `labels` / `metadata` | 有限允许 | 普通非敏感字段可合并；敏感凭证后续走 SecretRef/Credential Vault |
| `image` / `runtime` / `working_dir` | 普通客户端不允许 | 第一阶段仅由 profile 决定；管理员能力后续单独设计 |
| `allow_runtime_install` | 普通客户端不允许 | 默认 false；只允许构建期/受控 skill-build profile 开启 |

服务端必须返回或记录 `effective_policy`，让调用方知道请求被如何裁决：

```json
{
  "runtime_profile": "skill-python-basic",
  "exec_timeout_seconds": 180,
  "max_timeout_seconds": 600,
  "resource_limits": {"memory_mb": 512, "cpus": 1, "pids": 256},
  "network_policy": {"mode": "none"},
  "filesystem_policy": {"read_only_rootfs": false, "allowed_host_paths": []},
  "allow_runtime_install": false
}
```

兼容策略：现有 `spec` 字段继续保留，但语义调整为“旧客户端的策略请求”，不能绕过 `PolicyResolver`。后续 SDK 优先暴露 `PolicyRequest`，弱化直接暴露 `SandboxSpec`。

已有 sandbox 的资源、网络、文件系统属于创建期不可变策略；后续 ExecJob 只允许覆盖执行级参数（如 timeout、普通 env），不能在已有容器上变更 CPU/内存/网络/挂载策略。

OpenSandbox 的可借鉴点是：创建沙箱允许客户端传 `timeout/image/resourceLimits/env/metadata/networkPolicy`，执行命令允许传单次 `timeout`；但网络 egress policy 作为独立生命周期/sidecar API 管理。Genesis 第一阶段暂不实现 egress sidecar，但应保留 `PolicyRequest` 与 `EffectivePolicy`，避免将来 API 破坏。

## 26. 依赖安装、镜像构建与供应链安全策略（关键改进）

禁止运行时在沙箱中任意执行 `pip install` / `npm install`。默认的 `Code Sandbox` 和 `Tool Sandbox` 都不允许安装 Python 包、Node 包或系统包；依赖应在注册期或构建期完成，运行期只加载已审核产物。

只有专门的 **Build Sandbox** 可以执行依赖安装，而且必须满足：临时容器、独立网络策略、依赖源 allowlist、锁文件、SBOM/CVE 扫描、构建完成后销毁。运行沙箱只挂载构建好的只读依赖层，不把安装结果写回基础镜像或预热容器。

1. **镜像分层构建**：基础 OS -> Runtime -> 预置依赖层（Pandas, NumPy 等），镜像统一推送到私有仓库（Harbor）管理。
2. **依赖构建加速**：推荐在镜像构建中使用基于 Rust 的极速工具 **`uv`** 替代传统 `pip`，提升构建速度。
3. **镜像固定版本**：生产执行使用 image digest pinning（如 `image@sha256:...`），避免 tag 被覆盖导致不可复现。
4. **镜像签名与验证**：使用 cosign/Notary 类机制签名镜像，Sandbox Service 拉取前验证签名。
5. **SBOM 与 CVE 扫描**：镜像和 Skill bundle 都生成 SBOM，并在注册/构建阶段做漏洞扫描。
6. **依赖 allowlist**：第三方 Skills 的依赖安装走审批策略，禁止任意 postinstall 脚本或高风险 native extension。
7. **运行期安装开关**：`allow_runtime_install=false` 为默认值；只有隔离的 `skill-build-*` / `image-build-*` profile 可开启安装能力，普通 code/tool/browser 运行 profile 不开启。

## 27. Docker 安全基线

包含：非 root 用户、`read-only` 根文件系统、`cap-drop ALL`、`no-new-privileges`、`pids-limit`、`tmpfs` 挂载、seccomp profile 等。

必须明确禁止：

```text
--privileged
--net=host
--pid=host
--ipc=host
挂载 /var/run/docker.sock（Sandbox Service 可访问 Docker daemon；沙箱容器绝不能访问）
挂载宿主机用户目录或项目根目录（默认 allowed_host_paths=[]）
透传宿主机环境变量
访问云元数据地址、内网 CIDR、Docker bridge gateway
无限制 CPU / 内存 / PID / 磁盘 / stdout
```

高风险任务如果要求 `runsc` / gVisor，但当前 runtime 不可用，不允许静默降级为 `runc`。所有高级能力必须通过 capabilities/probe 显式返回，调用方和 Policy Engine 决定是否拒绝、排队或使用低风险降级策略。

```bash
# Linux 生产环境启用 gVisor 时使用 --runtime=runsc。
docker run --rm \
  --runtime=runsc \
  --network none \
  --cpus 1 \
  --memory 512m \
  --pids-limit 64 \
  --read-only \
  --cap-drop ALL \
  --security-opt no-new-privileges:true \
  --tmpfs /tmp:rw,noexec,nosuid,size=64m \
  -v /job/input:/workspace/input:ro \
  -v /job/output:/workspace/output:rw \
  genesis/sandbox-python:3.11
```

## 31. MCP / Skills 工具沙箱及脚本执行链路

针对 MCP 工具及 Skills 技能脚本（如 Python/Shell 扩展脚本），Sandbox Service 建立了一套标准的“声明-裁决-挂载-执行”完整链路：

### 31.1 Skills 脚本架构与 Bundle Staging 机制

不要长期设计为直接把宿主机 Skill 目录挂入沙箱。即使只读挂载，也会带来路径泄露、权限边界复杂、Windows/macOS 路径差异和分布式部署困难。

推荐采用 **Skill Bundle Staging**：

```text
Skill Registry
  → 生成 content-addressed bundle（skill_id + version + digest）
  → 静态扫描 / 签名 / SBOM / 依赖锁定
  → Sandbox Service 拉取 bundle 到 staging 区
  → 以只读 volume 挂载到容器 /sandbox/skills/{skill_id}@{version}
```

挂载分层：

- **技能代码层（只读 staging volume）**：`/sandbox/skills/{skill_id}@{version}`，防止脚本修改自身代码或注入持久化恶意代码。
- **技能依赖层（只读 vendor volume）**：`/sandbox/skills/{skill_id}@{version}/vendor`，由注册/构建阶段生成。
- **数据工作区（输入只读、输出可写）**：输入参数与依赖文件挂载至 `/workspace/input/`，运行结果输出至 `/workspace/output/`。

### 31.2 完整执行流程

```text
Agent Loop 决策调用 Skill 脚本 (如 run_skill_script)
          │
          ▼
Tool Router / Skill Registry 识别 skill_id、version、digest、风险等级
          │
          ▼
Policy Engine 动态生成 SandboxSpec
  - Runtime 映射
  - network allowlist / tool-proxy
  - 超时与资源限额
  - 是否允许 runtime 降级
          │
          ▼
Sandbox Service 准备 Bundle Staging
  - 校验 digest / 签名 / SBOM
  - 拉取只读 skill bundle
  - 挂载输入输出 staging
          │
          ▼
Sandbox Service 从对应 Runtime 池子获取容器并执行 (ExecJob)
  - 执行指令: python3 /sandbox/skills/{skill_id}@{version}/scripts/main.py --input /workspace/input/params.json
          │
          ▼
结果收集、脱敏处理、产物下载、按 profile 清理或销毁 -> 返回结构化 JSON 给 Agent Loop
```

### 31.3 第三方 Skills 依赖脚本的动态安装与加载机制 (生命周期双阶段方案)
针对第三方 Skill 自带 `requirements.txt` / `package.json` 等第三方依赖脚本的痛点，若直接在 Agent 运行期实时 `pip install`，会导致**严重的运行卡顿**以及**未知恶意包安装的安全威胁**。因此，平台采取 **“生命周期双阶段机制”** 解决：

1. **静态注册/构建阶段 (Build & Install Stage)**：
   - **一次性构建任务 (Disposable Build Task)**：当技能导入或更新时，后台从全局共享的 **Skill Build Queue** 中触发一个临时构建沙箱。该沙箱**并非永久常驻**，安装完成后**立即销毁释放**，零常驻资源开销。
   - **全局 Wheel 缓存与基础镜像预置 (Cache & Base Image)**：常用基础包（如 `requests`, `pydantic` 等）已预置在平台基础镜像中；构建时挂载宿主机全局 `uv` 缓存目录，相同依赖秒级命中缓存复制，网络与 CPU 开销极小。
   - **离线预编译打包**：使用 `uv pip install --target /sandbox/skills/{skill_name}/vendor -r requirements.txt` 将特有依赖隔离安装至 Skill 专属的 `vendor` 目录中，并进行 CVE 漏洞扫描。
2. **运行隔离执行阶段 (Runtime Execution Stage)**：
   - 真正对话执行时，优先使用专用 `skill-python-*` / `skill-node-*` 运行 profile。它们可以复用 Code Sandbox 的隔离能力，但池、镜像、依赖层和配额应与普通 code/tool 分开，避免第三方 Skill 污染通用沙箱池。
   - 沙箱保持高隔离性，阻止脚本篡改系统。
   - Sandbox Service 启动容器时，自动将预构建好的只读 `vendor` 注入系统环境变量（如 `PYTHONPATH=/sandbox/skills/{skill_name}/vendor:$PYTHONPATH`），无需在运行期安装即可秒级加载依赖。
   - 若 Skill 声明了 API 请求权限，Policy Engine 根据声明裁决是否允许基础 `allowlist` 域名访问；`tool-proxy` 和透明凭证代理只作为远期扩展点。

## 33. 本地轻量执行与沙箱开关

平台必须支持“是否启用容器沙箱”的配置开关，适配桌面端、开发者本机、私有化轻量部署等场景。推荐把执行后端抽象为 `ExecutionBackend`，由 Policy Engine 决定是否允许降级到本地执行。

```yaml
sandbox:
  enabled: true
  default_backend: docker
  allowed_backends: [docker, local_process, trusted_only]
  fallback:
    docker_unavailable: trusted_only
    allow_local_process_for_code: false
    allow_local_process_for_tools: true
  local_process:
    enabled: false
    allowed_profiles: [trusted_tool]
    allow_user_code: false
    allow_third_party_skill: false
    working_dir_root: .genesis/local-runtime
    env_allowlist: []
    network_policy: host_limited
    exec_timeout_seconds: 10
```

执行后端建议：

| Backend | 适用场景 | 默认策略 |
|---|---|---|
| `docker` | 标准 code/tool/skill/browser 沙箱 | 默认首选 |
| `local_process` | 桌面端轻量可信工具、开发调试 | 默认关闭，只允许低风险工具 |
| `trusted_only` | 无 Docker 环境，只运行平台内置可信工具 | Docker 不可用时可降级 |
| `remote_sandbox` | 桌面端调用远程沙箱服务 | 后续推荐 |
| `disabled` | 禁止代码/工具执行 | 管控场景 |

本地执行不能提供强隔离，不能默认运行用户代码、第三方 Skill、未知 MCP 工具、联网爬虫或浏览器自动化。若用户明确开启本地执行，应在 UI/API 上展示风险提示，并通过 allowlist 限制命令、环境变量、工作目录和网络访问。

## 35. 分阶段落地路线（Docker 稳定版优先）

### 第一阶段：MVP & 生产级 (当前实施重点)
- 实现 Go 独立 `sandbox-service` 与 `RuntimeDriver` 接口。
- Windows & macOS 支持 Docker (`runc`) 开发测试；Linux 生产优先 Docker + gVisor (`runsc`)，通过能力探测启用。
- 支持可配置容器预热池、队列、租约、超时、错误策略（销毁重建模式保证彻底干净）与基础文件挂载。

### 第二阶段：Docker 增强版（仍以稳定 Docker 为主）
- 探索 Snapshot / Restore 快照恢复机制，目标降低冷启动耗时，不作为固定 SLA。
- 可选实现 `genesis-execd` MVP：command / session / file proxy / metrics，替代直接 Docker exec 的薄封装。
- 保持 Tool-Proxy / Falco / containerd / K8s egress sidecar / Credential Vault 扩展点，但不进入当前主流程。

### 远期云原生阶段（暂不规划实现，等第三阶段或真实需求）
- 出现多节点调度、强多租户、公有云托管、K8s 原生集成或合规隔离需求后，再评估 containerd / K8s / RuntimeClass / Kata / Firecracker。
- 有明确受控出网和凭证代理需求后，再评估 Tool-Proxy、K8s egress sidecar、Credential Vault、Falco/eBPF。
- 有明确 GUI Agent 产品需求后，再实现 browser-desktop / desktop-agent profile，支持隔离浏览器画面实时查看。
