# Genesis Agent — 文件系统能力设计方案

> 本方案面向当前产品分发架构：一个 Go module、多个 `cmd/<product>` 入口、`products/<product>/bootstrap` 作为产品装配根。设计目标不是把 CLI、Desktop、Web/Enterprise 做成一套完全相同的文件系统实现，而是统一上层语义，隔离底层执行边界。

---

## 一、核心结论

文件系统能力应采用：**统一契约 + 产品注入 + 后端隔离 + 会话级安全状态**。

- CLI、Desktop、Web/Enterprise 的上层工具语义应统一：`read_file`、`write_file`、`edit_file`、`list_dir`、`walk_dir` 的输入输出、错误语义、权限决策、审计事件尽量一致。
- 具体文件访问方式不能统一实现：CLI/Desktop 默认访问本地宿主机或本地工作区，也可以选择 Docker/genesis-sandbox workspace；Web/Enterprise 必须通过远程 sandbox/workspace 服务访问，不能直接碰宿主机文件系统。
- 文件系统后端实现必须由产品 bootstrap 注入，`internal` 只放契约、工具逻辑、权限模型、路径解析模型和新鲜度追踪模型。
- 本轮不强行实现 DB、OIDC、Wails、Docker/genesis-sandbox 细节；但接口要能容纳这些产品差异。

## 二、三端是否统一

### 2.1 必须统一的部分

这些能力属于共享内核，放在 `internal/capabilities/filesystem` 或相关能力域：

| 统一项                         | 原因                                                                 |
| --------------------------- | ------------------------------------------------------------------ |
| Tool 输入/输出 schema           | 让 Agent、Prompt、测试、审计和 UI 呈现一致                                      |
| `FileSystemBackend` 契约      | 上层不关心本地 OS 还是远程 sandbox                                            |
| `PathResolver` 契约           | 所有路径都必须先解析、规范化、绑定 workspace/sandbox 语义                             |
| `Decision / Evaluation`     | UI 需要知道 Allow / Ask / Deny 以及原因                                    |
| `RuntimeFilePermissions` 模型 | CLI/Desktop 用得多，Enterprise 也可映射为会话级授权状态                            |
| `FreshnessTracker` 契约       | 防止 Agent 覆盖用户或其他进程修改                                               |
| bounded walk 限制             | 防止遍历大仓库、大目录、符号链接循环                                                 |
| 错误分类                        | 统一返回 permission denied、not found、modified externally、too large 等语义 |

### 2.2 不能统一实现的部分

这些能力属于产品边界，放在 `products`：

| 产品             | 文件系统实现                                                           | 权限交互                          | 路径边界                             |
| -------------- | ---------------------------------------------------------------- | ----------------------------- | -------------------------------- |
| CLI            | 默认 `shared/local/filesystem`；可选 Docker/genesis-sandbox backend   | TUI/终端确认                      | 本地工作区、显式授权目录，或 sandbox workspace |
| Desktop        | 默认复用 `shared/local/filesystem`；可选 Docker/genesis-sandbox backend | GUI 弹窗、native picker、画布/任务上下文 | 用户选择目录、项目工作区，或 sandbox workspace |
| Web/Enterprise | 通过 sandbox/workspace API，复用通用 sandbox client                     | Web 审批、RBAC/租户策略              | sandbox workspace，禁止宿主机 FS       |

因此设计原则是：**工具统一，后端按运行位置选择；权限模型统一，权限来源不统一；路径语义统一，路径解析实现可不同。**

## 三、参考项目中真正值得借鉴的点

### 3.1 Codex

Codex 的 `codex-rs/file-system` 有几个关键点非常适合 Genesis Agent：

- `ExecutorFileSystem` 是抽象文件系统，不要求调用方知道本地还是远程。
- 每次操作都可携带 `FileSystemSandboxContext`，把权限、cwd、workspace roots、Windows sandbox level 等信息传到后端边界。
- `walk` 是有边界的：最大深度、最大目录数、最大 entry 数、最大响应字节数都有限制。
- `execpolicy` 使用三值决策：Codex 命名是 `Allow / Prompt / Forbidden`；Genesis 内部可以命名为 `Allow / Ask / Deny`，但要保持语义一致。

### 3.2 Kode-CLI

Kode-CLI 的文件系统权限和新鲜度追踪值得借鉴，但要按 Go 后端并发场景重新设计：

- 运行时目录授权：`readFileAllowedDirectories` / `writeFileAllowedDirectories` 支持“一次授权，子目录生效”，并会合并冗余父子目录。
- 文件新鲜度：读取时记录 `mtime` 和 hash，写入前再次检查，避免覆盖外部编辑。
- 权限模式：`default / acceptEdits / dontAsk / yolo / bypass` 能覆盖交互式、自动化、企业强治理等场景。

注意：Kode 的实现是 CLI 进程内状态，Genesis 的 HTTP/Enterprise 场景存在多会话、多用户、多租户和并发，不能照搬全局变量。

## 四、目标目录设计

### 4.1 internal 共享能力

```text
internal/capabilities/filesystem/
  contract/
    backend.go              # FileSystemBackend、PathResolver、核心 options/types
    errors.go               # 统一错误分类
  model/
    path.go                 # PathRef、ResolvedPath、WorkspaceRef、SandboxContext
    stat.go                 # FileStat、DirEntry、WalkOutcome
  adapter/
    sandbox/                # 可选：基于 genesis-sandbox API 的产品无关 FileSystemBackend
  permission/
    runtime.go              # RuntimeFilePermissions，会话级授权集合
    protect.go              # 系统保护路径、敏感路径、symlink 策略
    rules.go                # 文件路径规则、操作类型 read/write/edit/list/walk
  freshness/
    tracker.go              # FreshnessTracker interface
    inmemory.go             # session scoped 实现，不使用全局变量
  tool/
    read_file/
    write_file/
    edit_file/
    list_dir/
    walk_dir/

internal/capabilities/execution/
  contract/
    runner.go               # SandboxRunner / CommandRunner port
    sandbox.go              # SandboxPolicy、SandboxProfile、SandboxTransform 纯模型
  adapter/
    sandbox/                # 可选：基于 genesis-sandbox API 的产品无关 SandboxRunner
  policy/
    command_policy.go       # 命令解析、前缀规则、风险判定
  tool/run_command/

internal/capabilities/sandbox/
  contract/
    client.go               # genesis-sandbox API 契约，隐藏 HTTP/gRPC 细节
    workspace.go            # sandbox workspace、文件版本、trace id 等模型
  adapter/
    http/                   # HTTP client adapter，产品 bootstrap 注入 endpoint/credential
    grpc/                   # 后续可选 gRPC client adapter

internal/capabilities/tool/
  permission/
    decision.go             # Allow / Ask / Deny
    evaluation.go           # 命中规则、原因、建议授权项
    mode.go                 # default / acceptEdits / dontAsk / yolo / bypass
  scheduler/
    queue.go                # 工具并发队列，按 ToolTraits 和资源锁调度
    locks.go                # session/run scoped 资源锁，不使用全局变量
  gateway/
    gateway.go              # 统一入口：可见性、权限、执行、审计
```

说明：

- `filesystem/permission` 只负责文件路径和文件操作规则。
- `execution/policy` 只负责命令和进程执行规则。
- `tool/permission` 放通用决策模型，不承载文件路径细节。
- `tool/gateway` 是所有工具执行的统一入口，后续接人审、审计、Usage、Trace。

    detector.go             # 本地轻量沙箱能力探测
    seatbelt/               # macOS sandbox-exec / Seatbelt
    bubblewrap/             # Linux bubblewrap / namespace / seccomp
    landlock/               # Linux Landlock 可选后端或降级路径
    windows/                # Windows Restricted Token / Job Object / AppContainer 适配
  workdir/
    manager.go              # 本地 WorkDir 生命周期

products/cli/
  bootstrap/
  profile/
  filesystem/
    prompts.go              # CLI/TUI 文件权限提示文案和授权策略

products/desktop/
  bootstrap/
  profile/
  filesystem/
    picker.go               # native 文件选择器授权适配
    prompts.go              # Desktop GUI 权限提示/审批适配

products/enterprise/
  pathresolver/
    resolver.go             # sandbox workspace 路径解析，禁止宿主机路径

说明：

- CLI 和 Desktop 默认共享本地后端，放 `shared/local`，避免重复实现本地 `os` 访问和 symlink 策略。
- CLI/Desktop 的交互方式不同，所以 permission prompt 不放 `local`，分别放 `products/cli` 与 `products/desktop`。
- 基于 genesis-sandbox API 的 backend/client 不是 Enterprise 专属，放在 `internal/capabilities/*/adapter/sandbox` 和 `internal/capabilities/sandbox/adapter`，由 CLI/Desktop/Enterprise bootstrap 按 profile 选择注入。

## 五、核心契约建议

### 5.1 路径模型

不要让 backend 直接接收未经处理的 `string path`。工具输入可以是 string，但进入 backend 前必须经过 resolver。

```go
type PathRef struct {
    Raw string
}

type ResolvedPath struct {
    DisplayPath  string // 给用户/UI看的路径
    BackendPath  string // 后端实际访问路径，本地绝对路径或 sandbox path
    WorkspaceRel string // 相对 workspace 的路径，可为空
}

type PathResolver interface {
    Resolve(ctx context.Context, ref PathRef, opts ResolveOptions) (ResolvedPath, error)
}
```

必要原因：

- Windows 路径大小写与盘符处理不同。
- symlink 可能绕过 workspace 边界。
- Enterprise 的路径应是 sandbox/workspace 内路径，不应该接受宿主机绝对路径。
- UI 展示路径和 backend 真实路径不是一回事。

### 5.2 FileSystemBackend

第一轮建议只放安全基础操作，不急着加入高风险操作。

```go
type FileSystemBackend interface {
    Read(ctx context.Context, path ResolvedPath, opts ReadOptions) ([]byte, error)
    Write(ctx context.Context, path ResolvedPath, content []byte, opts WriteOptions) error
    ListDir(ctx context.Context, path ResolvedPath, opts ListOptions) ([]DirEntry, error)
    Walk(ctx context.Context, path ResolvedPath, opts WalkOptions) (*WalkOutcome, error)
    Stat(ctx context.Context, path ResolvedPath) (*FileStat, error)
    MkdirAll(ctx context.Context, path ResolvedPath, opts MkdirOptions) error
}
```

`Remove / Copy / Rename` 不建议第一轮加入。长期最佳实践是把它们做成结构化文件工具，而不是鼓励 Agent 用 `run_command` 拼 shell 命令完成；结构化工具更容易做路径解析、权限确认、审计、新鲜度/版本校验和跨本地/sandbox 后端适配。第一轮只有在用户明确要求临时命令时，才允许通过 `run_command` 执行等价 shell 操作，并必须按高风险/破坏性命令走 approval、锁和审计。

### 5.3 WriteOptions

写入不是简单覆盖，应至少保留这些选项：

```go
type WriteOptions struct {
    CreateParents bool
    Overwrite     bool
    Atomic        bool
    ExpectedHash  string
}
```

- `Atomic`：本地 backend 用临时文件 + rename；远程 backend 由 sandbox/workspace API 保证或返回不支持。
- `ExpectedHash`：和新鲜度追踪联动，避免覆盖外部修改。
- `Overwrite=false`：支持“只创建新文件”。

### 5.4 WalkOptions

`Walk` 必须强制有边界：

```go
type WalkOptions struct {
    MaxDepth       int
    MaxDirs        int
    MaxEntries     int
    MaxBytes       int
    FollowSymlinks bool
}
```

默认值要保守，例如：`MaxDepth=8`、`MaxDirs=1000`、`MaxEntries=10000`、`MaxBytes=4MB`。产品 profile 可以放宽，但不能无限制。

文本读取与二进制预览必须分层：`read_file` 只承载文本读取，所有二进制/媒体/Office/PDF/压缩包内容通过 `filesystem/binarygate.ClassifyContent` 统一分类后只返回路径、大小、hash、MIME 和提示信息，不把原始字节塞进 Tool result、LLM 上下文或日志。需要内容级理解时，应走对应的文档/媒体检查工具，例如 Office extract/inspect/render、PDF parser、image preview，而不是让 `read_file` 兼任二进制查看器。

### 5.5 CommandRunner / SandboxRunner

命令执行不是“不需要 adapter”。更准确的边界是：上层只依赖 `CommandRunner` / `SandboxRunner` port；CLI/Desktop 的 local runner、Enterprise 的 sandbox runner 都是产品 adapter，由各自 `products/<product>/bootstrap` 注入。

文件工具和命令工具都遵循 ports-adapters。差别只是命令工具不应该为 CLI、Desktop、Enterprise 暴露三套不同 Tool；Tool 语义保持统一，runner 实现按产品隔离。

### 5.6 RunCommand / Command Tool 边界

`run_command` 通用命令行执行工具不属于 `filesystem/tool`，应放在 `internal/capabilities/execution/tool/run_command`。它和文件系统强相关，但边界不同：文件系统工具负责 `read_file/write_file/edit_file/list_dir/walk_dir` 这类结构化文件操作；命令工具负责进程启动、stdout/stderr、PTY/stdin、超时、退出码、环境变量、网络和 sandbox escalation。

命令工具可以使用文件系统相关能力，但只能通过 port 间接协作：

- `PathResolver`：解析 cwd、workspace roots 和用户输入路径。
- `CommandRunner` / `SandboxRunner`：执行本地命令、平台沙箱命令或 genesis-sandbox 命令。
- `ToolScheduler` / `ResourceLock`：根据命令风险申请 workspace/file/sandbox 锁。
- `PermissionEngine`：处理命令权限、额外文件权限、网络权限和人工确认。

因此 RunCommandTool 不应该直接 import `os/exec`、Docker SDK、genesis-sandbox HTTP client、Wails、RBAC 或 DB；这些都由产品 bootstrap 注入到 runner、resolver、permission adapter 中。后续如果支持 PTY、`write_stdin`、后台进程、任务输出订阅，也应继续留在 execution 能力域，而不是扩张 filesystem 能力域。

推荐执行链路：

```text
run_command tool
  → ToolGateway
  → CommandPolicy / CommandClassifier
  → PermissionEngine（命令权限、文件权限、网络权限、escalation）
  → ToolScheduler / ResourceLock
  → PathResolver（cwd、workspace roots、附加授权路径）
  → CommandRunner 或 SandboxRunner
  → Trace / Audit / Usage / TaskOutput
```

可复用能力：

- `ToolGateway`：统一可见性、权限入口、审计、Usage、Trace。
- `PermissionEngine`：复用 Allow / Ask / Deny、runtime grants、additional permissions、system protected paths、bypass 底线。
- `ToolScheduler`：复用 ToolTraits、barrier、workspace/file/sandbox 资源锁。
- `PathResolver`：复用 cwd、workspace roots、路径规范化、symlink/reparse point 处理。
- `SandboxProfile`：复用 filesystem/network/sandbox escalation 模型。
- `RuntimeFilePermissions`：命令请求额外读写目录时复用同一套授权状态。

RunCommandTool 需要额外拥有的逻辑：

| 额外逻辑                                  | 原因                                            | 放置建议                                          |
| ------------------------------------- | --------------------------------------------- | --------------------------------------------- |
| `CommandPolicy` / `CommandClassifier` | 判断命令风险、是否只读、是否需要网络、是否可能写文件                    | `internal/capabilities/execution/policy`      |
| shell 选择与平台差异                         | Windows PowerShell/cmd、Linux/macOS shell 参数不同 | `CommandRunner` adapter 或 product profile     |
| cwd / env 构造                          | 需要绑定 workspace、sandbox、产品 profile             | `execution` contract + product bootstrap 注入   |
| timeout / output limit                | 防止长时间运行和输出爆炸                                  | `execution/tool/run_command` + runner options |
| PTY / stdin / background session      | 支持交互命令、持续进程、`write_stdin`                     | 后续 `execution/session` 或 `execution/task`     |
| network permission                    | 命令可以访问网络，不能只按文件权限判断                           | `execution/policy` + `SandboxProfile.Network` |
| escalation 审批                         | `require_escalated`、扩大文件/网络权限必须人审和审计          | `PermissionEngine` + product approval adapter |

设计上，RunCommandTool 的复用边界是“共享治理，不共享实现细节”：权限、锁、路径、沙箱策略复用；命令解析、进程生命周期、PTY/stdin/output、平台 shell 差异是 execution 自己的逻辑。这样后续开发 bash 时不会复制文件系统工具的权限和沙箱代码，也不会把命令执行塞进文件系统 backend。

### 5.7 Tool 元数据与执行契约

早期分析中关于 Tool 接口的元数据值得保留，但不要让文件系统能力直接拥有整个工具框架。建议放在 `internal/capabilities/tool/contract` 或现有 tool contract 中，文件工具只消费这些通用字段：

```go
type ToolExposure string

const (
    ToolExposureDirect   ToolExposure = "direct"
    ToolExposureDeferred ToolExposure = "deferred"
    ToolExposureHidden   ToolExposure = "hidden"
)

type ToolTraits struct {
    Exposure                ToolExposure
    ReadOnly                bool
    ConcurrencySafe         bool
    RequiresUserInteraction bool
}
```

设计含义：

- `ReadOnly`：用于只读模式、审计和自动授权，不等价于无权限检查。
- `ConcurrencySafe`：给 runtime 并发调度使用，避免两个写工具同时改同一文件。
- `RequiresUserInteraction`：即使在 `yolo` 或 `bypass` 类模式下，也不能被静默放行。
- `Exposure`：区分直接暴露、延迟发现、隐藏内部工具，避免工具列表无限膨胀。
- 输入校验要独立于执行逻辑，校验失败不能进入权限申请、文件访问或 runner 执行阶段。

文件工具目录仍保持“一工具一目录”，但不强制每个工具一开始就拆很多文件。第一轮可以 `tool.go / types.go / spec.go` 起步；当执行、渲染、prompt 变复杂时再拆 `call.go`、`render.go`、`prompt.go`。

## 六、权限与安全边界

### 6.1 权限决策分层

```text
ToolGateway
  → tool/permission.Engine
      → filesystem/permission rules
      → execution/policy rules
      → runtime grants
      → product approval adapter
  → FileSystemBackend / SandboxRunner
```

评估顺序建议：

1. System protected paths：系统目录、凭据目录、`.git` 危险操作、用户 home 敏感文件。
2. Tool traits：如果工具标记 `RequiresUserInteraction`，不能被 `dontAsk / yolo / bypass` 静默放行。
3. Deny rules：显式拒绝优先。
4. Runtime grants：会话级已授权目录。
5. Allow rules：Profile 或项目配置允许。
6. PermissionMode：`acceptEdits / dontAsk / yolo / bypass` 的产品策略。
7. Ask：交给 CLI TUI、Desktop GUI 或 Web 审批。

补充原则：

- PermissionEngine 返回的不只是 `Decision`，还要包含命中的规则、原因、建议授权项，方便 CLI/Desktop/Web 展示给用户和审计系统。
- `bypass` 只能跳过普通提示，不能跳过 system protected paths、显式 deny、安全写保护和必须交互的工具。
- 策略需要支持分层叠加：System → Product Profile → Tenant/Project → Agent → User/Role → Session → Request。后层可以追加或覆盖前层，但必须保留 matched rules 以便解释。
- 命令规则可借鉴 Codex 的按程序名索引方式；先匹配精确程序名，再匹配解析后的 basename，最后走默认启发式策略。

### 6.2 symlink 策略

必须明确 symlink 处理，否则 WorkDir 限制容易被绕过。

建议默认策略：

- `Resolve` 时计算逻辑路径和真实路径。
- 对写操作，默认不允许通过 symlink 写出 workspace。
- 对读操作，是否允许跟随 symlink 由 Profile 控制，默认只允许 workspace 内真实路径。
- `Walk` 默认不跟随目录 symlink。
- Enterprise/sandbox backend 不暴露宿主机 symlink 语义，只处理 sandbox 内路径。

### 6.3 RuntimeFilePermissions

运行时授权必须是 session scoped，不用包级全局变量。

```go
type RuntimeFilePermissions interface {
    GrantRead(path ResolvedPath) error
    GrantWrite(path ResolvedPath) error
    CanRead(path ResolvedPath) bool
    CanWrite(path ResolvedPath) bool
    Clear()
}
```

实现上可借鉴 Kode 的父子目录合并算法：授权父目录后移除子目录授权，减少匹配成本。

## 七、新鲜度追踪

新鲜度追踪是文件系统能力的一等组件，不应只是 tool 内部 map。

```go
type FreshnessTracker interface {
    RecordRead(ctx context.Context, path ResolvedPath, stat FileStat, hash string) error
    CheckBeforeWrite(ctx context.Context, path ResolvedPath, current FileStat, currentHash string) (*FreshnessCheck, error)
    RecordWrite(ctx context.Context, path ResolvedPath, stat FileStat, hash string) error
}
```

设计原则：

- CLI/Desktop：默认用 session scoped in-memory tracker。
- Enterprise：第一轮也可以内存；后续任务恢复/长任务可接数据库或任务状态存储。
- 写前如果 mtime 变化且 hash 变化，返回 `file_modified_externally`，由 ToolGateway 或 Human-in-the-loop 处理。
- Agent 自己刚写入的文件要更新 tracker，避免误判为外部修改。

## 八、并发控制策略

并发控制不能只靠工具名判断，也不能把所有写操作全局串行。推荐采用三层控制：**ToolTraits 队列调度 + 资源级锁 + 新鲜度检查**。

### 8.1 从参考项目吸收的原则

Kode-CLI 的核心思路值得借鉴：

- 工具先声明 `isConcurrencySafe(input)`。
- 多个 concurrency-safe 工具可以同时启动。
- 非 concurrency-safe 工具是 ordering barrier：它执行期间，后续工具不能越过它产出最终结果。
- 如果输入校验失败，不能调用 `isConcurrencySafe`，应降级为不安全并走串行路径。
- run_command 类工具的并发安全可以等价于只读检测：`pwd`、`ls`、`cat` 可并行，`cat foo > bar`、`git commit`、`npm install` 不可并行。

Genesis 需要在这个基础上增加资源级判断，因为 Go 后端会同时承载 CLI、Desktop、Web/Enterprise 多会话：

- 两个读不同文件：可并行。
- 两个读同一文件：可并行。
- 写 `a.go` 与写 `b.go`：理论可并行，但第一轮建议在同一个 Run 内串行写，降低编辑冲突和审计复杂度。
- 写 `a.go` 与读 `a.go`：同一 Run 内必须有顺序；不同 Run 由文件锁与新鲜度检查兜底。
- 非文件状态工具，例如 todo/task/session 修改，默认视为 Run 级写锁。

### 8.2 Tool Scheduler 契约

建议新增通用调度器，不放在 filesystem 内：

```go
type ToolUse struct {
    ID      string
    Name    string
    Input   any
    Traits  ToolTraits
    Locks   []ResourceLock
}

type ResourceLock struct {
    Scope string // run / session / workspace / file / sandbox / tenant
    Key   string // runID、sessionID、workspaceID、ResolvedPath.WorkspaceRel 等
    Mode  LockMode
}

type LockMode string

const (
    LockRead  LockMode = "read"
    LockWrite LockMode = "write"
)

type ToolScheduler interface {
    Submit(ctx context.Context, uses []ToolUse) (<-chan ToolEvent, error)
}
```

调度规则：

1. 先做 schema/input 校验；校验失败的工具直接返回错误，不参与并发安全判定。
2. 计算 `ToolTraits` 和 `ResourceLock`。
3. `ConcurrencySafe=false` 的工具作为 barrier：前面工具完成后才能执行，后续工具不能越过它产出最终结果。
4. `ConcurrencySafe=true` 的工具仍必须经过资源锁检查；锁冲突时排队。
5. 结果输出保持 LLM tool_use 顺序；progress 事件可以提前流式输出，但最终 tool_result 不能破坏 barrier 顺序。
6. 任一 sibling tool 失败时，后续未开始工具应取消或返回 sibling_error，避免在错误上下文下继续写文件。
7. 调度器必须 session/run scoped，不使用包级全局队列。

### 8.3 文件工具锁策略

第一轮建议保守：

| 工具                  | Traits                                    | ResourceLock                               |
| ------------------- | ----------------------------------------- | ------------------------------------------ |
| `read_file`         | `ReadOnly=true`, `ConcurrencySafe=true`   | `file:<resolved>` read                     |
| `list_dir`          | `ReadOnly=true`, `ConcurrencySafe=true`   | `workspace:<id>` read                      |
| `walk_dir`          | `ReadOnly=true`, `ConcurrencySafe=true`   | `workspace:<id>` read，且受 bounded walk 限制   |
| `write_file`        | `ReadOnly=false`, `ConcurrencySafe=false` | `file:<resolved>` write + `run:<id>` write |
| `edit_file`         | `ReadOnly=false`, `ConcurrencySafe=false` | `file:<resolved>` write + `run:<id>` write |
| `run_command` 只读命令  | 根据命令解析                                    | workspace read 或 sandbox read              |
| `run_command` 非只读命令 | `ConcurrencySafe=false`                   | workspace write 或 sandbox write            |

写工具即使拿到锁，也必须执行 `FreshnessTracker.CheckBeforeWrite`。锁解决“本进程/本 Run 内并发”，新鲜度解决“外部编辑或其他进程修改”。

### 8.4 多会话与 Enterprise

CLI/Desktop 第一轮可以用进程内 `sync.RWMutex` + 路径归一化 key。Enterprise 后续需要把锁分成两层：

- 进程内锁：保护单实例并发。
- 分布式锁或数据库条件更新：保护多副本、多 worker、多 sandbox 的并发。

Enterprise 不应依赖本地文件锁表达安全性；最终一致性必须由 sandbox/workspace API、对象版本号、`ExpectedHash` 或 DB 乐观锁保证。

## 九、沙箱实现模式

沙箱能力要分成两层：`internal/capabilities/execution/contract` 只定义策略模型和 runner port；`shared/local/sandbox` 提供本机平台实现，`internal/capabilities/sandbox/adapter` 提供 genesis-sandbox API client，`products/enterprise` 只提供企业上下文和 bootstrap 注入。不要让 macOS/Linux/Windows、Docker 或 HTTP/gRPC 细节进入 runtime、tool 或 filesystem contract。

### 9.1 沙箱策略模型

参考 Codex 的 `PermissionProfile / SandboxEnforcement / SandboxPermissions`，Genesis 建议采用下面的语义：

```go
type SandboxEnforcement string

const (
    SandboxManaged  SandboxEnforcement = "managed"  // Genesis 负责构造本机沙箱
    SandboxExternal SandboxEnforcement = "external" // 外部 sandbox/workspace 已负责隔离
    SandboxDisabled SandboxEnforcement = "disabled" // 明确不加外层沙箱，高风险
)

type SandboxPermissionOverride string

const (
    SandboxUseDefault             SandboxPermissionOverride = "use_default"
    SandboxWithAdditionalPerms    SandboxPermissionOverride = "with_additional_permissions"
    SandboxRequireEscalated       SandboxPermissionOverride = "require_escalated"
)

type SandboxProfile struct {
    Enforcement SandboxEnforcement
    FileSystem  FileSystemSandboxPolicy
    Network     NetworkSandboxPolicy
}
```

关键规则：

- `managed`：CLI/Desktop 由 Genesis 选择平台沙箱。
- `external`：CLI/Desktop 可选 Docker/genesis-sandbox，Enterprise 使用远程 workspace；外部服务负责隔离，本地不再叠加宿主机沙箱。
- `disabled`：只能由明确配置或人工确认触发，且必须审计。
- `with_additional_permissions` 只扩大当前命令/当前 tool call 的权限，不应污染 session 默认权限。
- `require_escalated` 表示请求脱离沙箱执行，必须走人审和审计；Enterprise 默认应拒绝或映射为受控运维流程。

### 9.2 平台选择与降级原则

本地产品建议提供 `SandboxManager`：

```go
type SandboxType string

const (
    SandboxNone             SandboxType = "none"
    SandboxMacOSSeatbelt    SandboxType = "macos_seatbelt"
    SandboxLinuxBubblewrap  SandboxType = "linux_bubblewrap"
    SandboxLinuxLandlock    SandboxType = "linux_landlock"
    SandboxWindowsToken     SandboxType = "windows_restricted_token"
    SandboxWindowsElevated  SandboxType = "windows_elevated"
)

type SandboxPreference string

const (
    SandboxAuto    SandboxPreference = "auto"
    SandboxRequire SandboxPreference = "require"
    SandboxForbid  SandboxPreference = "forbid"
)
```

选择规则：

1. `SandboxForbid`：不启用平台沙箱，但仍保留权限、人审、新鲜度、审计。
2. `SandboxRequire`：如果当前平台不能满足策略，直接失败，不静默降级为无沙箱。
3. `SandboxAuto`：能启用则启用；如果不能启用，必须返回 degraded warning，并由产品 profile 决定允许继续还是失败。
4. 只要涉及网络隔离、写隔离或高风险命令，不能在没有记录的情况下静默运行 unsandboxed。

### 9.3 macOS

参考 Codex 的 Seatbelt 实现：

- 使用系统固定路径 `/usr/bin/sandbox-exec`，不要从 `PATH` 查找，避免被恶意同名命令劫持。
- 通过 profile/policy 文本表达可读根、可写根、deny glob、网络访问和 Unix socket 例外。
- 对 protected metadata，例如 `.git`、凭据目录，写权限要做额外 deny carveout。
- 网络默认关闭；如需要代理或托管网络，只开放明确 loopback 端口或受控 socket。

Genesis 第一轮不必完整实现 Seatbelt policy 生成，但契约要允许把 `FileSystemSandboxPolicy` 和 `NetworkSandboxPolicy` 转换成平台参数。

### 9.4 Linux

参考 Codex 的 Linux 设计：

- 首选 bubblewrap / user namespace / seccomp 组合。
- Landlock 可作为可选后端或老环境降级路径，但能力表达要清楚。
- 需要探测 WSL1：WSL1 不支持所需 user namespace 时，不能假装 sandbox 可用。
- 需要探测系统 `bwrap` 是否存在，以及是否能创建 user namespace。
- 如果需要受控网络代理，通常需要 bubblewrap 的 network namespace；不满足时应失败或走明确降级。
- 平台沙箱 helper 的路径解析要可信，不应从 workspace 内寻找可执行文件。

Linux 的执行包装建议由 `shared/local/sandbox/bubblewrap` 生成 argv，`CommandRunner` 只负责运行转换后的命令和流式收集 stdout/stderr。

### 9.5 Windows

Windows 不能简单照搬 Unix 沙箱，建议分层支持：

- 第一层：Restricted Token + Job Object，限制权限、进程树、资源和生命周期。
- 第二层：AppContainer 或类似隔离能力，后续按可用性评估。
- Elevated/provisioned sandbox 可作为增强模式，但需要 setup/preflight，不应在第一次执行时悄悄失败后降级。
- Windows 文件权限需要考虑盘符、大小写、短路径、junction/symlink/reparse point，必须通过 `PathResolver` 和 sandbox override 统一处理。
- 如果某些 read deny、split writable roots、网络限制无法由当前 Windows 后端直接表达，应拒绝执行或请求更强 sandbox 模式，不能无声放宽。
- 默认启用 private desktop / isolated process group 这类增强隔离时，要允许产品 profile 显式配置。

### 9.6 CLI/Desktop 可选 Docker / genesis-sandbox

CLI/Desktop 默认走本地工作区和本机平台沙箱，但必须预留 Docker/genesis-sandbox 模式。这个模式不是 Enterprise 专属，而是本地产品可选的更强隔离运行环境：

```text
CLI/Desktop profile
  → product bootstrap
  → sandbox client adapter（HTTP/gRPC）
  → genesis-sandbox service
  → Docker/container workspace
```

设计规则：

- Profile 中用运行环境选择 backend：`local_host`、`local_platform_sandbox`、`docker_sandbox`、`remote_sandbox`。
- `docker_sandbox` 与 `remote_sandbox` 都属于 `SandboxExternal`：外部服务负责文件和命令隔离，本进程不要再叠加宿主机 FS backend。
- CLI/Desktop 的 UI 权限提示仍留在 `products/cli/filesystem` 与 `products/desktop/filesystem`；它们只决定是否授权，不直接实现 sandbox API。
- `genesis-sandbox` client 是产品无关 adapter，放 `internal/capabilities/sandbox/adapter/http|grpc`；产品 bootstrap 只注入 endpoint、credential、workspace id、默认 profile。
- 文件工具仍依赖 `FileSystemBackend`，命令工具仍依赖 `SandboxRunner`。切到 Docker sandbox 时只是注入不同实现，工具代码不变化。
- Docker workspace 的路径必须用 sandbox path 或 workspace-relative path；不能把宿主机绝对路径泄漏给工具或 Web/desktop UI。
- Docker sandbox 返回的 file version、hash、trace id 要写入 FreshnessTracker、Trace、Audit。

这条路径适合：用户希望 CLI/Desktop 有更强隔离；项目依赖复杂、需要容器环境；或者本机平台沙箱能力不足但 Docker 可用。

### 9.7 Enterprise / Web

Enterprise 不直接使用宿主机平台沙箱，而是 `SandboxExternal`：

- 文件访问走通用 sandbox `FileSystemBackend`，通过 sandbox/workspace API；Enterprise 只注入租户、项目、RBAC、endpoint 和 credential。
- 命令执行走 enterprise `SandboxRunner`，调用 `genesis-sandbox` exec API。
- 权限、租户、项目、用户、角色在请求进入 sandbox 前完成裁剪。
- sandbox 返回的版本号/hash/trace id 要进入 FreshnessTracker、Audit、Trace。
- 浏览器端不传宿主机路径，只传 workspace resource id 或相对路径。

### 9.8 沙箱失败与审计

所有平台都要统一错误分类：

- `sandbox_unavailable`：平台能力不存在，例如 Linux 无 user namespace。
- `sandbox_policy_unsupported`：策略无法被当前平台表达，例如 Windows 无法表达某些 read deny。
- `sandbox_denied`：沙箱成功启动但拒绝访问。
- `sandbox_timeout`：命令超时。
- `sandbox_escalation_required`：工具请求脱离沙箱或扩大权限。

每次命令执行审计至少包含：sandbox type、enforcement、filesystem policy 摘要、network policy、是否 additional permissions、是否 require escalated、降级原因、平台错误码。

## 十、三端数据流

### 10.1 CLI

```text
read_file tool
  → ToolGateway
  → PermissionEngine
  → CLI approval prompt（需要时）
  → product-selected PathResolver
  → local filesystem backend 或 sandbox FileSystemBackend
  → FreshnessTracker.RecordRead
```

CLI 的特点：轻量、交互式、默认 WorkDir 限制，不需要 DB；但 profile 可以切到 Docker/genesis-sandbox workspace。

### 10.2 Desktop

```text
desktop UI / agent action
  → ToolGateway
  → PermissionEngine
  → Desktop native picker / GUI approval
  → product-selected PathResolver
  → local filesystem backend 或 sandbox FileSystemBackend
  → Desktop projection / task view
```

Desktop 和 CLI 共享 local backend，但不共享 UI 审批实现。Desktop 可以通过原生文件选择器扩大授权范围，也可以通过 profile 切到 Docker/genesis-sandbox workspace。

### 10.3 Web / Enterprise

```text
web request / agent action
  → HTTP interface
  → app / runtime
  → ToolGateway
  → PermissionEngine + tenant/project policy
  → enterprise PathResolver
  → sandbox FileSystemBackend / SandboxRunner
  → sandbox/workspace API
```

Web 前端不直接获得宿主机路径，也不应该让浏览器上传任意服务器路径。Enterprise 只处理 sandbox/workspace 内资源。

### 11.1 当前实现同步说明

截至当前实现，目录和契约有几处需要明确：

- 本地文件系统后端实际目录为 `shared/local/filesystem`，不是早期草案中的 `shared/local/fs_backend`。
- `FileSystemBackend` 和 sandbox contract 中已经预留 `Remove` 能力，但当前没有向 Agent 暴露 `remove_file` 工具；`Remove / Copy / Rename` 仍按本文原则后续单独设计，未完成权限、审计、人审链路前不作为默认工具开放。
- `ToolTraits` 已进入 `internal/capabilities/tool/contract`，并由 `ToolGateway` 读取；后续新工具必须显式声明或补充默认 traits，避免工具治理只靠工具名猜测。
- `ToolGateway` 已作为 runtime 工具调用入口承载 Profile 可见性、通用 Authorizer hook、基础资源锁、Trace、系统化 Audit/Usage sink；文件路径、命令风险、租户/RBAC 等领域级策略仍由具体能力域或产品 bootstrap 注入。
- `RuntimeFilePermissions` 已放在 `internal/capabilities/filesystem/permission`，用于会话/项目级目录授权、父子目录合并和 approval 包装；Enterprise 分布式授权和持久化仍未实现。
- Windows 本地沙箱当前已实现 `process_constrained` 执行路径：通过 Restricted Token 启动进程，并绑定 Job Object 以支持进程树生命周期收口；同时按 Codex 的安全策略，`process_constrained` 仍不能声明为文件系统/网络强隔离。AppContainer capability 已显式 fail-closed。**Windows L2（Unelevated ACL）及后续等级的落地规格**见 `docs/Windows本地平台沙箱实现设计.md`；WFP 尚未实现。
- 审计/Trace/Usage 已有 Gateway 级工具生命周期、traits、成功/失败、耗时、用量 sink；sandbox type、降级原因、权限决策、hash/version 等领域字段仍需由 execution/filesystem 能力域继续补充。
- 并发调度目前已有 `ResourceLocker`、Gateway 基础锁，以及 React loop sibling tool queue/barrier：连续只读且 `ConcurrencySafe=true` 的工具并行执行，写入/危险/未知工具作为 barrier 串行执行；跨进程/分布式队列和失败级联取消仍留给后续。

## 十二、落地顺序

建议按风险从低到高落地：

1. `filesystem/model` + `filesystem/contract`：路径、stat、walk、backend、resolver 契约。
2. `filesystem/permission`：protected path、runtime grants、symlink 策略。
3. `filesystem/freshness`：session scoped tracker。
4. `tool/scheduler`：ToolTraits、ResourceLock、barrier 队列，先用内存锁实现。
5. `shared/local/pathresolver` + `shared/local/filesystem`。
6. `read_file`、`list_dir`、`walk_dir` 工具。
7. `write_file`、`edit_file` 工具，必须带新鲜度检查和原子写策略。
8. `execution/contract` + `SandboxProfile` + 本地 runner port。
9. `internal/capabilities/sandbox/contract`：genesis-sandbox API 契约和 workspace 模型。
10. `shared/local/sandbox`：平台探测和 macOS/Linux/Windows wrapper 骨架。
11. `internal/capabilities/sandbox/adapter/http` + sandbox FileSystemBackend/SandboxRunner，用于 CLI/Desktop Docker 模式和 Enterprise。

第一轮不建议实现 `Remove / Copy / Rename`。这些操作应等权限、审计、人审链路完整后再做；后续应优先作为结构化文件工具实现，而不是依赖 `run_command`。

## 十三、关键决策汇总

| 决策点                             | 结论                                                                                                                                                                 |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 三端文件工具是否统一                      | 工具语义统一，后端按运行环境选择                                                                                                                                                   |
| CLI/Desktop 是否共享实现              | 默认共享 local backend 和 pathresolver；可选共享 sandbox API backend；不共享 UI 权限提示                                                                                             |
| Web/Enterprise 是否访问宿主机          | 不允许，只访问 sandbox/workspace                                                                                                                                          |
| 路径是否直接用 string                  | Tool 输入可以是 string；backend 必须用 ResolvedPath                                                                                                                         |
| 新鲜度追踪放哪里                        | `filesystem/freshness`，session scoped，不用全局变量                                                                                                                       |
| Remove/Copy 是否第一轮做              | 不做，后续单独设计                                                                                                                                                          |
| symlink 是否需要第一轮考虑               | 必须考虑，否则 WorkDir 可被绕过                                                                                                                                               |
| bounded walk 是否必须               | 必须，且由 backend 强制执行                                                                                                                                                 |
| Tool 元数据是否需要                    | 需要，至少包含 Exposure、ReadOnly、ConcurrencySafe、RequiresUserInteraction                                                                                                  |
| 并发是否只靠 bool 控制                  | 不是，`ConcurrencySafe` 只决定 barrier；真正冲突还要靠 `ResourceLock`                                                                                                            |
| 写工具是否可以并行                       | 第一轮同一 Run 内写工具串行；后续可按文件锁细化                                                                                                                                         |
| 锁是否替代新鲜度检查                      | 不能，锁管本进程/本 Run，新鲜度管外部修改和多进程                                                                                                                                        |
| bypass 是否能跳过所有检查                | 不能，system protected paths、显式 deny、安全写保护和必须交互工具不可绕过                                                                                                                 |
| 权限策略是否支持叠加                      | 支持 System → Product Profile → Tenant/Project → Agent → User/Role → Session → Request                                                                               |
| 沙箱策略放哪里                         | `internal/capabilities/execution/contract` 放模型；本机平台实现放 `shared/local/sandbox`；genesis-sandbox client 放 `internal/capabilities/sandbox/adapter`                     |
| 沙箱不可用时能否静默降级                    | 不能，`Require` 直接失败，`Auto` 必须产生 degraded warning 并由 profile 决定                                                                                                       |
| run_command/命令行工具是否属于文件系统       | 不属于；放 `execution/tool/run_command`，通过 `ExecutionRunner` / `CommandRunner` / `SandboxRunner` port 执行，并复用 PermissionEngine、ToolScheduler、PathResolver、SandboxProfile |
| macOS/Linux/Windows 是否统一实现      | 不统一实现，只统一 `SandboxProfile`、`CommandRunner`、错误分类和审计字段                                                                                                               |
| CLI/Desktop 是否支持 Docker sandbox | 支持，可通过 profile 选择 genesis-sandbox API backend，不改变 Tool 代码                                                                                                          |

---

## 十四、设计反思与开发约束

整体方案是优雅且适合开发落地的，原因是它把变化点放在正确边界上：

- Tool 语义稳定：Agent、Prompt、测试、审计只认识统一工具，不关心本地、Docker、远程 sandbox。
- Backend 可替换：`FileSystemBackend`、`PathResolver`、`SandboxRunner` 是 port，产品 bootstrap 决定注入本地实现还是 genesis-sandbox 实现。
- 产品差异收敛：CLI、Desktop、Enterprise 的差异集中在 profile、bootstrap、approval UI、endpoint/credential，不扩散到工具代码。
- 安全状态独立：权限、运行时授权、新鲜度、资源锁、sandbox profile 分层，不互相替代。
- 并发策略可演进：第一轮保守串行写，后续可按 `ResourceLock` 细化，不需要重写工具接口。

需要特别守住的边界：

- `internal/capabilities/sandbox/adapter` 只能放产品无关 API client，不放 CLI/Desktop/Enterprise 的默认 endpoint、credential、租户/RBAC 策略。
- `products/enterprise` 不应该拥有 genesis-sandbox client；它只负责企业上下文、治理策略和 bootstrap 注入。
- `shared/local` 只放宿主机本地能力和平台沙箱能力；Docker/genesis-sandbox API backend 不属于 local host backend。
- 文件工具不直接 import `os`、Docker SDK、HTTP client、Wails、RBAC、DB；只能依赖 port。
- Docker/genesis-sandbox 模式下，工具和 UI 都不能泄漏宿主机绝对路径，只能使用 workspace-relative path、sandbox path 或 resource id。
- `SandboxAuto` 不能静默降级为无沙箱；所有降级都要进入 warning、trace 和 audit。

开发上建议按“能编译的小纵切”推进：先契约和内存实现，再接本地 backend，再加工具，最后接 sandbox API。不要一开始就实现 Docker、Windows elevated、Enterprise 分布式锁，否则会把边界搅在一起。
