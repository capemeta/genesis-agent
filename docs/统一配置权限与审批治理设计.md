# 统一配置、权限与审批治理设计

> 本文档设计 Genesis Agent 的配置体系、权限策略、审批授权与运行时配置边界。它覆盖 CLI、Desktop、Enterprise 三类产品，但不强行要求所有产品使用同一种交互方式；统一的是配置模型、策略决策和审计语义，差异留在产品 bootstrap 和 requester/adapter 中。

## 一、问题与结论

当前 `configs/config.yaml` 已有 LLM、HTTP Client、Secrets、Agent、Log、Server 等基础配置，但文件系统、网络、命令执行、沙箱、MCP、Skill、连接/密钥、审批授权等高风险能力还没有统一配置面。代码中已经出现了几个能力雏形：

- `internal/capabilities/approval/model` 已定义 `allow / ask / deny`、`once / turn / session / project / tenant / global` 授权范围。
- `internal/capabilities/filesystem/permission` 已有运行时文件授权缓存和文件风险上下文。
- CLI bootstrap 已有 `quiet` 模式、终端审批 requester、sandbox flag 最小配置。
- `configs/config.yaml` 当前只配置了 `secrets.data_dir`、`secrets.master_key_env`，还没有权限策略、路径策略、网络策略和沙箱策略。

结论：

1. 应新增统一的 `policy` 配置域，表达“什么动作、什么资源、什么主体、什么环境下 allow/ask/deny”。
2. 文件系统、网络、命令、MCP、Skill、连接、Sandbox 不各自发明权限模型，而是统一转成 `approval.Request` 和 `PolicySubject/PolicyResource`。
3. CLI/Desktop/Enterprise 共享策略模型与 evaluator，差异只在配置来源、默认 posture、审批 requester、授权持久化范围、管理界面。
4. 密钥和连接只通过 `credential_ref` / `connection_ref` 引用；配置文件可以声明 provider/connection 元数据，但不应保存明文密钥。
5. `config.yaml` 应保持可提交的安全默认值；`config.local.yaml` 只适合本地开发覆盖；Enterprise 应优先使用 DB/KMS/Vault/环境注入，不依赖本地文件作为治理事实源。

## 二、配置源与优先级

推荐配置源从低到高：

| 层级 | 来源 | 适用产品 | 是否可含密钥 | 说明 |
|---|---|---|---|---|
| 1 | 内置默认值 | 全部 | 否 | 代码里的保守默认值，保证缺省安全。 |
| 2 | `configs/config.yaml` | CLI/Desktop/Enterprise dev | 否 | 可提交版本库，只放非敏感默认配置。 |
| 3 | 项目配置 `.genesis/config.yaml` | CLI/Desktop | 否 | 项目级策略，例如允许读取某些项目外路径、默认 sandbox mode。 |
| 4 | 用户配置 `~/.genesis/config.yaml` | CLI/Desktop | 否/仅引用 | 用户偏好、常用 provider id、审批体验，不放明文密钥。 |
| 5 | `configs/config.local.yaml` | 本地开发 | 可以但不推荐 | 已 gitignore，只用于开发调试；生产不依赖它。 |
| 6 | 环境变量 | 全部 | 可以 | 适合 CI、容器、临时密钥、主密钥。 |
| 7 | Credential/Connection Store | 全部 | 是 | 密钥与连接事实源，文件加密/DB/Vault/KMS 可替换。 |
| 8 | 运行时授权 Grant Store | 全部 | 否 | 用户批准后的 once/turn/session/project 授权记录，只能在上层策略允许的范围内生效。 |
| 9 | Enterprise Admin Policy | Enterprise | 引用 | 租户/角色/项目/Agent 级策略，最终优先级最高，可撤销或压制运行时 grant。 |

合并原则：

- `deny` 优先于 `allow`，高优先级策略可以进一步收紧低优先级策略。
- 低优先级配置和运行时 grant 都不能解除 Enterprise Admin Policy 的 deny。
- 明文密钥只允许出现在 CredentialStore、环境变量或本地开发覆盖中；普通配置中只允许 `credential_ref`。
- `config.yaml` 中出现 `${ENV_NAME}` 时，只做环境展开，不把展开后的值写回文件。

## 三、统一策略模型

### 3.1 决策结果

统一决策仍沿用现有 `approval.model.PolicyType`：

| 决策 | 含义 |
|---|---|
| `allow` | 自动允许，不询问用户。 |
| `ask` | 需要产品侧 requester 询问用户或进入企业审批流。 |
| `deny` | 直接拒绝，不允许模型或用户在当前上下文绕过。 |

### 3.2 授权范围

现有 scope 基本合理，但语义需要统一：

| Scope | 适用场景 | 是否建议持久化 |
|---|---|---|
| `once` | 本次工具调用 | 否 |
| `turn` | 本轮用户交互内同类操作 | 会话内短暂缓存 |
| `session` | 当前会话 | 可持久化到会话状态 |
| `project` | 当前项目/工作区 | CLI/Desktop 可写项目 grant；Enterprise 需管理员允许 |
| `tenant` | 租户范围 | 仅 Enterprise 管理员策略，不由普通交互审批产生 |
| `global` | 全局用户/系统 | 仅系统/管理员策略，默认不在交互审批中提供 |

建议补充两个派生维度，但不新增为 `GrantScope`：

- `profile`：CLI/Desktop/Enterprise 或自定义 profile，作为 policy match 条件。
- `environment`：local/dev/test/prod/sandbox，作为 policy match 条件。

### 3.3 主体、动作、资源、环境

统一策略输入：

```go
type PolicySubject struct {
    TenantID  string
    UserID    string
    Roles     []string
    ProjectID string
    AgentID   string
    Product   string // cli / desktop / enterprise
}

type PolicyAction struct {
    Name     string // file.read / web.fetch / command.exec / credential.resolve
    ToolName string
    Risk     string
}

type PolicyResource struct {
    Type     string // file / directory / url / domain / command / connection / credential / sandbox
    URI      string
    Display  string
    Metadata map[string]string
}

type PolicyEnvironment struct {
    Profile     string
    WorkspaceID string
    WorkspaceRoot string
    SandboxMode string
    NetworkMode string
    RuntimeEnv  string // local/dev/test/prod
}
```

各能力域负责把自身上下文转成统一输入：

| 能力 | Resource URI 示例 | 关键 metadata |
|---|---|---|
| 文件系统 | `workspace://src/main.go`、`file:///D:/data/a.txt` | path_scope、backend、workspace_rel、operation、protected |
| 命令执行 | `command://go test ./...` | cwd、program、args_hash、network_intent、destructive |
| 网络搜索 | `web-search://query/<hash>` | query_preview、provider、domains、region |
| 网络获取 | `https://example.com/page` | domain、resolved_ip_class、redirect_host、content_type |
| HTTP API | `connection://crm_prod` 或 URL | connection_ref、method、host、path_template |
| MCP | `mcp://server/tool` | server、tool、connector_id |
| Skill | `skill://authority/package` | authority、trusted、external_dependencies |
| Credential | `credential://cred_id` | purpose、connection_ref、secret_type |
| Sandbox | `sandbox://profile/name` | provider、filesystem_mode、network_mode |

## 四、策略配置 Schema 建议

新增 `policy` 顶层配置域，第一版可以先只实现子集，但文档和结构应一次设计清楚。

```yaml
policy:
  defaults:
    unknown: ask
    dangerous: ask
    critical: deny
    deny_overrides_allow: true
    allowed_grant_scopes: [once, turn, session, project]

  grants:
    store: memory       # memory | file | db
    persist_scopes: [session, project]
    project_grants_file: .genesis/grants.yaml
    session_ttl: 12h

  files:
    default: ask
    workspace:
      read: allow
      list: allow
      walk: allow
      write: allow
      edit: allow
      delete: ask
    external:
      read: ask
      list: ask
      walk: ask
      write: ask
      edit: ask
      delete: deny
    protected:
      default: deny
    allow_paths:
      - path: D:\workspace\go
        operations: [read, list, walk]
        scope: project
      - path: D:\tmp
        operations: [read, write, edit, delete, list, walk]
        scope: session
    deny_paths:
      - path: C:\Windows
        operations: [read, write, edit, delete, list, walk]
      - path: ${USERPROFILE}\.ssh
        operations: [read, write, edit, delete, list, walk]
    workspace_metadata:
      write: deny
      paths: [.git, .agents, .codex]

  commands:
    default: ask
    allow:
      - program: go
        args_prefix: [test]
        cwd_scope: workspace
      - program: npm
        args_prefix: [run, build]
        cwd_scope: workspace
    deny:
      - program: rm
      - program: powershell
        args_contains: [Remove-Item, -Recurse, -Force]
    network:
      default: ask

  web:
    search:
      default: ask
      providers:
        searxng: allow
        brave: ask
        baidu_ai_search: ask
      allowed_domains: []
      blocked_domains: [localhost, 127.0.0.1]
    fetch:
      default: ask
      allowed_domains: []
      blocked_domains: [localhost, 127.0.0.1, 169.254.169.254]
      allow_private_ip: false
      cross_host_redirect: ask
      browser_render: ask

  connections:
    default: ask
    allow_refs:
      - connection_ref: github_public
        tools: [http_request, web_fetch]
    deny_refs: []

  credentials:
    resolve_default: deny
    allow_purposes: [llm_provider_call, connection_auth, web_provider_call]
    expose_plaintext_to_model: deny

  sandbox:
    default_mode: disabled
    required_for:
      - command.exec
      - web.fetch.browser
    allow_network: false
    allow_host_paths: []
```

关键规则：

- `allow_paths` 表示“不拦截/自动允许”的路径，但仍要经过路径解析、symlink 防护、工作区映射和审计。
- `deny_paths` 永远优先，不能被交互审批覆盖。
- `ask` 的 requester 可以给用户选择 `once/turn/session/project` 等 scope，但只能在 `allowed_grant_scopes` 内选择；`tenant/global` 不应由普通用户弹窗直接产生。
- 对外网络、命令执行、密钥解析、浏览器渲染默认不应 `allow`，除非项目或企业策略显式配置。

## 五、路径与资源匹配最佳实践

路径匹配是最容易出事故的地方，不应只做字符串前缀。

要求：

1. 所有路径配置先解析为绝对路径并 `Clean`。
2. Windows 下比较时统一大小写、处理盘符、UNC 路径、长路径前缀。
3. symlink/junction/reparse point 的最终目标要参与 deny/allow 判断。
4. allow 只对明确操作生效，例如允许 read 不代表允许 write/delete。
5. 工作区内 `.git`、`.agents`、`.codex` 等元数据默认禁止写入，但可允许只读。
6. 外部路径授权应优先建议 `once/session`，不默认建议 `project`。
7. `delete`、递归写、跨盘移动、覆盖大量文件应提升风险等级。

推荐资源匹配器：

```go
type ResourceMatcher interface {
    Match(ctx context.Context, subject PolicySubject, action PolicyAction, resource PolicyResource, env PolicyEnvironment) (MatchResult, error)
}
```

文件系统使用专用 matcher；URL/domain、command、connection、credential 也各自有 matcher，但输出统一 `PolicyResult`。

## 六、审批交互设计

### 6.1 用户可选项

当策略为 `ask` 时，产品侧 requester 应展示：

- 动作：例如 `file.write`、`web.fetch`、`command.exec`。
- 资源：路径、域名、命令、连接名。
- 风险：low/medium/high/critical。
- 原因：为什么要执行。
- 影响：读取/写入/删除/联网/发送凭据/执行命令。
- 可选授权范围。

推荐选项：

| 选项 | Decision | Scope |
|---|---|---|
| 允许本次 | `approved_for_scope` | `once` |
| 允许本轮 | `approved_for_scope` | `turn` |
| 允许本会话 | `approved_for_scope` | `session` |
| 允许本项目 | `approved_for_scope` | `project` |
| 拒绝本次 | `denied` | `once` |
| 中止任务 | `abort` | `once` |

可选但不建议默认展示：

- `tenant`：只给 Enterprise 管理员策略，不给普通交互审批。
- `global`：只给用户设置页或管理员后台，不在工具调用弹窗中展示。
- “记住对同类操作的拒绝”：需要单独 deny grant store，避免误伤后续正常操作。

### 6.2 产品差异

| 产品 | Requester | 默认 posture | 授权持久化 |
|---|---|---|---|
| CLI | 终端确认；quiet 模式自动 deny ask | 本地开发友好，但外部路径/网络/命令仍 ask | memory + 可选项目 `.genesis/grants.yaml` |
| Desktop | 弹窗/权限中心 | 更强调可视化和撤销 | 用户配置目录 + 项目 grant |
| Enterprise | 审批流/RBAC/策略引擎；一般不现场问模型操作者 | 默认最小权限，预配置 allowlist | DB，按 tenant/project/user/role 管理 |

## 七、各能力域配置边界

### 7.1 文件系统

统一放在 `policy.files`，不要在每个文件工具里写自己的 allowlist。

工具只负责：

- 解析路径。
- 识别操作类型。
- 构造 approval request。
- 执行前二次确认 freshness/lock。

策略负责：

- workspace/external/protected 分类。
- allow_paths/deny_paths。
- operation 级控制。
- 授权记忆范围。

### 7.2 命令执行与沙箱

命令执行策略应和 sandbox 配置联动：

- `command.exec` 默认 ask。
- 危险命令 deny 或 ask once，不允许 project scope。
- 需要网络的命令必须额外触发 network policy。
- `SandboxAuto` 不能静默降级为无沙箱；降级必须有 warning/trace/audit。
- `SandboxRequire` 不满足时必须失败。

### 7.3 网络搜索与网络获取

`web_search` / `web_fetch` 的 provider key 走 CredentialStore，provider 选择走配置，URL/domain 安全走 WebPolicy。

- Search query 可以 ask/allow；provider 凭据不暴露给模型。
- Fetch URL 必须做 SSRF、DNS、redirect、private IP 校验。
- Browser render、Crawl4AI、Firecrawl、Playwright 都是 backend adapter，不改变统一 policy。

### 7.4 连接与 HTTP API

`http_request` 是高风险工具，推荐只允许通过 `connection_ref` 调用业务系统。

- 直接 URL 默认 ask。
- 带 `connection_ref` 时先校验调用主体是否能使用该 connection。
- `connection.allowed_tools`、`allowed_agents` 只是连接元数据；最终授权由 policy evaluator 结合 subject 决定。

### 7.5 MCP 与 Skill

- MCP server/tool 应作为 `mcp://server/tool` 资源参与策略。
- Skill 加载本身可低风险 allow，但声明外部依赖、MCP、脚本、网络能力时必须 ask 或由管理员预授权。
- Skill 不应携带项目固定路径、Genesis 专属分层边界或真实密钥；这些应放配置和 AGENTS.md。

## 八、配置与代码目录建议

新增能力域：

```text
internal/capabilities/policy/
  contract/
    evaluator.go        # PolicyEvaluator
    model.go            # Subject/Action/Resource/Environment/Decision
    matcher.go          # ResourceMatcher
  service/
    evaluator.go        # 合并 rules、grants、product defaults
    grants.go           # GrantStore 编排
  adapter/
    config/             # 从 platform/config 映射 policy 配置
    memory/             # session grant store
    file/               # CLI/Desktop project grant store
    db/                 # Enterprise grant/rule store
  matcher/
    filesystem/
    command/
    web/
    connection/
    credential/
    mcp/
```

`approval` 与 `policy` 的关系：

```text
Tool -> BuildApprovalRequest -> ApprovalService
ApprovalService -> PolicyEvaluator -> allow/ask/deny
ask -> Product Requester -> Decision
Decision -> GrantStore -> Audit/Trace
```

`approval` 负责人机交互编排；`policy` 负责规则判断。现有 `approval/adapter/static` 可以逐步迁移为 `policy/adapter/config + matcher`。

## 九、配置文件演进计划

### Phase 1：文档与最小配置结构

- 新增 `policy` 顶层配置结构体，但先不要求完整实现全部 matcher。
- 将现有 static approval 策略映射为默认 policy。
- `config.yaml` 增加安全默认样例，保持不含密钥。
- CLI 继续支持 `--sandbox`、`--quiet`，但内部映射到 policy/default posture。

### Phase 2：文件系统、命令、网络策略落地

- 实现 filesystem matcher 和 allow_paths/deny_paths。
- 实现 command matcher。
- 实现 web domain/url matcher。
- GrantStore 支持 session/project。
- Desktop 加权限中心，CLI 加配置命令。

### Phase 3：Enterprise 治理

- DB PolicyStore / GrantStore。
- RBAC、租户、角色、项目策略。
- 审批流和审计查询。
- Connection/Credential 使用策略统一接入。
- 策略版本、变更审计、回滚。

## 十、推荐安全默认值

- 未分类动作：`ask`。
- critical/protected：`deny`。
- workspace read/list/walk：`allow`。
- workspace write/edit：开发期可 `allow`，Enterprise 默认可按项目策略 `ask`。
- workspace metadata write：`deny`。
- external path：默认 `ask`，delete 默认 `deny`。
- command.exec：默认 `ask`。
- web.fetch/web.search：CLI 可 `ask`，Enterprise 默认需 allowlist。
- credential.resolve：默认 `deny`，只允许 resolver 内部按 purpose 调用。
- browser render / Crawl4AI / Firecrawl / Playwright：默认 `ask` 或禁用。
- sandbox required 时不可静默降级。

## 十一、验证要点

后续实现配置能力时，至少测试：

- 配置优先级：default < config.yaml < project/user config < local < env < credential/connection store < runtime grant < admin policy。
- deny 覆盖 allow。
- Windows 路径大小写、盘符、symlink/junction。
- once/turn/session/project grant 命中与过期。
- quiet 模式下 ask 自动 deny。
- Enterprise 下普通用户不能提升到 tenant/global。
- Credential 明文不进入日志、Trace、工具输出。
- SandboxRequire 降级失败。
