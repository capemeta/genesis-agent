# Genesis Agent 沙箱与执行引擎终极设计方案 (Unified Sandbox & Execution Architecture)

## 1. 架构目标与总体定位

本设计定义 Genesis Agent 运行引擎中的沙箱隔离（`shared/local/sandbox`）与沙箱路由（`internal/capabilities/execution`）体系。

```text
shared/local/sandbox/
├── seatbelt/        # macOS Seatbelt (/usr/bin/sandbox-exec) 宿主进程约束
├── bubblewrap/      # Linux bwrap 宿主进程约束
├── landlock/        # Linux Landlock 备用平台约束
└── windows/         # Windows Restricted Token & Job Object (KillOnJobClose) 进程约束
```

### 核心设计原则
1. **本质区分（不可混淆）**：
   - **本地 OS 沙箱 (`local_platform_sandbox`)**：宿主机 OS 内核级进程约束（Windows Token/Job, macOS Seatbelt, Linux bwrap）。物理绝对路径直读直写，零复制、零网络传输开销，毫秒级响应。
   - **远程容器沙箱 (`remote_sandbox`)**：独立 Docker 容器服务（`genesis-sandbox`）。通过**沙箱 Worker 子 Agent** 隔离调度，两次狭窄穿越（输入资源注入、输出成品/Patch 对账），海量中间日志隔离在容器内部。
2. **绝对路径原则**：Agent 引擎与本地工具 100% 使用规范物理绝对路径（`filepath.Abs`），禁止任何模棱两可的模糊相对路径转化。
3. **四层链式策略评估器 (`4-Tier Policy Pipeline`)**：
   - **Layer 1 (系统顶层硬门控)**：全局 `sandbox.remote.enabled` 拥有最高一票否决权，关闭时严禁启动远程沙箱。
   - **Layer 2 (Skill / Task 级配置)**：支持静态声明与集中重写（`execution_mode`, `preferred_backend`, `allow_degradation`, `trust_level`）。
   - **Layer 3 (通用降级法则)**：远程不可用时，若 `allow_degradation = true` 通用优雅降级为本地 OS 沙箱，带 `degraded: true` Warning 审计；若 `allow_degradation = false` 则 Fail Closed 阻断。
   - **Layer 4 (选定 Backend 执行)**：调度选定沙箱后端（Local OS Sandbox 或 远程 Sub-Agent 容器）。
4. **复合脚本 AST 防御 (`ScriptASTAnalyzer`)**：
   - 对 Shell/Python/Node/PowerShell 脚本做词法与 AST 解析，只要发现一条敏感指令（`pip install`, `curl`, `eval` 等），整个复合脚本**一票强切远程容器沙箱**。
5. **Skill 配置与覆盖优先级 (Override Hierarchy)**：
   - `外部配置文件覆盖 (skills_override) > SKILL.md Frontmatter 声明 > 系统缺省配置`。
6. **三端独立与内核统一 (CLI / Desktop / Enterprise Balance)**：
   - 沙箱与评估引擎在能力层强统一，CLI、Desktop、Enterprise 三端产品保留独立的 Prompts、专属 Tools 和审批交互。

## 2. 沙箱后端分类与执行模型

| 维度 | 本地 OS 沙箱 (`local_platform_sandbox`) | 远程容器沙箱 (`remote_sandbox`) |
| --- | --- | --- |
| **本质归属** | **宿主机 OS 内核级进程约束 (Native OS Process Confinement)** | **独立的远程 Docker 容器 / 云端 Sandbox VM** |
| **底层实现** | macOS Seatbelt, Linux bwrap/Landlock, Windows Token + Job Object | 独立容器服务 (`genesis-sandbox` API, Session + WorkspaceFS) |
| **文件系统与路径** | **绝对路径直读直写 (Zero Copy)**，直接运行在宿主真实磁盘 | 容器内部隔离文件系统 `/workspace`，通过网络传输与 Diff 对账 |
| **执行开销** | **毫秒级启动，零文件传输开销** | 涉及容器启动、环境装配与增量 Sync 开销 |
| **适用场景** | 本地代码构建与调试 (`go test`, `npm run build`, `git status`) | 不可信第三方代码、高危 Python/Bash 脚本、多步骤生成式 Skill (PPTX) |

## 3. 全局配置文件 `configs/config.yaml` 架构

消灭旧的单模式互斥 `mode:` 节点，采用 `local` 与 `remote` 并行解耦架构：

```yaml
sandbox:
  # 1. 本地 OS 平台沙箱配置 (Process Confinement)
  local:
    enabled: true
    preference: auto         # auto (默认,优先使用可降级) | required (强隔离) | disabled
    default_level: process_constrained

  # 2. 远程容器沙箱配置 (genesis-sandbox API / Docker Service)
  remote:
    enabled: false            # Layer 1 硬门控开关：未开启时禁止任何远程调度
    base_url: ${GENESIS_SANDBOX_BASE_URL}
    api_key: ${GENESIS_SANDBOX_API_KEY}
    api_key_env: GENESIS_SANDBOX_API_KEY
    workspace_id: ""
    default_runtime_profile: code-polyglot-basic

  # 3. 全局路由与调度配置
  routing:
    default_execution: optional       # disabled | optional | required
    allow_session_override: true      # 允许每次对话或 HTTP run 请求覆盖 sandbox 策略
    auto_route_risk: true             # 允许 CommandRiskEvaluator 自动按风险切分远程沙箱

  # 4. 外置技能沙箱集中覆盖
  skills_override:
    office-ppt:
      sandbox:
        preferred_backend: remote_sandbox
        allow_degradation: true
```

## 4. 四层链式策略评估器与通用降级法则 (`4-Tier Policy Pipeline`)

```text
[Skill / Command 触发]
         │
         ▼
【Layer 1: 系统顶层硬策略 (System Hard Guardrails)】
 • 若系统全局关闭远程沙箱 (sandbox.remote.enabled = false)
   即使 Skill 声明要 remote_sandbox，也【绝不允许】擅自启动远程沙箱！
         │
         ▼
【Layer 2: Skill / Task 级配置策略 (Skill Execution Spec)】
 • 解析 Skill Frontmatter 与 skills_override：
   - execution_mode: per_call | sandboxed_session
   - preferred_backend: local_platform_sandbox | remote_sandbox
   - allow_degradation: true | false
   - trust_level: verified | untrusted
         │
         ▼
【Layer 3: 通用降级法则 (General Degradation Rule)】 ──► 全局通用法则
 • 若首选后端 (如 remote_sandbox) 不可用 / 未就绪 / 被 Layer 1 阻断：
   - 若 allow_degradation == true  ──► 自动优雅降级为【本地 OS 沙箱】，带 degraded: true 与 Warning 上报！
   - 若 allow_degradation == false ──► 【Fail-Closed 阻断报错】，坚决拒绝降级运行！
         │
         ▼
【Layer 4: 选定 Backend 调度执行】
 • 调度选定的 Backend (Local OS Sandbox 或 派遣 Remote Sub-Agent 容器) 执行。
```

## 5. 复合脚本 AST 词法语法安全扫描 (`ScriptASTAnalyzer`)

针对包含 Shell 脚本或 Python/Node/PowerShell 脚本文件/字符串的多步骤执行场景，防范隐蔽命令注入与包装攻击：

1. **深层 AST 词法与语法分析**：
   - 集成 `ScriptASTAnalyzer`，解析包含在 `sh -c`, `bash -c`, `python -c`, `eval` 以及多行脚本体内部的全量指令树。
2. **全脚本“一票强切远程沙箱”规则**：
   - **纯安全脚本**：若脚本内部 100% 为安全构建/查询命令（如 `echo`, `cat`, `go test`, `git diff`），判定为 `RiskLevelLocalSafe` ──► 在 **本地 OS 沙箱 (`local_platform_sandbox`)** 极速运行。
   - **复合/敏感脚本**：只要脚本中**存在任意一条敏感命令**（如 `pip install`, `npm install`, `curl`, `wget`, `nc`, `eval`）：
     - **全脚本强切**：整个复合脚本被判定为 `RiskLevelUntrustedRemote`！
     - **强制隔离**：**整个脚本只能在【远程容器沙箱】中执行**，绝不允许在本地 OS 沙箱中暴露任何漏洞隐患！

## 6. Skill 配置元数据与解析优先级 (`Skill Config Schema & Hierarchy`)

### 1. SKILL.md 内置 Frontmatter 声明 (作者视角)
```yaml
---
name: office-ppt
description: 用于自动生成与编辑 PPTX 演示文稿
sandbox:
  execution_mode: sandboxed_session       # per_call (默认) | sandboxed_session
  preferred_backend: local_platform_sandbox # local_platform_sandbox | remote_sandbox
  allow_degradation: true                 # true (默认, 允许优雅降级) | false (不可用时 Fail Closed)
  trust_level: verified                   # verified (自研/已审核) | untrusted (社区/第三方)
inputs:
  - declared_by_caller
outputs:
  - "*.pptx"
---
```

### 2. 解析覆盖优先级
1. **最高优先**：外部配置文件 / CLI 参数 (`skills_override[skill_name].sandbox`)
2. **次高优先**：`SKILL.md` 顶层 Frontmatter 声明中的 `sandbox` 字段
3. **缺省兜底**：系统默认配置 (`execution_mode: per_call`, `preferred_backend: local_platform_sandbox`, `allow_degradation: true`, `trust_level: untrusted`)

## 7. 沙箱 Worker 子 Agent 调用与返回数据契约 (`SubAgent Delegation Contract`)

针对 `execution_mode: sandboxed_session` 的生成式 Skill（如 PPTX）：

### 1. 调用请求契约 (`SubAgentDelegationRequest`)
```go
type SubAgentDelegationRequest struct {
    TaskID          string            `json:"task_id"`          // 唯一任务 ID
    Goal            string            `json:"goal"`             // 任务目标
    SkillName       string            `json:"skill_name"`       // 绑定的 Skill 名称
    InputManifest   []ResourceBinding `json:"input_manifest"`   // 边界穿越 1: 输入文件快照
    SessionID       string            `json:"session_id"`       // 绑定的远程沙箱 Session ID
    MaxTurns        int               `json:"max_turns"`        // 子 Agent 最大交互轮数
}
```

### 2. 返回结果契约 (`SubAgentDelegationResult`)
```go
type SubAgentDelegationResult struct {
    TaskID          string            `json:"task_id"`          // 对应任务 ID
    Status          string            `json:"status"`           // success | failed | timeout
    Summary         string            `json:"summary"`          // 给主 Agent 的精简汇报 (Prompt Token 低)
    ProducedOutputs []ResourceBinding `json:"produced_outputs"` // 边界穿越 2: 交付物成品列表
    WorkspacePatch  *WorkspacePatch   `json:"workspace_patch"`  // 增量 Git Diff 补丁
    Warnings        []string          `json:"warnings"`         // 运行审计 Warning 说明
}
```

### 3. 实时控制面与日志透传 (`SubAgentProjection`)
子 Agent 内部运行的海量 Shell 日志存放在内部 log，底层通过现有的 `SubAgentProjectionHandler` 和 `ProgressEvent` SSE 向前端渲染进度，保证主 Agent 上下文绝对干净。

## 8. 安全防护与运行时管辖

1. **环境变量净化 (`EnvSanitizer`)**：
   - 过滤剥离敏感凭证 Key（`SECRET`, `PASSWORD`, `PRIVATE_KEY`, `TOKEN`, `CREDENTIAL`）。
   - 保留与补全代理变量（`HTTP_PROXY`, `HTTPS_PROXY`, `NO_PROXY`）。
2. **孤儿进程树销毁**：
   - Windows 绑定 `windows.CreateJobObject` 与 `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`。
   - Linux bwrap 传入 `--die-with-parent`。
3. **工作区并发锁 (`Workspace Concurrency Locker`)**：
   - 基于 `ResourceLocker` 对目标工作区绝对路径申请排他写锁，防范并发写冲突。
