# 统一执行工作空间、文件权限与产物规范

> 状态：目标规范（后续实现整改基线）  
> 适用范围：CLI、Desktop、Enterprise；Agent App、Run、子智能体与工作流中的文件工具、命令执行、Skill、MCP、Sandbox、Artifact 交付  
> 设计依据：本文从目标、不变量、安全边界和产品价值推导目标契约，不继承旧实现或旧设计的偶然结构。旧文档仅作为迁移清单和问题证据，不构成本文语义的约束；仓库级稳定硬约束和已确认的产品边界仍必须遵守

## 1. 目标与边界

本文解决以下问题：

- 输入文件如何经过授权、快照和 staging 进入执行环境；
- 项目文件、中间过程文件、临时文件和最终产物分别放在哪里；
- 宿主直跑、本地平台沙箱、远程 genesis-sandbox API 如何保持相同语义；
- 文件工具、普通命令、Skill 脚本和 MCP 如何复用统一的权限与产物能力；
- Agent App、Task Template、Run、子智能体、Workflow 和协作空间如何绑定、继承或隔离执行工作空间；
- CLI、Desktop、Enterprise 哪些能力必须共享，哪些策略由产品独立注入。

本文不规定具体 UI，不把宿主绝对路径作为跨产品业务协议，也不要求第三方 Skill 或 MCP 修改自身协议才能被安装。适配必须由 Genesis 运行时完成。

本文也不把所有业务结果强行文件化：客服回复、告警确认、工单状态、数据库事务和业务办理回执可以是结构化 Tool/MCP/领域结果，其事务、幂等和补偿由对应能力域负责。只有需要持久交付、下载、预览、跨执行引用或留存证据的文件/对象才进入 Artifact 体系；外部系统资源使用 ResourceRef 或专用领域引用。无文件任务仍遵守 binding、权限、审计和 sandbox 规则，但无需为了统一形式制造空目录或伪 Artifact。

工作空间是 **Run 的执行资源边界**，不是业务场景分类，也不是 Agent App 的身份。`code`、`doc-review`、`ops-diagnosis` 等 Agent App 可以根据任务意图使用不同工作空间模式；同一个 App 的不同 Run 也可以不同。Agent App 只能声明默认值、允许范围和最低隔离要求，最终工作空间必须在 Run 创建时由可信运行时解析并固化。

本文区分三个容易混淆的概念：`Skill` 是加载知识包的固定网关 Tool；Skill Runtime Harness 是产品无关的运行期编排服务；`run_command`、`write_file`、`publish_artifact` 等是受 Tool Gateway 治理的通用原语。Skill 名永远不能成为 Tool 名，Harness 也不能绕开 Tool Gateway 直接获得执行权限。

## 2. 第一性原则与不变量

### 2.1 核心不变量

1. **路径语义与物理环境解耦**：业务层只使用 WorkspacePath、ResourceRef、InputRef、ArtifactRef；物理路径由 backend 映射。
2. **先区分执行资源形态**：已有项目原位修改、隔离型单次任务、会话式连续任务使用不同路径纪律；它们不等于编程、审查、客服等业务类型，也不能由一个含糊的 `run_command` 自行选择。
3. **输入必须受控**：外部文件进入任务执行环境前必须经过 PathResolver、权限审批、大小限制、hash 和 staging。
4. **中间状态与交付物分离**：`work` 只保存可继续迭代的状态；最终文件必须由调用契约声明具体路径或受限匹配规则，或显式发布为 Artifact，仅写入某个目录不构成交付声明。
5. **产物不靠猜测**：目录 diff 只能提供候选文件提示，不能自动把所有新增文件认定为最终产物。
6. **三个执行环境语义等价**：同一任务切换宿主、本地平台沙箱、远程 sandbox API 后，输入名、相对工作路径和 ArtifactRef 语义不变。
7. **权限在每次边界转换时重新判断**：本地路径进入 Run、Run 文件进入 sandbox、Artifact 导出到项目目录都必须经过对应权限边界。
8. **产品不复制内核规则**：Workspace、staging、artifact、路径校验和执行语义属于统一内核；产品只注入 backend、策略和存储。
9. **Enterprise 不依赖宿主文件系统**：企业服务不得把进程 cwd、宿主盘符或本地 `.genesis` 当成租户业务存储。
10. **可审计、可重放、可清理**：输入和产物有 manifest；工作目录有生命周期、配额和清理策略。
11. **Skill 原包保持通用**：第三方 Skill 的 `SKILL.md`、references、scripts、assets 不写入 Genesis 专用路径、工具名或权限规则；平台强化全部由通用 Skill Runtime Harness 完成。
12. **App 与工作空间正交**：Agent App、App Type、产品端和模型参数都不能直接授予文件访问；有效工作空间是 App 需求、请求意图、产品/租户策略、资源授权和 backend 能力共同解析出的只读绑定。
13. **多执行主体默认隔离**：父 Run、子 Run、Workflow step 和不同 Agent App 成员不得隐式共享可写 cwd；共享必须通过明确的只读视图、受锁共享资源或 ResourceRef/ArtifactRef 交接。
14. **执行位置与交付位置分离**：WorkspaceMode 决定任务在哪里、按什么纪律执行；DeliveryTarget 决定用户最终在哪里看到结果。内部 Run/output 和 ArtifactStore 不是要求用户浏览的交付目录。

### 2.2 失败条件

出现以下任一情况即视为契约失败：

- 模型看到的 `path_map` 与进程实际环境变量不一致；
- 远程命令执行时仍把宿主绝对路径作为业务输入；
- 文件工具读取本地项目，但命令在远程 `/workspace` 中执行且没有 staging；
- Skill 或 MCP 的缓存、缩略图、日志被自动当成最终交付物；
- 同名输入或产物被静默覆盖；
- MCP stdio 绕过所选 sandbox 和文件权限策略；
- `SandboxRequire` 不可用时静默降级到宿主；
- Enterprise 将租户文件写到服务实例本地目录并作为正式交付；
- Run 被清理后，已交付 Artifact 随之丢失。
- 同一 Skill 的两个并发会话共享可写 cwd、临时文件或输出命名空间。
- 权限审批绑定的是可变路径字符串，而不是已解析的资源身份与版本/hash。
- 根据 `AgentAppType=code|business|workflow` 硬编码工作空间模式或文件权限。
- 用 `local_*`、`sandbox_*` 等 backend 前缀定义 WorkspaceMode，把资源生命周期与执行环境混成一个枚举。
- 切换 Agent App 后沿用上一 App 的可写 cwd、输入挂载、临时凭证或审批结果。
- 父子智能体、并行 Workflow step 或协作空间成员隐式共享可写目录并发生覆盖、越权读取或竞态。

## 3. 三类执行工作空间模式

目录纪律首先由本次 Run 的执行资源形态决定，而不是由“是否启用沙箱”、产品端或 Agent App 类型决定。三种模式是统一 Runtime 的基础模式，不是只为编码设计的场景枚举。

三种模式的**互斥粒度是单个 `ExecutionBinding`**，不是项目、Agent App、Session 乃至整个 Run：

- 一个 binding 必须且只能有一个主模式，避免同一个 cwd 同时遵守两套冲突的输入/输出纪律。
- 一个根 Run 有一个 primary binding；Skill、子 Run、Workflow step 可以在父级允许范围内创建自己的 child binding，并选择不同模式。
- 一个项目不绑定固定模式。同一项目中的代码修改、单次报告生成、长期审查会话可以分别使用三种模式。
- 多个 binding 之间不因属于同一项目或 Run 就自动共享可写目录；资源通过受限视图、InputRef、ArtifactRef 或显式项目写授权交接。

| 模式 | 典型场景 | cwd 语义 | 文件纪律 | 产物方式 | AI 判定提示 |
| --- | --- | --- | --- | --- | --- |
| `project_workspace` | 修改已授权的源码库、文档库、IaC、Notebook、知识库或其他项目资源 | 用户授权的项目资源根；本地可是真实目录，远程是等价 WorkspaceFS 视图 | 原位读写、Freshness、锁和保护路径优先，不强制 `input/output` | 项目资源变更或显式 Artifact | 用户要求在一个既有项目/资源集合上持续读取、修改、验证 |
| `task_job` | 文件转换、批量数据处理、一次性审查、独立业务作业 | 隔离的 job 工作目录 | 输入、临时、输出严格分离 | 从 `OUTPUT_DIR` 回收已声明路径，或显式发布 | 输入边界明确、执行一次、结果可独立交付且无需保留可变工作状态 |
| `session_workspace` | 报告迭代、长周期分析、客服/运维调查、多步业务办理、Skill 或 Agent 连续工作 | 持久的 session 工作目录 | 中间状态跨命令/Turn 保留，外部资源仍显式绑定 | 从 workspace 显式发布 Artifact 或通过受治理 connector 提交业务结果 | 同一会话需要跨多步保留草稿、索引、检查点或恢复状态 |

### 3.1 `project_workspace`

- 适用于任何需要原位操作既有授权项目资源的 Agent App；AI 编程只是最常见示例，不是该模式的定义。
- cwd 是项目资源语义的一部分，不得强制改成 `<state_root>/runs/<run>/work`。远程和 Enterprise 中的“项目”可以是租户 WorkspaceFS、受版本控制的数据集或文档集合，不要求存在宿主目录。
- 工作区内读写经过 PathResolver、权限、资源锁和 FreshnessTracker。
- 工作区外路径必须审批。
- 构建缓存写项目约定缓存或受控 `TMPDIR`，不能冒充最终产物。
- 生成独立报告、压缩包或 Office 文件时，应显式发布 Artifact，或按用户明确授权导出到指定目录。
- 产物语义：`project_workspace` 的默认结果是项目资源本身的受追踪变更；独立报告、压缩包、导出物等应显式发布为 Artifact。项目内新建的文件不因“是代码”或“不是代码”而区别对待，是否发布只由交付契约决定。

### 3.2 `task_job`

- 输入只来自 InputRef；执行进程读取 `INPUT_DIR`。
- `WORK_DIR` 仅用于本 job 的工作状态；如果不需要跨 job，不应长期保留。
- `TMPDIR` 为 call/job 级，执行后清理。
- `OUTPUT_DIR` 只是允许回收的边界；默认仍仅回收 `ExpectedOutputs` 声明的具体相对路径。
- 只有调用方显式选择 `collect_all_declared_output_dir` 策略时，才允许收集该目录内全部合规文件；该策略必须进入权限决策和审计。
- 执行成功但要求产物且没有输出时，返回结构化 `NO_OUTPUT_ARTIFACTS`。

### 3.3 `session_workspace`

- 同一 Run、同一执行主体的多次命令共享工作状态。
- 第三方 Skill 可以继续使用相对 cwd，不强制改写脚本。
- `produced[]` 只用于提示本次新增或修改了哪些文件。
- 最终交付必须通过 `PublishArtifact` 或等价显式动作声明具体相对路径。
- 发布前执行格式门禁、大小限制、hash、MIME 检测和权限检查。

### 3.4 模式解析与 Agent App 的关系

Agent App 的 `RuntimeSpec`（或等价配置）可以声明 `workspace.default_mode`、`allowed_modes`、`requires_project`、持久化需求、只读/读写需求和最低 sandbox 等要求，但这些字段都是**需求或收窄项**，不是授权。`AgentAppType` 只用于分类，禁止据此推断工作空间。

Run 创建时由 `WorkspaceResolver` 根据以下可信输入生成不可变 `ExecutionBinding`：

```text
任务/请求意图 + 已选择的 EffectiveAgentAppProfile
  + Product Profile / Enterprise Policy / RBAC
  + 已授权 ResourceRef / Project / InputRef
  + Workflow / SubAgent 委派约束
  + backend 与 sandbox 实际能力
→ WorkspaceMode + ExecutionBinding（Run 内固化）
```

模式选择分为“候选选择”和“安全裁剪”两步，不能简单理解为某个配置字段最终覆盖一切。

**第一步：选择候选模式**：

1. 用户/API 对本次任务的显式执行意图，例如“直接修改当前项目”或“在隔离副本中处理”；
2. Task Template、Workflow step、Skill 或子智能体契约声明的必要资源形态；
3. `WorkspaceResolver` 根据是否需要原位修改、输入输出是否有界、是否需要跨 Turn 持久化进行判定；
4. 无法从任务判定时，才使用 Agent App、项目或产品配置的默认偏好。

**第二步：安全裁剪与可运行性校验**：候选模式必须同时命中 Agent App `allowed_modes`、Product Profile、Enterprise Policy/RBAC、资源授权、sandbox 和 backend 能力；任何配置层只能收窄，不能借“默认模式”扩权。显式选择不兼容时明确失败或要求用户改变执行意图，禁止静默换成权限更宽的模式。

普通用户通常不需要配置技术模式。CLI/Desktop 应优先询问或展示业务含义，例如“直接修改项目”“隔离处理并导出”“保留连续工作区”；高级用户、Task Template 和 Enterprise 管理员可以配置默认偏好与允许范围。项目级默认只代表该项目中新 Run 的 fallback，绝不把项目内所有任务锁死为一种模式。

解析纪律：

- 已存在且需要原位修改的授权项目资源选择 `project_workspace`；是否为代码无关紧要。
- 输入和交付边界明确、无需跨执行保留状态的请求选择 `task_job`。
- 需要跨 Turn、暂停/恢复或多步迭代状态的请求选择 `session_workspace`。
- `code` App 可把 `project_workspace` 作为默认偏好，但显式的独立代码生成/分析 job 仍可使用 `task_job`；文档审查 App 既可对文档库使用 `project_workspace`，也可对单份上传文件使用 `task_job` 或在迭代审查时使用 `session_workspace`。
- 客服、RAG、监控或业务办理即使不使用文件，也仍获得逻辑上的 job/session binding；backend 可惰性创建物理目录，不能退化为“使用进程 cwd”。
- 模型只能请求允许范围内的模式；模式选择、切换和资源挂载由控制面完成。Run 期间如确需改变模式，必须创建新的 binding 并重新做权限决策，不能就地改写 cwd。

同一项目示例：

| 任务 | primary binding | 项目资源如何进入 | 结果如何返回 |
| --- | --- | --- | --- |
| 修复代码并运行测试 | `project_workspace` | 项目作为受锁、可写资源根 | 原位项目变更；独立报告可发布 Artifact |
| 读取项目数据生成一次性 PDF | `task_job` | 选定文件以只读 InputRef 快照进入 job | PDF 作为 Artifact；不污染项目目录 |
| 连续数日审查文档并保留草稿 | `session_workspace` | 项目文档以只读或明确授权的受限视图挂载 | 草稿留在 session；最终报告发布 Artifact；需要回写项目时单独审批 |
| 主 Agent 修改项目，同时委派子智能体生成报告 | 根 binding 为 `project_workspace`，子 binding 可为 `task_job` | 子智能体只获得声明的项目文件快照或受限只读视图 | 子 ArtifactRef 回传父 Run，不共享父级可写 cwd |

### 3.5 交付目标独立解析

WorkspaceMode 与最终交付位置是两个正交决策。`task_job` 可以在隔离目录执行，然后把通过 gate 的 Artifact 原子导出到源文件平级目录、项目内指定路径、用户下载目录或仅保留在 ArtifactStore；不能因为最终写回项目或源目录，就要求生成过程直接在该目录运行。

```go
type DeliveryTargetKind string

const (
    DeliveryArtifactOnly DeliveryTargetKind = "artifact_only"
    DeliverySourceSibling DeliveryTargetKind = "source_sibling"
    DeliveryProjectRoot   DeliveryTargetKind = "project_root"
    DeliveryProjectPath   DeliveryTargetKind = "project_path"
    DeliveryExplicitPath  DeliveryTargetKind = "explicit_path"
    DeliveryProductInbox  DeliveryTargetKind = "product_inbox"
)

type DeliveryTarget struct {
    Kind     DeliveryTargetKind
    Resource ResourceRef // 目标目录或文件；由控制面解析，不能由模型伪造物理路径
    Name     string
}
```

本地 CLI/Desktop 的推荐解析顺序：

1. 用户明确给出的输出目录或完整输出路径；
2. 存在明确的主目标文档/源文档时，默认使用该文档的平级目录 `source_sibling`，并生成语义清晰且不冲突的文件名；
3. 没有明确目标文档、存在多个不同目录的输入，或者目标平级目录不可写/越界/无法解析时，一律回退到当前已授权的 `project_root`；
4. Task Template / Agent App 可以指定语义化文件名或在上述范围内进一步收窄，但不能把默认交付改到用户不可见的内部目录；
5. 只有确实不存在项目上下文时，才使用用户配置的产品级可见输出根目录（CLI output root / Desktop Inbox）或返回可点击 Artifact；禁止把进程 cwd 或 `.genesis/runs` 冒充项目根目录。

`source_sibling` 和 `project_root` 是可解释的产品默认，不是隐式扩权。系统必须在执行前解析目标、检查写权限和命名冲突；产品策略要求审批时只询问一次业务化问题，例如“将在原文件旁生成 `Ultra5产品对比总结.pptx`，是否允许？”。平级目标不可用时自动尝试项目根目录；项目根目录也不可用时，不得把 `.genesis/runs/...` 暴露为替代交付位置，而应返回产品级可见输出位置/Artifact 入口并说明原因。

这里的 `project_root` 指当前 `ProjectRef`/授权项目资源的业务根目录，不是 StateRoot，不是 `<project>/.genesis`，也不是进程启动 cwd。Enterprise 中它对应租户项目 WorkspaceFS 的逻辑根；没有 ProjectRef 时必须诚实进入“无项目”分支，不能虚构项目根目录。

典型结果：

| 请求 | 执行模式 | 默认 DeliveryTarget |
| --- | --- | --- |
| 总结 `E:\资料\3-人资部.pdf` 并新建 Markdown | `task_job` | `E:\资料\3-人资部-年度总结.md`（源目录可写且策略允许时） |
| 根据当前目录的 `ultra5-comparison-summary.md` 生成 PPT | `task_job` | 与源文件平级的 `ultra5-comparison-summary.pptx` 或语义化名称 |
| 没有指定目录、也没有明确目标文档的生成任务 | `task_job` 或当前 Run 模式 | 当前已授权项目根目录 |
| 目标文档平级目录不可写或越界 | 不改变执行模式 | 自动回退到当前已授权项目根目录 |
| 修改项目源码 | `project_workspace` | 项目变更本身；额外报告按显式目标或 Artifact 交付 |
| Enterprise API 批处理 | `task_job` | 租户 ArtifactStore / 业务对象存储引用，不存在宿主平级目录 |

## 4. 统一资源与路径模型

### 4.1 禁止裸路径跨边界传播

跨能力域使用以下抽象：

```go
type ResourceScope struct {
    TenantID  string
    ProjectID string
    UserID    string
}

type ResourceRef struct {
    Authority string // host, executor, orchestrator, tenant
    Scheme    string // project, run, workspace, artifact, mcp, object
    ID        string
    Path      string // scheme 内相对路径
    Version   string // SHA-256、etag、generation 或其他不可变版本
    MediaType string // 可选 MIME type；跨系统传递时消费方据此决定处理方式
    Scope     ResourceScope
}

type InputRef struct {
    ID         string
    Name       string
    Size       int64
    SHA256     string
    MIME       string
    Source     ResourceRef
    StagedPath WorkspacePath
}

type ArtifactRef struct {
    ID         string
    Name       string
    Kind       string
    Size       int64
    SHA256     string
    MIME       string
    Producer   string
    RunID      string
    Scope      ResourceScope
    StorageRef ResourceRef
}

type ExecutionOwnerRef struct {
    AgentAppID          string
    AgentAppVersion     string
    TaskID              string
    RunID               string
    ParentRunID         string
    SubAgentInstanceID  string
    WorkflowStepID      string
    CollaborationSpaceID string
    MemberID            string
}

type WorkspaceMode string

const (
    WorkspaceModeProject WorkspaceMode = "project_workspace"
    WorkspaceModeTask    WorkspaceMode = "task_job"
    WorkspaceModeSession WorkspaceMode = "session_workspace"
)

type ExecutionBinding struct {
    ID      string
    Mode    WorkspaceMode
    Owner   ExecutionOwnerRef
    // Roots、访问模式、backend、sandbox、配额和生命周期均由可信运行时填充；
    // LLM/Skill/MCP 只能消费该绑定，不能自行构造或修改。
}
```

`ResourceScope` 表达资源的租户/项目/用户授权域；`ExecutionOwnerRef` 表达本次执行由哪个 App 版本、Run、子智能体、Workflow step 或协作成员产生。二者不可互相替代：同租户不等于可在不同 App/成员之间共享文件，同一 App 也不等于可以跨项目访问。完整 owner/binding 快照应进入 Run、Input 和 Artifact manifest；对外引用只保留稳定 ID，避免把全部内部拓扑塞进路径。

Authority 枚举由实现者按产品扩展，但必须满足：不同 Authority 下的同一 Path 是不同资源；跨 Authority 使用必须经过 staging 或显式引用。本地平台沙箱中的执行器属于 `host` authority（因为仍在宿主机命名空间内）；MCP remote HTTP 返回的资源属于 `executor` 或服务自声明的 authority。

推荐的模型可见形式：

```text
project:/src/main.go
run:/input/input-123/report.pdf
run:/work/binding-123/skills/office-ppt/output.pptx
artifact:artifact-456
mcp:server/resource-id
```

禁止把以下内容作为跨 backend 业务路径：

```text
D:\workspace\...
/Users/alice/...
/home/alice/...
.genesis/runs/... 的宿主绝对展开结果
```

### 4.2 逻辑目录

| 逻辑目录 | 语义 | 生命周期 | 是否可交付 |
| --- | --- | --- | --- |
| `PROJECT_DIR` | 已授权项目工作区 | 项目级 | 仅显式发布或用户要求的项目修改 |
| `WORK_DIR` | 当前 execution/session 工作状态 | execution/session 级 | 否，必须显式发布 |
| `INPUT_DIR` | 当前 job 的只读输入视图 | job 级 | 否 |
| `OUTPUT_DIR` | 当前 job 允许回收的内部输出边界，不是用户寻找结果的目录 | job 级 | 仅调用契约声明后可发布，并按 DeliveryTarget 交付 |
| `TMPDIR` | 当前 call/job 临时文件 | call/job 级 | 否 |
| `SKILL_DIR` | Skill 包只读视图或受保护目录 | Skill 版本级 | 否 |

`path_map` 必须来自实际创建的 ExecutionWorkspace，不能根据 backend 名称静态拼接；返回给模型的映射必须与进程实际 env 完全一致。

## 5. 本地目录布局

本地实现的逻辑布局如下，`state_root` 必须由产品显式解析，不能把进程 cwd 隐式当根目录：

```text
<state_root>/
├── runs/
│   └── <run_id>/
│       ├── manifest.json
│       ├── input/
│       │   └── <input_id>/
│       │       └── <original_name>
│       ├── work/
│       │   └── <execution_binding_id>/
│       │       ├── skills/
│       │       │   └── <skill_execution_id>/
│       │       └── state/
│       ├── tmp/
│       │   └── <call_id>/
│       └── output/
│           └── <producer_id>/
│               └── <artifact_name>
└── artifacts/
    └── <artifact_id>/
        ├── manifest.json
        └── <artifact_name>
```

### 5.1 StateRoot 选择

`StateRootResolver` 根据产品、工作负载、项目身份和用户配置在 Run 创建时一次性决定根目录，并把选择结果记录在 Run manifest；Run 运行期间不得随 cwd 或 UI 当前目录变化。

| 场景 | 默认 state root | 说明 |
| --- | --- | --- |
| CLI `project_workspace` | 授权项目的 `<project>/.genesis`，或显式用户级 state root | 项目选择必须稳定；目录应加入忽略规则且使用 owner-only 权限 |
| CLI `task_job/session_workspace` | `$GENESIS_HOME/state` 或平台用户数据目录 | 没有明确项目时不得偶然写入当前目录 |
| Desktop | OS 应用数据目录下的用户级 Genesis state | 默认不污染用户打开的项目；可显式选择项目共享会话 |
| Enterprise | 租户 WorkspaceFS、对象存储和数据库形成的逻辑 state root | 不对应服务实例本地路径 |

本地 UI 可以向用户展示解析后的 Artifact 文件位置，但模型、跨 backend 协议和审计业务字段只使用 ResourceRef/ArtifactRef。`GENESIS_HOME`、项目 `.genesis` 和 Desktop 应用数据目录是不同物理策略下的实现，不得在能力内核硬编码。

规则：

- `runs/<run>/input` 是可审计输入快照，不直接复用用户原文件作为任务输入。
- `runs/<run>/work/<execution_binding_id>` 可以在 Run 内持久，但不是正式交付目录。根执行、子 Run、并行 Workflow step 默认使用不同 binding 目录；禁止用 Agent 名或可重名 step 名直接充当物理隔离键。
- `skill_execution_id` 是一次绑定后的 Skill 执行标识（不是顶层 Agent Session，也不是 `package_id`）；同一包的并发执行不得共享可写目录。
- `runs/<run>/tmp/<call>` 执行完成后可删除。
- `runs/<run>/output` 是交付暂存区。
- `<state_root>/artifacts/<artifact_id>` 是本地持久 ArtifactStore；Run 清理不影响已发布 Artifact。
- Enterprise 使用对象存储或受控 Artifact Store，不使用服务实例本地 `.genesis/artifacts`。

## 6. 三种执行环境映射

| 语义 | 宿主直跑 | 本地平台沙箱 | 远程 genesis-sandbox API |
| --- | --- | --- | --- |
| 项目工作区 | 宿主授权目录 | 同一宿主目录，由 OS sandbox 限制 | 远程 workspace 或显式同步后的项目视图 |
| execution/session work | `<state_root>/runs/<run>/work/<binding>/...` | 同左 | `/workspace/work/<binding>/...` |
| job input | `<state_root>/runs/<run>/input/...` 的只读视图 | 同左 | `/workspace/input/...` |
| job output | `<state_root>/runs/<run>/output/...` | 同左 | `/workspace/output/...` |
| call tmp | `<state_root>/runs/<run>/tmp/<call>` | 同左 | `/workspace/tmp/...` |
| ArtifactStore | `<state_root>/artifacts/...` | 同左 | 下载后进入本地 ArtifactStore；Enterprise 进入对象存储 |

### 6.1 宿主直跑

- 文件工具使用 Local FileSystemBackend。
- 命令执行使用真实 cwd。
- 外部路径必须由 PathResolver 标记 scope 并审批。
- 禁止 Tool 内直接调用 `os`/`filepath` 绕过 port。

### 6.2 本地平台沙箱

- 文件系统仍是宿主命名空间；只是被执行进程受到 Seatbelt、bubblewrap、Landlock、Windows Token/ACL 等限制。
- Agent 运行时负责 staging、Artifact 发布和 manifest，命令进程只能访问策略允许的 roots。
- `SandboxRequire` 无法满足所需文件/网络隔离时必须失败。
- optional 降级必须返回 warning、trace 和 audit。

### 6.3 远程 sandbox API

- 宿主文件必须先变成 InputRef，再上传为 input artifact 或写入 session WorkspaceFS。
- command 中只使用 sandbox-relative cwd 和进程环境变量。
- 客户端不得猜测容器路径，也不得扫描容器根目录兜底。
- 无状态 job 的回收边界仅限 `/workspace/output`，且默认只读取调用契约声明的相对路径。
- 会话式 workspace 通过 session scoped WorkspaceFS 显式读取声明的发布文件。
- 下载后的产物重新执行本地/服务端 Artifact gate，再写 ArtifactStore。

上传与下载均须端到端校验 size/hash；远端返回的路径、MIME 和 hash 只能作为声明，不能替代控制面的重新验证。Sandbox API credential、签名 URL 和租户 token 只能由产品 bootstrap 通过最小作用域 secret channel 注入，不能写入 workspace、模型上下文或普通 trace。

## 7. 权限与安全模型

### 7.1 权限层次

```text
产品身份与 Scope
→ Tool Gateway 权限
→ PathResolver / ResourceResolver
→ Workspace 与 Artifact 操作权限
→ Execution sandbox
→ Freshness / hash / version 校验
→ Audit / Trace / Usage
```

任何一层允许都不能替代下一层检查。

有效权限按最小交集计算：

```text
EffectiveCapability
= ProductProfile
∩ Identity/RBAC Grant
∩ EffectiveAgentAppProfile
∩ Project/Run Scope
∩ Workflow/SubAgent/Collaboration Member Policy
∩ Skill allowed-tools（非空时）
∩ Tool Policy
∩ Sandbox Effective Capability
∩ Resource-specific Approval
```

Agent App、Task Template、SubAgent、Workflow step、Skill 声明、MCP 元数据、RuntimeProfile 和模型参数都只能提出需求或进一步收窄能力，不能扩大上层授权。审批必须绑定操作、资源版本、目标 namespace、ExecutionBinding/调用主体、有效期；不能把一次“允许读取”复用为写入、导出、另一个 App/成员使用或后续版本读取权限。

### 7.2 文件权限

- 工作区内读取：按产品策略允许或审批。
- 工作区内写入：需要文件锁和 Freshness 检查。
- 工作区外读取：逐资源审批后才允许转为 InputRef。
- 工作区外写入：只能通过显式 Export/Publish 操作，不能由 sandbox adapter 直接覆盖。
- `.git`、凭证目录、配置密钥、系统目录属于 protected paths，默认禁止写入。
- symlink、junction、reparse point 必须解析真实边界，禁止借链接越界。
- 本地 state root、Run snapshot、ArtifactStore 和临时目录默认使用当前用户专属权限；Enterprise 数据使用租户隔离和静态加密，密钥由产品 KMS/secret port 注入。任何 backend 都不得把 secret 写入输入快照或 Artifact manifest。

### 7.3 输入 staging 安全

每次 staging 必须：

1. 解析真实来源；
2. 检查权限和 scope；
3. 流式读取，应用单文件和总量限制；
4. 计算 SHA-256；
5. 分配 Input ID；
6. 检查 basename 重名和保留路径冲突（两个同名输入必须自动重命名并在 staged manifest 中保留 source → staged name 映射，模型只能使用映射后的逻辑名称——详见 §10.4）；
7. 写入只读输入快照；
8. 记录 source → staged name 映射；
9. 向执行环境 stage/bind；
10. 审计来源、hash、大小、目标 execution。

权限决策必须绑定规范化后的 `ResourceRef(authority, scheme, id, version, scope)`，不能只绑定用户提交的路径字符串。实现应使用 no-follow/受控句柄读取，或在复制前后验证 identity、size、mtime/version 与 hash；资源在审批后发生变化时返回冲突并重新审批，不能把新内容沿用旧授权。Windows reparse point、Unix symlink 和远程对象 generation 均按此规则处理。

输入只读必须由 backend 强制执行：优先只读挂载或只读 WorkspaceFS ACL；不具备强制能力时使用独立副本，并在执行后复核 hash。仅设置进程环境变量或约定“不要修改”不算隔离。

禁止直接以 `filepath.Base()` 覆盖到 Skill cwd。

### 7.4 并发与一致性

- InputRef 内容不可变，相同 ID 的 hash 必须稳定。
- Work 文件写入使用 workspace/file 资源锁。
- 同一 session 的可变命令默认通过 workspace lease 串行执行；只有声明为只读且 backend 提供一致快照时才允许并发。取消、超时或进程异常退出后必须释放 lease，并由孤儿回收流程兜底。
- 项目文件写入同时使用 FreshnessTracker 或 expected hash。
- Artifact 发布以 `run_id + producer + source + hash` 做幂等键。
- 同名 Artifact 不静默覆盖；返回冲突或分配新 ID。
- Enterprise 多副本使用对象存储条件写、DB 乐观锁或分布式锁，不能依赖进程内 mutex。

### 7.5 子智能体、Workflow 与协作空间

多智能体编排沿用 Agent App 设计中的三层模型，但统一复用本文件的 `ExecutionBinding`、ResourceRef 和 ArtifactRef：

| 层级 | 默认工作空间关系 | 允许的共享方式 | 禁止行为 |
| --- | --- | --- | --- |
| L1 App 内子智能体委派 | 子 Run 继承父级 tenant/project/session 关联，但获得独立 binding；权限、工具、mount 和写范围重新取交集 | 父项目的只读或受限视图、显式授予的细粒度可写路径、InputRef、ArtifactRef | 因共享 `session_id` 就继承父级全部可写 cwd、secret 或审批 |
| L2 App 内 Workflow | 串行 step 默认独立 work namespace；并行 step 必须独立可写 namespace | 显式 step output、ArtifactRef、版本化共享状态、带锁共享资源 | 多个并行 step 写同一目录后靠目录 diff 猜归属 |
| L3 App 间协作 | 每个 Agent App 成员拥有独立 binding 和有效策略快照 | Blackboard 中的 ResourceRef、消息附件 InputRef、ArtifactRef、显式 Handoff | 将一个成员的宿主路径、MCP credential 或可写 workspace 直接暴露给另一成员 |

补充规则：

- `fork_context`、共享 transcript 或 App-as-member 投影只传上下文，不自动传文件权限；资源可见性必须单独投影。
- 子智能体若要修改父项目，委派契约必须声明具体资源范围和写意图，并复用 FreshnessTracker/锁；默认只读是更安全的基线。
- 父子 Run 的 Artifact 交接使用稳定引用并在消费时重新鉴权；不得复制回父 cwd 作为隐式结果。
- Workflow reducer 可以汇总多个 step 的 ArtifactRef，但不能因为 orchestrator 可见所有 step 就绕过 producer/scope 校验。
- CollaborationSpace 只负责成员、消息与路由治理，不拥有另一套文件系统；其 Blackboard/附件最终仍落到 Workspace、Input 和 Artifact 的统一 port。

## 8. 文件工具规范

文件工具只依赖：

- `FileSystemBackend`
- `PathResolver` / `ResourceResolver`
- `FreshnessTracker`
- `PermissionEngine`
- `ResourceLocker`
- Audit/Trace port

产品装配规则：

| 产品/环境 | Backend |
| --- | --- |
| CLI/Desktop 宿主或本地平台沙箱 | Local FileSystemBackend |
| CLI/Desktop 远程 session workspace | Remote WorkspaceFS Backend |
| Enterprise | 租户 WorkspaceFS/Object Backend，禁止 Local Backend |

若 CLI 采用“项目仍在本地、任务在远程”模式，必须显式区分 `project:` 与 `workspace:`，不能让同一个相对路径有两个物理含义。

## 9. 普通命令执行规范

### 9.1 工具边界

- `run_command` 必须消费控制面注入的 `ExecutionBinding`，不能根据当前 Agent App 名、工具参数或进程 cwd 临时推断模式。
- CLI/Desktop 的交互式 `code` App 在已明确选择并授权项目时，可由产品入口创建 `project_workspace` binding；这是一项显式产品默认，不是 `run_command` 的内建语义。
- 未绑定任务上下文、未授权项目且没有可解析工作空间时必须返回结构化错误，禁止偶然落到进程 cwd。
- Skill 加载后由运行时绑定到对应的 execution/session workspace，模型参数不能改变该绑定。
- 任务型文件处理使用独立 `run_task`，或由运行时注入不可伪造的 `workspace_mode=task_job`。
- LLM 不得通过自行选择较弱 mode 绕过安全策略；最终 mode 是用户意图、工具类型和产品策略的交集。

### 9.2 `run_task` 最小契约

```go
type TaskExecutionRequest struct {
    Command         Command
    Binding         ExecutionBinding // 控制面生成；模型不可覆盖
    Inputs          []InputRef
    ExpectedOutputs []WorkspacePath
    RuntimeProfile  RuntimeProfile
    Sandbox         SandboxProfile
    CollectionPolicy ArtifactCollectionPolicy
}
```

远程执行时：

- `Inputs` 转成 `input_artifact_ids` 或 session WorkspaceFS 写入；
- cwd 与 env 由 ExecutionWorkspace 生成；
- `ArtifactCollectionPolicy` 默认是 `declared_outputs_only`；`OUTPUT_DIR` 只限定可发布边界，不代表目录中的所有文件都已被声明；
- 输出下载、gate、发布为 ArtifactRef；
- 仅当请求声明“必须产生 Artifact”且没有声明路径或没有回收到合规产物时，返回可修复诊断；纯计算命令不能因没有文件输出而失败。

## 10. Skill 脚本执行规范

### 10.1 原始 Skill 包不侵入原则

- Skill 是知识和流程包，不是工具原语。
- 第三方 `SKILL.md`、references、scripts、assets 必须按来源字节原样保存并以不可变版本执行；解析、索引和运行期 bridge 都只能生成包外派生数据。
- 禁止为了适配 Genesis，把 Genesis Tool 名、`$WORK_DIR`、`$OUTPUT_DIR`、镜像名、审批语义或产品配置写回第三方 Skill。
- 禁止通过修改第三方脚本来绕过输入 staging、权限、sandbox、Artifact gate 或依赖治理。
- Skill 包内容视为版本化只读资产；运行中产生的文件必须写入独立工作视图，不能污染已安装原包。
- Harness 必须先把来源解析为不可变 `SkillPackageRef(authority, resource, version, digest)`；禁止对一个可能被并发更新的“当前目录”先审批、后执行。
- 确需修复上游 Skill 时，应作为有来源、有版本、有 diff 的派生包发布，不能在 materialize 阶段静默改写。

### 10.2 Skill Runtime Harness

Genesis 使用统一的 `Skill Runtime Harness` 把可移植 Skill 映射到受控运行时：

```text
原始 Skill Package（只读、通用）
→ Harness 解析 frontmatter / dependencies / references / QA 要求
→ 分配 session workspace
→ materialize 分层 Skill 执行视图
→ InputStager 注入受控输入别名
→ Permission + Approval
→ RuntimeProfile + Sandbox
→ verbatim command 执行
→ produced 候选与 QA 状态跟踪
→ 显式 PublishArtifact
→ Gate + ArtifactStore + Audit
```

Harness 的统一职责：

| 职责 | 要求 |
| --- | --- |
| 包解析 | 将 Source 解析为不可变 SkillPackageRef；校验 frontmatter、依赖声明、资源路径和包完整性；生成版本化 manifest 与内容 hash，作为运行前后不可变性依据 |
| materialize | 构造“原包只读层 + session 可写层 + input 只读层 + call tmp 层”的统一执行视图；安装源与工作文件分离 |
| 运行期 bridge | 只在模型上下文中说明宿主适配，不修改 Skill 正文 |
| 输入 | InputRef staging、重命名、冲突检测、只读约束 |
| 权限 | Skill 加载、资源读取、命令执行、外部输入、Artifact 导出分别审批 |
| 依赖 | 通过可插拔 metadata adapter 规范化不同 Skill 格式的声明；只准备已声明且策略允许的 runtime 依赖，不允许脚本旁路装包 |
| 执行 | 选择 RuntimeProfile、Sandbox、cwd、env、timeout 和资源限额 |
| 路径 | 验证相对路径、符号链接、保留目录和跨 namespace 使用 |
| QA | 跟踪 Skill 声明的必读资源与 QA 步骤，不伪造成功结果 |
| 产物 | produced 仅提示；最终文件必须显式发布并经过格式门禁 |
| 治理 | Audit、Trace、Usage、配额、并发锁、取消和清理 |

Harness 不承担：

- 把任意 Skill 命令翻译成 Genesis 私有 DSL；
- 根据 Skill 名称在 Engine 中写硬编码分支；
- 静默改写脚本中的路径或命令；
- 为了“方便”自动扫描整个 workspace 并交付全部新增文件；
- 在 sandbox 失败时绕过策略改为宿主直跑。
- 从 Skill 正文自然语言中猜测依赖后静默联网安装；未声明依赖应返回结构化诊断，由受治理的依赖流程处理。

Harness 属于产品无关内核。CLI、Desktop、Enterprise 必须复用同一实现，仅注入不同的执行 backend、审批器、ArtifactStore、凭证、配额和租户上下文。

依赖声明和命令前缀只用于选择 RuntimeProfile、做依赖预检和决定是否需要审批，不是 shell 安全边界。包含管道、重定向、子命令或解释器动态加载的命令仍以完整命令、有效权限和 sandbox 约束为准，不能通过“前缀已声明”自动放行。

### 10.3 加载与运行时适配

- `Skill(...)` 只加载知识和绑定 Skill，不立即执行命令。
- `Skill(...)` 的 `allowed-tools` 只能收窄当前 Profile，不能扩权；后续每个执行原语仍须经过 Tool Gateway、PermissionEngine 和 Approval。
- Skill 加载后第一次执行通用 `run_command` 时，Harness 在当前 Run 的允许模式内惰性打开独立 Skill execution workspace，并通过运行时生成、模型不可伪造的 ExecutionBinding 固定 owner、session、cwd、权限、sandbox 和审计上下文。Skill 不强制所有 App 使用 `session_workspace`：短作业可绑定 `task_job`，需连续状态时绑定 `session_workspace`，在授权项目原位工作的 Skill 可使用受限 `project_workspace` 视图。模型参数不能改变该绑定。
- Harness 将 Skill 包 materialize 为逻辑分层视图：包 manifest 中已有路径始终只读；session work 只允许创建或修改非保留工作路径；InputRef 别名只读；call tmp 独立可写。第三方脚本仍看到稳定的相对路径和 cwd，不需要知道物理分层方式。
- backend 可使用 overlay mount、逐路径 ACL、WorkspaceFS 路由或等价机制实现分层。包只读层约束分级：
  - **强要求**（`SandboxRequire` 或不可信外部 Skill）：必须在写入发生时拒绝修改包层，而不是只在命令结束后发现 hash 已变化。无法提供该能力时必须失败。
  - **弱要求**（`SandboxAuto` + 可信内置 Skill）：可以使用"独立副本 + 执行后 hash 复核"的方案，但必须返回 warning、trace 和 audit，且受产品策略允许。
  - **Windows 特别说明**：Windows 本地平台沙箱当前无 overlay 能力，对内置 Skill 使用独立副本方案；对不可信外部 Skill 需要 Docker/genesis-sandbox 后端。
- Skill 正文要求执行的普通相对命令由 Harness 映射到统一命令执行原语，命令内容本身保持 verbatim。
- Runtime bridge 是宿主侧临时上下文，不属于 Skill 包内容，也不得被持久化回安装源。

### 10.4 输入

```text
外部资源
→ InputRef
→ Run input snapshot
→ Skill workspace alias/bind
```

Skill 正文中的包内相对资源以 Skill package root 解析，命令中的工作文件以 session cwd 解析；Harness 在执行视图中保持这些路径稳定。两个同名输入必须改名并在 staged manifest 中保留映射，模型只能使用映射后的逻辑名称，不能获知不同 backend 的物理路径。

### 10.5 中间状态

- 解包目录、草稿、分析 JSON、缩略图、QA 日志保留在 Skill work。
- `TMPDIR` 只放无需跨命令的缓存。
- `produced[]` 是诊断信息，不能自动进入 ArtifactStore。

### 10.6 产物发布

Skill session 的最终交付统一使用显式工具：

```text
publish_artifact(
  source="workspace-relative/output.pptx",
  name="产品对比报告.pptx"
)
```

发布流程：存在性检查 → 路径边界检查 → 格式门禁 → hash/MIME → ArtifactStore → ArtifactRef。

发布时必须再次解析真实对象并拒绝 symlink/junction/reparse 越界；压缩类与 Office/PDF 格式的门禁还应限制条目数、解压后总量、嵌套深度和压缩比，防止校验阶段资源耗尽。

禁止：

- 全目录 diff 自动发布；
- 扁平化 basename 导致层级丢失；
- 复制所有 QA 文件到 output；
- 用 `write_file` 伪造二进制 Office/PDF；
- Artifact 已发布后再次复制到仓库根。

## 11. MCP 执行与文件规范

### 11.1 MCP Placement

MCP Transport 必须声明进程/服务放置位置：

| Placement | 用途 | 文件权限 |
| --- | --- | --- |
| `local_stdio` | CLI/Desktop 可信本地 server | 仍须受最小环境和目录策略限制 |
| `local_sandbox_stdio` | 文件、浏览器、代码类 stdio server | 通过 ExecutionRunner 启动 |
| `remote_sandbox_stdio` | Enterprise 或强隔离场景 | 在 genesis-sandbox session 中启动 |
| `streamable_http` | 外部或企业内部 MCP 服务 | 使用服务自身资源协议，不传宿主路径 |

stdio 禁止直接绕过 ExecutionRunner 使用裸 `exec.CommandContext`。子进程环境采用 allowlist 注入，不能默认继承全部宿主 secret。

### 11.2 MCP 文件绑定

不能根据 JSON 字段名猜测路径。文件型 MCP tool 的绑定声明有两种方式：

**方式 A：json_pointer 显式声明（adapter 级低层机制）**

```yaml
tools:
  convert_file:
    input_bindings:
      - json_pointer: /input
        kind: input_ref
    output_bindings:
      - json_pointer: /result/path
        kind: artifact
```

**方式 B：schema 自动识别（高层便捷方式）**

MCP tool 参数 schema 中包含 `type: string, format: uri` 或 `type: string, format: file-path` 或扩展属性 `x-genesis-kind: input_ref|artifact` 的字段，Harness 自动识别为文件绑定候选。首次识别时需用户确认后生效，确认结果缓存到 MCP 配置。方式 B 生成的 binding 最终等价于方式 A 的结构。

运行时只改写声明过的字段：

- InputRef → MCP 所在环境可访问的路径、URL 或 resource ID；
- MCP 输出路径/资源 → 下载、gate、ArtifactRef；
- 未声明 binding 的参数和结果保持原样。

Streamable HTTP MCP 优先使用签名 URL、resource ID 或上传 API；不支持文件上传时必须明确返回“不支持本地文件输入”，不能发送宿主绝对路径。

## 12. Artifact 规范

Artifact 是需要持久交付、下载、预览、跨执行引用或留存证据的正式文件/对象，不等同于任意 output 文件，也不等同于所有业务结果。外部工单、订单、告警、数据库记录等由对应领域 ID/ResourceRef 表达；如需留痕，可另发布经过 gate 的回执 Artifact，但不能用 ArtifactStore 替代业务事务系统。

最小 manifest：

```json
{
  "id": "artifact-...",
  "run_id": "run-...",
  "scope": {"tenant_id": "", "project_id": "project-...", "user_id": "user-..."},
  "producer": "skill:office-ppt",
  "producer_version": "office-ppt@sha256:...",
  "name": "产品对比报告.pptx",
  "kind": "pptx",
  "size": 173661,
  "sha256": "...",
  "mime": "application/vnd.openxmlformats-officedocument.presentationml.presentation",
  "inputs": [{"id": "input-...", "sha256": "..."}],
  "gate_version": "office-gate/v1",
  "storage_ref": {"scheme": "artifact", "id": "artifact-..."},
  "created_at": "..."
}
```

要求：

- 二进制格式执行魔数和结构校验；
- 文本格式至少校验编码、大小和非空策略；
- ArtifactRef 不暴露 Enterprise 宿主路径；
- 导出到用户目录是独立操作，重新经过权限、Freshness 和审计；
- Artifact Store 支持按租户、项目、Run 和过期策略查询。
- Artifact manifest 记录 producer/version、输入版本与 gate 版本，支持追溯和重放；重放不等于保证字节级确定性。
- 发布采用两阶段流程：先写不可见 quarantine/CAS 临时对象并完成 size/hash/MIME/结构门禁，再以条件写或事务原子提交 manifest 与可见引用；失败或取消不得留下可查询的半成品 Artifact。
- 导出到已有用户文件时使用 expected version/hash 与同目录原子替换；目标在审批后发生变化必须返回冲突，不能覆盖新版本。
- Artifact 发布与 DeliveryTarget materialize 是两个阶段：先进入受治理 ArtifactStore，再以 ArtifactRef 为源原子导出；导出失败不破坏已发布 Artifact，也不能留下半写目标文件。
- CLI/Desktop 成功结果必须直接返回用户可见的导出路径、可点击 Artifact 或产品 Inbox 入口；`.genesis/runs/...`、quarantine、CAS 和内部 output 路径只用于运行时，不作为用户操作说明。
- 默认 `source_sibling` 和回退 `project_root` 都必须执行目标路径解析、写权限、Freshness、同名冲突和 protected path 检查；用户只授权读取源文件时，产品策略仍可要求一次写入确认。

## 13. 生命周期、配额与清理

| 对象 | 默认生命周期 | 清理规则 |
| --- | --- | --- |
| call tmp | 单次调用 | 调用结束清理；失败可短期保留诊断 |
| job input/output | 单 job | 回收/发布后清理执行副本 |
| session work | Run 或 session | 完成后进入 TTL；活跃会话不得清理 |
| Run manifest/input snapshot | 可配置 | 审计保留期后清理 |
| Artifact | 独立保留策略 | 不随 Run 自动删除；引用计数/保留期决定 |
| Skill package cache | Skill 版本级 | 版本过期 + LRU + 磁盘配额；不可变版本可长期缓存 |

必须配置：

- 单文件输入上限；
- 单 Run 输入总量；
- work/tmp/output 配额；
- 最大文件数量和遍历深度；
- session idle TTL；
- Run retention；
- Artifact retention；
- 崩溃和孤儿 session 清理任务。

| Local FileSystemBackend | 允许 | 允许 | 禁止作为租户业务 backend |

| 身份与 Scope | 本地用户/项目 | 本地用户/项目 | tenant/project/user/role 全量 |
| 配额 | 本地配置 | 本地配置 | 租户配额与计费 |

### 14.3 产品红线

CLI：

- remote mode 下必须明确区分本地 `project:` 和远程 `workspace:`；
- 不能让 `read_file` 看到本地文件、`run_command` 却默认在远程找同名路径；
- 用户可看到本地 Artifact 路径，但内部仍以 ArtifactRef 为准。
- 切换 Agent App 或从 `code` 切到业务 App 时必须重新解析 binding；不得沿用旧 App 的 cwd、mount 或审批缓存。
- 成功后直接打印实际交付路径或 Artifact 入口；不得要求用户进入 `.genesis/runs` 查找文件。

Desktop：

- 不复制 CLI 的 workspace 实现，只复用内核并提供 UI；
- ArtifactRef 可投影为画布、预览或下载入口；
- 桌面自动化 MCP 必须声明 Placement 和权限范围。
- UI 的“当前项目”“当前 App”“当前会话”是三个正交选择；不得用选中 App 暗中扩大目录权限。
- 生成结果必须出现在当前任务卡片/Artifact 面板，并展示实际导出位置；内部 Run 目录不进入普通用户导航。

Enterprise：

- 默认 `SandboxRequire`，禁止宿主直跑租户代码；
- 禁止租户文件依赖实例本地 cwd；
- 所有查询和 manifest 带 `tenant_id`；
- ArtifactStore、InputStore、WorkspaceFS 使用租户隔离的远程实现；
- stdio MCP 默认禁止，除非运行在租户隔离的远程 sandbox；
- headless `ask → auto approve` 只能作为开发过渡，生产必须接 RBAC/工单。
- Agent App、Run、子 Run、Workflow step、CollaborationSpace member 与 Workspace/Artifact manifest 的关联均须可按 `tenant_id` 审计；App 间交接只传受治理资源引用。

### 14.4 Agent App 配置边界

统一内核只接收已经解析的 `EffectiveAgentAppProfile`，不直接读取 `.genesis/app.yaml`、Desktop 状态或 Enterprise DB。Agent App 与工作空间的职责边界固定如下：

| 对象 | 可以声明 | 不能声明/授予 |
| --- | --- | --- |
| Agent App | 默认/允许模式、是否要求项目、持久化需求、所需工具/MCP/Skill、最低 sandbox、默认读写姿态 | 宿主绝对路径、租户外资源、绕过审批的 mount、强制降低 sandbox |
| Task Template / Run Preset | 在 App 允许范围内进一步选择模式、输入槽位、ExpectedOutputs、超时/配额 | 扩大 App/Product/Enterprise 上界、复用别的 Run 审批 |
| SubAgent / Workflow step | 继承并收窄父 binding，申请明确的资源投影 | 默认继承父级全部可写视图或 secret |
| CollaborationSpace member | 声明可接收的附件/Artifact 类型和交接契约 | 直接引用另一成员的物理 cwd 或存储凭证 |

因此 `code` 是内置默认 Agent App，`project_workspace` 是通用执行模式；二者不得合并成一个概念。未来新增客服、运维、数据分析、文档审查或业务办理 App 时，无需新增场景专属 workspace 类型，除非发现了无法由“项目原位 / 隔离 job / 持久 session + policy”表达的全新资源生命周期。

## 15. 架构依赖方向

```text
agentapp resolver / runtime / multiagent / tool / skill / mcp
        ↓ 只依赖 contract
workspace + filesystem + execution + artifact capabilities
        ↓
shared/local                 sandbox/adapter/http
宿主与本地平台实现             genesis-sandbox API 实现
        ↑                         ↑
products/<product>/bootstrap 选择并注入
```

禁止：

- Tool 直接 import Docker SDK、HTTP client、Wails 或产品 RBAC；
- MCP transport 自己实现一套文件 staging；
- Skill service 自己维护与 ArtifactService 不一致的最终目录；
- Enterprise bootstrap 回退到 Local Backend 处理租户文件；
- 产品层复制 Workspace 业务规则。
- agentapp service 直接创建物理目录、拼接 sandbox 路径或绕过 WorkspaceResolver；
- runtime/multiagent 为子智能体或协作空间另造一套 staging/Artifact 协议。

## 16. 结构化错误

至少提供以下稳定错误码：

| 错误码 | 含义 |
| --- | --- |
| `PATH_NAMESPACE_MISMATCH` | 控制面路径与执行面路径混用 |
| `STATE_ROOT_UNAVAILABLE` | 产品无法为当前 Run 解析稳定、受控的 state root |
| `RESOURCE_VERSION_CONFLICT` | 资源在审批、staging、导出或发布期间发生版本变化 |
| `INPUT_PERMISSION_DENIED` | 输入来源未授权 |
| `INPUT_NAME_CONFLICT` | staged 输入重名 |
| `INPUT_RESERVED_PATH_CONFLICT` | 输入覆盖执行包保留路径 |
| `INPUT_TOO_LARGE` | 输入超过限制 |
| `WORKSPACE_NOT_AVAILABLE` | 目标 workspace/session 不可用 |
| `WORKSPACE_MODE_NOT_ALLOWED` | 请求模式不在 App/Product/Policy 允许交集内 |
| `EXECUTION_BINDING_REQUIRED` | 执行原语缺少可信控制面生成的 binding |
| `EXECUTION_BINDING_CONFLICT` | Run 中尝试篡改模式、owner、root 或复用不相容 binding |
| `CROSS_EXECUTION_RESOURCE_DENIED` | 子 Run、Workflow step 或协作成员未获授权就访问其他执行主体资源 |
| `DELIVERY_TARGET_DENIED` | 交付目标不可写、未授权或违反 protected path/产品策略 |
| `DELIVERY_TARGET_CONFLICT` | 交付目标已存在或在审批后发生版本变化，不能静默覆盖 |
| `DELIVERY_MATERIALIZE_FAILED` | Artifact 已发布，但导出到用户目标失败；错误中必须保留可访问 ArtifactRef |
| `SANDBOX_REQUIRED_UNAVAILABLE` | required sandbox 不可用 |
| `NO_OUTPUT_ARTIFACTS` | 任务要求产物但没有输出 |
| `ARTIFACT_PATH_INVALID` | 发布路径越界或不存在 |
| `ARTIFACT_INVALID` | 格式门禁失败 |
| `ARTIFACT_NAME_CONFLICT` | 产物命名冲突 |
| `MCP_FILE_BINDING_MISSING` | MCP 文件参数没有绑定契约 |
| `MCP_PLACEMENT_UNAVAILABLE` | MCP 所需放置环境不可用 |
| `OUTPUT_NOT_DECLARED` | 文件位于输出边界内，但未被调用契约声明为产物 |
| `SKILL_PACKAGE_CHANGED` | Skill 来源在解析、materialize 或执行校验期间发生变化 |
| `SKILL_EXECUTION_VIEW_UNSUPPORTED` | backend 无法提供策略要求的只读包层和可写工作层隔离 |
| `SKILL_DEPENDENCY_UNDECLARED` | 运行需要未声明依赖，Harness 拒绝猜测或旁路安装 |

错误必须包含可修复建议，但不能自动放宽权限或隐式搬运文件兜底。

## 17. 可观测性

每次边界操作至少记录：

- tenant/project/session/run/call ID；
- Agent App ID/version、Task Template/Preset（若有）、parent/child Run、SubAgent instance、Workflow step、CollaborationSpace/member ID；
- workspace mode 和 backend；
- ExecutionBinding ID、owner、继承/收窄来源和共享资源方式；
- sandbox requested/effective/degraded；
- 输入 source、Input ID、hash、size；
- cwd 与逻辑目录映射（不得记录敏感宿主路径到业务事件）；
- command/tool/skill/MCP producer；
- Artifact ID、hash、size、gate 结果；
- 审批决定、scope 和原因；
- staging、执行、上传、下载、校验耗时；
- 清理结果与失败重试。

命令参数、环境变量、签名 URL、MCP 参数和审批上下文必须先按敏感字段策略脱敏；“可审计”不等于记录完整 secret 或宿主绝对路径。

## 18. 验收矩阵

### 18.1 三环境一致性

同一任务分别在宿主、本地平台沙箱、远程 sandbox API 执行，必须满足：

- 输入 manifest 的 name/hash 一致；
- 脚本使用相同相对文件名；
- 中间文件不出现在 Artifact 列表；
- Artifact name/kind、格式门禁和业务内容验收结果一致；仅对声明为确定性生成或经过规范化的格式要求 hash 一致，Office 元数据、时间戳等非确定性内容不得让跨环境验收产生假失败；
- 模型、跨 backend 协议和 Enterprise UI 不出现宿主绝对路径；CLI/Desktop 本地 UI 只有在路径属于当前用户可访问的本地 ArtifactStore 或已授权导出目标时才可展示物理路径；
- required/optional 降级语义一致且可审计。

### 18.2 能力验收

- 文件工具：本地和远程 WorkspaceFS backend 契约测试。
- StateRoot：CLI 项目/非项目、Desktop 和 Enterprise 根选择稳定；改变进程 cwd 或 UI 当前目录不得迁移既有 Run。
- App/Workspace 正交：`code` 可运行 project/job/session；非 code App 可使用 project workspace；切换 App 后旧 binding 不可复用；`AgentAppType` 不影响权限判定。
- 普通命令：project/task/session 三种模式不能互相污染；无 binding 时不落到进程 cwd。
- 多智能体：父子 Run、并行 Workflow step 和 L3 成员默认可写隔离；只读投影、受锁项目写、Artifact/Handoff 三类显式共享均有权限与竞态测试。
- Skill 原包：同一不可变 SkillPackageRef 的 manifest/hash 在加载、materialize、执行前后保持一致；来源版本漂移必须失败并重新解析，运行期 bridge、状态和输出不得回写原包。
- Skill Harness：输入重名、保留路径冲突、嵌套产物、QA 中间文件、显式发布、权限拒绝和沙箱差异。
- MCP：local stdio、本地沙箱 stdio、远程 stdio、HTTP 文件绑定。
- Artifact：二进制门禁、文本校验、同名幂等、Run 清理后仍可读。
- Delivery：显式目标优先；明确目标文档时默认平级交付；无目标文档或平级不可用时回退项目根目录；项目根也不可用时保留 Artifact 并返回产品级可见入口；任何成功路径都不要求用户浏览内部 runs 目录。
- 权限：资源版本漂移、链接越界、只读输入篡改、Skill allowed-tools 扩权、审批跨操作复用均被拒绝并可审计。
- Enterprise：tenant 隔离、对象存储、required sandbox、stdio deny。

## 19. 分阶段实施建议

### Phase 0：冻结契约

- 将本文作为目标基线；
- 统一旧文档中 `OUTPUT_DIR` strict job 与 session workspace 显式发布的适用范围；
- 禁止新增依赖目录 diff 自动交付的代码。
- 将历史 `coding_workspace` 统一更名为场景无关的 `project_workspace`，不保留双轨模式名。
- 将当前 `local_coding_workspace` / `local_task_workspace` / `sandbox_session_workspace` 拆成正交的 WorkspaceMode（project/task/session）与 backend/sandbox 字段；同一 mode 必须可映射到三种执行环境。
- 冻结 WorkspaceResolver/ExecutionBinding 与 Agent App 配置边界。

### Phase 1：修复现有高风险不一致

- `path_map` 改为读取实际 ExecutionWorkspace；
- 实现 StateRootResolver，先消除依赖进程 cwd 的 Run 根漂移；
- Skill 输入增加 hash、大小、重名和保留路径检查；
- 本地产物不再 basename 扁平化；
- 增加 tmp/run 清理策略。

### Phase 2：统一 Input/Artifact

- 实现 InputStager、ArtifactPublisher、ArtifactStore；
- 由通用 Skill Runtime Harness 接入不可变 SkillPackageRef、分层执行视图、InputStager 与显式 Artifact 发布，不修改原始 Skill 包；
- 返回 ArtifactRef，逐步移除最终路径字符串作为唯一交付依据。

### Phase 3：普通命令三环境闭环

- 拆分 project workspace、task job 与 session workspace；
- 接通远程 input artifacts 和声明输出回收；
- 增加三环境端到端等价测试。

### Phase 4：MCP Placement 与 FileBinding

- stdio 统一经过 ExecutionRunner；
- 支持本地沙箱和远程 sandbox stdio；
- 实现显式文件输入/输出绑定。

### Phase 5：三产品存储与治理

- CLI/Desktop 本地 ArtifactStore；
- Enterprise 租户 InputStore/WorkspaceFS/ArtifactStore；
- 配额、retention、GC、审计和运营指标闭环。

### Phase 6：Agent App 与多智能体资源闭环

- EffectiveAgentAppProfile 接入 WorkspaceResolver，App 只声明需求与上界内偏好；
- L1 子 Run、L2 Workflow step、L3 CollaborationSpace member 统一生成独立 ExecutionBinding；
- 完成只读资源投影、受锁共享项目写、Artifact/Handoff 交接和跨成员重新鉴权测试；
- CLI、Desktop、Enterprise 用同一契约验证 App 切换、恢复和分布式执行，产品层只注入 Store/Policy/backend。
