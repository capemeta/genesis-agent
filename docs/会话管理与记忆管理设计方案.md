## 一、设计目标与要回答的问题

会话管理与记忆管理是 Agent Runtime 的第二根内核支柱（第一根是提示词分层，见 `docs/提示词分层设计方案.md`）。提示词决定"把通用内核塑造成哪类工作"，而会话/记忆决定"这类工作在长程、多轮、跨会话中如何保持连贯、如何积累经验、如何在有限上下文里做最优取舍"。

本方案要回答用户提出的核心问题：

1. **记忆分层**：工作记忆、短期记忆、长期记忆、用户画像如何分层？各自生命周期、存储、检索有何不同？（§五）
2. **长期记忆如何形成**：从哪来、抽取什么、如何去重合并、如何支持"不同项目沉淀不同内容"的自定义抽取与结构？（§七）
3. **会话管理**：摘要、压缩、上下文如何组织？完整消息链如何持久化与恢复？（§六）
4. **上下文分配**：历史记录、摘要、长期记忆、全局提示词、工具、当前问题如何在有限窗口内分配？（§八）
5. **提示词构建是否要考虑模型总上下文**：是。需要（§八）：
   - 模型配置文件增加"最大上下文"字段（私有化部署模型难以自动探测，必须可配置）；
   - 一套预算计算公式（预留输出长度后，剩余额度在各部分间按比例分配，但比例可弹性伸缩、带区间限制）；
   - 历史记录取多少条、单条最大长度、最大长度占比的约束。
6. **科学的压缩机制**：超过多少触发压缩？压缩多少轮、保留多少轮？长期记忆在压缩链路的哪个环节沉淀？（§六 + §七）
7. **三产品差异**：CLI / Desktop / Enterprise 在存储与检索上如何统一接口、差异化后端？（§十）
8. **多智能体兼容**：不同 Agent / Agent App / 子智能体 / 协作空间下，会话与记忆如何兼容（至少预留字段、隔离范围）？（§九）

一句话结论（先给出，后文论证）：

> **会话 = 一条可追加、可摘要、可恢复的消息流（每条带 `Role` + `Kind`）+ 一份上下文预算契约**；**记忆 = 四层生命周期不同的知识资产**。运行时内核只负责"预算分配 + 上下文装配 + 压缩编排 + 记忆注入顺序"这套与产品无关的算法；消息/记忆/画像的**存储后端**（文件 / SQLite / DB / 向量库）和**检索方式**（关键词 / 结构化 / 语义）通过统一契约接口由产品端口注入。模型最大上下文是可配置字段，预算按"刚性段优先满足 + 弹性段按比例分配并带区间 clamp 与额度回流"的算法平衡。长期记忆通过"可配置抽取器（两阶段：抽取 → 合并）"从会话中沉淀，不同项目可自定义抽取提示词与结构。三端同源算法，异构后端。`Message.Kind` 是 UI 投影、压缩用户轮边界、LTM 过滤的唯一主开关（详见 §6.2）；不采用必填 `Origin`。

## 二、第一性原理：会话与记忆到底要解决什么

抛开当前实现，把问题还原到本质。

### 2.1 三个不可回避的物理约束

| 约束 | 本质 | 带来的必然设计 |
| --- | --- | --- |
| **上下文窗口有限** | LLM 单次能看到的 token 有上限（私有化模型甚至更小、且难探测） | 必须有"预算 + 裁剪 + 压缩"，不能把全量历史无脑塞进去 |
| **对话是增量、单调增长的** | 每轮都在累积消息，长会话必然溢出 | 必须有摘要/滑窗/选择性召回，把"旧而重要"的信息用更少 token 表达 |
| **经验值得跨会话复用** | 用户偏好、项目约定、失败教训不该每次从零开始 | 必须有跨会话的长期记忆与画像，且"记什么"是可控、可编辑、可审计的业务决策 |

### 2.2 信息按"寿命"分层（与提示词分层同一方法论）

会话/记忆承载的信息，其**寿命**和**归属**完全不同，必须分层，不能塞进同一个存储或同一段上下文：

| 信息类别 | 寿命 | 存储 | 检索方式 | 谁决定"记什么" |
| --- | --- | --- | --- | --- |
| **工作记忆** | Run 内，结束即释放 | 内存（RunContext） | 直接访问 | 框架 |
| **短期记忆（对话历史）** | Session 生命周期 | 文件/DB/Redis | 按时间顺序 + 摘要 | 框架 |
| **长期记忆（经验）** | 跨 Session，永久（可过期） | 文件/DB/向量库 | 关键词/结构化/语义 | **业务**（框架只负责读取注入与提供抽取器骨架） |
| **用户画像** | 跨 Session，相对稳定 | 文件/DB/画像服务 | 结构化字段读取 | 业务（可编辑可审计，不靠不可解释的 Prompt 总结） |

### 2.3 必须满足的不变量（验收基线）

1. **完整性**：短期记忆必须能保存**完整消息链**（user / assistant / tool_calls / tool result / reasoning / **runtime 注入的 skill 全文**），而不是只存 user + 最终回答（当前实现的缺陷）——否则恢复会话、复现工具链、做摘要都缺料。
2. **可恢复**：会话可从持久化介质完整或按摘要边界恢复（resume / fork）。
3. **预算安全**：装进 LLM 的上下文**永不超**模型有效窗口；刚性段（system + 当前问题 + 工具 schema）优先满足；溢出时优雅降级（压缩/裁剪/明确报错），不静默截断关键信息。
4. **压缩可解释、可持久化**：压缩后的历史是"替换"而非"丢失"——摘要作为一条消息留在流里，原始历史在持久层可追溯（借鉴 codex `CompactedItem.replacement_history`）。
5. **消息语义可区分（Message.Kind）**：同一 `role=user` 下必须能区分「真人输入 / Skill 注入 / 会话摘要 / reminder」等；UI 投影、压缩「保留真实用户轮」、长期记忆抽取**以 Kind 为唯一主开关**，禁止靠猜 content 前缀做策略（标签仅作可读双保险）。详见 §6.2。
6. **记忆可控可编辑**：长期记忆与画像"记什么"由业务决策，且可查看、编辑、删除、审计；`secret/pii` 默认不进上下文。
7. **接口与实现分离**：内核只依赖 `ShortTermMemory` / `LongTermMemory` / `UserProfileStore` / `TokenEstimator` / `SessionStore` 契约；文件/DB/向量库/tiktoken 都是可替换 adapter。
8. **三端同源**：预算算法、装配顺序、压缩编排零 `if product == ...`；差异只经存储/检索/token 后端注入。
9. **多智能体兼容**：Session/Message/Memory 数据模型预留 `app_id`/`agent_instance_id`/`scope`，为 Agent App 与协作空间留出隔离维度，当前阶段只记录不编排。

### 2.4 不做会怎样（失败条件）

- 无预算 → 长会话直接触发模型报错 `context_length_exceeded`，或被网关静默截断丢失关键上下文。
- 无压缩 → 要么频繁报错，要么被迫限制对话轮数，长程任务无法进行。
- 只存 user+final → 无法恢复工具执行链，无法复盘，摘要缺失中间证据。
- 无 Message.Kind → Skill 注入 / 摘要 / reminder 与真人 user 混为一谈 → UI 把说明书当用户话展示、压缩把 Skill 当「最近用户问题」保留、LTM 误抽取 SKILL.md → 企业产品观感与记忆污染。
- 长期记忆由框架自动写 → 记忆库膨胀、污染、不可审计、隐私泄露。
- 存储写死本地文件 → Enterprise 无法多租户隔离、无法向量检索、无法接画像服务。

### 3.1 当前会话/记忆链路

```text
RunOnce(sessionID, input)                         // internal/app/run_service.go
  └─ ReactLoopEngine.Start / loop                 // internal/runtime/strategy/react/react_loop.go
       ├─ prompt.BuildSystem()  → system 消息（仅 Run 开始一次，不落短期记忆）
       ├─ ShortTermMemory.GetRecent(sessionRef, budget) → 历史（含既往 tool/skill；加载后 EnsureKind）
       ├─ append 当前 user 输入 + 本 Run 轨迹（tool / skill_injection / assistant…）
       ├─ loop: transcript.ProjectForModel(rc.Messages) → llm.StreamGenerate(..., tools)
       └─ 终态/incomplete：SessionMessagesFromRun → ShortTermMemory.Append（本 Run 增量全链）
```

### 3.3 方向正确、可复用的部分

- `domain.Message` 是不依赖外部框架的自有消息模型，已具备 `ToolCalls / ToolCallID / ReasoningContent / Kind / Source`，可承载完整消息链和语义投影。
- `RunContext` 已作为运行时上下文载体，`TokenUsed` 与 `Run.TotalTokens` 在收到模型 usage 后累加。
- Skill 注入遵循 `role=user + Kind=skill_injection`；模型投影保留、UI 投影默认隐藏，避免与真实用户输入混淆。
- `LongTermEntry`、`MemoryScope` 和 `UserProfile` 已完成领域建模与文件后端接线；后续可按契约替换为 DB/向量/画像服务。
- `LLMModelConfig` 已具备 `ContextWindow`、`EffectiveContextRatio`、`OutputReserveTokens` 等预算字段，并在 Runtime 中使用。

## 五、记忆四层模型（分层与职责）

沿用 `docs/agent loop设计.md §11` 的四层分类，本方案补齐**接口契约与形成机制**。

```text
┌─────────────────────────────────────────────────────────────────────────┐
│ L0 工作记忆 Working Memory   —— Run 内，内存，框架直管                       │
│    RunContext.WorkingMemory: current_plan / retrieved_chunks / tool_results │
│    生命周期：Run 结束即释放；无独立接口                                       │
├─────────────────────────────────────────────────────────────────────────┤
│ L1 短期记忆 ShortTermMemory  —— Session 内，可持久化，框架直管               │
│    完整对话消息链 + 滚动摘要(summary) + leaf 指针                            │
│    接口：ShortTermMemory（append/get_recent/summarize/get_summary/clear）    │
│    存储：内存 → 文件 JSONL → DB/Redis                                        │
├─────────────────────────────────────────────────────────────────────────┤
│ L2 长期记忆 LongTermMemory   —— 跨 Session，业务决定记什么，框架负责读注入    │
│    episodic / semantic / procedural / negative；scope=user/agent/workspace  │
│    接口：LongTermMemory（save/search/list/update/delete/consolidate/decay）  │
│    存储：文件(关键词) → DB → 向量库(语义)                                     │
├─────────────────────────────────────────────────────────────────────────┤
│ L3 用户画像 UserProfileStore —— 跨 Session，稳定偏好，可编辑可审计            │
│    locale/communication_style/tool_preferences/risk_preference...           │
│    接口：UserProfileStore（get/update/list_fields/delete_field）             │
│    存储：文件 JSON → DB → CRM/画像服务                                        │
└─────────────────────────────────────────────────────────────────────────┘
```

### 5.1 层间协作（一次 Run 的记忆生命周期）

```text
Run 启动（一次性，缓存到 RunContext）
  1. UserProfileStore.Get(scope)        → 画像片段（注入 system，见提示词分层 L3 user_prefs）
  2. LongTermMemory.Search(user_input)  → 相关经验（注入 system 的 long_term_memories 块）
  3. ShortTermMemory.GetRecent(budget)  → 对话历史（按预算裁剪）+ GetSummary() 摘要

Run 进行中
  - 工作记忆在 RunContext.WorkingMemory 累积（工具结果、plan 进度）
  - 上下文接近阈值 → 触发压缩（§六）

Run 结束（on_run_complete）
  - ShortTermMemory.Append(完整消息链)  // 修正现状：存全链，非仅 user+final
  - （可选，业务）MemoryExtractor 触发长期记忆抽取（§七）
```

> **注入顺序与预算**由 §八的 ContextBudgetPlanner + ContextAssembler 统一编排；本节只定义"有哪些层、各层接口"。

### 5.2 接口契约（Go，落 `internal/capabilities/memory/contract`）

```go
// ---- 公共类型定义（所有契约共用，落 contract 包顶部） ----

// MemoryScopeType 记忆作用域类型（类型安全枚举，禁止使用裸字符串）。
type MemoryScopeType string

const (
    MemoryScopeUser          MemoryScopeType = "user"           // 用户级：跨 Agent 共享
    MemoryScopeAgentInstance MemoryScopeType = "agent_instance" // Agent 实例级：Agent 私有
    MemoryScopeWorkspace     MemoryScopeType = "workspace"      // 工作区级：同工作区共享
    MemoryScopeProject       MemoryScopeType = "project"        // 项目级：经业务声明才跨 Agent 共享
)

// MemoryScope 记忆作用域（Type + ID 组合唯一标识一个隔离域）。
// 替代原 LongTermEntry.ScopeType/ScopeID 裸字符串组合，避免拼写错误。
type MemoryScope struct {
    Type MemoryScopeType
    ID   string // 对应 UserID / AgentInstanceID / WorkspaceID / ProjectID
}

// MemoryEntryType 长期记忆类型（类型安全枚举）。
type MemoryEntryType string

const (
    MemoryTypeEpisodic   MemoryEntryType = "episodic"   // 情节记忆：特定事件/交互经历
    MemoryTypeSemantic   MemoryEntryType = "semantic"   // 语义记忆：事实/领域知识
    MemoryTypeProcedural MemoryEntryType = "procedural" // 程序记忆：操作步骤/工作流偏好
    MemoryTypeNegative   MemoryEntryType = "negative"   // 负面记忆：失败教训/禁止事项
)

// ---- 接口定义 ----

// ShortTermMemory 短期记忆：管理 Session 级完整对话历史与滚动摘要。
// 演进自现有 ShortTermStore（保留 Append/Get/Clear，补齐预算读取与摘要）。
type ShortTermMemory interface {
    Append(ctx context.Context, ref SessionRef, msgs []*domain.Message) error
    // GetRecent 从新到旧返回消息，受条数与 token 预算双约束；返回是否发生截断。
    GetRecent(ctx context.Context, ref SessionRef, opt RecentOptions) (RecentResult, error)
    // Summarize 将 keepRecent 之外的旧历史压缩为摘要并落盘（返回新的 summary 与 leaf 指针）。
    Summarize(ctx context.Context, ref SessionRef, opt SummarizeOptions) (SummaryResult, error)
    GetSummary(ctx context.Context, ref SessionRef) (*domain.SessionSummary, error)
    Clear(ctx context.Context, ref SessionRef) error
}

// SessionRef 所有请求必携带隔离维度（租户/用户/会话 + 多智能体维度）。
// 使用方式：调用方可通过 context.WithValue(ctx, SessionRefKey{}, ref) 在业务入口注入，
// 内核 helper SessionRefFromCtx(ctx) 从 ctx 取出，避免多层透传；接口仍显式传参以保证可测试性。
type SessionRef struct {
    TenantID        string
    UserID          string
    SessionID       string
    AppID           string // 多智能体：Agent App（可空）
    AgentInstanceID string // 多智能体：Agent 实例（可空）
}

type RecentOptions struct {
    MaxTurns    int   // 条数/轮数上限（0=不限）
    TokenBudget int   // token 预算（0=不限）
    IncludeToolChain bool // 是否包含 tool_calls/tool result 完整链
}

type RecentResult struct {
    Messages  []*domain.Message
    Truncated bool // 是否因预算被裁剪（供 UI/摘要判断）
    UsedTokens int
}

// LongTermMemory 长期记忆：字段对齐 domain.LongTermEntry（§七.4）。
type LongTermMemory interface {
    Save(ctx context.Context, entry *domain.LongTermEntry) (string, error)
    Search(ctx context.Context, q MemoryQuery) ([]*domain.LongTermEntry, error)
    List(ctx context.Context, q MemoryListQuery) (MemoryPage, error)
    Get(ctx context.Context, id string) (*domain.LongTermEntry, error)
    Update(ctx context.Context, id string, patch MemoryPatch) error
    Delete(ctx context.Context, id string) error
    Consolidate(ctx context.Context, scope MemoryScope) error // 合并去重
    Decay(ctx context.Context, scope MemoryScope, now time.Time) error
}

// UserProfileStore 用户画像：可编辑、可审计、字段级敏感度。
type UserProfileStore interface {
    Get(ctx context.Context, scope ProfileScope) (*domain.UserProfile, error)
    Update(ctx context.Context, scope ProfileScope, patch ProfilePatch) (*domain.UserProfile, error)
    ListFields(ctx context.Context, scope ProfileScope) ([]domain.ProfileField, error)
    DeleteField(ctx context.Context, scope ProfileScope, field string) error
}

// SessionStore 会话"元数据"存储，与 ShortTermMemory（消息流）职责分离：
// - ShortTermMemory 管的是"消息本体 + 摘要"；
// - SessionStore 管的是 Session 记录本身（标题、状态、summary_leaf 指针、token 统计、列表检索）。
// 二者可由同一后端实现（如都落文件/同一 DB），但契约分开，便于会话列表/管理端独立演化。
type SessionStore interface {
    Create(ctx context.Context, s *domain.Session) error
    Get(ctx context.Context, ref SessionRef) (*domain.Session, error)
    Update(ctx context.Context, ref SessionRef, patch SessionPatch) error // 更新 status/leaf/token/title
    List(ctx context.Context, q SessionListQuery) (SessionPage, error)     // 用户/管理端分页
    Delete(ctx context.Context, ref SessionRef) error
}
```

- **兼容策略**：现有 `ShortTermStore`（3 方法）作为 `ShortTermMemory` 的最小子集，Phase 1 先把 `react_loop` 对接 `ShortTermMemory`，`InMemoryStore` 升级实现新接口；旧接口在过渡期后删除（对齐"开发阶段不兼容旧代码"）。现有 `internal/app/session_service.go` 的 `NewSession`/`ClearSession` 改为委托 `SessionStore`。
- **依赖方向说明**：以上契约（`ShortTermMemory`/`LongTermMemory`/`UserProfileStore`/`SessionStore`/`MemoryExtractor`）与 `SessionRef` 等类型定义在 `internal/capabilities/memory/contract`，属**端口（接口）包**。运行时（`runtime/strategy/react` 已 import `memory.ShortTermStore`、`runtime/context` 的 Compactor/Assembler）依赖这些**契约端口**是合法的；被禁止的是依赖 `adapter/*` 具体实现与 `products/*`（见 §十一）。

## 六、会话管理：消息流、摘要、压缩、恢复

### 6.1 会话生命周期与状态

```text
create → active ⇄ compacting → idle → archived → deleted
```

- `create`：首次 RunOnce 隐式创建，或显式 `NewSession`。
- `active`：有进行中的 Run。
- `compacting`：正在做 auto-compact（防止并发 Run 读到半压缩态，加会话级锁）。
- `idle`：无活跃 Run，超过 TTL（短期记忆默认 7 天）可迁冷。
- `archived / deleted`：Enterprise 合规留存或用户删除。

**状态转换守卫**（每条转换边的触发条件 + 执行动作 + 错误处理）：

| 转换 | 触发条件 | 执行动作 | 错误处理 |
| --- | --- | --- | --- |
| `create → active` | 首次 Run 启动 | 原子写 Session 记录（CAS status） | 写失败则 Run 拒绝，返回错误 |
| `active → compacting` | MaybeAutoCompact 触发（使用率≥阈值）且获得会话 CAS 锁 | CAS `active→compacting`；成功才执行 LLM 摘要 | CAS 失败说明并发压缩已在进行，**跳过**本次 |
| `compacting → active` | 压缩完成（摘要写盘，leaf 指针更新） | CAS `compacting→active` | 异常时强制重置为 `active` + 告警，保证会话可继续使用 |
| `active → idle` | Run 正常结束 + 超过 `idle_after`（默认 30 min 无新 Run） | 定时器/结束钩子触发 CAS `active→idle` | — |
| `idle → active` | 用户发起新 Run（resume） | CAS `idle→active` | — |
| `idle → archived` | 超过 `archive_after`（默认 7 天）或显式归档 API | CAS + 冷存迁移（消息流转冷层） | 迁移失败保持 `idle` + 告警，下次重试 |
| `archived → active` | 用户显式恢复（restore API） | CAS `archived→active`；短期记忆从冷存重载 | 冷存不可用则报错，不静默降级 |
| `any → deleted` | 用户显式删除 | 软删标记 + 异步清理消息/记忆 | — |
| `compacting`（已在压缩中） | 第二个 Run 触发压缩判断 | **跳过**本轮压缩，继续正常 LLM 调用 | 下一轮重新判断使用率 |

### 6.2 消息语义类型 `Message.Kind`（契约；避免后续推翻）

> **问题**：API 协议层只有 `role`（system/user/assistant/tool）。Skill 全文、会话摘要、reminder 都合法使用 `role=user`（对齐 Kode/Codex），但它们**不是**用户亲手发送的问题。若无语义类型，企业 UI、压缩「保留最近用户轮」、长期记忆抽取都会误伤。

#### 6.2.1 命名决策（为何是 Kind，不是 Origin+Kind）

| 候选 | 结论 | 理由 |
| --- | --- | --- |
| **`Kind`（采用，必填）** | 消息**语义类型**，策略主开关 | 一张表即可推导：是否进模型、是否进聊天气泡、是否算真实用户轮、是否进 LTM |
| **`Origin`（不采用为必填）** | 曾考虑 `user/runtime/model` | 与 Kind **高度冗余**：`user_turn⇒user`、`skill_injection⇒runtime`、`assistant⇒model`；双字段易漂移（Kind=skill 却 Origin=user） |
| **`Source`（可选，审计）** | 细粒度生产者 | 如 `skill_gateway` / `skill_mention` / `compactor` / `prompt_composer`；**不**参与 UI/压缩主策略，仅审计/调试 |
| **`Displayable` 持久布尔（不采用）** | 是否展示 | 与 Kind 策略表重复；展示策略变更时要改历史数据。改为 **`IsChatVisible(kind) bool` 纯函数** |

**结论（冻结）**：

1. 契约主键：`domain.Message.Kind`（`MessageKind`）。  
2. 协议角色继续用现有 `Role`（发给 LLM 的 OpenAI/Anthropic 角色）。  
3. 内容标签（`<skill_injection>` / `<conversation-summary>`）保留作**可读双保险**，策略实现**禁止**只靠正则猜标签。  
4. 可选 `Source string`：细粒度出处；缺省为空。

#### 6.2.2 `MessageKind` 枚举与策略表

```go
// 落 internal/domain/message.go
type MessageKind string

const (
    MessageKindUserTurn             MessageKind = "user_turn"              // 真人输入
    MessageKindAssistant            MessageKind = "assistant"             // 模型回复（可含 tool_calls）
    MessageKindToolResult           MessageKind = "tool_result"           // role=tool
    MessageKindSkillInjection       MessageKind = "skill_injection"       // runtime 注入的 SKILL 全文（role=user）
    MessageKindConversationSummary  MessageKind = "conversation_summary"  // compact 回填摘要（role=user）
    MessageKindReminder             MessageKind = "reminder"              // 提示词分层 L4 turn reminder 等
    MessageKindSystem               MessageKind = "system"                // 稳定 system / 宿主诊断（如 skill_mention_error）
)

type Message struct {
    Role             RoleType
    Content          string
    ToolCalls        []ToolCall
    ToolCallID       string
    ReasoningContent string
    Kind             MessageKind // 必填（新消息）；读旧数据缺省时按 Role 回退推导，见下
    Source           string      // 可选：skill_gateway | skill_mention | compactor | ...
}
```

| Kind | 典型 Role | 进模型上下文 | 默认聊天气泡（ForUI） | 算「真实用户轮」（压缩保留） | 进 LTM 抽取 |
| --- | --- | --- | --- | --- | --- |
| `user_turn` | user | ✅ | ✅ | ✅ | ✅（按业务策略） |
| `assistant` | assistant | ✅ | ✅ 仅当 `Content` 非空；纯 `tool_calls` 中间步默认 ❌ | — | ❌ |
| `tool_result` | tool | ✅ | 摘要/可展开，默认不刷屏 | — | ❌ |
| `skill_injection` | user | ✅ | ❌ | ❌（摘要时可只记 skill 名进 snapshot） | ❌ |
| `conversation_summary` | user | ✅ | 可选一行「上下文已压缩」 | 特殊：作为摘要锚点保留 | ❌ |
| `reminder` | user | ✅ | ❌ | ❌ | ❌ |
| `system` | system | ✅（装配时） | ❌ / 诊断条 | — | ❌ |

兼容旧数据：JSONL/内存中 `Kind==""` 时回退——`RoleUser→user_turn`（保守：宁可多展示，不可把真人话藏掉）、`RoleAssistant→assistant`、`RoleTool→tool_result`、`RoleSystem→system`。**新写入必须带 Kind**。

#### 6.2.3 双投影（与企业「界面只看最终对话」对齐）

```text
ShortTermMemory / RunContext.Messages   ← 完整真相（含 Kind）
        │
        ├─ ForModel(msgs)  → ContextAssembler / LLM 输入：按预算保留全链；含 skill_injection / tool 轨迹
        └─ ForUI(msgs, UIPolicy) → 产品会话列表：默认 user_turn + assistant（+ 可选 summary）
                                   （调试/审计模式可开「完整轨迹」）
```

**统一（内核）**

| 层 | 职责 |
| --- | --- |
| `domain.Message.Kind` | 语义真源；`IsChatVisible` / `ForModel` / `ForUI` 纯函数 |
| `internal/runtime/transcript` | 投影端口 `Projector` + `UIPolicy`；默认实现委托 domain，供三端共用 |
| `react_loop` | 调用 LLM 前走 `transcript.ProjectForModel` |
| `app.ListSessionMessages` | 只返回完整链（EnsureKind）；**不**在 app 层写死 UI 策略 |

**独立（产品）**

| 产品 | 策略入口 | 当前状态 |
| --- | --- | --- |
| CLI | `transcript.DefaultCLIPolicy()` → `projectUIMessages` 水合 TUI | ✅ 已接：历史水合 + 本轮仍用 progress 流式渲染 |
| Desktop | 同 Projector，可换 `UIPolicy` / 自有气泡组件 | ⏳ 未实现 UI，契约已就绪 |
| Enterprise | `DefaultEnterprisePolicy` 或租户级策略注入 | ⏳ 未实现会话页，契约已就绪 |

- **企业应用默认 = ForUI**；**续聊/ReAct = ForModel**。二者同源，禁止维护两套互相漂移的存储。  
- Skill 注入：模型必须看见，UI 默认不可见——这就是「contextual」的产品含义。
- 纯 `tool_calls`、无 `Content` 的中间 assistant：**进 ForModel，默认不进 ForUI**（避免空气泡；本轮轨迹仍由 progress / 调试视图承载）。
- Desktop/Enterprise 后续只换 **UIPolicy / 渲染适配**，不改 ShortTerm 存储与 Kind。

#### 6.2.4 与 Skill / 提示词分层的边界（冻结，防回摆）

| 内容 | 放置 | Kind |
| --- | --- | --- |
| 人格 + 短硬规则（含 `skills_instructions`） | 初始 **system**（PromptComposer） | `system` |
| Catalog | Skill 工具 description | （非 Message） |
| SKILL.md 全文 + `skill_runtime_bridge` | **user** 消息，标签 `<skill_injection>` | `skill_injection` |
| Skill 网关短确认 | **tool** 结果 | `tool_result` |
| 禁止 | 整段替换初始 system；无 Kind 的裸 user 灌说明书 | — |

多 Skill 同 Run 累加：本阶段**允许**，由 §6.5 L1/L2 压缩回收；单活跃 Skill 窗口为可选增强，不阻塞本契约。

### 6.3 消息持久化（修正现状最关键的一处）

**现状**：`react_loop.persistRunSessionMessages` 已按 `SessionMessagesFromRun` 追加本 Run 增量全链（含 tool_calls 对应的 assistant、tool result、skill_injection；不含每轮重建的基线 system）。仍缺 JSONL envelope / leaf 摘要等（见 Phase 1 其余项）。

**目标**：短期记忆持续为完整消息链；文件/DB 后端带 envelope（`session_id / run_id / uuid / parent_uuid / role / kind / source? / …`）。

采纳 codex/Kode 的**追加式 JSONL**：每条消息一行，带 envelope：

```text
session_id / run_id / uuid / parent_uuid / role / kind / source? / app_id / agent_instance_id / created_at
```

```text
{sessionRoot}/{sessionID}.jsonl        # 主消息流（追加写）
{sessionRoot}/{sessionID}.summary.json # 当前滚动摘要 + leaf 指针
{sessionRoot}/{sessionID}/subagents/agent-{instanceID}.jsonl # 子智能体旁链（借鉴 Kode）
```

> JSONL 追加对文件后端天然友好、易调试、易迁移；DB 后端则映射为 `agent_message` 表（`docs/agent loop设计.md §11.3`，列需含 `kind`）。二者都实现同一 `ShortTermMemory` 契约。

### 6.4 恢复（resume / fork）

- **resume**：加载 `.summary.json` 得 leaf 指针，从 JSONL 中 leaf 之后的消息重建历史（借鉴 Kode `loadForResume` + codex `apply_rollout_reconstruction`）；leaf 之前用摘要代表（`Kind=conversation_summary`）。
- **fork**：复制到新 sessionID，指向同一 leaf，之后独立演化。
- 未压缩过的会话：leaf 为空，全量加载。
- 恢复后 `ForModel` / `ForUI` 仍按 Kind 投影，行为与在线一致。

### 6.5 两级压缩机制（科学的压缩：触发、保留、回填）

借鉴 Kode 双层 + codex 替换语义，分两级：

#### L1 — Micro-Compact（工具结果卸载，轻量、无 LLM）

- **目的**：大工具输出（文件读取、命令 stdout、检索结果）是 token 消耗大户，先就地瘦身，往往就能避免触发昂贵的 LLM 摘要。
- **触发**：单条工具结果 token > `tool_result_max_tokens`（默认 8k），或上下文使用率 ≥ `warn_ratio`（默认 0.75）。
- **策略**：把旧工具结果正文外置到 `{sessionRoot}/artifacts/`，原位替换为预览 + `<persisted-output ref="...">` 占位；**保留最近 `keep_last_tool_uses` 次（默认 3）**工具结果完整。
- **可逆**：占位保留 ref，模型或用户可按需重新读取（借鉴 Kode `toolResultPersistence`）。
- **范围**：主要针对 `Kind=tool_result`；**不**把 `skill_injection` 当工具结果外置（Skill 正文走 L2 摘要或保留集自然淘汰）。

#### L2 — Auto-Compact（对话摘要，LLM 生成）

- **触发**：估算的输入上下文使用率 ≥ `compact_ratio`（默认 0.85，即有效窗口的 85%）且历史轮数 ≥ `min_turns_to_compact`（默认 4，防抖）。
- **压缩多少轮 / 保留多少轮**（轮数是下界、token 是上限）：
  - 「轮」的边界以 **`Kind=user_turn`** 为准（**不是**任意 `role=user`）；`skill_injection` / `reminder` **不**单独算作用户轮。
  - 保留集 = 从最新往回，**至少**保留 `keep_recent_turns`（默认 6~10 轮，可配）以保证近期连贯；在满足该下界后继续往回多留，直到累计 token 触达 `keep_recent_token_budget`（默认有效窗口的 30%）这一**硬上限**为止。
  - 若仅 `keep_recent_turns` 轮就已超过 token 上限（近期消息很大），先对这些消息走 L1 micro-compact 瘦身，仍超则以轮数下界为准（保连贯优先）。
  - 其余"更老"的历史 **全部**送入摘要器，压成一段结构化摘要；其中旧 `skill_injection` 可被吞并，摘要模板可含可选「曾加载技能名」轻量快照（借鉴 Kode compaction skill snapshot，**不**保留全文）。
- **滚动摘要**：若已存在旧摘要，新摘要**吸收**旧摘要（rolling summary），避免摘要丢失更早信息（借鉴 codex summary_prefix 承接语义）。
- **摘要模板**：结构化分段（借鉴 Kode 8 段），落 `internal/runtime/context/templates/compact_prompt.md`，可被 Agent App / Hook 覆盖 `compact_instructions`：
  ```text
  技术上下文 / 任务目标 / 关键改动 / 调试与决策 / 当前状态 / 待办 / 用户偏好 / 关键结论
  ```
- **压缩用模型**：复用**已存在**的 `llm.yaml → llm.router.summarization`（默认 `fast`）选择便宜模型（借鉴 Kode compact 指针）。
- **回填（替换而非丢弃）**：
  - 内存：`history = [摘要消息] + [保留的最近消息]`；摘要作为一条带标记的消息（`role=user`，`Kind=conversation_summary`，`<conversation-summary>` 包裹，借鉴 codex 把摘要作 user 消息以便模型当"既有背景"）。
  - 持久层：写一条 `summary` 记录 + 更新 leaf 指针；**原始 JSONL 不删除**（可追溯、可 fork 回旧点）。
- **压缩后处理（可选）**：关键工件恢复——把最近读过的少量文件/工件重新注入（借鉴 Kode fileRecovery，受独立小预算约束），避免摘要丢失代码上下文。
- **长期记忆沉淀钩子**：压缩时"即将被摘要吞并的原始历史"是长期记忆抽取的天然输入点（§七.3）；抽取输入**过滤掉** `skill_injection` / `reminder` / 过大 `tool_result`。

#### 压缩编排接口

```go
// CompactionKind 压缩类型枚举（与 MessageKind 风格一致，禁用裸字符串）。
type CompactionKind string

const (
    CompactionKindNone  CompactionKind = "none"  // 未触发压缩
    CompactionKindMicro CompactionKind = "micro" // L1：工具结果卸载，无 LLM，幂等
    CompactionKindAuto  CompactionKind = "auto"  // L2：LLM 结构化摘要，持会话 CAS 锁
)

// Compactor 编排两级压缩，落 internal/runtime/context（内核，产品无关）。
//
// 并发安全约定：
//   - MaybeMicroCompact 可在每轮 LLM 调用前调用，幂等、无外部锁（仅修改 RunContext 内存态）。
//   - MaybeAutoCompact 通过 SessionStore CAS（active→compacting）实现分布式互斥；
//     同一 SessionRef 同时只能有一个 Auto 压缩在执行，并发请求直接返回 Triggered=false。
//   - 调用方（react_loop）须保证每轮调用顺序：
//     ① MaybeMicroCompact → ② 重新估算使用率 → ③ 若仍超阈才 MaybeAutoCompact；
//     确保 micro 卸载后使用率可能已低于 auto 阈值，避免不必要的 LLM 摘要。
type Compactor interface {
    // MaybeMicroCompact 在每轮 LLM 调用前尝试卸载大工具结果（就地、无 LLM、幂等）。
    MaybeMicroCompact(ctx context.Context, rc *runtime.RunContext) (Compaction, error)
    // MaybeAutoCompact 在使用率超阈时用 LLM 生成摘要并替换历史（CAS 会话状态，串行安全）。
    MaybeAutoCompact(ctx context.Context, rc *runtime.RunContext, ref SessionRef) (Compaction, error)
}

type Compaction struct {
    Triggered   bool
    Kind        CompactionKind // 使用类型安全枚举，禁止裸字符串
    TokensSaved int
    SummaryID   string // auto 时的 summary 记录 ID（对应 SessionSummary.LeafID）
}
```

- **react_loop 改造点**：每轮 `StreamGenerate` 前调 `MaybeMicroCompact`（函数 `runLoopIteration` 或等价入口）；micro 完成后重估使用率，超阈则调 `MaybeAutoCompact`。压缩期间通过 CAS 置会话 `compacting`，压缩完成后置回 `active`（见 §6.1 状态转换表）。

## 七、长期记忆：如何形成、存储与检索

长期记忆是本方案最大的子课题。核心纪律（对齐 `docs/agent loop设计.md §11.6`）：**框架不自动、无审批地把 Run 结论写进长期记忆**——"记什么"是业务决策；框架只提供**抽取器骨架、可配置提示词/结构、存储契约、读取注入**。

### 7.1 三种形成来源

| 来源 | 触发 | 内容 | 审批 |
| --- | --- | --- | --- |
| **用户显式** | `#记住...` 命令 / "记住这个偏好"工具（借鉴 Kode hashCommand） | 用户原话/偏好/项目约定 | 无需（用户主动） |
| **抽取器沉淀** | 会话压缩时 / Run 结束时 / 会话归档时（业务选择） | 从对话中提炼的经验、结论、失败教训 | 可配"直接入库"或"入待审队列" |
| **项目文档** | 目录链发现 `AGENTS.md`（借鉴 codex/Kode，与提示词分层 `ProjectDocInjector` 共用） | 项目级稳定约定 | 文件即真相，可编辑 |

### 7.2 两阶段抽取（借鉴 codex memories Phase1/Phase2）

```text
Phase 1 抽取（per-Run / per-压缩批次，异步）
  输入：被摘要吞并的原始历史 或 整个 Run 的消息链
       （过滤：跳过 Kind=skill_injection / reminder；过大 tool_result 先截断或只保留预览——对齐 Codex 跳过 contextual user）
  过程：用 extraction_prompt 让 LLM 产出候选记忆条目（结构化 JSON）
  输出：raw LongTermEntry[]（带 source_run_id / source_message_id 证据）

Phase 2 合并（周期性 / 阈值触发，异步）
  过程：与现有同 scope 记忆做相似度比较 → merge / supersede / 丢弃低价值
  输出：去重、更新 importance/confidence 后的稳定记忆集
```

- **异步**：抽取与合并都走后台（对齐 §11.6 方式 B），不阻塞对话响应。
  - **Phase 2 实现**：in-process goroutine + buffered channel（`MemoryExtractQueue`）；任务失败记录错误日志，不影响主流程；进程退出时未处理任务允许丢弃（下次压缩/Run 结束重触发）。
  - **Phase 3 替换**：持久化任务队列（Enterprise：PG job table / Redis Stream），支持重试、死信告警、多副本分发。
  - **待审队列**（`ReviewMode=review_queue`）：候选条目以 `Status=pending_review` 写入 `LongTermEntry`，不参与检索注入；需用户/管理端审批后置 `Status=active`；超时未审批可配置自动过期。
- **不同项目沉淀不同内容 → 可自定义抽取**：

```go
// MemoryExtractor 业务可自定义的抽取器（框架提供默认实现 + 骨架）。
type MemoryExtractor interface {
    // Extract 从一段消息中抽取候选长期记忆。
    Extract(ctx context.Context, in ExtractInput) ([]*domain.LongTermEntry, error)
}

// ExtractSpec 声明式配置：不同 Agent App / 项目可各自定义。
type ExtractSpec struct {
    Enabled       bool
    Trigger       string            // "on_compact" | "on_run_complete" | "manual"
    Prompt        string            // 自定义抽取提示词（决定"抽什么"）
    Schema        map[string]any    // 自定义 custom_data 结构（JSON Schema，决定"存成什么结构"）
    Scope         MemoryScope       // 沉淀到 user / agent_instance / workspace / project
    ReviewMode    string            // "auto" | "review_queue"
    MaxPerBatch   int               // 单批最多抽取条数（防噪声）
}
```

- **默认实现**：`LLMMemoryExtractor` 用内置通用抽取提示词（"提炼用户稳定偏好、项目约定、关键结论、失败教训，忽略一次性事实"）。
- **自定义**：`ExtractSpec.Prompt` + `Schema` 允许"客服项目沉淀常见问题解法"、"编码项目沉淀仓库约定与踩坑"，各存各的结构化 `custom_data`。
- **配置来源**：`app.yaml` 的 `memory.extract` 节 → `ExtractSpec`（与 Agent App `AgentSpec` 对齐，见 §九）。

### 7.3 检索与注入

```text
Run 启动：LongTermMemory.Search(MemoryQuery{
    Query: user_input, Scopes: [user, agent_instance, project],
    TopK: memory.inject_top_k(默认 5), MinConfidence: 0.5,
    Filters: {memory_type, sensitivity ≤ 允许级别},
    SortBy: MemorySortByComposite,  // 见下方注入排序策略
})  → 注入 system 的 <long_term_memories> 块（受 §八 预算 clamp）
```

- **检索退化链**：向量库(语义) → 无向量则关键词/标签/importance 排序（文件后端）。同一 `Search` 契约，后端能力不同。
- **敏感度过滤**：`secret/pii` 默认不注入（`docs/agent loop设计.md §11.9`）。
- **可选检索工具**：Enterprise 可暴露 `memory_search`/`memory_read` 工具让模型主动查（借鉴 codex ext/memories tools），而非只在启动时静态注入。

#### 7.3.1 LTM 注入内部排序策略（防御 Lost in the Middle 效应）

> **认知科学背景**：研究（arXiv:2307.x）表明，在多条独立信息/文档的检索任务中，模型对首位的注意力最强（71.2%），中间位置最弱（48.3%），末位次之（70.8%），呈 U 形曲线，中间性能差距高达 23%。
>
> **适用边界**：该效应对**独立事实列表**影响显著（LTM 条目正是此类场景），对**连贯叙事类内容**（对话历史）影响较小——对话历史有顺序语义关联，模型不做独立片段检索，因此不需要对历史做位置优化。LTM 注入顺序是真正需要防御的点。

```go
// MemorySortKey 检索结果排序维度枚举。
type MemorySortKey string

const (
    MemorySortByRelevance  MemorySortKey = "relevance"  // 纯语义相似度（向量后端）
    MemorySortByImportance MemorySortKey = "importance" // importance 字段（文件后端回退）
    MemorySortByRecency    MemorySortKey = "recency"    // 最近访问时间（动态记忆偏好）
    MemorySortByComposite  MemorySortKey = "composite"  // 推荐：复合得分（见下方公式）
)

// CompositeWeights 复合排序得分的权重参数。
// 默认值：w_rel=0.6, w_imp=0.3, w_rec=0.1
// 含义：语义相关性是主导因子；importance 是内容价值；recency 防止陈旧记忆压制新近相关记忆。
type CompositeWeights struct {
    Relevance  float64 // w_rel：语义相似度得分权重
    Importance float64 // w_imp：importance 字段权重
    Recency    float64 // w_rec：时效衰减因子权重
}

// MemoryQuery LTM 检索请求（完整定义）。
type MemoryQuery struct {
    Query            string
    Scopes           []MemoryScope
    TopK             int
    MinConfidence    float64
    Filters          MemoryFilters
    SortBy           MemorySortKey    // 默认 MemorySortByComposite
    // CompositeWeights 覆盖复合得分权重（nil 则用默认值）
    CompositeWeights *CompositeWeights
}

// 复合得分计算（在 LongTermMemory 实现内部）：
// composite_score = w_rel × relevance_score
//                 + w_imp × entry.Importance
//                 + w_rec × recency_factor(entry.LastAccessedAt)
// recency_factor = exp(-λ × days_since_access)，λ 默认 0.05（半衰期约 14 天）
```

**注入内部排序规则**：

- 检索返回 Top-K 条目后，按 `composite_score` **降序**排列再注入 `<long_term_memories>` 块。
- **得分最高的条目放首位**（利用首因效应，模型对该条目关注度最强）。
- 如果 LTM 预算充足、所有 Top-K 条目都注入，仍按得分降序线性排列。
- **互斥冲突处理**：存在 `supersedes` 关系时，只保留 `Status=active` 且得分较高的一方；被取代的旧条目不注入（无论其 importance 多高）。
- **不做首末交替置换**（书中将"最重要文档放首末"的做法仅适用于 RAG 多文档列表，LTM 条目是结构化分段而非文档列表，顺序排列更清晰）。



### 7.4 数据模型（落 `internal/domain`）

```go
// LongTermEntry 长期记忆条目（对齐 docs/agent loop设计.md §11.4，补齐字段）。
// - Scope 使用 MemoryScope 结构体（类型安全，替代原 ScopeType/ScopeID 裸字符串）。
// - MemoryType 使用 MemoryEntryType 枚举（类型安全，见 contract 包顶部定义）。
// - CustomData 写入时须通过 ExtractSpec.Schema 做 JSONSchema 校验，校验失败则丢弃该条目。
// - Status 支持 pending_review（待审队列），只有 active 条目才参与检索注入。
type LongTermEntry struct {
    ID            string
    TenantID      string
    Scope         MemoryScope     // 作用域（类型安全，替代原 ScopeType+ScopeID 裸字符串）
    MemoryType    MemoryEntryType // 记忆类型（类型安全枚举）
    Content       string
    CustomData    map[string]any  // 业务自定义结构（由 ExtractSpec.Schema 约束，Save 时校验）
    Embedding     []float32       // 向量后端写入；文件后端为空
    Importance    float64  // 0~1
    Confidence    float64  // 0~1
    Status        string   // active / pending_review / superseded
    SensitivityLevel string // public/internal/confidential/secret/pii
    DecayPolicy   string   // none/time_decay/access_decay/custom
    Tags          []string
    SourceRunID   string
    SourceMessageID string
    SupersedesID  string   // 被本条目取代的旧条目 ID
    LastAccessedAt time.Time
    ExpiredAt     *time.Time
    domain.RuntimeAudit
}

// UserProfileBuiltin 内置画像字段（强类型结构体，保证 IDE 提示与编译期检查）。
// 新增内置字段须修改本结构体并同步更新持久化 schema；自定义业务字段用 UserProfile.CustomFields。
type UserProfileBuiltin struct {
    Locale             string   // BCP-47 语言标签，如 zh-CN / en-US
```

### 8.2 预算计算公式

分两类段：**刚性段**（必须完整放入，否则任务无法进行）与**弹性段**（可伸缩，按预算取舍）。

```text
# 第 1 步：可用输入预算
usable          = context_window × effective_context_ratio        # 扣协议/tokenizer 误差头
output_reserve  = clamp(output_reserve_tokens or max_tokens, MIN_OUT, MAX_OUT)  # 为模型回复预留
input_budget    = usable − output_reserve                          # 全部输入段共享的总预算

# 第 2 步：先满足刚性段（按实际测量值扣除）
# 注意：system_prompt 这里只含"稳定 prompt 块"（L0~L2 persona/instructions/安全/工具协议），
#       不含 LTM/摘要/历史等动态内容——后者是第 3 步的弹性段（见 8.4 分工说明）。
rigid     = tok(stable_system_blocks) + tok(tools_schema) + tok(current_user)
remaining = input_budget − rigid
若 remaining < 0 → 降级（见 8.5）

# 第 3 步：先扣安全缓冲，再把可分配额度按"目标比例 + 区间 clamp + 额度回流"分给弹性段
safety_buffer = remaining × buffer_ratio            # 默认 5%，吸收估算误差，永不分配
distributable = remaining − safety_buffer
弹性段 = { history(含本 Run 工作观察), summary, long_term_memory }   # 三段权重之和 = 100%
for each seg:
    target_i = distributable × weight_i
    alloc_i  = clamp(target_i, min_i × distributable, max_i × distributable)  # 区间限制
# 回流：clamp 后（命中 max 或实际内容不足）会有结余；
#       结余按"未达 max 的可扩张段"权重再分配，优先补给 history（最近最相关），迭代至收敛
reflow(结余) → 优先 history → 再 summary → 再 ltm
```

**默认权重与区间**（占 `distributable` 的比例，三段权重之和 = 100%，全部可配）：

| 弹性段 | 目标权重 | 最小 | 最大 | 说明 |
| --- | --- | --- | --- | --- |
| `history`（含工作观察） | 60% | 20% | 90% | 最近对话最相关，回流优先补给它 |
| `summary` | 15% | 0% | 30% | 有摘要才占用；无摘要额度回流 |
| `long_term_memory` | 25% | 0% | 40% | 检索命中才占用；受 TopK 与 confidence 双限 |

> `safety_buffer`（`remaining × 5%`，可配 5%~10%）在分配前独立扣除，**不参与**三段权重，故三段权重严格求和为 100%，避免"权重不足 100% 导致预算浪费"。
>
> **关键**：这正是用户要的"不严格按比例、某部分少了另一部分可以多些、但有区间限制"——通过 **clamp（区间）+ reflow（回流）** 实现。例如某会话还没有长期记忆和摘要，`history` 可回流吃到 90% 上限；反之长期记忆很多时，`ltm` 最多到 40%，多出的挤不进去（保证 history 不被饿死）。

#### 8.2.1 场景化预算权重配置（Context Profile）

> **设计动机**：不同类型的 Agent（RAG 知识库问答 / 纵向对话 / 编码 Agent）在弹性段需求上差异显著——RAG Agent 需要大量 LTM/文档预算，对话 Agent 则以历史为主。通过命名 Profile 配置权重默认值，再由 `clamp + reflow` 动态适应实际内容量，比硬编码百分比表更优雅。
>
> **与 Session.ContextStrategy 的关系**：`domain.Session.ContextStrategy`（已有字段）存储 Profile 名称；`ContextBudgetPlanner` 在 Run 启动时将其解析为具体权重表。两者职责分离，字段先行、实现后置。

`config.yaml` 中初始化 Profile 默认值（全部可被 Agent App 级配置覆盖）：

```yaml
context_profiles:
  default:                    # 通用默认（无特定 Agent 类型）
    weights: { history: 0.60, summary: 0.15, ltm: 0.25 }
    clamp:
      history: [0.20, 0.90]
      summary: [0.00, 0.30]
      ltm:     [0.00, 0.40]

  rag:                        # RAG 重检索 Agent（知识库问答、文档搜索）
    weights: { history: 0.30, summary: 0.10, ltm: 0.60 }
    clamp:
      history: [0.15, 0.60]
      summary: [0.00, 0.20]
      ltm:     [0.20, 0.80]  # ltm 最小保证 20%（RAG 场景无命中时发出告警）

  chat:                       # 纵向对话 Agent（客服、助手、聊天）
    weights: { history: 0.75, summary: 0.15, ltm: 0.10 }
    clamp:
      history: [0.40, 0.95]
      summary: [0.00, 0.30]
      ltm:     [0.00, 0.20]

  coding:                     # 编码 / 代码 Agent（工具多、任务长、摘要价值高）
    weights: { history: 0.55, summary: 0.20, ltm: 0.25 }
    clamp:
      history: [0.20, 0.85]
      summary: [0.05, 0.35]  # 编码 Agent 摘要保底 5%（工具链摘要价值高）
      ltm:     [0.00, 0.40]
```

**为什么不用书中的静态百分比表**？

书中 `rag_heavy: docs=50%, history=15%` 是预先分配——当某次运行 LTM 无命中时，50% 的预算闲置浪费。我们的 Profile 只是**权重默认值 + 允许区间**，真实分配由 `clamp + reflow` 按实际内容量驱动，LTM 无命中时预算自动全部回流给 history，不存在浪费。

Profile 的**三种启动方式**（按优先级从低到高）：

1. **配置文件默认**：Agent App 的 `agent_spec.yaml` 中配置 `context_profile: coding`，作用于该 App 下所有新会话。
2. **会话创建时指定**：`NewSession(spec)` 时 `spec.ContextStrategy = "rag"`，写入 Session 记录。
3. **运行时覆盖**：管理端/Hook 通过 `SessionStore.Update(patch)` 动态修改，立即生效于下一次 Run。

> **注意**：Profile 修改只影响**预算分配权重**，不改变 `ContextAssembler` 的消息装配顺序和 `CompactionKind` 的触发阈值——这些是全局统一配置，与 Profile 无关。

### 8.3 历史记录的取用规则（条数 + 长度 + 占比）


`ShortTermMemory.GetRecent` 在 `history` 段预算内从新到旧取：

- **条数上限**：`max_history_turns`（默认 40 轮），防止极多短消息塞爆。
- **token 预算**：`history` 段分配到的 token（§8.2）。
- **单条最大长度**：单条消息超 `per_message_max_tokens`（默认 4k）时，先走 micro-compact 卸载（§6.5 L1），而非整条丢弃。
- **取用顺序**：从最新消息回溯累加，直到触达条数或 token 上限；**保留完整 tool 链的成对性**（assistant.tool_calls 与其 tool result 不拆散）。
- **用户轮定义**：计数与「首条保护」仅针对 `Kind=user_turn`；`skill_injection` / `reminder` / `conversation_summary` 不计入「用户轮数」，但可按预算随邻近 tool 链一并保留或淘汰。
- **首条保护**：会话首条 **`user_turn`**（任务定义）在有摘要时由摘要代表，无摘要时优先保留（借鉴 codex/§4.8）。

#### 8.3.1 内容感知的截断策略（TruncateStrategy）

> **设计动机**：单条消息超过 `per_message_max_tokens` 时，必须就地裁剪。一刀切的截断方式（如纯截头或纯截尾）会系统性地损失特定类型内容的关键信息。不同内容类型的信息分布规律完全不同，截断策略必须按 `Kind` 区分。
>
> **认知科学依据**：Lost in the Middle 研究表明，对于独立信息片段列表，中间段注意力最弱。但对于 **顺序叙事内容**（对话历史）和 **工具执行结果**，最优截断逻辑截然不同。照搬"首尾各45%"是对该研究的误用——那个策略仅适合长篇独立文档，不适合工具输出（结果在尾部）或对话历史（应删最老的整轮）。

```go
// TruncateStrategy 单条消息超限时的就地截断策略枚举。
// 落 internal/runtime/context/truncate.go，与 TokenEstimator 同包。
type TruncateStrategy string

const (
    // TruncateHeadTail 保留首尾各 HeadRatio/TailRatio 比例，丢弃中间。
    // 适用：user_turn（长输入）、conversation_summary（摘要正文）、长篇 LTM Content。
    // 原理：长文本中间段（背景展开、论证过程）的模型注意力最弱；首尾包含主题/结论。
    // HeadRatio 默认 0.5（对称），可调。
    TruncateHeadTail TruncateStrategy = "head_tail"

    // TruncateTailOnly 只保留尾部最多 budget 个 token，截去头部。
    // 适用：tool_result（命令输出、代码运行结果）、assistant（长回复）。
    // 原理：命令行/代码执行结果的关键信息（最终输出、错误栈）在最后几行；
    //        截去头部（中间过程日志）损失最小。
    TruncateTailOnly TruncateStrategy = "tail_only"

    // TruncateTurnBoundary 按对话轮边界删除最老的完整 user+assistant 对。
    // 适用：conversation_history 整体超限时（由 GetRecent 内部使用）。
    // 原理：对话是顺序叙事，不能在消息对中间截断；最老的轮次价值最低。
    // 注意：此策略作用于消息列表层面，而非单条消息，是 GetRecent 的淘汰策略。
    TruncateTurnBoundary TruncateStrategy = "turn_boundary"

    // TruncateSectionAware 保留完整 Markdown 段落，从优先级最低的段落删起。
    // 适用：skill_injection（SKILL.md 全文注入）、structured_prompt 块。
    // 原理：结构化内容内部各段之间有语义依赖（指令→示例→约束），
    //        任意位置截断会破坏逻辑；应按段落优先级整段删除。
    // 实现：按 ## / ### Markdown 标题分段，各段可配置 priority（整数，越小越先删）；
    //        未配置的段默认 priority=50（居中）。
    TruncateSectionAware TruncateStrategy = "section_aware"
)

// TruncateConfig 截断策略的完整配置。
// 落 MessageKindConfig（每种 Kind 一份，通过 config.yaml 覆盖默认值）。
type TruncateConfig struct {
    Strategy     TruncateStrategy
    HeadRatio    float64         // 仅 TruncateHeadTail 有效；默认 0.5（对称截断）
    // SectionPriority 仅 TruncateSectionAware 有效；key=段标题关键词，value=优先级整数
    // 示例：{"## Examples": 10, "## Instructions": 90}（10 先被删，90 最后被删）
    SectionPriority map[string]int
}
```

**各 Kind 的默认截断策略对照表**（可通过 `config.yaml` 全局覆盖）：

| Kind | 默认策略 | HeadRatio | 原因 |
| --- | --- | --- | --- |
| `tool_result` | `TruncateTailOnly` | — | 尾部 = 执行结果/错误栈，是模型最需要的部分 |
| `user_turn`（长输入） | `TruncateHeadTail` | 0.5 | 首部=问题上下文，尾部=具体问题，中间=背景展开 |
| `assistant`（长回复） | `TruncateTailOnly` | — | 尾部=最终结论/行动，比开头的「好的，我来……」更有价值 |
| `skill_injection` | `TruncateSectionAware` | — | 段落结构有语义依赖，整段删除比任意截断损失少 |
| `conversation_summary` | `TruncateHeadTail` | 0.6 | 尾部摘要更新鲜，头部保留任务定义；中间相对次要 |
| `reminder` | `TruncateTailOnly` | — | reminder 通常是追加的条目，末尾最具时效性 |

**与 L1 micro-compact 的职责分工**：

- `TruncateStrategy` 是**单条消息内容就地裁剪**：一条 tool_result 太长，直接截取尾部。操作对象是 `string`，无 LLM 调用，无 I/O，纯内存操作。
- L1 micro-compact 是**将工具结果外置到文件存储并留占位符**：管理的是多条消息累积占用的总量问题，是「卸载」而非「裁剪」，会产生持久化 I/O。
- **两者互补不重叠**：先尝试 `TruncateStrategy` 就地裁剪（零成本）；若仍超过 `per_message_max_tokens`，则触发 L1 micro-compact 外置；若整个 history 段仍超限，由 `TruncateTurnBoundary` 在 `GetRecent` 层删最老轮次。

### 8.4 上下文装配顺序（ContextAssembler）

```text
最终 messages（对齐 docs/agent loop设计.md §4.8 与 §11.7）：
  [1] system      = PromptComposer 产出（含 long_term_memories / user_profile 块，见提示词分层方案）；Kind=system
  [2] history     = summary（Kind=conversation_summary，若有）+ GetRecent(history 段预算) 裁剪后的历史
                    （可含 skill_injection / tool 轨迹；按 Kind 计入预算）
                    ⚠ GetRecent 返回已持久化的跨 Run 历史，不含本 Run 新增消息，与 [3] 无重叠。
  [3] working_obs = RunContext.Messages 中本 Run 当前已产生的工具轨迹（micro-compact 后）
                    ⚠ 来源：RunContext.Messages 是本 Run 运行中累积的消息，尚未持久化，与 [2] 严格互斥。
                      装配时按顺序追加 [2]+[3]，不做去重，也不混合。
  [4] user        = 当前用户输入（Kind=user_turn）+ 可选 reminder（Kind=reminder，提示词分层 L4）
```

产品侧另提供 **`ForUI(history)`** 投影（§6.2.3）：默认不含 `skill_injection` / `reminder` / 原始 tool dump。

> **分工（消除两方案职责重叠）**：
> - `ContextBudgetPlanner`（本方案）**先**算出每个弹性段（LTM/摘要/历史）各能占多少 token。
> - `PromptComposer`（提示词分层方案）负责 system 段的分层 Block 拼装；LTM 块、画像块虽在文本上位于 system 消息内，但其**内容量**由 Planner 下发的 LTM 预算约束（Composer 只按给定预算渲染，不自行决定塞多少）。
> - `ContextAssembler`（本方案）负责 history/摘要/工作观察的裁剪与最终 messages 拼装。
> - 因此 §8.2 把"稳定 system 块"算作刚性、把 LTM/摘要/历史算作弹性，与"它们最终渲染在哪条消息里"是两个正交维度：**预算归 Planner，渲染位置归 Composer/Assembler**。
>
> **命名对齐**：本方案的 `ContextAssembler` + `ContextBudgetPlanner` 合起来即 `docs/agent loop设计.md §4.8/§11.7` 所称的 `ContextBuilder`；本方案将其拆为"预算"与"装配"两个职责更单一的组件。

### 8.5 溢出降级（刚性段都放不下时）

按顺序尝试，每步有 trace/audit：

1. 强制 auto-compact（把历史压到最小保留集）。
2. 裁剪工具 schema（仅保留本轮可能用到的工具，借鉴按需暴露）。
3. 缩减长期记忆/摘要到各自 min。
4. 当前 user 输入过长 → 提示用户拆分，或对输入本身做摘要（谨慎，保留原文引用）。
5. 仍不行 → 明确报错（不静默截断关键信息，满足 §2.3 不变量 3）。

### 8.6 Token 估算（TokenEstimator 端口）

```go
// TokenEstimator Token 估算端口（预算决策用；实际计费以 provider usage 为权威）。
// model 参数允许按模型选择正确 tokenizer（如 cl100k_base for GPT-4o、qwen for Qwen 系列）；
// 实现可选忽略 model 参数（HeuristicEstimator 统一启发式），但接口须兼容以便未来替换。
type TokenEstimator interface {
    // Estimate 估算单段文本的 token 数。
    Estimate(ctx context.Context, text string, model string) int
    // EstimateMessages 批量估算消息列表（含 role/分隔符 per-message overhead，比逐条调用 Estimate 更准）。
    EstimateMessages(ctx context.Context, msgs []*domain.Message, model string) int
}
```

- 默认 `HeuristicEstimator`：借鉴 codex `bytes/4` + Kode 开销系数；**CJK 加权**（中文字符按 ~0.6~1 token/字估算，比纯 bytes/4 更准）+ 每消息固定 overhead（role/分隔符）。
- 可选 `TiktokenEstimator`（Enterprise 精确计费）、`ProviderUsageEstimator`（用 API 返回的 usage 校准，作权威计费；估算仅用于"装之前"的预算决策）。
- **接线**：`RunContext.AddTokens`（现有但未调用）在每次 LLM 调用后用 provider usage 更新；预算决策用 estimator 的估算值（借鉴 Kode/codex 双轨）。

## 九、多智能体兼容（对齐 Agent App 三层模型）

`docs/Agent App ...md` 定义了 L1（App 内委派子智能体）/ L2（App 内 Workflow 编排）/ L3（App 间 CollaborationSpace）。会话与记忆**当前阶段只需预留维度、正确隔离，不实现编排**。

### 9.1 Session 模型扩展

```go
// domain.Session 扩展（现状仅 6 字段）
type Session struct {
    ID        string
    TenantID  string
    UserID    string
    AgentID   string   // 现有
    AppID     string   // 新增：Agent App（哪个应用发起的会话）
    Title     string
    // 上下文/压缩元数据
    Status       string // active/idle/compacting/archived
    SummaryLeaf  string // 压缩边界指针（恢复用）
    TotalTokens  int64  // 累计 token（计费/展示）
    ContextStrategy string // sliding_window/summary/selective（§4.8，Agent 可配）
    // 时间
    CreatedAt    time.Time
    LastActiveAt time.Time
}
```

### 9.2 隔离维度：`SessionRef` 与 `MemoryScope`

- **会话内多智能体**：一个 Session 可涉及主 Agent + 子智能体（L1）或 Workflow 多角色（L2）。消息 envelope 记 `agent_instance_id`；子智能体消息走**旁链 JSONL**（`{sessionID}/subagents/agent-{id}.jsonl`，借鉴 Kode），主流只保留委派摘要，避免子链爆主上下文（对齐 `docs/子智能体设计.md` 预算熔断）。
- **记忆隔离**：`MemoryScope` 支持 `user / agent_instance / workspace / project`。不同 Agent 实例默认记忆隔离（`scope=agent_instance`）；项目级经验用 `scope=project` 跨 Agent 共享（需业务显式声明）。
- **L3 协作空间**：多个独立 App 成员的会话/记忆由 CollaborationSpace（`internal/runtime/multiagent`，Phase 3+）统筹；本方案只保证 `SessionRef`/`LongTermEntry` 带 `app_id`/`scope` 字段，L3 落地时无需迁移数据模型。

### 9.3 兼容原则

- 单 Agent（当前默认 `code` 应用）：`app_id`/`agent_instance_id` 可为空或取默认，行为等价现状。
- 字段先行、编排后置：数据模型一次到位，编排逻辑分阶段接入，避免未来改表/改协议。

## 十三、关键决策摘要

| 决策点 | 结论 |
| --- | --- |
| 记忆分层 | 四层：工作(RunContext) / 短期(Session 对话) / 长期(跨会话经验) / 画像；生命周期与存储各异 |
| 消息语义类型 | **`Message.Kind` 必填**（user_turn / assistant / tool_result / skill_injection / conversation_summary / reminder / system）；**不**采用必填 Origin；可选 Source 仅审计 |
| 双投影 | `ForModel`=完整链（含 Skill）；`ForUI`=默认聊天气泡（藏 skill_injection/reminder）；同源存储、禁止双写 |
| 消息持久化 | 存**完整消息链**（含 tool_calls/tool result/reasoning/**skill_injection**），追加式 JSONL（envelope 含 kind）+ 摘要 leaf 指针；修正现状只存 user+final |
| 会话恢复 | 按 summary leaf 截断重建（resume）；支持 fork；原始历史不删可追溯 |
| 会话状态机 | 7 个状态，每条转换边有 CAS 守卫 + 错误处理；`compacting` 并发冲突直接跳过不等待；`archived→active` 不静默降级 |
| 压缩机制 | 两级：micro-compact(工具结果外置，无 LLM，幂等) + auto-compact(LLM 结构化摘要，CAS 互斥)；摘要"替换非丢弃"；**用户轮边界=Kind=user_turn** |
| 压缩调用时序 | react_loop 每轮：① MaybeMicroCompact → ② 重估使用率 → ③ 若仍超阈才 MaybeAutoCompact；确保 micro 卸载后可跳过昂贵 LLM 摘要 |
| 压缩触发 | 使用率 ≥ compact_ratio(默认 0.85) 且轮数 ≥ min_turns；micro 在 warn_ratio(0.75) 触发 |
| 保留/压缩多少 | 保留最近 keep_recent_turns(默认 6~10) 且不超 keep_recent_token_budget；其余全部摘要（轮数+token 双约束）；旧 skill 全文可被摘要吞并 |
| Skill 与历史 | Skill 全文 `role=user` + `Kind=skill_injection`；短硬规则留 system；多 Skill 累加先靠压缩回收 |
| 模型最大上下文 | 配置新增 `context_window` 字段（私有化模型必填）+ `effective_context_ratio`(默认 0.92) |
| 预算公式 | usable=窗口×有效比例；input_budget=usable−输出预留；刚性段优先，弹性段按权重 clamp + 回流 + safety_buffer |
| 比例弹性 | "目标权重 + [min,max] 区间 clamp + 结余回流(优先 history)"——不严格按比例但有区间限制 |
| 历史取用 | 条数上限 + token 预算 + 单条超限先就地截断(TruncateStrategy)再考虑 micro-compact + tool 链成对保留 + **user_turn** 首条保护 |
| 上下文装配分区 | [1]system / [2]history(已持久化跨Run历史) / [3]working_obs(RunContext本Run工作轨迹,与[2]严格互斥) / [4]user；三段严格不重叠 |
| 类型安全 | `MemoryScopeType`/`MemoryScope` struct 替代裸 ScopeType+ScopeID；`MemoryEntryType` 替代裸 string；`CompactionKind` 替代 "micro"/"auto" 字符串；与 `MessageKind` 风格一致 |
| 用户画像结构 | `UserProfileBuiltin` 强类型内置字段（locale/style/timezone等）+ `CustomFields map[string]any`；两者独立持久化，互不影响 |
| 会话摘要模型 | `SessionSummary`(Content/LeafID/TokensCount/Iteration)；Rolling Summary = 旧摘要全文注入 {previous_summary} 占位，LLM 产综合新摘要 |
| LTM 异步边界 | Phase 2：in-process goroutine + buffered channel，进程退出可丢失；Phase 3：持久化任务队列；pending_review 条目不参与检索，需审批后置 active |
| 长期记忆形成 | 三来源(用户显式/抽取器/项目文档)；两阶段(抽取→合并) 异步；框架不自动无审批写入；**抽取跳过 skill_injection**；CustomData Save 时 JSONSchema 校验 |
| 自定义抽取 | `ExtractSpec`(prompt+schema+scope+review) 支持不同项目沉淀不同结构化内容 |
| Token 估算 | `TokenEstimator` 端口含 `model` 参数（支持按模型选 tokenizer）；默认 Heuristic(CJK 加权)，可选 tiktoken/provider usage 双轨 |
| 三产品 | 内核同源；存储/检索/token/画像后端经端口注入差异化；Enterprise 禁宿主机绝对路径 + RBAC + pii 不注入 |
| 多智能体 | Session/Message/Memory 预留 app_id/agent_instance_id/scope；子智能体旁链；字段先行编排后置 |
| 与提示词分层 | PromptComposer 管 system 段分层块；本方案 ContextAssembler 管 history/memory/预算；两方案对接不重叠 |
| 复用现有路由 | 压缩模型复用已存在的 `llm.yaml llm.router.summarization` |
| LTM 注入排序 | 按 `composite_score`（语义×0.6 + importance×0.3 + 时效×0.1）降序，得分最高条目放首位（首因效应）；superseded 条目不注入；不做首末交替置换（LTM 是分段而非文档列表）|
| Context Profile | 命名 Profile（default/rag/chat/coding）配置弹性段默认权重+区间，由 Session.ContextStrategy 选择；clamp+reflow 动态适应实际内容量，比静态百分比表不浪费空预算 |
| 截断策略多态 | `TruncateStrategy`（head_tail/tail_only/turn_boundary/section_aware）按 Kind 配置默认策略；tool_result 用 tail_only，skill_injection 用 section_aware，对话历史用 turn_boundary；与 L1 micro-compact 互补不重叠 |

## 十四、残留风险（不阻塞 Phase 1）

1. **摘要质量与信息丢失**：结构化模板 + 关键工件恢复缓解，但仍需在真实长会话上验证；建议保留原始 JSONL 以便回退/评估。
2. **Token 估算误差**：Heuristic 对代码/多语言/工具 JSON 误差较大，靠 safety_buffer + provider usage 校准兜底；Enterprise 上 tiktoken 提精度。
3. **CJK 估算系数**需按实际模型 tokenizer 标定（不同模型中文分词差异大）。
4. **长期记忆污染/隐私**：抽取默认可走 review_queue；`secret/pii` 分级与 RBAC 需与统一权限治理（`docs/统一配置权限与审批治理设计.md`）对齐后定稿。
5. **压缩与并发 Run**：同会话并发 Run 需会话锁 + `compacting` 状态防脏读；多副本部署下需分布式锁（Enterprise Phase 3）。
6. **差量注入**（Phase 3）与摘要/历史序列的交互需验证不破坏对话连续性（借鉴 codex 需谨慎处理 compaction 后强制全量 reinject）。
7. **Kind 回填**：旧会话无 Kind 时按 Role 保守回退为 `user_turn`；若历史上曾把 skill 全文当裸 user 持久化，回退后可能短暂进 ForUI——需一次性迁移脚本或按 `<skill_injection>` 标签补标（仅迁移期）。

## 附录 D：2026-07-15 未完成项（明确不在本轮处理）

以下事项因涉及 DB/外部基础设施或产品认证体系，本轮不实现；当前文件后端已经通过契约隔离，可在不改运行时核心算法的前提下替换：

1. **Enterprise DB/向量后端**：`SessionStore`、`ShortTermMemory`、`LongTermMemory`、`UserProfileStore` 仍未提供 PostgreSQL/Redis/pgvector/Qdrant adapter；文件 LTM 仅关键词排序，不能承担大规模语义检索。
2. **多副本可靠抽取队列**：当前 `MemoryExtractWorker` 是进程内队列，进程退出时未消费任务可丢失；需替换为 PG job table 或 Redis Stream，并补重试、死信和监控。
3. **Enterprise 身份与租户来源**：HTTP 请求暂允许 `user_id` 作为开发期身份占位，`tenant_id` 仍为 `dev`；生产环境必须由认证中间件/JWT 注入可信身份，禁止采信客户端声明的租户或用户。
4. **分布式会话锁与原子持久化**：文件实现只覆盖单进程互斥；多副本运行需要以租户+会话为键的分布式锁、数据库事务和乐观版本控制。
5. **产品管理入口**：app 层已经提供会话回放/fork、长期记忆与画像管理用例；Enterprise HTTP 和 Desktop bridge/UI 暂未接入。
6. **Token 精度与硬限校准**：当前使用启发式估算器，并在 Assembler 后执行最终硬限；仍需用 provider usage 校准模型 tokenizer，并在请求层补 provider-aware 的精确复核。
