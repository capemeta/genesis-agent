# 密钥与连接管理设计

> 本文档描述 Genesis Agent 的密钥、凭证与业务连接管理方案。当前阶段不接入 K8S，但设计上预留 K8S Secret、Vault、云密钥管理服务等实现。

---

## 1. 结论

密钥管理应作为系统级能力建设，而不是散落在 `skill`、工具参数、提示词、`config.yaml` 或业务代码里。

当前阶段推荐方案：

- 底层定义统一的 `CredentialStore` 与 `ConnectionStore` 接口。
- 第一版使用本地加密文件存储，后续可替换为 DB、K8S Secret、Vault、云 Secret Manager。
- Agent、Skill、Tool 运行时只使用 `credential_ref` / `connection_ref`，不直接持有真实密钥。
- `http_request` 工具保持通用 HTTP 能力，但业务接口调用优先通过 `connection_ref` 注入认证与默认配置。
- 密钥管理做成系统能力，提供后台菜单、HTTP API 和 CLI 三种入口；当前第一版先落 HTTP API + 文件存储，后台菜单和 CLI 随后补。

一句话：

> 密钥真实值由系统保管，Skill 只描述业务接口，Agent 只引用连接，Tool 在执行前由运行时安全注入凭证。

## 3. 设计目标

### 3.1 要解决的问题

1. **不能把密钥写进 Skill**
   - Skill 可能被导入、导出、分享、审查、版本化。
   - 如果 Skill 内含真实 token，泄露面会非常大。

2. **不能让模型直接长期持有明文密钥**
   - 模型上下文、日志、Trace、工具参数都有泄露风险。
   - 模型只应看到 `connection_ref` 或必要的非敏感描述。

3. **不能每接一个业务接口都改平台代码**
   - 业务接口千差万别，平台应通过连接配置、接口描述和 Skill 组合扩展。

4. **当前不能依赖 K8S**
   - 本项目当前主要是本地开发 / 单机 / 普通部署阶段。
   - 但接口必须为未来 K8S / Vault / 云原生部署留好替换点。

### 3.2 非目标

当前阶段不做：

- K8S Secret Controller。
- Vault 动态凭证签发。
- OAuth 全自动授权码流程。
- 透明 MITM Credential Vault。
- 沙箱 egress sidecar 凭证代理。
- 多租户细粒度 RBAC 全量实现。

这些能力可进入 Phase 2.5 或云原生阶段。

---

### 3.3 设计反思与边界修正

这套设计的核心边界是正确的，但需要明确几个原则，避免后续实现走偏：

1. **底层 Store 不做业务判断**
   - `CredentialStore` 只负责安全存取密钥。
   - `ConnectionStore` 只负责存取连接配置。
   - 是否允许某个 Agent / Tool / Skill 使用某个连接，应由上层治理和 resolver 判断。

2. **Resolver 是运行时安全入口**
   - `CredentialResolver` 不是简单查库工具，而是运行时凭证注入的唯一入口。
   - 它应接入权限、审计、用途校验、环境校验和 allowlist。

3. **Connection 是业务集成边界**
   - Skill 不直接写 `base_url + auth`，而是引用 `connection_ref`。
   - 业务接口可以通过 Skill / Connector 描述，但认证和默认连接配置仍由 Connection 管。

4. **本地文件只是第一版实现，不是领域模型**
   - 文件、K8S Secret、Vault、云 Secret Manager 都只是 adapter。
   - 上层不能感知当前密钥到底来自文件还是 K8S。

5. **明文只在极短运行窗口出现**
   - 创建、轮换、Resolve 时短暂出现。
   - 不落日志、不进 Trace、不进 Skill、不进 Prompt、不进入持久化普通字段。

### 3.4 威胁模型

第一版设计至少要防住这些常见泄露路径：

- Skill 被导出或共享时泄露密钥。
- Agent 工具参数进入日志或 Trace 时泄露密钥。
- LLM 上下文中出现真实 token。
- 开发者误把 `config.local.yaml`、`.env`、测试样例提交到仓库。
- 普通用户通过列表接口看到密钥明文。
- 某个 Agent 知道 `connection_ref` 后越权调用其他业务系统。
- 本地文件被复制走后直接解密。

对应约束：

- 所有展示接口只返回脱敏元数据。
- `Resolve` 必须带 `purpose`、调用主体和运行上下文。
- 审计日志记录引用和用途，不记录明文。
- 文件存储必须加密；生产环境主密钥必须来自部署系统或外部密钥管理。
- 后续治理层必须校验 tenant / project / agent / tool / skill / environment。

## 3. K8S Secret 的设计启发

K8S 的 Secret 体系给我们的启发不是“现在就必须接 K8S”，而是几个重要原则：

1. **配置与密钥分离**
   - Deployment 引用 Secret 名称。
   - 真实密钥值由 Secret 对象保存。

2. **运行时注入**
   - Secret 可以以环境变量或文件挂载方式注入容器。
   - 应用不需要把密钥写死在配置文件中。

3. **引用稳定，值可轮换**
   - 应用引用 `secretRef`。
   - Secret 的实际值可以更新或轮换。

4. **存储实现可替换**
   - K8S Secret 可以来自原生 Secret、External Secrets Operator、Vault、云 Secret Manager。

但 K8S Secret 本身也有局限：

- 默认只是 base64 编码，不等于强加密。
- 需要额外开启 etcd encryption 才更安全。
- 它更擅长“分发 Secret”，不擅长细粒度审计、动态授权和短期凭证签发。

因此，本项目应学习它的 `SecretRef` 思路，但不要把架构绑定在 K8S 上。

## 4. 核心概念

### 4.1 Credential

`Credential` 表示真实认证材料。

典型类型：

- API Key
- Bearer Token
- Basic Auth
- Access Key / Secret Key
- OAuth Refresh Token
- 自定义 Header Secret
- mTLS 证书引用

`Credential` 不描述业务接口，只描述“怎么认证”。

### 4.2 Connection

`Connection` 表示一个可被 Tool / Skill / Connector 使用的业务连接。

例如：

- `crm_prod`
- `erp_test`
- `github_org`
- `weather_api`
- `sandbox_admin`

它描述：

- 连接名称
- base_url
- 默认 headers
- auth 类型
- `credential_ref`
- 默认 timeout / retry
- allowlist scope
- 所属 tenant / project / environment

`Connection` 是 Agent 和 Skill 最常引用的对象。

### 4.3 ConnectionRef

运行时传递的安全引用。

示例：

```json
{
  "connection_ref": "crm_prod",
  "method": "POST",
  "path": "/v1/customers",
  "body": {
    "name": "Alice"
  }
}
```

模型和 Skill 不需要知道真实 token。

### 4.4 CredentialRef

`Connection` 内部引用的密钥标识。

示例：

```yaml
connection:
  id: crm_prod
  base_url: https://crm.example.com
  auth:
    type: bearer_token
    credential_ref: cred_crm_prod_token
```

## 6. 接口设计

### 6.1 CredentialStore

```go
type CredentialStore interface {
    Create(ctx context.Context, req CreateCredentialRequest) (*CredentialMeta, error)
    Update(ctx context.Context, req UpdateCredentialRequest) (*CredentialMeta, error)
    GetMeta(ctx context.Context, ref CredentialRef) (*CredentialMeta, error)
    Resolve(ctx context.Context, ref CredentialRef, purpose ResolvePurpose) (*ResolvedCredential, error)
    Delete(ctx context.Context, ref CredentialRef) error
    List(ctx context.Context, filter CredentialFilter) ([]*CredentialMeta, error)
}
```

原则：

- `GetMeta` 只返回脱敏元数据。
- `Resolve` 才返回运行时可用的真实值。
- `Resolve` 必须经过权限、审计和用途校验。
- 普通列表接口永远不返回明文。

### 6.2 ConnectionStore

```go
type ConnectionStore interface {
    Create(ctx context.Context, req CreateConnectionRequest) (*Connection, error)
    Update(ctx context.Context, req UpdateConnectionRequest) (*Connection, error)
    Get(ctx context.Context, ref ConnectionRef) (*Connection, error)
    Delete(ctx context.Context, ref ConnectionRef) error
    List(ctx context.Context, filter ConnectionFilter) ([]*Connection, error)
}
```

原则：

- `Connection` 可以包含 base_url、headers、timeout、retry、credential_ref。
- `Connection` 不保存真实 secret。
- `Connection` 可以被 Skill、Tool、Connector 引用。

### 6.3 CredentialResolver

运行时建议额外抽象 resolver：

```go
type CredentialResolver interface {
    ResolveForHTTP(ctx context.Context, ref ConnectionRef, req HTTPResolveRequest) (*ResolvedHTTPConnection, error)
}
```

它负责：

- 根据 `connection_ref` 查连接。
- 根据 `credential_ref` 解密密钥。
- 按 auth 类型生成 header/query/basic auth。
- 合并默认 headers、base_url、timeout、retry。
- 返回给 `http_request` 或业务 adapter 使用。

## 8. 第一版存储方案

当前没有 K8S，推荐使用 **本地加密文件存储**。

### 8.1 文件布局

```text
data/
  credentials/
    tenants/
      <tenant_id>/
        <credential_id>.json.enc
  connections/
    tenants/
      <tenant_id>/
        <connection_id>.yaml
```

### 8.2 主密钥来源

第一版支持：

```text
GENESIS_MASTER_KEY
```

要求：

- 主密钥不写入仓库。
- 主密钥不写入 `config.yaml`。
- 本地开发可以放在 `.env` 或系统环境变量。
- 生产环境由部署系统注入。

未来可扩展：

- Windows Credential Manager
- macOS Keychain
- Linux Secret Service
- K8S Secret
- Vault Transit
- 云 KMS

### 8.3 加密算法

推荐：

- AES-256-GCM
- 每条密钥独立 nonce
- metadata 明文，secret value 密文
- 密文文件可备份，但不能离开主密钥单独解密

### 8.4 文件存储的限制

文件存储适合第一阶段，但不是最终生产方案：

- 多实例并发写入能力弱。
- 审计能力有限。
- 权限模型依赖应用层。
- 主密钥轮换需要额外流程。

未来进入多租户生产后，应迁移到 DB + KMS/Vault 或云 Secret Manager。
### 8.5 主密钥与数据密钥

第一版虽然可以只用 `GENESIS_MASTER_KEY`，但接口上建议按 envelope encryption 思路设计：

```text
MasterKey / KEK
  -> 加密 DataKey / DEK
    -> 加密具体 Credential secret value
```

当前文件实现可以简化为：

- `GENESIS_MASTER_KEY` 作为 KEK。
- 每条 Credential 生成独立随机 DEK。
- DEK 加密 secret value。
- KEK 加密 DEK。
- 文件保存 `encrypted_dek`、`nonce`、`ciphertext`、`key_id`、`algorithm`。

这样未来接 KMS / Vault Transit 时，不需要推翻数据模型，只需要把 KEK 加解密替换为外部服务。

### 8.6 主密钥轮换

主密钥轮换不能要求重建所有业务连接。推荐流程：

1. 引入新的 `key_id`。
2. 使用旧 KEK 解开 DEK。
3. 使用新 KEK 重新加密 DEK。
4. 不重新加密真实 secret value，除非需要完整重封装。
5. 记录轮换审计。

第一版可以先不实现自动轮换，但文件格式必须包含 `key_id`，避免以后无法演进。

### 8.7 本机安全存储优先级

本地开发和桌面端场景下，`GENESIS_MASTER_KEY` 不是唯一选择。推荐优先级：

1. 桌面端：Windows Credential Manager / macOS Keychain / Linux Secret Service。
2. 服务端单机：环境变量或部署系统注入的 master key。
3. 容器环境：容器 Secret / 挂载文件 / 环境变量。
4. 云原生：K8S Secret + etcd encryption，或 External Secrets + 云 Secret Manager。

不要把 master key 写入仓库、普通配置文件或 Skill。

## 9. K8S 预留实现

未来接 K8S 时，不改变上层接口，只新增 adapter：

```text
internal/capabilities/credential/adapter/k8s/
```

映射关系：

```text
CredentialRef -> Kubernetes Secret name + key
TenantID      -> namespace 或 label selector
ProjectID     -> label / annotation
Version       -> Secret annotation
```

K8S Secret 示例：

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: genesis-cred-crm-prod
  namespace: tenant-a
  labels:
    genesis.io/tenant: tenant-a
    genesis.io/project: sales
type: Opaque
data:
  token: <base64>
```

注意：

- 生产必须开启 etcd encryption。
- 推荐结合 External Secrets Operator 或 CSI Secret Store。
- 不建议业务代码直接读取 K8S Secret；仍应走 `CredentialStore` 接口。

## 11. 系统功能入口

密钥和连接管理应该做成系统功能，而不是只靠配置文件。

推荐入口优先级：

### 11.1 后台菜单

适合普通用户和管理员。

菜单建议：

```text
系统设置
  连接管理
    连接列表
    新增连接
    编辑连接
    测试连接
    绑定凭证
  密钥管理
    密钥列表
    新增密钥
    轮换密钥
    禁用密钥
    删除密钥
    查看使用记录
```

后台页面原则：

- 列表只展示脱敏信息。
- 新增密钥时只允许输入一次明文。
- 编辑时不回显密钥明文。
- 支持“测试连接”，但测试结果不能输出密钥。
- 支持按 tenant / project / environment 过滤。

### 11.2 CLI

适合开发、运维、CI。

建议命令：

```bash
genesis credential create
genesis credential list
genesis credential rotate
genesis credential disable
genesis credential delete

genesis connection create
genesis connection list
genesis connection test
genesis connection bind-credential
genesis connection delete
```

CLI 适合第一版先落地，因为：

- 实现成本低。
- 本地开发立刻可用。
- 可用于初始化环境和调试。

### 11.3 HTTP API

后台菜单和 CLI 最终都应该调用同一套应用服务 / API。

当前第一版 API：

```text
POST   /v1/credentials
GET    /v1/credentials
GET    /v1/credentials/{id}
PATCH  /v1/credentials/{id}
DELETE /v1/credentials/{id}

POST   /v1/http-connections
GET    /v1/http-connections
GET    /v1/http-connections/{id}
PATCH  /v1/http-connections/{id}
DELETE /v1/http-connections/{id}
POST   /v1/http-connections/{id}/test
```

后续再补 rotate、disable、审计记录、CLI 和后台菜单。

原则：

- CLI 和后台菜单都调用同一套 service。
- HTTP API 只做协议转换、鉴权、参数校验和响应序列化。
- 第一版没有后台菜单时，CLI 可以直接调用 service，但不要绕过 service 写文件。
- 后台菜单上线后，不改变底层 store 和 resolver 接口。

### 11.5 当前阶段推荐入口

当前第一版已按“先暴露接口、后补菜单”的方式落地，推荐顺序调整为：

1. **HTTP API 优先**
   - 服务后台菜单、外部管理调用和自动化初始化。
   - 当前已暴露密钥元数据与 HTTP Connection 的 CRUD 接口。

2. **CLI 后补**
   - 适合本地开发、CI 初始化和运维脚本。
   - CLI 应复用同一套 service 或 HTTP API，不直接改文件。

3. **后台菜单产品化**
   - 适合管理员管理连接、轮换密钥、查看使用记录。
   - 放到 Phase 1B 更合理。

因此，不建议只靠手写配置文件。当前用 HTTP API 先闭环，后续 CLI 与菜单复用同一套能力。

## 12. 与 HTTP 工具的关系

`http_request` 工具应支持两种模式：

### 12.1 低级模式

直接传 `url`、`headers`、`auth`。

适用：

- 本地开发。
- 临时调试。
- 低风险公开 API。

风险：

- 模型可能看到明文 token。
- 不适合作为生产默认模式。

### 12.2 连接引用模式

传 `connection_ref`。

示例：

```json
{
  "connection_ref": "crm_prod",
  "method": "POST",
  "path": "/v1/customers",
  "body": {
    "name": "Alice"
  }
}
```

运行时流程：

1. `http_request` 收到 `connection_ref`。
2. 调用 `ConnectionResolver`。
3. 解析 base_url、默认 headers、timeout、retry。
4. 通过 `CredentialStore.Resolve` 获取真实密钥。
5. 注入到 `platform/httpclient.Request`。
6. 执行请求。
7. 审计记录使用情况。

生产推荐优先使用连接引用模式。
### 12.3 限制放在上层，不放在底层 HTTP Client

`platform/httpclient` 应保持通用和可复用，不内置业务 allowlist、租户策略或连接权限。限制应放在更靠近业务语义的位置：

```text
platform/httpclient     # 万能 HTTP 请求能力
http_request tool       # 工具参数映射和 connection_ref 支持
ConnectionResolver      # 凭证注入、连接默认值合并、用途校验
Governance / Policy     # 租户、Agent、Skill、Tool、Host、Path 权限裁决
```

这样做的好处：

- 底层可以服务 LLM Provider、Sandbox Client、Webhook、业务 API 等不同场景。
- 不同租户、不同 Agent 的策略可以不同。
- 未来接 K8S/Vault 不影响 HTTP Client。
- 安全策略不会散落到每个 HTTP 调用点。

生产环境推荐默认禁用低级明文 auth 模式，或要求高风险调用进入审批流程。

### 12.4 文件上传与下载

`http_request` 第一版已支持受控文件上传和下载，但它仍然不是任意文件系统工具。

上传使用 `multipart`：

```json
{
  "connection_ref": "oss_api",
  "method": "POST",
  "path": "/upload",
  "multipart": {
    "fields": {
      "bucket": "demo"
    },
    "files": [
      {
        "field": "file",
        "path": "workspace/report.pdf",
        "filename": "report.pdf",
        "content_type": "application/pdf"
      }
    ]
  }
}
```

下载使用 `download`：

```json
{
  "connection_ref": "oss_api",
  "method": "GET",
  "path": "/files/report.pdf",
  "download": {
    "save_as": "downloads/report.pdf",
    "max_bytes": 104857600
  }
}
```

安全边界：

- 上传文件和下载保存路径必须位于 HTTP 工具工作区内。
- 工作区默认是当前进程工作目录，可通过 `GENESIS_HTTP_TOOL_WORKSPACE` 指定。
- 拒绝 `../` 路径穿越和工作区外绝对路径。
- 上传读取普通文件；下载拒绝写入符号链接或目录，避免通过 symlink 写出工作区。
- 下载不会把二进制内容回显到 `body_text`，只返回保存路径和字节数。
- 当前下载仍走 `httpclient.Do` 的受限响应体读取；大文件流式下载后续应扩展底层 `Download` 接口。

## 13. 与 Skill 的关系

Skill 应描述业务接口的用法，不保存真实密钥。

Skill 可以包含：

- 使用哪个 `connection_ref`。
- 接口路径。
- 参数 schema。
- 返回值解释。
- 调用前置条件。
- 多接口编排顺序。

Skill 不应包含：

- API Key。
- Bearer Token。
- Basic Auth 密码。
- 生产数据库连接串。
- 长期访问密钥。

示例：

```yaml
name: crm-create-customer
tools:
  - http_request

connections:
  crm: crm_prod

endpoints:
  create_customer:
    method: POST
    path: /v1/customers
    connection_ref: crm_prod
    body_schema:
      type: object
      required: [name, phone]
```

---

## 14. 权限与治理

密钥能否被使用，不应只由是否知道 `credential_ref` 决定。

至少需要校验：

- tenant
- project
- environment
- agent
- user / role
- tool
- skill
- connection
- host / path allowlist

第一版可以先预留字段，简单校验 tenant / project / tool。

未来治理策略示例：

```yaml
connection_policy:
  connection_ref: crm_prod
  allowed_tools:
    - http_request
  allowed_agents:
    - sales_agent
  allowed_hosts:
    - crm.example.com
  allowed_paths:
    - /v1/customers
    - /v1/orders
  require_approval_for:
    - DELETE
    - /v1/refunds
```

## 15. 审计与日志

必须记录：

- 谁使用了哪个 `connection_ref`。
- 哪个 Agent / Skill / Tool 发起。
- 访问的 host / path。
- 请求方法。
- 响应状态码。
- 是否成功。
- 错误分类。
- 使用的 `credential_ref` 元数据。

绝不记录：

- 明文 token。
- 密码。
- secret key。
- 完整 Authorization header。

日志中只允许：

```text
credential_ref=cred_crm_prod_token
connection_ref=crm_prod
auth_type=bearer_token
```

## 16. 轮换与版本

Credential 需要支持版本：

```text
cred_crm_prod_token@v1
cred_crm_prod_token@v2
```

轮换流程：

1. 创建新版本。
2. 标记新版本为 active。
3. 测试关联 connection。
4. 保留旧版本一段时间。
5. 禁用旧版本。
6. 最终删除旧版本。

Connection 默认引用 active 版本，也可以在高级场景固定版本。

## 17. 当前阶段推荐落地顺序

### Phase 1A

- 定义 `CredentialStore` / `ConnectionStore` / `CredentialResolver` 接口。
- 实现本地加密文件 CredentialStore。
- 实现本地文件 ConnectionStore。
- HTTP API 支持创建、列表、更新、删除、测试连接解析。
- `http_request` 支持 `connection_ref`。
- CLI 后续复用同一套 service 或 HTTP API 补齐。

### Phase 1B

- 后台菜单维护连接和密钥。
- 审计日志持久化。
- Tool / Agent / Skill 能力范围策略接入。
- Secret 脱敏展示和使用记录查询。

### Phase 2.5

- DB 存储。
- K8S Secret adapter。
- Vault / 云 Secret Manager adapter。
- 动态短期凭证。
- 密钥轮换任务。
- 更细粒度 RBAC。

## 18. 最终判断

当前没有 K8S 时，最佳实践不是把 K8S Secret 硬搬进项目，而是先做一个 K8S 风格的引用模型：

```text
Skill / Agent / Tool
  -> connection_ref
    -> ConnectionStore
      -> credential_ref
        -> CredentialStore
          -> file / env / k8s / vault / cloud
```

这样第一版可以轻量落地，未来又能自然迁移到 K8S、Vault 或云密钥服务。

系统入口上，推荐：

- 第一版先做 HTTP API，方便后续后台菜单和自动化接入。
- CLI 作为开发和部署初始化入口随后补齐。
- 后台菜单作为正式产品化入口，在 Phase 1B 补齐。
