# 混合执行：路径身份与产物交接最佳实践

> 状态：父子产物交接（Adoption）已落地并通过全仓库 build/vet/test；路径身份/远程状态生命周期等其余条目仍为最佳实践基线
> 日期：2026-07-21（父子 Adoption 实现版）
> 适用范围：CLI、Desktop、Enterprise；主 Agent、子 Agent（context:fork）、Skill、run_command、sandbox_exec、Artifact 交付
> 关联：`docs/统一执行工作空间、文件权限与产物规范.md`、`docs/Skill三模式执行与依赖闭环设计.md`、`docs/superpowers/specs/2026-07-17-produced-resource-artifact-delivery-architecture.md`、`docs/子智能体设计.md`
>
> **本轮实现摘要（以此为准，正文历史推演见 §2/§3）**：跨 Run 产物可读性不再来自「创建即全局登记」的隐式索引（旧 `GlobalCandidateRegistry`，已删除），而来自**唯一父子边界（`Controller.finish`）的一次显式 `Adopt`**，落成显式、版本锁定、可审计、按消费者隔离的 `AdoptionRecord`（`artifactservice.AdoptionStore`，磁盘持久化于 `runtime/adoptions/`）。消费方（`view_image` / `finalizer` / `select`）跨 Run 读时按 `AdoptionRecord` 解析真实所属子 Run，再**按资源所属 binding 的 backend** 读取；未接纳则不可跨 Run 读（消除「假跨 Run 可读」）。交付归属遵循「谁声明 deliverable 谁 finalize+deliver」：子在其子 Run 内交付，父只接纳为**只读引用**并总结，不在父 Run 重新 publish/select 子产物。
>
> **补充实现（2026-07-22，子 Run 交付契约改为证据驱动，见 §7.1）**：双模态 Skill（如 office-ppt 读+写）下，按 Prompt/`ArtifactRequired` 预建 Spec 会误伤只读任务。现改为：spawn 只注入 `EvidenceQAHints`；写出可交付 office 产物后由 `EnsurePrimaryDeliverableFromProduced`（`FinalizeRequired` + `EvaluateCompletion`）再建 Spec 并交付；只读无产物无门禁可完成。父仍只 Adopt 只读引用 + 总结。NLP IntentResolver 不再自动置 `ArtifactRequired`。

## 1. 产品需求（与 Codex / Claude Code 的本质差异）

Codex / Claude Code 的执行模型是**单一世界**：要么全在本地宿主/本地沙箱，要么代码整体进远程沙箱。它们的路径与产物之所以简单，是因为「思考的环境」和「执行的环境」几乎同构。

本项目的目标是**混合执行（hybrid execution）**，且默认重心在本地：

- **默认**：本地宿主 / 本地平台沙箱。日常读写项目、看文件、改代码都在这里。
- **仅在必要时下沉远程沙箱**：高危命令、技能代码执行、第三方/不可信代码、需要镜像工具链的 shell 命令。

这意味着一次会话里会**同时存在两个执行世界**，而用户和主 Agent 仍希望「像在本地一样」表达意图。难点不在任何单一世界，而在**世界之间的边界**。

## 2. 三次尝试复盘与根因

| 尝试 | 做法 | 暴露的问题 | 根因 |
| --- | --- | --- | --- |
| 1 | 主 Agent 同时驱动宿主与远程命令 | 宿主/沙箱命令路径经常混乱 | 一个大脑维护两套物理路径宇宙，模型必须逐条判断「这条在哪跑、路径怎么写」 |
| 2 | 主 Agent 只写逻辑路径，工具自行拼宿主/沙箱 | 用户说「看下 D 盘 Temp 的 xxx」时，主 Agent 仍要关注真实路径，照样错乱 | 缺少「外部路径 → 稳定身份」的强制入口；逻辑路径只是换了写法，没换语义 |
| 3 | 统一绝对路径，无前缀默认工作区根；需要沙箱的多步技能用 `context:fork` 子 Agent 承担 | 单步远程 OK；多步技能（office-ppt）子 Agent 执行成功，但**父 Agent 选择候选产物失败** | fork 正确解耦了「执行面 cwd 与认知」，但**结果面缺少跨执行主体的产物 Adoption 协议**，父拿到的是 child-run-owned 的 `produced-*`，按父 Run 鉴权必然 `CROSS_EXECUTION_RESOURCE_DENIED` |

**统一结论：**

> 路径混乱与产物失败，都不是「绝对 vs 相对」的写法问题，而是 **「路径」没有升级为「身份」，以及「执行主体之间没有正式交接契约」**。

`context:fork` 是对的战术，但它只覆盖「执行面解耦」这半段，缺了「结果面交接」那半段。

## 3. 第一性原理与不变量

系统真正要保证的是：

> 在一次会话里，用户以本地直觉表达意图；运行时在正确的执行世界里完成动作；跨世界传递的永远是**稳定身份**，而不是某个世界的物理路径；结果最终以用户可见的交付形态回到本地世界。

不变量（违反任一即视为契约失败）：

- **I1 身份先于路径**：跨执行边界传递的只能是 `WorkspacePath`（相对名）/ `InputRef` / `ProducedResource(candidate_id)` / `ArtifactRef`，绝不能是宿主绝对路径或 `/workspace` 物理路径。
- **I2 单执行面单符号集**：任一执行面（宿主主 Agent、远程 skill 会话）只暴露**一套**路径符号；模型不在多套符号间自行翻译。
- **I3 隔离单位是 Binding**：远程/隔离的最小单位是 `ExecutionBinding + backend`，不是 SubAgent。是否 fork 由「认知/会话隔离需求」决定，不由「是否要远程」决定。
- **I4 外部路径必须显式入境**：任何项目外/宿主绝对路径在被任一执行面消费前，必须经过 **授权 + staging → `InputRef`** 这道确定性闸门；之后任何执行面都不再出现该绝对路径。
  - **闸门 ≠ 解析**：把自然语言/模糊引用（"D 盘 Temp 那个对比文件"）解析成某个**具体候选路径**，是**上游交互/认知层**（用户↔系统、主 Agent）的职责，允许 glob、列目录、歧义消解、反问用户确认；PathResolver 只对**已确定的具体路径**做规范化 + 授权 + staging，不负责理解自然语言。解析不确定时应向用户澄清，而不是让 PathResolver 猜。
- **I5 跨主体产物必须 Adoption**：子 Run / 远程执行产出的资源，进入父 Run / Session 的交付面前，必须经过显式提升（Adoption / Handoff），并在消费主体的 scope 内重新鉴权。
- **I6 消费方 scope 完成交付**：Publish / Delivery / view 只能在「资源已归属或已被 adopt 到的执行主体」上进行。
- **I7 降级可观测**：`SandboxRequire` 不满足必须失败；任何降级必须有 warning / trace / audit。

## 4. 核心模型：三个正交轴

绝不能把下面三件事绑成一件事。历史上每次绑死都直接导致混乱。

```text
① 谁在思考 (Cognition)   主 Agent │ Skill-fork 子 Agent │ 纯 Harness 流水线
② 在哪执行 (Execution)   host │ local sandbox │ remote session
③ 资源身份 (Identity)    WorkspacePath │ InputRef │ ProducedResource │ ArtifactRef
```

- Codex/CC 简单：① 与 ② 通常同构。
- 本项目：① 可停在宿主，② 局部跳远程，③ 必须全程用稳定身份贯穿。

**最佳实践口径：**

- **远程 = 换 ②（Binding/backend）**，默认不换 ①。
- **fork = 换 ①（独立认知 + 会话）**，仅当技能需要多轮、有状态、强工具链闭环时才用。
- **③ 永远是跨轴通信的唯一语言。**

### 4.1 执行归属 × backend 正交矩阵（主 agent 直接远程 vs 子 agent 远程）

远程沙箱有两种进入方式，二者产物交接路径**本质不同**，必须分开建模。**Adoption 的唯一触发条件是"跨 Run（跨执行主体）"，与 backend 是不是 remote 无关。**

| 情况 | 进入方式 | 执行归属 | 产物归属 | 需要 Adoption？ | finalize | 读产物（view_image / reader） |
| --- | --- | --- | --- | --- | --- | --- |
| **A 主 agent 直接远程** | `sandbox_exec` / `run_command`(remote backend)，`PrepareExecution` 在**同 Run** `AddExecution` 追加 remote binding | 同 Run，跨 binding | 同 Run，不同 binding | **否**（同 Run scope，鉴权天然通过） | 同 Run finalize（已实现） | remote binding 在**同一** manifest → 按资源所属 binding backend 读取即可命中 |
| **B 子 agent 远程** | `context:fork` 子 Run，子 Run 内再建 remote binding | 跨 Run，跨 binding | 子 Run，子 binding | **是** | 子 Run finalize + 父 **Adopt** | 子 binding **不在父** manifest → 必须按资源所属 binding（来自子 Run manifest / grant）的 backend 读取，禁止父 manifest 硬匹配 |

要点：

- **Case A 基本已 work，不要给它套 Adoption**。它只依赖两件事：①同 Run finalize（`sandbox_exec` 已做）；②按资源所属 binding 的 backend 读取——而该 binding 就在同一 manifest 内。
- **Case B 才是 §7 Adoption 的适用对象**，且叠加 §7.2 的"view_image 不得从父 manifest 硬匹配 binding"。
- **待验证风险（Case A）**：`view_image` 读到的 `prepared.Manifest` 若为 PrepareRun 时旧快照、未包含 `sandbox_exec` 经 `AddExecution` 追加的 remote binding，则 Case A 的 view_image 也会 `binding not in run manifest`。需确认"上下文 manifest 是否随 `AddExecution` 刷新"，否则 Case A 也需按资源所属 binding 走 registry/最新 manifest 读取。

## 5. 路径身份契约（每个执行面暴露什么）

这是根治「路径混乱」的核心表。目标是让**每个执行面只有一套符号**，模型永不跨面翻译。

| 执行面 | 模型可见符号 | 物理映射（仅 Harness 知道） | 禁止出现 |
| --- | --- | --- | --- |
| 主 Agent · 宿主/本地（project/task/session） | `.` + workspace 相对路径；`input://<id>`（外部文件授权后的引用） | `WorkDir` = 项目根或 `.genesis/.../work/<binding>` | 宿主绝对路径当业务路径、`/workspace`、`$OUTPUT_DIR` 等执行面符号 |
| 远程 Skill / sandbox 会话 | `$INPUT_DIR`/`$OUTPUT_DIR`/`$WORK_DIR`/`$TMPDIR`/`$SKILL_DIR`（环境变量，脚本内读取）；相对文件名 | `/workspace/...`（远程） | 宿主绝对路径、把逻辑变量当字面量写进 command |
| 跨面传递（工具入参/返回） | `InputRef` / `candidate_id` / `ArtifactRef` | descriptor + backend locator | 任一物理路径、locator 明文 |
| 用户 ↔ 系统（仅 CLI 交互层） | 允许 `D:\Temp\xxx`、相对路径、**自然语言引用** | 交互层先解析出具体路径（可歧义消解/问用户），再经 PathResolver 授权 + staging 转 `InputRef` | 让绝对路径/未澄清的模糊引用穿透到执行面或模型下游 |

关键规则：

1. **用户绝对路径只活在「用户↔系统」这一层**。一旦授权，转成 `InputRef`，之后主 Agent、子 Agent、沙箱都只见引用，不见 `D:\`。
   - 这直接修复尝试 2 的痛点：主 Agent 不再「关注真实路径」，它只持有 `input://xxx`。
   - **解析与闸门分层（见 I4）**：自然语言/模糊引用 → 具体候选路径的解析属于本交互层，允许灵活、可反问、可失败；PathResolver 只是把**已确定的具体路径**授权 + staging 成 `InputRef` 的确定性闸门，不承担自然语言理解。现状的正则抽取（`ResourceRegistry.PlanRequestInputs`）只能当便捷提示，命中不确定时必须向用户澄清，不得当权威解析。
2. **主 Agent 面永不出现 `$OUTPUT_DIR` / `/workspace`**。这些是执行面（沙箱脚本）内部符号。
3. **跨面只用身份**。工具入参校验：若在远程 command 里检测到宿主绝对路径或 `/workspace` 明文，直接结构化失败（已有 `pathcontract` 校验器，应保证覆盖 skill/子 Run 路径）。
4. **裸目录名约定（`output/`、`input/`）要么显式文档化，要么收紧**。当前 `StripLogicalDirPrefix` 允许裸名映射到逻辑目录，属于隐式约定，是串味来源之一；最佳实践是**只保留 `$VAR` 显式前缀**，或在对模型的契约里明确枚举。

## 6. 决策树：远程与 fork 如何选

```text
收到一步动作
│
├─ 需要访问项目外/用户绝对路径的文件？
│     → 先 PathResolver + 授权 + staging → InputRef（与是否沙箱、是否 fork 无关）
│
├─ 动作性质：
│   ├─ 读写已授权项目、常规低危 → 主 Agent · 宿主/本地，相对路径
│   │
│   ├─ 单次高危命令 / 单次技能脚本（无多轮 QA）
│   │     → 同一主 Agent + 临时/复用 Skill 或 sandbox Binding（backend=remote）
│   │     → 产出 Produced → 同 Run 内 finalize → 交付
│   │     → 不 fork
│   │
│   └─ 多轮、有状态、强工具链技能（office-ppt：写脚本→跑→markitdown/thumbnail→改→再跑→视觉 QA）
│         → context:fork 子 Agent（独立认知）+ task_job binding（backend=remote session）
│         → 子 Run 内 finalize 或 父 Run Adoption（见 §7）
```

判据一句话：

- **要不要远程？** 看动作是否高危/第三方/需镜像工具链 → 决定 ② backend。
- **要不要 fork？** 看是否需要**独立的多轮认知上下文 + 有状态会话** → 决定 ①。
- 两者**独立判断**，不能「因为要远程所以 fork」。

`Skill` 的 `sandbox` / `context: fork` 声明表达的是**执行与会话需求**，不是交付实现；Harness 据此选择「同 Run 换 binding」还是「fork 子 Run」，但**无论哪条路径，产物都走同一套 §7 交接**。

## 7. 产物交接：Produced → Parent/Session Adoption

这是修复「子成功、父选候选失败」的核心，也是当前实现最大的缺口。

### 7.1 两阶段边界

**阶段 A — 执行主体内部（子 Run / 远程会话）**

- **交付契约：证据驱动（已实现）**。fork 子 Run **不再**按 Prompt/`IntentResolver.ArtifactRequired` 预建 `DeliverableSpec`（双模态 Skill 如 office-ppt 读+写并存时会误伤只读）。改为：
  1. spawn 时经 `prepareChildArtifactContext` 注入 `EvidenceQAHints`（Skill frontmatter QA）与完成门禁可观测告警；
  2. 命令写出可交付 office 产物（`.pptx/.docx/.xlsx/.pdf`，排除 QA 预览图）后，`FinalizeRequired` → `EnsurePrimaryDeliverableFromProduced` 再建 primary Spec 并交付；
  3. `EvaluateCompletion` 同样会补建 Spec，保证「有产物证据 ⇒ 有门禁」不变量；
  4. 只读无产物 → 无 Spec → 完成门禁直接放行。
- 继续登记每次命令的 `ProducedResource`；视觉 QA 产物标 `role=qa_asset`（投影层）并以文件名白名单排除证据建约。
- `FinalizeRequired` 只在**该执行主体自己的 Run** 上运行。
- 唯一 required 匹配 → 在该 Run 内自动 Gate→Publish→Delivery（或至少落 Selection）。
- 多 required 候选 → `selection_required`，由该主体内部或经受控 API 处理。
- **硬化**：子完成门禁依赖从父 ctx 隐式继承的 `CompletionPolicy`；一旦缺失将静默不交付，故 `prepareChildArtifactContext` 若探测不到即 `Warn` 告警（fail-loud）。

**阶段 B — 交接到消费主体（`Controller.finish`，唯一插入点，已实现）**

`Controller.finish` 对归约后 `TaskResult.Artifacts`（QA/中间物已在 `reducer.deliverablesForParent` 剔除）逐项调用 `artifactservice.GlobalAdoptionStore.Adopt(...)`，写下 `AdoptionRecord`：

| 产出分类 | Harness 行为 | 父/Session 可见投影 |
| --- | --- | --- |
| 子交付候选（`role` 空） | **Adopt**：写 `AdoptionRecord{consumer=父, owner=子, produced_id, content_hash}`；父随后可按该记录跨 Run **只读引用**（view_image 等） | `candidate_id`（引用子产物，读取按子 binding backend） |
| 子已在子 Run 内 Delivery 的交付 | 子 Run 内 finalize+deliver 已完成，父只接纳其候选作只读引用并总结 | 交付位置由子 Run 完成；父只总结 |
| `qa_asset` / 中间文件 | **归约层直接剔除**，根本不进入 `TaskResult.Artifacts`，也就不会被 Adopt | 不进入 `select_deliverable_candidate` 候选集 |
| supersede 旧版 | produced store 只暴露 logical head | 避免出现十几个重复候选 |

**`AdoptionRecord`（实际字段，见 `internal/capabilities/artifact/service/registry.go`）**：`consumer_tenant_id` / `consumer_run_id` / `produced_id` / `owner_tenant_id` / `owner_run_id` / `agent_id` / `content_hash` / `role` / `created_at`。

- **按消费者隔离**：磁盘键 = `sha256(consumer_run_id \x00 produced_id)`，不同父各自独立接纳同一子产物，互不影响。
- **版本锁 + 幂等**：同 `(consumer_run, produced_id)` 已存在且 `content_hash` 不同 → 版本冲突（防子被 resume 重跑后父静默读旧版本）；相同内容再次 Adopt 幂等返回既有记录。
- **授权来自派生关系**：父 spawn 子、同 scope，接纳只发生在受信的父子边界（`Controller.finish`），无需再引入独立 Authorizer。
- **无「创建即登记」**：`produced_registrar` / skill 不再在产出时做任何全局登记；跨 Run 可读性**只**来自这一次显式 Adopt。

### 7.2 父侧工具契约

- `select_deliverable_candidate`：只接受**已 adopt / 已授权到父 Run** 的 ID；跨 scope 返回明确错误码（如 `ADOPTION_REQUIRED`），不伪装成 `NOT_FOUND`。
- `view_image`：打开时用**资源所属 binding 的 backend**读取，不用父 manifest 硬匹配 BindingID（当前 `binding not in run manifest` 就是硬匹配导致）。
- Skill/Task 回传的 `artifacts`：**由子执行面（skill / 子 Run 的 harness）在回传前自行过滤**，只放父可操作的交付候选；QA 资产、中间文件、旧版本留在子执行面内。父 Agent 不感知、不接管、不管理子技能内部——**父不是"分离 QA"，而是根本收不到 QA**。
- `next_action`：仅当父侧 Deliverable 真为 `selection_required` 时才提示 select；**禁止**用 `len(artifacts) > 1` 粗判。

### 7.3 fork 技能默认交付策略

对 office-ppt 这类**单一主交付**任务：

1. 子 Run 结束时 Harness 尽量自动 finalize（唯一 pptx head）。
2. Adoption 进父 Session 并完成 Delivery。
3. 父 ToolResult：`status=completed` + summary + 已交付引用；`next_action` 为空。
4. 父 Agent **禁止**再 glob / 探测子 cwd / 手动 select；直接向用户总结。

**决策（已拍板 R1=A，本轮实现）**：多产物时父 Agent 是否仍可 `select_deliverable_candidate`？

- **A（已采纳，默认）**：父不 select 子产物，只总结；父对跨 Run 子候选 `select` 返回 `ADOPTION_REQUIRED`。
- **B（未实现，留作后续开关）**：允许父发布「已提升 ownership 到父 Run」的子产物；需要时再引入。

✅ = 本轮已实现并通过 build/vet/test；⬜ = 仍为待办。

## 9. 评审意见与待决（2026-07-21）

> 结论（更新）：架构层面接近最佳实践（三轴正交、身份先于路径、Adoption、fail-closed 均正确）。**R1/R2/R3/R6 已在本轮落地**（见 §9.1）：`Controller.finish` 已做显式 `Adopt`，跨 Run 读改由 `AdoptionRecord` 解析并按资源所属 backend 读取，隐式 `GlobalCandidateRegistry` 已删除。**R4/R5/R7（远程状态生命周期、staging 成本）仍为待定契约**。

### 残留 / 明确取舍（residual）

- **版本锁依赖 `content_hash`**：当前 `TaskResult.Artifacts.ContentHash` 在真实技能流里多为空（`RegisterArtifact` 未回填内容哈希），此时版本锁退化为「按身份幂等」——正常流（子只跑一次）不受影响，仅子被 resume 产出不同版本时无法自动判冲突。回填内容哈希为可选增强。

## 11. 一页速记

- 需求：本地为主，高危/技能/第三方才远程 —— **混合执行是一等公民，不是降级模式**。
- 三轴正交：**谁思考 / 在哪跑 / 什么身份**，永不绑死。
- 远程换 Binding，fork 换认知；**要不要远程 ≠ 要不要 fork**。
- 跨面只传身份（相对名 / InputRef / candidate_id / ArtifactRef），物理路径与 `$VAR` 由 Harness 兜底。
- 用户绝对路径只活在交互层，入境即变 `InputRef`。
- 子产出进父世界必须 **Adoption**（本轮已落地：`AdoptionRecord`，`Controller.finish` 唯一边界，版本锁 + 审计 + 按消费者隔离）；跨 Run 读按资源所属 backend；未接纳不可跨 Run 读。
- 交付遵循「谁产出谁 finalize+deliver」：**证据驱动**——有可交付产物再建 Spec；子在子 Run 内交付，父只接纳只读引用并总结。只读无产物不建门禁（见 §7.1）。
- `context:fork`（office-ppt 等）是正确战术，配合 Adoption 即为完整最佳实践。
