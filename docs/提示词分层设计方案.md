## 一、设计目标与要回答的问题

Genesis 是通用 Agent 平台，**编码只是默认应用场景**（见 Agent App 设计 §一）。同一套 Runtime 内核靠不同提示词 / Skills / Tools / MCP / 子智能体 / 编排策略去满足代码、文档审查、客服、运维等业务。提示词是「把通用内核塑造成某类工作」的核心手段之一，因此它必须**规范化、可分层、可被每个项目/应用自定义**，而不是今天这样一坨扁平字符串。

本方案要回答：

1. **公共部分与业务部分如何分离**：安全、工具使用总则、Skill 网关边界、失败熔断等「所有应用都要遵守」的内容，与「编码约定 / 审查方法论 / 客服话术」等业务特定内容，怎么拆、怎么组装、怎么复用？（§五、§六）
2. **提示词如何做成"规范"**：用什么模型描述一段提示词？业务应用如何声明式地覆盖/追加而不是复制粘贴整份？（§十一 数据模型）
3. **默认编码应用（**`code`**）的提示词由哪些块组成**，它凭什么"只是一个应用"而不是"产品本身"？（§八）
4. **CLI / Desktop / Enterprise 如何统一又保留差异**：哪些提示词内核三端同源，哪些靠端口注入差异化？（§九）
5. **与 Agent App 的** `AgentSpec.PromptSpec`**、子智能体提示词、Hook 注入如何对接**，不另造平行体系。（§十）
6. **可借鉴 Kode-CLI / codex 的哪些点**，对应源码文件在哪，未来开发能直接查。（§四 + 附录 A/B）

一句话结论（先给出，后文论证）：

> **提示词 = 分层 Block 的有序合成**。公共内核块（安全 / 工具协议 / Skill 网关 / 失败熔断）由 Runtime 统一提供且**强制存在（只收窄不放宽）**；业务块（身份人格 / 领域方法论）由 Agent App 的 `PromptSpec` 声明式提供；环境上下文与项目文档（AGENTS.md）由产品端口注入。`code` 编码应用 = 公共内核 + 编码人格块 + 编码约定块的一份具体实例。组装由与产品无关的 `PromptComposer` 完成，三端同源、后端异构。

## 二、第一性原理：提示词到底要解决什么

抛开当前实现，把系统提示词还原到本质，它承载四类**寿命完全不同**的信息：


| 信息类别         | 寿命      | 是否可被业务改            | 示例                                                         |
| ------------ | ------- | ------------------ | ---------------------------------------------------------- |
| **安全与协议底线**  | 全局恒定    | **否**（只能追加，不能删除）   | 拒绝恶意代码、Skill 只能经 `Skill()` 网关加载、危险操作需审批                    |
| **通用工作纪律**   | 全局稳定    | 可裁剪部分              | 工具使用总则、Task/Todo 计划、失败熔断（`repeated_failure`/`no_progress`） |
| **业务身份与方法论** | 随应用变    | **是**（这正是"自定义"的入口） | "你是 AI 编程助手" + 编码约定 / "你是文档审查助手" + 审查清单                    |
| **上下文快照**    | 随会话/回合变 | 由运行时/产品注入          | cwd、git 状态、平台、日期、AGENTS.md、plan 进度、本轮 reminder             |


**关键洞察**：当前 `builder.go` 把这四类塞进同一个扁平字符串（时间 + `Agent.SystemPrompt` + injectors + 硬编码「行为规则」），导致：

- 安全/协议底线**没有强制保证**——业务方把 `Agent.SystemPrompt` 一换，「行为规则」还在但 Skill 网关规则其实在 `skillstack` injector 里，二者割裂，换个产品/应用就可能漏。
- 业务自定义**只有一个** `SystemPrompt` **字符串**，要么整段覆盖、要么什么都做不了，无法"沿用公共块 + 只改业务块"。
- 上下文快照与稳定规则**混在一起每次全量重建**，无法缓存、无法做 per-turn 差量（codex/Kode 的核心优化）。

**必须满足的不变量（本方案的验收基线）**：

1. **安全/协议底线块恒定存在**：无论哪个应用、哪个产品、哪层覆盖，安全块、Skill 网关块、失败熔断块**一定在最终提示词里**；下层只能追加约束，不能移除（与 Agent App §6.1「只收窄不放宽」、子智能体权限纪律同源）。
2. **公共块定义一次、全应用复用**：不允许每个应用复制一份安全/工具规则文案。
3. **业务块声明式可覆盖**：应用通过 `PromptSpec` 提供人格 + 方法论，无需感知公共块实现。
4. **三端同源**：组装器与块模型零 `if product == ...`；产品差异只经端口注入（项目文档源、环境上下文源、治理策略）。
5. **对齐既有代码与设计**：复用 `internal/runtime/prompt` 的 `Builder`/`ContextInjector`、`BuildOptions.PromptInjectors`、Agent App 的 `AgentSpec.PromptSpec` 与两阶段合成链。

**失败条件（不做会怎样）**：

- 每个业务应用各写一份安全规则 → 漂移，总有应用漏掉 Skill 网关或危险操作约束 → 越权 / 提示注入。
- 业务方自定义时误删安全块 → 运行时无兜底。
- 环境上下文每回合拼超长字符串 → token 浪费，且 Anthropic prompt caching 无法命中稳定前缀。
- 编码提示词被写死进产品 → 无法切换到 `doc-review` 等其他应用（违背 Agent App §2.3 克制原则）。

## 三、现状盘点（本项目当前实现与差距）



### 3.1 当前提示词链路（2026-07-19 代码对照）

```text
react_loop.loop()
  └─ prompt.Builder.BuildSystem(Audience, CollabMode, tools)
       ├─ "当前时间: ..."
       ├─ Agent.SystemPrompt 或默认一句          // config.yaml；子 Run 则为 ComposeChildSystem 结果
       ├─ range injectors                     // Environment / skills_instructions（产品各异）
       ├─ <plan_mode_rules> XOR <task_management>
       ├─ <delegation>                        // 仅 Audience=root 且 Task 可用
       └─ "## 行为规则"（按 Audience 裁剪）
  └─ Assembler 合并 LTM/profile 进首条 system
  └─ 主循环旁路：Skill injection / tasklist·plan·@agent reminder / Hook 额外 system
```

- `BuildSystem` 主要在 Run 开始调用；规划模式切换时会重建。主循环内不每 turn 重建 system。
- 业务自定义入口仍是单一 `domain.Agent.SystemPrompt`；**尚无** `domain.PromptSpec` / `PromptComposer`。
- 公共约束仍分散：行为规则硬编码 + `skills_instructions` injector（熔断文案可能重复）。
- 子智能体过渡：`AudienceSubAgent` + `capabilities/subagent/prompt` SSOT；完整 Composer 见 §14.0 冻结说明。
- `profilemodel.Profile` 无 prompt 字段；能力上界靠 Tools/Skills 过滤，与提示词分层正交。



### 3.2 现有可复用资产（方向是对的）

```10:40:internal/runtime/prompt/interface.go
type BuildRequest struct {
	Agent   *domain.Agent
	Run     *domain.Run
	UserID  string
	TurnID  string
```

`Builder` + `ContextInjector` 插件模型与 codex 的 `ContextualUserFragment` trait、Kode 的 injector 思路**方向一致**，是本方案演进的基座，无需推倒重来。

## 五、核心设计：分层 Block 模型（公共 vs 业务分离）

提示词的最小单元是 **Block（提示词块）**。最终系统提示词 = 若干 Block 按角色分组、按顺序拼接。

### 5.1 Block 分层（按来源与寿命）

```text
L0 公共内核层（Runtime 提供，全应用复用，产品无关）
   ├─ [强制] security          安全与拒绝：恶意代码、越权、危险操作需审批
   ├─ [强制] skill_gateway      Skill 只能经 Skill() 网关加载；$skill/skill:// 引用规则
   ├─ [强制] failure_policy     repeated_failure / no_progress / dependency_missing 处置纪律
   ├─ [可关] tool_protocol      工具使用总则：先判断必要性、并行调用、结果需整合
   ├─ [可关] task_management     Task/Todo 计划管理（启用 plan 时）
   └─ [可关] tone               沟通/语气风格（业务可关或替换）

L1 产品基座层（产品 bootstrap 注入，三端各异）
   └─ product_persona           产品身份微调（CLI 简洁 / Enterprise 合规口径）

L2 业务应用层（Agent App 的 PromptSpec 提供，随应用变）
   ├─ persona                   身份人格："你是 AI 编程助手" / "你是文档审查助手"
   ├─ instructions              领域方法论：编码约定 / 审查清单 / 客服话术
   └─ task_template_hint        当前 Task Template 的片段（可选）

L3 项目/用户覆盖层（端口注入）
   ├─ project_docs              AGENTS.md 栈（项目级，高优先级）
   └─ user_prefs                用户级偏好（~/.genesis-agent/...）

L4 会话/回合动态层（运行时注入）
   ├─ environment_context       cwd / git / 平台 / 日期（会话级，可缓存）
   ├─ plan_reminder             计划进度（→ 迁往 user 前缀，见 §六.C）
   └─ turn_reminder             本轮提醒：文件变更、重复失败 guard（→ user 前缀）
```

**"公共 vs 业务"的边界就是 L0/L1 与 L2 的分界线**：

- **公共部分 = L0 + L1**：一次定义，所有应用（含 `code`）继承。L0 三端同源，L1 三端各异但仍是"产品级公共"。
- **业务部分 = L2**：每个 Agent App 各自声明，这就是"每个项目可以自定义"的入口。
- L3/L4 是正交的上下文，不属于"业务方法论"，由产品/运行时负责。



### 5.2 Block 的角色（Role）与投放通道

对齐 codex 的 role 分层与 Kode 的双通道：


| Role          | 投放位置                               | 缓存性                 | 典型块                                |
| ------------- | ---------------------------------- | ------------------- | ---------------------------------- |
| `system`      | System 消息稳定前缀                      | 高（可 prompt caching） | L0/L1/L2/L3 全部                     |
| `user_prefix` | 最后一条 user 消息前缀 `<system-reminder>` | 低（每轮变）              | L4 的 plan_reminder / turn_reminder |


> **为什么 reminder 要放 user 前缀而非 system**（借鉴 Kode/codex）：稳定规则进 system 才能被模型当"系统底线"并利用缓存；每轮变化的提醒若混进 system 会破坏缓存前缀，且模型更易把 user 前缀的 `<system-reminder>` 当"系统附加信息"而非"用户原话"。第一轮可先全放 system（兼容现状），Phase 2 拆出 user 通道。



### 5.3 强制块不变量（安全底线）

```text
最终 system 提示词 ⊇ { security, skill_gateway, failure_policy }
```

- 强制块由 Runtime **无条件注入**，任何层的 `PromptSpec` / AGENTS.md / CLI flag **只能追加约束，不能移除或改写其安全语义**。
- 业务块可以覆盖 `persona`/`instructions`/`tone`，但覆盖动作**不触及**强制块内容——这与 Agent App §七「阶段二硬约束收窄，任何层不可放宽」是同一条纪律在提示词维度的落地。
- 实现上：合成器先放强制块，再叠加可覆盖块；`Overrides` 只能命中非强制块名，命中强制块名时**报错或忽略并告警**（audit）。

## 六、PromptComposer 组装流水线

把现有 `DefaultBuilder`（一次性扁平拼接）演进为**三阶段** `PromptComposer`，对齐 Kode 的三个时间点。**接口向后兼容**：`Builder.BuildSystem` 保留为 Phase 1 入口，内部改由 Composer 驱动。

### 6.1 三阶段职责

```text
阶段 A  BasePromptProvider   会话级/Run 级：产出 L0+L1 公共块（稳定、可测试、可缓存）
阶段 B  AppPromptResolver    Run 级：把 AgentSpec.PromptSpec 解析为 L2 业务块，与公共块合成
阶段 C  TurnEnhancer         每 turn：注入 L3/L4（环境/项目文档/reminder），拆 system 增量与 user 前缀
```



### 6.2 时序（对齐 react_loop 演进）

```text
Run 开始（react_loop.loop）
  ├─ A: BasePromptProvider.Blocks()      → 强制块 + 可关公共块（按 Profile 开关）
  ├─ B: AppPromptResolver.Resolve()      → 合入 AgentApp.Spec.Prompt（persona/instructions/append/overrides）
  ├─ (会话级 memoize) EnvContext / ProjectDocs 快照
  └─ 合成初版 system 消息（Phase 1 到此为止，等价现状但已分层）

每个 agentic turn（Phase 2 起）
  ├─ C.EnhanceSystem()      → 若环境/项目文档变化则更新对应块（Phase 3 做差量）
  └─ C.EnhanceUserPrefix()  → plan/turn reminder 注入最后一条 user 消息
```

> **react_loop 改造点**：`react_loop.go:166` 现在只调一次 `BuildSystem`。Phase 2 在主循环内（`react_loop.go:193` 的 for 前后）增加 `EnhanceUserPrefix` 调用，把 `planReminderInjector`（`builder.go:241`）与失败熔断提醒从 System 迁到 user 前缀。System 稳定块只在会话开始或环境变化时重建。



### 6.3 接口草案（放 `internal/runtime/prompt`）

```go
// Block 是提示词最小单元。
type Block struct {
    Name      string   // 段名 → <name>...</name>；空则裸拼
    Role      Role     // system | user_prefix
    Content   string
    Order     int      // 组内排序
    Mandatory bool     // 强制块：下层不可删除/改写
    Source    string   // 来源标注（builtin/product/app/project/user/run），供诊断
}

type Role string
const ( RoleSystem Role = "system"; RoleUserPrefix Role = "user_prefix" )

// BasePromptProvider 提供 L0+L1 公共块（产品 bootstrap 注入具体实现）。
type BasePromptProvider interface {
    Blocks(ctx context.Context, req BuildRequest) ([]Block, error)
}

// AppPromptResolver 把 App 的 PromptSpec 解析为 L2 块并与公共块合成。
// spec 来源：Phase 1 取自 req.Agent.Prompt（domain.PromptSpec）；Agent App 落地后取自 EffectiveAgentAppProfile.Prompt。
type AppPromptResolver interface {
    Resolve(ctx context.Context, base []Block, spec domain.PromptSpec, req BuildRequest) ([]Block, error)
}

// TurnEnhancer 每 turn 注入 L3/L4（Phase 2 起启用）。
type TurnEnhancer interface {
    EnhanceSystem(ctx context.Context, blocks []Block, req TurnRequest) ([]Block, error)
    EnhanceUserPrefix(ctx context.Context, req TurnRequest) (string, error)
}

// Composer 编排三阶段，产出最终 system 字符串（+ Phase 2 的 user 前缀）。
type Composer interface {
    ComposeSystem(ctx context.Context, req BuildRequest) (Composed, error)
}

type Composed struct {
    System     string
    UserPrefix string          // Phase 2 起非空
    Blocks     []Block         // 可解释：每块来源，供 `agent current --explain-prompt`
}
```

- 现有 `ContextInjector`（`interface.go:26`）**保留并复用**：环境上下文、项目文档、plan reminder 都以 injector 形式实现，被 `TurnEnhancer` 编排；`skillstack` 的 injector 迁为 L0 `skill_gateway` 强制块的 provider。
- 合成规则：按 `Role` 分组 → 组内按 `Order` → 强制块置顶且不可被 `Overrides` 命中 → 拼接时套 `<Name>` 标签（沿用 `builder.go:49-60`）。

## 七、合成与覆盖优先级（对齐 Agent App §七）

提示词合成**复用 Agent App 的两阶段合成链**，不另造一套。

**阶段一：偏好覆盖（低→高，后者覆盖前者同名可覆盖块）**

```text
Builtin 公共块(L0)
  → Product 基座块(L1)
  → App PromptSpec(L2：persona/instructions/tone override)
  → Project AGENTS.md(L3)
  → User 偏好(L3)
  → CLI/Run --append-system-prompt / --system-prompt(L4 一次性，优先级最高)
```

> L2 的 `PromptSpec` 来源：优先取 App 的 `PromptSpec`；App 未提供时**兜底**取 `domain.Agent.SystemPrompt`（映射为 `Instructions`）。二者是"同一层的来源选择"，不是独立的覆盖层——因此不出现在覆盖链的顶端。

**阶段二：强制块收窄（不可放宽）**

```text
最终块集 = 阶段一结果 ∪ {security, skill_gateway, failure_policy}   // 强制块恒在
覆盖动作 ∩ 非强制块名                                              // Overrides 不得命中强制块
```

- 与 Agent App §七完全同构：**偏好可层层覆盖，安全底线只能层层追加**。
- `--system-prompt`（整段替换业务块）仍**不能替换强制块**——即便用户要求"完全自定义 system prompt"，安全/网关/熔断块仍拼在最终结果里（可参数化位置，但不可移除）。
- 每块 `Source` 字段记录来源层，支撑 `genesis agent current` 输出 SourceChain（对齐 Agent App §10.2 的 `agent current` 诊断）。

## 九、三产品统一与差异（端口注入，对齐 Agent App §十三）

CLI / Desktop / Enterprise **共用同一套** `prompt` **内核**（Block 模型、Composer、合成链、强制块不变量），差异只经端口后端注入，内核**零** `if product == ...`。


| 端口                   | 职责                  | CLI/Desktop 后端                | Enterprise 后端                                                       |
| -------------------- | ------------------- | ----------------------------- | ------------------------------------------------------------------- |
| `BasePromptProvider` | L0+L1 公共块           | 内置常量 + 产品基座块（本地口径）            | 内置常量 + 租户合规口径基座块                                                    |
| `ProjectDocSource`   | L3 AGENTS.md 栈发现/读取 | 本地文件系统（git root→cwd，字节预算）     | 沙箱工作区 / 仓库快照（同规则，经 FileSystemBackend）                               |
| `EnvContextSource`   | L4 环境上下文            | 宿主机 cwd/git/平台                | **workspace-relative / sandbox path**（禁宿主机绝对路径，对齐 AGENTS.md 文件系统边界） |
| `UserPrefsSource`    | L3 用户偏好             | `~/.genesis-agent/<product>/` | 租户/用户画像存储                                                           |


- **统一什么**：Block 模型、Composer 三阶段、两阶段合成、强制块不变量、reminder 双通道、SourceChain 可解释性。
- **各自独立什么**：公共基座文案口径（CLI 简洁 vs Enterprise 合规）、项目文档/环境上下文的**来源后端**、Desktop 的可视化配置投影、Enterprise 的审计（记录用了哪个 App 的哪个 PromptSpec 版本）。

> **Enterprise 关键约束**：环境上下文块**不得**注入宿主机绝对路径（对齐 AGENTS.md「Docker/sandbox 模式下工具与 UI 不得展示宿主机绝对路径」）；`EnvContextSource` 的 Enterprise 后端只输出 workspace-relative / sandbox path / resource id。

## 十、与 Agent App / SubAgent / Hook 的关系



### 10.1 与 Agent App 的 `AgentSpec.PromptSpec`

本方案**就是** Agent App §6.1.1 里 `AgentSpec.PromptSpec` 的内容定义。`PromptSpec` 不再是"一段字符串"，而是 §十一 的结构化模型。`EffectiveAgentAppProfile.Prompt`（Agent App §七）经两阶段合成后，由 `AppPromptResolver` 解析为 L2 块。

> **落地顺序解耦（重要）**：Agent App 是后续阶段的工作，但提示词系统**不必等它**。`PromptSpec` 定义在 `internal/domain`（见 §十二），`domain.Agent` 直接持有它；Agent App 落地后，其 `AgentSpec` 复用同一个 `domain.PromptSpec`，无需迁移。因此本方案 Phase 1 可**独立先行**：`code` 的业务块来自内置/`config` 映射到 `domain.Agent.Prompt`，Agent App 到位后只是换了一个更丰富的 `PromptSpec` 来源。



### 10.2 与子智能体提示词

子智能体（`docs/子智能体设计.md`）复用**同一个 `prompt.Builder`**，但按 `BuildRequest.Audience` 裁剪主侧块。当前实现公式（SSOT：`capabilities/subagent/prompt`）：

```text
# 子 Agent.SystemPrompt（Spawn 前由 Task 写入）
ChildSystem = ChildBase + InheritedRuntimeContract + Definition.persona

# 子 Run BuildSystem（Audience=subagent）
BuildSystem = 时间 + ChildSystem + injectors
            +（plan_mode_rules | task_management，按协作模式）
            + 精简行为规则（熔断 / 工具纪律；跳过 <delegation> 与主侧冗长腔调）

# 子 User 输入（Spawn 前由 contextsnapshot 写入）
ChildUser = [过滤背景?] + BoundaryMessage + DelegationEnvelope(objective/expected/return)
```

- 主 Agent：`Audience=root` 时注入 `<delegation>`（`delegation_posture`）与完整行为规则。
- 子智能体**保留**失败熔断与工具纪律；**跳过**主侧委派纪律，避免「教子 Agent 如何再委派」噪声。
- L4：根会话 `@agent-*` / `@run-agent-*` → `injectAgentMentions` reminder（仍经 `Task`）；子 Run 不解析 mention。
- 完整 Composer / `PromptSpec` 分层仍是目标态；当前以 Audience + 子侧 SSOT 作为过渡实现，不另造平行体系。



### 10.3 与 Hook 机制

`docs/Hook机制设计方案.md` 的 SessionStart / UserPromptSubmit Hook 产出的附加上下文，作为 **L4 动态块**经 `TurnEnhancer` 注入（借鉴 codex `hook_runtime` post-hook developer messages、Kode `drainHookSystemPromptAdditions`）：稳定的 Hook 追加 → system；本轮 Hook 提醒 → user 前缀。Hook 不得覆盖强制块。

## 十一、数据模型（PromptSpec）

`PromptSpec` 是"业务如何自定义提示词"的规范载体。它是**声明式配置模型**，被 `runtime/prompt`（内核，消费它）、`agentapp`（能力域，其 `AgentSpec` 内嵌它）、`subagent`、配置反序列化共同引用。

> **落位决策（经依赖方向校验）**：`PromptSpec` 放 `internal/domain`，**不放** `agentapp/model`。理由：边界文档规定 `internal/capabilities -> internal/domain/internal/platform`，若放 `agentapp/model` 会迫使内核 `runtime/prompt` 反向依赖 capability 域（`runtime → capabilities`，方向错误），且 `agentapp` 尚未落地会绑死提示词系统的实现顺序。放 `internal/domain`（`domain.Agent` 已持有 `SystemPrompt`，同域一致）后，`runtime → domain`、`capabilities/agentapp → domain` 均为合法方向，且两处零耦合。

```go
// internal/domain/prompt_spec.go
// PromptSpec 声明一个可执行智能单元的业务提示词（L2），以及对公共块的有限调整。
type PromptSpec struct {
    // 业务身份人格（L2）。空 → 沿用产品基座/兜底。
    Persona string

    // 业务方法论/约定（L2），等价演进自旧 Agent.SystemPrompt。
    Instructions string

    // 对"非强制"公共块的开关与覆盖（不可命中 security/skill_gateway/failure_policy）。
    Blocks BlockOverrides

    // 末尾追加（对应 --append-system-prompt / app.yaml prompt.append）。
    Append string

    // 模板变量（渲染 persona/instructions 中的 {{.Var}}）。
    Vars map[string]string
}

type BlockOverrides struct {
    Disabled  []string          // 关闭的可关块，如 ["tone"]
    Overrides map[string]string  // 覆盖具名可关块内容；命中强制块 → 校验报错
    Order     map[string]int     // 可选：调整块顺序
}
```

- **只暴露业务维度**：业务方看到的是 persona/instructions/append，不需要理解 L0 实现。
- **覆盖受控**：`Overrides`/`Disabled` 只能作用于非强制块；校验器拒绝命中强制块名（对齐 §五.3 不变量）。
- **与现状兼容**：`domain.Agent` 增加 `Prompt PromptSpec` 字段；旧 `domain.Agent.SystemPrompt` 作为 `PromptSpec` 缺失时的兜底映射（等价 `Instructions`），过渡期不破坏现有 CLI，Phase 2 后清理。
- **配置映射**：`app.yaml` 的 `prompt` 节（Agent App §16.3）直接反序列化为 `PromptSpec`。

## 十二、目录与边界（遵循 `docs/项目目录与边界说明.md`）

```text
internal/domain/
  prompt_spec.go      # 新增：PromptSpec / BlockOverrides（共享内核，runtime 与 capabilities 均可依赖）
  agent.go            # 现有：domain.Agent 增加 Prompt PromptSpec 字段

internal/runtime/prompt/
  interface.go        # 现有：BuildRequest / Fragment / ContextInjector / Builder（保留）
  block.go            # 新增：Block / Role / 分组合成 / 强制块校验
  composer.go         # 新增：Composer 三阶段编排（演进自 builder.go）
  base.go             # 新增：BasePromptProvider 接口 + 默认公共块常量（L0，强制块唯一真源）
  turn.go             # 新增（Phase 2）：TurnEnhancer 接口

internal/capabilities/agentapp/model/
  agent_spec.go       # 后续：AgentSpec 内嵌 domain.PromptSpec（不另造平行 prompt 模型）

internal/capabilities/agentapp/adapter/embedded/code/

products/<product>/bootstrap/

```

边界约束：

- **依赖方向**：`PromptSpec` 在 `internal/domain`；`runtime/prompt → domain`、`capabilities/agentapp → domain` 均合法；**禁止** `runtime → capabilities`（内核不反向依赖能力域）。

- L1 产品基座块、项目文档源、环境上下文源的**具体后端**在 `products/*/bootstrap` 或 `shared/local/prompt` 注入。
- Runtime 只消费合成后的 Block 集，不直接读 App 存储（对齐 Agent App §十四「Runtime 只接收 EffectiveAgentAppProfile」）。
- 强制块常量放 `internal/runtime/prompt/base.go`（产品无关），确保三端唯一真源。

## 十四、分期落地

### 14.0 实施冻结说明（2026-07-19）

经代码对照与产品优先级决策：**现阶段暂不启动** `PromptSpec` / `PromptComposer` / 薄 Agent App 能力合成的实现。原因：

1. 当前过渡实现（扁平 `DefaultBuilder` + `Audience` 裁剪 + 子侧 `ComposeChildSystem` + L4 reminder）已支撑 Phase 1 子智能体与多产品运行。
2. 完整 Composer 应与「自由组装 Prompt / Tools / Skills / MCP / SubAgent / Workflow」的 Agent App 合成链一并推进，避免为子智能体单独造半套组装器。
3. `code` 只是默认场景包，不是产品本身——落地时必须以可切换 App 为前提，而不是继续加厚 `config.yaml` 的编码文案。

**冻结期间维持**：场景人格仍走 `domain.Agent.SystemPrompt`；能力上界仍走 Product Profile；主/子差异继续用 `Audience` + 子侧 SSOT，不得再扩散第二套平行拼装路径。

**解冻条件（任一满足即可重启 Phase 1）**：需要多业务场景切换（非 coding）、或强制块/重复文案开始阻碍演进、或 Agent App Effective 快照开始裁剪 Tools/Skills。

解冻后目标分期仍适用：

| 阶段 | 内容 | 验收 |
| --- | --- | --- |
| **Phase 1** | Block 模型 + `BasePromptProvider`（L0 强制块 + 可关块）；`Composer` 驱动 `BuildSystem`；`code` 内置 `PromptSpec`；迁移硬编码规则与 `skillstack` 重复文案为 L0 块 | 零配置下 `code` 行为不回退；强制块恒在；能列出块来源 |
| **Phase 2** | `TurnEnhancer` + reminder 双通道；`ProjectDocSource`（AGENTS.md）；`EnvContextSource`；`--append-system-prompt` | per-turn 增强生效；AGENTS.md / 环境上下文注入 |
| **Phase 3** | 稳定前缀缓存 + 差量；按 model family 选基座；Desktop 可视化 / Enterprise 版本化审计 | 缓存命中；企业可审计「用了哪版 prompt」 |

每阶段结束删除被替换的临时实现。配套建议：Composer 解冻时同期或紧随落地薄 `EffectiveAgentAppProfile`（PromptSpec + Tools/Skills/MCP/SubAgents 只收窄；Workflow 可仍固定 `react`）。

## 十五、关键决策摘要


| 决策点             | 结论                                                                                              |
| --------------- | ----------------------------------------------------------------------------------------------- |
| 提示词组织方式         | 分层 Block 的有序合成，取代扁平字符串拼接                                                                        |
| 公共/业务如何分离       | 公共 = L0 内核块(产品无关) + L1 产品基座；业务 = L2 App 的 PromptSpec；分界在 L1/L2 之间                               |
| 安全底线保证          | `security/skill_gateway/failure_policy` 为强制块，下层只能追加不能删改（对齐"只收窄不放宽"）                             |
| 业务自定义载体         | 结构化 `PromptSpec`(persona/instructions/blocks/append/vars)，非单一字符串                                |
| `PromptSpec` 落位 | `internal/domain`（共享内核），非 `agentapp/model`；避免内核 runtime 反向依赖能力域，且解耦落地顺序                         |

| 默认应用            | `code` = L0 + 编码 persona/instructions 的实例；是应用不是产品                                               |

| 与子智能体关系         | 复用同一 Composer 与 L0 强制块；子 Agent 可跳过 tone/OutputStyle                                             |
| reminder 投放     | 稳定规则→system；本轮/事件提醒→user 前缀 `<system-reminder>`（Phase 2）                                        |
| 组装时机            | 三阶段：Base(会话) / AppResolve(Run) / TurnEnhance(每 turn)，对齐 Kode 三时间点                               |
| 三端统一/独立         | 内核(Block/Composer/合成/强制块)同源；差异靠 BasePromptProvider/ProjectDocSource/EnvContextSource 后端注入，零产品分支 |
| Enterprise 路径约束 | 环境上下文禁宿主机绝对路径，只输出 workspace-relative/sandbox path                                               |
| 与现状兼容           | `Builder`/`ContextInjector` 保留；`Agent.SystemPrompt` 作为 PromptSpec 兜底；分期迁移                       |
| 参考取舍            | 采纳 codex 双层分离/fragment marker/优先级链、Kode 三时间点/reminder 双通道；不照搬大文案与远程 catalog                     |

## 十六、残留风险（不阻塞 Phase 1）

1. **强制块的最小文案与措辞**需与安全/合规评审对齐后定稿（尤其 Enterprise 合规口径）。
2. **per-turn 差量注入**（Phase 3）与现有 `memory.GetHistory`/消息序列的交互需在实现时验证不破坏对话连续性。
3. **多模型基座选择**依赖多模型网关落地（当前 LLM 路由见 `internal/bootstrap/builder.go:230`），Phase 3 前保持单基座。
4. **AGENTS.md 与本项目根** `AGENTS.md`**（AI 编程助手上下文）的关系**：项目根 AGENTS.md 面向开发者，注入运行时的 AGENTS.md 面向被驱动的 Agent，二者语义需在 Phase 2 实现时明确区分（避免把开发规范当运行时业务约定）。
