# 本地沙箱设计

> [!NOTE]
> **架构澄清与最新集成**：本文档专门描述 `shared/local/sandbox` 本地主机 OS 内核隔离层（macOS Seatbelt / Linux bubblewrap / Windows Job Object）的底层实现细节。
> 关于四层策略评估管道 (`PolicyPipeline`)、`configs/config.yaml` 最新拆分架构、复合脚本 AST 分析 (`ScriptASTAnalyzer`) 以及沙箱 Worker 子 Agent 契约，请参阅全局权威文档：[docs/沙箱与执行引擎终极设计方案.md](file:///D:/workspace/go/genesis-agent/docs/%E6%9C%BA%E5%9C%B0%E6%B2%99%E7%AE%B1%E4%B8%8E%E6%87%A7%E8%A1%8C%E5%BC%95%E6%93%CE%E7%BB%88%E6%9E%81%E8%AE%BE%E8%AE%A1%E6%96%B9%E6%A1%88.md)。

## 1. 目标

本设计用于后续落地 `shared/local/sandbox`：

```text
shared/local/sandbox
shared/local/sandbox/seatbelt
shared/local/sandbox/bubblewrap
shared/local/sandbox/landlock
shared/local/sandbox/windows
```

它只覆盖 CLI/Desktop 可复用的本地主机平台沙箱，不覆盖企业远程 sandbox、Docker/genesis-sandbox API、RBAC/tenant/audit，也不在本轮替代现有 approval、PathResolver、FileSystemBackend、CommandRunner。沙箱是运行时隔离层，不能成为工具直接访问系统资源的捷径。

核心目标：

- command execution 后续可以在 macOS/Linux/Windows 上尽量使用平台沙箱执行。
- 文件系统能力后续可以选择通过沙箱化 helper 进程执行高风险操作，但第一优先仍是 PathResolver、权限、新鲜度、并发锁和 backend port。
- `shared/local/sandbox` 只提供本地主机 sandbox plan 和平台能力，不 import `products/*`，不 import enterprise RBAC/DB，不 import Docker/genesis-sandbox HTTP client。
- 不宣称跨平台安全能力等价。不同 OS 的能力差异必须显式暴露给上层策略。
- `SandboxRequire` 必须 fail closed；`SandboxAuto` 不能静默降级为无沙箱，必须返回 warning/trace/audit 信息，由产品策略决定是否继续。

### 2.1 Codex

参考路径：

- `D:\workspace\go\go-project\codex\codex-rs\sandboxing\src\seatbelt.rs`
- `D:\workspace\go\go-project\codex\codex-rs\sandboxing\src\landlock.rs`

- `D:\workspace\go\go-project\codex\codex-rs\linux-sandbox\src\bwrap.rs`
- `D:\workspace\go\go-project\codex\codex-rs\linux-sandbox\src\landlock.rs`
- `D:\workspace\go\go-project\codex\codex-rs\windows-sandbox-rs\src\lib.rs`
- `D:\workspace\go\go-project\codex\codex-rs\core\src\safety.rs`

Codex 的关键策略：

- macOS 使用固定 `/usr/bin/sandbox-exec`，避免 PATH 注入。Seatbelt profile 按文件读写 root、不可读 glob、受保护 metadata、网络策略动态生成。
- Linux 当前默认不是纯 Landlock，而是 bubblewrap 建文件系统视图，内部再应用 `no_new_privs` + seccomp；Landlock 是 legacy fallback/backup。Codex 明确写到：bubblewrap 默认、Landlock legacy。
- Linux bwrap 语义接近 macOS：根文件系统默认只读，可写 root 叠加，`.git/.codex/.agents` 等 metadata 即使在可写 root 内也保持保护。
- Linux 网络限制不是只靠 bwrap `--unshare-net`，还配合 seccomp 和 proxy-only routing 模式，避免网络策略被 Unix socket 等路径绕过。
- Windows 不是“Restricted Token + Job Object”这么简单。Codex 代码里包含低权限身份、ACL、deny-read ACL、workspace capability SID、WFP、防隐藏用户目录、private desktop、helper materialization、process/conpty capture 等模块。它的 Windows sandbox 是一套专门系统，不应被简化成等价于 Linux/macOS 的轻量 wrapper。
- 安全判断会检查“是否真的有平台沙箱”。例如 patch 看起来只写 workspace，也仍然倾向在平台沙箱内执行，因为硬链接等情况可能突破简单路径判断；没有可用沙箱时会 ask 或 reject，而不是假装安全。

可借鉴：

- 平台能力必须有可检测标签和 metrics/audit tag。
- Linux 以 bubblewrap 为默认强隔离路线，Landlock 只作为受限 fallback。
- macOS 固定系统路径调用 Seatbelt，不从 workspace/PATH 找可执行文件。
- Windows 必须区分轻量约束和强沙箱等级，不能把 restricted token/job object 说成完整文件系统沙箱。
- 文件系统和网络策略要拆开表达，再映射到平台沙箱实现。

### 2.2 Kode-CLI

参考路径：

- `D:\workspace\go\go-project\Kode-CLI\docs\system-sandbox.md`
- `D:\workspace\go\go-project\Kode-CLI\packages\runtime\src\shell\sandboxCommand.ts`
- `D:\workspace\go\go-project\Kode-CLI\packages\runtime\src\shell\linuxSandbox.ts`
- `D:\workspace\go\go-project\Kode-CLI\packages\runtime\src\shell\macosSandbox.ts`
- `D:\workspace\go\go-project\Kode-CLI\packages\core\src\sandbox\systemSandbox.ts`
- `D:\workspace\go\go-project\Kode-CLI\packages\core\src\sandbox\bunShellSandboxPlan.ts`

Kode-CLI 的关键策略：

- safe mode 或环境变量开启 system sandbox；`KODE_SYSTEM_SANDBOX=required` 时无法启动 sandbox 则 fail closed。
- Linux 使用系统 `bwrap`，文件系统策略是 `--ro-bind / /` 后对 allow write roots 做 `--bind`；deny read 目录用 `--tmpfs`，文件用 `/dev/null` 覆盖。
- Linux 网络默认 `--unshare-net`，可借助 host proxy + socat bridge 提供受控网络。
- macOS 使用 sandbox-exec 动态 profile，并用 `<sandbox_violations>` side-channel 标记拒绝，便于 UI/日志结构化展示。
- Settings sandbox 提供 `enabled/excludedCommands/allowUnsandboxedCommands/autoAllowBashIfSandboxed`，强调低摩擦 UX。
- Windows 明确不提供等价系统 sandbox，建议外置 execution kernel，例如 WSL2/VM worker/MCP。

可借鉴：

- required/auto/disabled 三态非常适合 Genesis 的产品 Profile。
- sandbox 可用性探测和执行计划要分离，便于 CLI/Desktop 做不同 UX。
- macOS sandbox violation 应结构化输出。
- Linux bwrap 参数可作为 Genesis 第一版实现参考。

不直接采用：

- Kode 的 best effort 在普通 auto 模式可接受，但不能作为 Genesis 的强安全默认。Genesis 的 `SandboxRequire` 必须失败，`SandboxAuto` 也必须记录降级。
- Kode 在 macOS 上可能回退到 PATH 查找 `sandbox-exec`，Genesis 应采用 Codex 的固定 `/usr/bin/sandbox-exec` 策略。
- Windows 不能只写“不支持”，Genesis 可以先实现轻量进程约束等级，但必须明确它不是完整文件系统/网络沙箱。

## 3. 技术路线选择

最终选择：Codex 安全语义为基线，Kode-CLI 执行参数与 UX 开关为参考。

具体路线：

- macOS：Seatbelt/sandbox-exec 是本地平台沙箱首选。固定 `/usr/bin/sandbox-exec`，动态 profile，默认 network restricted，允许通过产品策略打开网络或代理。
- Linux：bubblewrap 是默认强隔离路线；Landlock 只作为 fallback 或补充能力，不作为默认强沙箱。强网络隔离需要 bwrap network namespace，后续再叠加 seccomp/proxy routing。
- Windows：第一版只做清晰能力分级。Restricted Token + Job Object 属于 `process_constrained`，不是完整文件系统沙箱。强沙箱路线需要低权限身份/ACL/WFP/AppContainer/Windows Sandbox 或外置 execution kernel，不能在文档中承诺轻量实现已经等价。
- **Windows 专项落地规格**（分阶段 L1/L2/L3、对齐 Codex `windows-sandbox-rs`）：见 `docs/Windows本地平台沙箱实现设计.md`。
- 通用 manager 只构造 sandbox plan，不直接执行命令；真实进程启动仍由 `shared/local/execution` 的 CommandRunner 做。

## 4. 包边界

目标包职责：

```text
shared/local/sandbox/
  manager.go       # 能力探测、平台选择、Preference/Requirement 解析、降级语义
  model.go         # SandboxProfile、CapabilityReport、SandboxPlan、EnforcementLevel
  errors.go        # sandbox_unavailable / sandbox_policy_unsupported / sandbox_denied

shared/local/sandbox/seatbelt/
  builder.go       # macOS sandbox-exec argv/profile builder
  detector.go      # /usr/bin/sandbox-exec 检测
  profile.go       # 文件系统/网络策略转 SBPL

shared/local/sandbox/bubblewrap/
  builder.go       # Linux bwrap argv builder
  detector.go      # bwrap/user namespace/WSL/container 能力检测
  filesystem.go    # ro-bind/bind/tmpfs/dev-null/proc/dev 参数构造
  network.go       # unshare-net/proxy-only 计划，后续接 seccomp/proxy bridge

shared/local/sandbox/landlock/
  detector.go      # kernel/ABI 检测
  builder.go       # legacy/fallback plan；只表达 Landlock 可覆盖的策略
  notes.go         # 不支持项说明：deny read、网络、mount view 等

shared/local/sandbox/windows/
  detector.go      # OS/version/token/job/WFP/AppContainer 能力检测
  builder.go       # 第一版 process_constrained plan
  policy.go        # 判断策略是否可由当前 Windows 等级真实支持
```

Go 跨平台约束：

- 平台 syscall、`x/sys/unix`、Windows token/job object、Seatbelt profile 细节必须放在带 build tag 的文件中，例如 `detector_darwin.go`、`builder_linux.go`、`policy_windows.go`。
- 每个平台包都要提供非目标平台 stub，例如 `detector_stub.go`，返回 `sandbox_unavailable` 或 `sandbox_policy_unsupported`，保证 `go test ./...` 在 Windows/Linux/macOS 上都能编译。
- 根包 `shared/local/sandbox` 不直接调用平台 syscall，只根据 `runtime.GOOS` 和平台 package 的 detector 结果选择 backend。
- 平台包之间不能互相 import，避免 Linux 构建时拉入 Windows API 或 macOS Seatbelt 细节。

依赖方向：

```text
products/cli|desktop/bootstrap
  -> shared/local/sandbox
  -> internal/capabilities/sandbox/model 或 execution model
  -> 标准库/平台 syscall
```

禁止方向：

- `internal/capabilities/*` 不 import `shared/local/sandbox`。
- 文件工具和 execution tool 不 import `os/exec`、platform sandbox 实现、Docker/genesis-sandbox client。
- `shared/local/sandbox` 不 import `products/*`，不 import Enterprise RBAC/DB/OIDC。
- `shared/local/sandbox` 不保存产品默认 endpoint、credential、租户策略。

## 5. 统一模型

建议统一模型不是“是否启用 sandbox”一个 bool，而是四层：

```text
SandboxPreference
  disabled     # 明确不使用本地平台沙箱
  auto         # 尝试使用；不可用必须返回 warning，由产品策略决定继续/审批/失败
  required     # 不满足策略必须失败

SandboxKind
  none
  macos_seatbelt
  linux_bubblewrap
  linux_landlock
  windows_process_constrained
  windows_appcontainer
  external

EnforcementLevel
  none                 # 没有系统隔离
  process_constrained  # token/job/process tree 等，不能代表 FS/network 完整隔离
  filesystem           # 可实际限制 FS read/write
  filesystem_network   # 可同时限制 FS 与网络

SandboxPlan
  kind
  enforcement_level
  command argv/env/cwd transform
  filesystem_policy_effective
  network_policy_effective
  helper_path
  warnings
  unsupported_reasons
  audit_tags
```

这样做的原因：

- Windows 第一版只能诚实表达 `process_constrained`。
- Landlock 不能表达 bwrap 的 mount view，也不能可靠支持 deny read/网络隔离。
- macOS Seatbelt 和 Linux bubblewrap 能表达的策略也不完全相同，需要通过 effective policy 反映差异。

## 6. 执行流程

命令执行推荐流程：

```text
execution tool
  -> CommandRiskAnalyzer 分析命令风险
  -> approval.Service 判断是否允许
  -> PathResolver 解析 cwd/workspace
  -> ResourceLocker 加 workspace/run 锁
  -> SandboxRunner 根据产品 Profile 构造 SandboxRequest
  -> shared/local/sandbox.Manager.BuildPlan
  -> shared/local/execution.CommandRunner 执行 plan.command
```

关键点：

- approval 发生在 sandbox 前。沙箱是额外隔离，不替代用户确认。
- `SandboxRequire` 下 BuildPlan 失败即工具失败。
- `SandboxAuto` 下 BuildPlan 如果降级，必须把 warning 传回上层；CLI 可提示或按配置继续，Desktop 可弹窗，Enterprise 本地模式默认不启用。
- CommandRunner 只执行已构造好的 argv/env/cwd，不把策略散落进工具。
- SandboxPlan 必须保留结构化 argv，不能把命令提前拼成单个 shell 字符串。只有当上层 execution policy 明确选择 shell 执行时，shell、`-c`、PowerShell 参数等平台差异才进入 runner adapter；sandbox builder 只包裹 argv，不重新解释用户命令。
- helper path 必须进入 audit，便于确认实际使用的是系统 sandbox、受信任 bwrap，还是产品随包 helper。

## 7. 文件系统策略映射

统一策略先表达为平台无关 policy：

```text
FileSystemPolicy
  readable_roots
  writable_roots
  read_only_roots
  unreadable_globs
  protected_metadata_names
  include_platform_defaults
  temp_roots

NetworkPolicy
  disabled
  full_access
  proxy_only
  loopback_only

ProcessPolicy
  kill_process_tree
  timeout
  pty_allowed
  env_allowlist / env_denylist
```

映射规则：

- writable root 不意味着其中所有 metadata 都可写。`.git/.codex/.agents` 默认保持写保护，除非未来用户明确批准并且产品策略允许。
- unreadable glob 在 Seatbelt 可转 regex deny，在 bwrap 需要展开成具体 mount mask；无法完整展开时不能声称完全生效。
- network proxy-only 在 macOS 可通过 Seatbelt loopback/proxy port 规则表达；Linux 需要 bwrap 网络 namespace + proxy bridge；Windows 需要 WFP 或外置执行环境，restricted token/job object 不足以表达。
- path 一律先规范化，处理 symlink/junction/reparse/case/short path，再进入 sandbox builder。

## 8. macOS Seatbelt 设计

采用策略：

- 只使用 `/usr/bin/sandbox-exec`，不从 PATH 或 workspace 查找。
- 生成 SBPL profile，不写临时 profile 文件，优先通过 `-p` 传递。
- 基础策略 deny default，显式 allow process/sysctl/mach lookup/必要系统读。
- 文件读写分别从 `FileSystemPolicy` 生成：
  - read：只允许 readable roots 或平台默认只读 roots。
  - write：只允许 writable roots、sandbox temp、`/dev/null` 等必要 sink。
  - protected metadata：在 writable root 内额外 deny。
  - unreadable glob：转换为 anchored regex，覆盖 read 和 unlink-style write。
- 网络默认 disabled；proxy-only 只允许 loopback proxy port；允许 Unix socket 必须显式配置。
- stderr 中解析 sandbox denial，可统一包装成 `<sandbox_violations>` 或 Genesis 自己的结构化事件，但不让普通 stderr 文本承载唯一语义。

风险与限制：

- `sandbox-exec` 是 macOS 现存机制但并非 Apple 推荐的新 API，未来可能变化；因此 detector 必须返回版本/可用性。
- Seatbelt profile 很容易因系统工具需要读取平台路径而过严；应保留 `include_platform_defaults`，但默认只加系统路径，不加用户目录。

## 9. Linux Bubblewrap 设计

采用策略：

- 默认 Linux 强隔离选 bubblewrap，不默认选 Landlock。
- bwrap 可执行文件必须来自可信路径。第一版可以做 PATH 探测，但 resolved path 不能位于 workspace、writable roots、临时目录或用户可写的不可信目录；不满足时视为 `sandbox_unavailable`。长期优先支持产品配置的绝对路径或随分发物 vendor/bundled helper。
- 基础参数参考 Codex/Kode：
  - `--new-session`
  - `--die-with-parent`
  - `--unshare-user`
  - `--unshare-pid`
  - `--unshare-uts`
  - `--unshare-ipc`
  - 需要网络限制时 `--unshare-net`
  - 可用时 mount `/proc`，受限容器里允许显式 no-proc fallback，但必须 warning。
- 文件系统：
  - 默认 `--ro-bind / /`。
  - writable roots 用 `--bind root root` 叠加。
  - protected metadata 在可写 root 内用 read-only bind、synthetic empty target 或 create monitor 保护。
  - deny read 目录用 tmpfs mask；deny read 文件用 `/dev/null` ro-bind mask。
  - `/dev` 使用最小必要视图。
- 网络：
  - disabled：unshare network + seccomp deny socket/connect 类调用。
  - proxy-only：unshare network + host proxy bridge；第一版可只设计接口，真实 proxy bridge 后续实现。
  - full access：不 unshare network，但仍可保留 FS sandbox。

风险与限制：

- rootless/user namespace 在某些发行版、容器、WSL1 上不可用。
- bwrap 可能依赖 setuid，因此 seccomp/no_new_privs 必须在 bwrap 完成 mount view 后再应用。Codex 的两阶段 helper 设计值得参考。
- unreadable glob 展开可能代价高，需要上限、超限失败或降级 warning。

## 10. Linux Landlock 设计

采用策略：

- Landlock 作为 fallback 或补充，不作为默认强隔离。
- 只在 kernel/ABI 检测满足时启用。
- 适合表达“读全盘、写仅限 writable roots”的一部分策略。
- 不适合表达：
  - 完整 deny read。
  - mount view 隐藏。
  - 网络隔离。
  - proxy-only。
  - 已打开 fd 的全部绕过场景。

使用原则：

- 当 policy 只要求限制写入且允许全盘读取时，Landlock 可以作为 fallback。
- 当 policy 要求 unreadable roots/globs、network disabled/proxy-only、protected metadata 强保证时，Landlock 不满足，应返回 `sandbox_policy_unsupported`。
- 若用户要求 `SandboxRequire`，Landlock 不满足策略时必须失败，不可降级成 partial enforcement。

## 11. Windows 设计

Windows 需要单独诚实建模，不应照搬 Unix 语义。

### 11.1 第一版可做能力

第一版建议实现 `windows_process_constrained`：

- Restricted Token：降低权限、禁用高危 privilege。
- Job Object：限制进程树生命周期、内存/CPU/子进程行为、确保取消时杀进程树。
- 环境变量与 cwd 清理。
- 可选：ConPTY/stdio capture 边界。
- 路径规范化：drive letter、UNC、case-insensitive、8.3 short path、junction、symlink、reparse point。

这一级只能声明为 process constrained，不能声明为 filesystem sandbox。

### 11.2 强隔离路线

要接近 Codex Windows sandbox，需要后续至少考虑：

- 低权限专用用户或 AppContainer identity。
- workspace ACL / capability SID。
- deny-read ACL 或临时 ACL overlay。
- WFP 网络策略。
- private desktop。
- helper materialization，避免从不可信 workspace 启动 helper。
- 持久 setup/provisioning 和回滚。

### 11.3 策略规则

- 如果 policy 要求限制文件读写，而当前只有 Restricted Token + Job Object，则返回 `sandbox_policy_unsupported`。
- 如果 policy 只要求进程树收束、超时、降低权限，可以返回 `process_constrained`。
- Windows 下不允许 silent fallback 到普通 direct exec。
- 如果产品需要强 FS/network 隔离，优先建议 Docker/genesis-sandbox API、WSL2/VM worker 或后续 AppContainer/WFP 方案。

## 12. 降级与错误语义

统一错误：

```text
sandbox_unavailable
  当前平台没有可用 sandbox backend，或必要二进制/内核能力缺失。

sandbox_policy_unsupported
  平台有 sandbox，但不能表达请求的 policy，例如 Windows restricted token 不能限制 FS read。

sandbox_denied
  sandbox 启动成功，命令运行时被 OS sandbox 拒绝。

sandbox_init_failed
  sandbox backend 启动失败，例如 bwrap namespace/proc mount 失败。
```

Preference 处理：

- disabled：不构造本地平台 sandbox，但仍走 approval/permission/locker。
- auto：尽量构造；不可用或部分策略不支持时返回 warning 和 effective enforcement，由产品策略决定继续、询问或失败。
- required：任何不可用、策略不完整、初始化失败都 fail closed。

审计字段：

```text
sandbox.kind
sandbox.enforcement_level
sandbox.preference
sandbox.effective_filesystem_policy
sandbox.effective_network_policy
sandbox.degraded
sandbox.warnings
sandbox.backend_version
sandbox.helper_path
```

## 13. 与现有架构的关系

- `internal/capabilities/execution` 继续定义 `CommandRunner` / `SandboxRunner` port。`shared/local/sandbox` 是 CLI/Desktop 本地适配实现，不进入 internal。
- `internal/capabilities/filesystem` 的工具仍只能依赖 FileSystemBackend/PathResolver/Approval/Freshness/Locker。若未来要让文件操作也进入 OS sandbox，应实现 sandboxed backend/helper，而不是让工具 import 本地 sandbox。
- Docker/genesis-sandbox API 不属于 `shared/local/sandbox`。它应走 `internal/capabilities/sandbox/contract` 和对应 adapter，由产品 bootstrap 注入；具体 SDK/HTTP 对接和 runtime profile 选择规则见 `docs/沙箱API对接与Profile选择规则.md`。
- Enterprise 默认不接 `shared/local` 本地沙箱；企业强隔离应使用远程 sandbox、容器或受管 worker。

两条沙箱路径必须保持分离：

| 路径 | 代码位置 | 适用场景 | 不应该做什么 |
| --- | --- | --- | --- |
| 本地平台沙箱 | `shared/local/sandbox` | CLI/Desktop 本机执行的 defense-in-depth | 不引入 genesis-sandbox HTTP/SDK、endpoint、credential、租户策略 |
| 外部 sandbox API | `internal/capabilities/sandbox/adapter/*` + 产品 bootstrap | Docker/genesis-sandbox、Enterprise 远程 workspace、CLI/Desktop 可选强隔离 | 不放进 `shared/local`，不让工具直接 new SDK client |

`SandboxProfile` 是两条路径共同理解的执行意图：本地平台沙箱读取其中的 mode、filesystem/network metadata；外部 sandbox adapter 读取 runtime profile、task type、operation、language、risk level。adapter 不应在最后一层重新猜“这是 Skill 还是 Office 还是普通代码”。

## 14. 后续落地顺序

建议后续编码按以下顺序落地：

1. `shared/local/sandbox/model.go`：定义 Preference、Kind、EnforcementLevel、Policy、Plan、CapabilityReport、错误类型。
2. `shared/local/sandbox/manager.go`：实现平台选择、required/auto/disabled 语义、统一降级处理。
3. `seatbelt` builder/detector：只生成 argv/profile 和能力探测，先不接执行。
4. `bubblewrap` builder/detector：先实现 FS read-only + writable roots + network disabled 基础参数；proxy-only/seccomp 预留接口。
5. `landlock` detector/policy check：明确 fallback 支持范围，策略不支持时返回错误。
6. `windows` detector/policy check：先实现 process_constrained plan，不承诺 FS/network enforcement。
7. 接入 `shared/local/execution` 的 SandboxRunner，使用 plan 执行命令。
8. 增加跨平台测试：policy mapping、降级语义、路径规范化、required fail closed。

## 15. 验证矩阵

后续编码完成后至少需要这些验证：

| 维度 | 必测点 |
| --- | --- |
| 编译 | Windows/Linux/macOS 都能 `go test ./...`，非目标平台 stub 不拉入错误 syscall |
| manager | disabled/auto/required 三态；auto 降级有 warning；required fail closed |
| argv | 空 argv、含空格参数、以 `-` 开头参数、PowerShell/cmd/bash 参数不被 sandbox builder 重解释 |
| helper 信任 | bwrap/sandbox helper 位于 workspace 或用户可写目录时拒绝；固定 `/usr/bin/sandbox-exec` |
| macOS | readable/writable roots、protected metadata、network disabled/proxy-only、denial 结构化 |
| Linux bwrap | ro root、writable root 叠加、deny read mask、WSL1/user namespace 不可用、network namespace |
| Linux Landlock | 只允许支持范围内策略；deny read/network/proxy-only 返回 unsupported |
| Windows | process constrained 与 filesystem/network enforcement 明确区分；策略不支持时不静默运行 |
| 审计 | kind、enforcement、effective policy、degraded、helper path、warnings 都可观测 |

## 16. 对 `docs/文件系统设计方案.md` 的修正意见

该文档中的大方向是对的：沙箱应通过 port 接入，工具不能直接依赖平台实现；`SandboxAuto` 不能静默降级，`SandboxRequire` 必须失败。

需要在后续同步修正或澄清的点：

- Linux 不应把 bubblewrap 和 Landlock 视为同级首选。最佳实践是 bubblewrap 作为默认强隔离，Landlock 作为 fallback/补充。
- Windows 不应写成 Restricted Token + Job Object 就能提供完整文件系统沙箱。第一版只能是 process constrained；强隔离需要 AppContainer/ACL/WFP/低权限用户或外置执行环境。
- macOS 应固定 `/usr/bin/sandbox-exec`，不要从 PATH 查找。
- 本地平台沙箱与 Docker/genesis-sandbox API 是两条路径：前者在 `shared/local/sandbox`，后者走 `internal/capabilities/sandbox` contract + adapter。
- 文件系统工具的直接 backend 权限控制仍然必要；OS sandbox 是 defense-in-depth，不替代 approval、resolver、freshness、lock。
