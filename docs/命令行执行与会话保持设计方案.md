# Genesis Agent — 命令行执行与会话保持设计方案

> 本文档针对当前 `run_command` 通用命令行执行工具的单次同步阻塞局限，分析参考项目 Codex 与 Kode-CLI 的实现逻辑，推演 Genesis Agent 在本地（CLI/Desktop）与远程（Enterprise）架构下的最佳会话保持与 PTY 交互设计。

---

## 一、 背景与痛点

当前 Genesis Agent 的 `run_command` 工具运行机制是**同步、单次且无状态**的：
1. **单次同步阻塞**：一旦大模型启动一个命令（如编译或测试），整个 Agent 循环必须挂起等待进程结束；无法支持启动后台常驻服务（如 `npm run dev`）。
2. **缺乏二次交互输入**：在运行过程中，一旦命令需要用户输入（如 `git push` 要求账号密码，或 `npm init` 的问答），由于没有终端输入桥接，进程会永久卡死或抛出 I/O 阻塞错误。
3. **无状态会话**：每次调用 `run_command` 都会开启并销毁一个独立的 Shell。大模型无法在同一个上下文会话中，先后执行 `export VAL=1` 和 `echo $VAL`。

## 三、 本项目架构的最佳设计（统一与独立平衡）

在 Genesis Agent 的产品族（CLI 独立黑窗口、Desktop 本地客户端、Enterprise 分布式 Web 服务）中，我们必须遵循 **“统一核心契约与抽象，独立实现 UI 与进程适配层”** 的原则，确立最佳设计：

### 3.1 统一核心与执行模式整合（Mode Integration）
- **统一工具 Schema**：对外仅暴露一个统一的 `run_command` 工具（在 [internal/capabilities/execution/tool](file:///d:/workspace/go/genesis-agent/internal/capabilities/execution/tool) 中），通过内部参数控制执行通道：
  - **模式 A (默认, `use_pty = false`)**：传统管道重定向。用于 95% 的静态编译、测试与状态查询，其输出为干净的 Plain Text，**StdOut 与 StdErr 物理隔离**，且绝无 ANSI 控制符污染，对大模型阅读极度友好。
  - **模式 B (主动, `use_pty = true`)**：PTY 伪终端模式。用于常驻守护进程或强交互动作。
- **统一会话契约**：在 `internal` 内核中定义统一的 PTY 会话接口，工具层只与契约交互，不知晓本地或远程细节：
  1. 定义 `InteractiveSessionRunner` 契约接口：
     ```go
     type InteractiveSessionRunner interface {
         StartSession(ctx context.Context, sessionID string, cmd Command) error
         WriteStdin(ctx context.Context, sessionID string, data []byte) error
         // SubscribeOutput 支持多路复用订阅输出字节流，返回只读通道 and 取消函数
         SubscribeOutput(ctx context.Context, sessionID string) (<-chan []byte, context.CancelFunc, error)
         // KillSession 强制终止会话，且必须以进程树级（Cascade Tree-Kill）清理所有衍生子进程
         KillSession(ctx context.Context, sessionID string) error
     }
     ```
  2. **CLI/Desktop**：注入 `LocalPTYRunner` 适配器（通过本地 `creack/pty` 维持物理进程，并在销毁时配合系统信号或 Windows Job Object 进行进程树清理）。
  3. **Enterprise**：注入 `SandboxRemotePTYRunner` 适配器（将 `WriteStdin` 与 `SubscribeOutput` 转换为 HTTP/gRPC WebSocket 长连接流，由沙箱远端来维持进程生命周期与资源回收）。

### 3.2 独立适配三端接管界面（Human Takeover）
当 Agent 因安全管控（命中危险前缀）或大模型遇到阻碍请求人类接管时，三端产品各自使用独立的优雅方案进行物理和界面接管，保障最优体验：

1. **CLI 终端（TUI 物理流直接嫁接）**：
   - **设计**：暂退当前 Bubbletea 绘屏流，调用 Go 的 `term.MakeRaw(os.Stdin)` 临时接管键盘，再用 `io.Copy(pty, os.Stdin)` 和 `io.Copy(os.Stdout, pty)` 直接在系统内核层面把用户的物理控制台和后台 PTY 进行嫁接绑定。
   - **优势**：用户能够无延迟地体验原生终端里的 `Vim`、`Tab自动补全` 和 `Ctrl+C/R`。输入 `exit` 退出后自动恢复 Agent 控制流。
2. **Desktop 客户端（Wails IPC 桥接 + xterm.js）**：
   - **设计**：前端渲染 `xterm.js`。利用 Wails 的本地 IPC 总线（Wails Events），实现进程内微秒级键盘输入回传与 PTY 日志推流。
   - **优势**：隐藏式“抽屉终端（Terminal Drawer）”，在需要时一键滑出并接管。
3. **Web / Enterprise 企业端（WebSocket 桥接 + xterm.js）**：
   - **设计**：前端部署 `xterm.js`，通过 WebSocket 与远程后端服务保持长连接。后端在验证租户角色与会话有效性（防跨租户注入）后，拉起 Goroutine 进行 WS 字节流与远程 Docker 容器（Genesis-Sandbox）PTY 端点的双向映射。

## 四、 核心安全机制设计：动态行扫描器（Dynamic Line Scanner）

> [!CAUTION]
> **安全漏洞**：如果大模型首先开启一个安全的 `bash`，随后通过 `write_stdin` 工具向这个 bash 管道连续写入 `rm -rf /`，传统的静态 `policy.go` 前缀扫描将被完全绕过。

为了堵死这一后门，在 `WriteStdin` 适配器层必须建立 **“行级动态命令扫描器”**：
1. **行缓冲（Line Buffer）**：在内存中缓冲大模型输入的数据流，并根据换行符（`\n` 或 `\r`）提取出完整的一行。
2. **动态过滤（On-The-Fly Classification）**：对拆分出的每一个新行命令，执行 `policy.Classify(lineCommand)` 安全研判。
  3. **阶梯提权拦截**：一旦发现命令的危险等级发生了越级（例如从 `ReadOnly` 升至 `Destructive`），立即截断 PTY 数据流并挂起，调起三端的 `TerminalRequester` 人审流，经用户确认允许后再行将字符送入 PTY。

### 4.2 鲁棒性与高可用防卫：会话存活校验与背压缓冲控制

为确保 PTY 交互会话的长周期执行不会导致内存溢出、死锁或向失效管道写入数据，系统在 `session_manager.go` 中必须贯彻以下防御策略：

1. **会话存活检查（Session Preflight Check）**：
   - 每次调用 `WriteStdin` 或 `SubscribeOutput` 前，必须前置检查会话标识的物理存活状态及状态机 `SessionStatus`。
   - 若进程已终止（由于退出或被强杀），应立即抛出强类型错误（如 `ErrSessionClosed`），防止向已关闭的文件描述符写入导致 `SIGPIPE` 崩溃或内存句柄泄漏。
2. **多路复用广播与无阻塞背压（No-Blocking Backpressure）**：
   - PTY 输出的单向读取必须经过 `Multicast Broadcaster` 多路复用。每个订阅者（Log 文件流、WS 客户端、Wails IPC）均持有隔离的流式通道。
   - 为了防止消费过慢的订阅者（如因网络卡顿的浏览器端）拖慢或锁死整个 PTY 的执行，订阅通道必须设置为**有界环形缓冲区（Bounded Ring Buffer）**或设定丢弃窗口（Drop-oldest）。
   - 一旦订阅者缓冲区写满，广播器抛出警告并直接丢弃旧日志，绝对不允许阻塞底层的物理 PTY 读取循环，从而确保后台编译/测试进程的绝对高可用。

3. **Windows 平台 ConPTY 兼容与优雅降级回退**：
   - 在 Windows 环境下，`LocalPTYRunner` 必须依赖 Windows Pseudo Console (ConPTY) 系统调用来实现伪终端（利用 Kernel32.dll 的 `CreatePseudoConsole` 与命名管道桥接）。
   - 由于低版本 Windows 系统（Windows 10 Build 1809 之前，或部分未更新的 Windows Server 容器环境）缺乏 ConPTY 支持，`LocalPTYRunner` 必须具备优雅的主动探测与降级能力。
   - 一旦检测到当前系统不支持 ConPTY，系统严禁发生 Panic，而必须在日志中输出 degraded warning 降级警告，并自动将 `use_pty` 执行通道降级退回到 **“模式 A (基础非 PTY 模式)”**。此时通过拦截提示，告知大模型当前环境不支持实时交互，防止运行陷入死锁。

## 六、 演进路线规划 (Roadmap)

- **Phase 1 (异步基础版)**：实现 `background=true` 非 PTY 重定向输出与文件归档，解除编译阻塞痛点。
- **Phase 2 (本地 PTY 交互版)**：开发 `LocalPTYRunner` 及 CLI/Desktop 前端桥接，完成 **动态行扫描器** 以解决安全漏洞。
- **Phase 3 (远程沙箱网络版)**：对接独立仓库 `genesis-sandbox` 长连接 API，通过 Web Socket 接入 Enterprise Web 审批接管。
