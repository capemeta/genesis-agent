# 项目目录与边界说明

本文档说明 `genesis-agent` 当前代码目录、主要职责、依赖边界和目录变更规则。它是编码时判断“代码应放在哪里、哪些依赖不该出现”的直接参考。

完整架构、数据模型和路线图仍以 `docs/agent loop设计.md` 为准；产品分发细节参考 `docs/产品分发架构设计.md`；文件系统与沙箱细节参考 `docs/文件系统设计方案.md`。

## 阅读原则

- 当前代码可能落后于目标架构。开发时以实际代码为事实，以本文档和架构文档为目标边界。
- 目录说明分为“当前存在”和“目标约束”。当前存在不代表推荐继续扩张。
- 新增代码优先放到最窄、最稳定的能力域或产品域，不为方便跨层 import 打破边界。
- 当前是开发阶段，不需要兼容旧目录或旧数据结构；如果发现旧结构阻碍演进，应提出清理方案。

## 当前一级目录

```text
.
  api/             # API 契约、协议或生成物预留区；当前内容很少，不承载业务实现
  bin/             # 本地构建产物或辅助二进制输出
  cmd/             # 正式产品入口，保持薄壳
  configs/         # 配置样例、默认配置和环境配置
  deploy/          # 部署相关配置与脚本
  distributions/   # 分发、打包、版本产物相关内容
  docs/            # 架构、设计、计划和专项说明文档
  internal/        # 产品无关的核心运行时、能力域和平台基础设施
  products/        # CLI / Desktop / Enterprise 等产品接入面与产品私有实现
  scripts/         # 检查、构建、迁移、生成等开发脚本
  shared/          # 多产品共享但不属于 internal 核心的平台适配
  test/            # 跨包、端到端或集成测试资源
```

以下目录属于本地开发或工具输出，不应被业务代码依赖：

```text
.agents/          # Agent 本地技能与辅助配置
.gocache/         # 仓库内 Go 构建缓存
.gomodcache/      # 仓库内 Go module 缓存
.gotmp/           # Go 工具临时目录
.testdata/        # 本地测试数据或运行样本
genesis-cli.exe   # 本地构建产物
```

## 产品入口目录

`cmd/<product>` 是正式产品进程入口，当前包括：

```text
cmd/
  genesis-cli/
  genesis-desktop/
  genesis-enterprise/
```

边界要求：

- `cmd/<product>` 只 import 标准库和自己的 `products/<product>/bootstrap`。
- `cmd` 不直接 import `internal` 具体实现，不直接 import `products/<product>/internal`。
- 不恢复旧兼容入口；新增正式产品入口必须走 `cmd/<product>`。

## 产品目录

当前产品目录：

```text
products/
  cli/
    bootstrap/
    internal/
  desktop/
    bootstrap/
    internal/
  enterprise/
    bootstrap/
    internal/
    web/
```

职责说明：

- `products/<product>/bootstrap` 是产品级 Composition Root，负责选择产品 Profile、持久化、Trace、Auth、Sandbox、工具适配、外部接口等具体实现。
- `products/<product>/internal` 放产品私有实现，只能被同一产品内部和该产品 bootstrap 使用。
- `products/enterprise/web` 是企业版 Web 前端目录。
- Desktop 前端未来如独立落地，默认放到 `products/desktop/web`。

产品边界：

- CLI 默认不引入 PostgreSQL、Wails、企业 OIDC、企业 sandbox client。
- Desktop 可以引入桌面和本地主机能力，但产品私有能力仍放在 `products/desktop/internal`。
- Enterprise 默认不引入 `shared/local` 的本地主机执行、文件系统实现。
- `products/enterprise` 只负责企业上下文、治理策略、endpoint/credential 注入和 bootstrap，不拥有通用 genesis-sandbox client。
- 其他产品不得 import 某个产品的 `internal` 包。

## internal 目录

当前 `internal` 目录：

```text
internal/
  app/
  bootstrap/
  capabilities/
  domain/
  interfaces/
  platform/
  runtime/
```

目标职责：

- `internal/runtime`：Agent Runtime 内核，包括 action、context、stop、strategy、session、`collab`（规划模式协作状态与工具闸门）等运行时概念。不得直接依赖 Eino、OpenAI、具体工具、MCP Client、Docker/WASM、PostgreSQL/Redis 或 HTTP/CLI。
- `internal/capabilities`：LLM、Tool、Memory、Sandbox、Trace、Usage、Profile 等能力域。能力内部可按 `contract / adapter / service / model` 细分，但优先保持域内聚合。
- `internal/domain`：核心领域模型。不得依赖外部框架、产品层或基础设施实现。
- `internal/platform`：跨域可复用基础设施，例如 config、db、cache、idgen、httpclient、errors、observability。不得承载具体业务能力。
- `internal/app`：少量跨能力域应用编排。能力自己的应用服务优先进入对应 `internal/capabilities/*/service`，不要继续无边界扩张。
- `internal/bootstrap`：产品无关 shared builder。不得 import `products/*` 或 `shared/local/*`，不得承载 CLI/Desktop/Enterprise 产品语义。
- `internal/interfaces`：当前仍存在的历史接入面目录。目标上不继续扩张，产品接入面应迁移到各自 `products/<product>/internal`；新增 HTTP/SSE/CLI/TUI/Wails 接入代码不得放入这里。

当前能力域目录：

```text
internal/capabilities/
  approval/
  audit/
  capability/
  connection/
  credential/
  execution/
  filesystem/
  llm/
  memory/
  plan/
  package/
  policy/
  profile/
  sandbox/
  skill/
  tool/
  trace/
  usage/
  web/
```

能力域边界：

- `filesystem` 能力不得直接依赖 `os`/`filepath` 执行真实文件读写，不得直接 import Docker SDK、HTTP/gRPC client、Wails、RBAC、DB；只能依赖 `FileSystemBackend`、`PathResolver`、`FreshnessTracker`、权限/审计等 port。
- `execution` 能力负责命令执行抽象，通过 `CommandRunner` / `SandboxRunner` port 执行，并复用 `PermissionEngine`、`ToolScheduler`、`PathResolver`、`SandboxProfile`；工具层不得直接调用 `exec.Command` 或产品 sandbox 实现。
- `sandbox` 能力只放产品无关的 sandbox contract/model/adapter。`internal/capabilities/sandbox/adapter` 可以放产品无关 genesis-sandbox API client，但不得放产品默认 endpoint、credential、租户/RBAC 策略。
- `capability` 是运行时能力索引和 RuntimeAdapter 端口的通用能力域；不得反向依赖 Package Marketplace、Skill、Tool 的具体实现。
- `package/marketplace` 是 Package Marketplace 的产品无关模型、契约和编排服务；不得放在 `skill` 能力域下，也不得直接依赖 CLI/Desktop/Enterprise 产品实现。
- `tool` / `mcp` / `skill` / `sandbox` / `memory` / `llm` / `trace` / `usage` 等能力应可独立演进，通过契约接口装配。

## shared 目录

当前共享本地适配目录：

```text
shared/
  local/
    execution/
    filesystem/
    pathresolver/
    plan/
    sandbox/
    skill/
    skillmarket/
```

职责说明：

- `shared/local` 只放 CLI/Desktop 共享的本地主机适配实现。
- 本地文件系统、路径解析、本机命令执行、本机 sandbox、Package Marketplace 本地 JSON/cache/store 等偏宿主机能力可以放在这里。

边界要求：

- `shared/local` 不得 import `products/*`。
- `shared/local` 不得引入 Wails、PostgreSQL、企业 OIDC/RBAC。
- `shared/local` 可以依赖 `internal/capabilities/*` 的 contract/model，但不得反向让 `internal` 依赖 `shared/local`。
- Docker/genesis-sandbox API backend 不属于 local host backend，不应放在 `shared/local`。

## 依赖方向

推荐依赖方向：

```text
cmd/<product>
  -> products/<product>/bootstrap
  -> internal / products/<product>/internal / shared/local（仅 CLI/Desktop）

internal/app
  -> internal/runtime / internal/capabilities / internal/domain

internal/capabilities
  -> internal/domain / internal/platform

shared/local
  -> internal/capabilities/* contract/model
```

禁止方向：

- `internal/...` 不得 import `products/...`。
- `internal/...` 不得 import `shared/local/...`。
- `internal/bootstrap` 不得 import 产品目录或本地共享适配。
- `domain` 不得依赖框架、数据库、HTTP、CLI、产品层或 sandbox 实现。
- 产品之间不得互相 import 对方的私有实现。

## 沙箱与路径展示边界

- 沙箱服务已拆分为独立仓库 `D:\workspace\go\genesis-sandbox`；本仓库只保留对接契约和客户端适配层。
- 主平台不得直接依赖 `genesis-sandbox` 仓库的 `internal/...`。
- **三端业务路径展示**（详见 `docs/Skill三模式执行与依赖闭环设计.md` §4.3.1）：
  - **无沙箱 / 本地平台沙箱**：共用宿主工作区相对路径（`.genesis/runs/...`）；本地平台沙箱只做进程隔离，不另开 `/workspace` 展示空间，也不得回传宿主机绝对路径。
  - **Docker / genesis-sandbox 远程**：执行态可用 sandbox path（`/workspace`）；正式交付不得以宿主绝对路径或旁路 `DownloadArtifact` 落盘表示，须经 ProducedResource → Artifact Gate → Delivery（见下方权威文档）。
- 工具和 UI 的业务路径只能是 workspace-relative、sandbox path 或 resource id（candidate_id / artifact id）。
- `SandboxAuto` 不能静默降级为无沙箱；降级必须有 warning/trace/audit。
- `SandboxRequire` 不满足时必须失败。
- **产物与 Artifact Delivery（权威）**：[`docs/superpowers/specs/2026-07-17-produced-resource-artifact-delivery-architecture.md`](superpowers/specs/2026-07-17-produced-resource-artifact-delivery-architecture.md)。旧 `artifacts[].path` / `LocalArtifactRoot` 自动回收表述已废止。

## 文档与目录变更规则

目录结构不是永久不可变，但跨层级、跨产品或大规模迁移必须先说明并等待确认。说明至少包括：

- 问题现象：当前目录或依赖哪里影响扩展、测试、复用或治理。
- 为什么现有结构不适合：具体到依赖方向、职责混杂、生命周期或产品边界。
- 候选方案：至少给出推荐方案和主要备选方案。
- 影响范围：涉及目录、包、入口、配置、脚本、文档、测试。
- 迁移步骤：如何分批移动、如何清理旧代码和旧路径。
- 验证方式：构建、测试、产品隔离脚本和必要的人工检查。

新增产品、入口或产品特有能力后，必须同步维护 `scripts/check_product_isolation.ps1`，用 import graph 检查边界，而不是只靠目录约定。

## 验证建议

涉及后端目录或依赖边界修改时，优先运行：

```powershell
$env:GOCACHE='D:\workspace\go\genesis-agent\.gocache'; $env:GOMODCACHE='D:\workspace\go\genesis-agent\.gomodcache'; go test ./...
```

涉及产品隔离时，额外运行：

```powershell
.\scripts\check_product_isolation.ps1
```

如只修改文档，也应至少读回相关片段，确认 Windows 路径和 `$env:` 示例没有被错误插值。
