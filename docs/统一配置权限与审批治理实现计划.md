# 统一配置、权限与审批治理实现计划

> 本计划对应 `docs/统一配置权限与审批治理设计.md`。目标是先落地可运行、可测试、可渐进替换现有 static approval 的最小闭环，再逐步扩展到文件系统、命令、网络、连接与企业治理。

## 1. 实现原则

1. **先兼容现有审批链路**
   - 现有工具已经依赖 `approvalcontract.Service`、`approvalcontract.PolicyEngine` 和 `approvalmodel.Request`。
   - 第一批实现不破坏这些接口，而是在新 `policy` 能力域中提供一个兼容 `approvalcontract.PolicyEngine` 的 adapter。

2. **先配置结构，再策略执行**
   - `internal/platform/config.Config` 先增加 `Policy PolicyConfig`，并能解析 `configs/config.yaml` 的安全默认值。
   - 策略执行先覆盖现有 static policy 的能力：default、critical/protected、workspace/external、危险操作、文件 allow/deny path。

3. **产品差异留在 bootstrap**
   - CLI 先使用配置驱动 policy engine。
   - Desktop/Enterprise 暂时不强制接入，保留 shared builder / 产品 bootstrap 后续注入点。
   - Enterprise admin policy、DB GrantStore、审批流只预留接口方向，不在第一批实现。

4. **deny 永远优先**
   - 每次授权前都重新评估 policy，确保 hard deny 能压制已有 session/project grant。
   - Runtime grant 只能放宽 ask，不能绕过 deny。

5. **密钥不进入普通配置**
   - 本计划只实现 policy/config 框架，不把 provider API key、Credential 明文或 Connection secret 放入 `policy`。

## 2. 当前代码事实

| 模块 | 当前状态 | 对实现的影响 |
|---|---|---|
| `internal/platform/config` | 已加载 `config.yaml`、`config.local.yaml`、`AGENT_` 环境变量；已有 `SecretsConfig` 但 `applySecretsDefaults` 未在 `validate` 中调用 | 新增 `PolicyConfig` 时一并补默认值测试，并修复 secrets 默认值应用 |
| `internal/capabilities/approval/model` | 已有 Action、Risk、GrantScope、PolicyType、Request、PolicyResult | 第一批复用，不新增平行模型，避免重复概念 |
| `internal/capabilities/approval/service` | 每次先 evaluate，再查 session cache；只缓存 session | 保留该安全顺序；第一批不扩展通用缓存 scope，避免 grant 语义半成品 |
| `internal/capabilities/approval/adapter/static` | metadata 驱动策略：critical/protected deny，trusted/workspace allow，external/dangerous ask | 新 policy engine 先行为兼容，再增加配置覆盖 |
| `internal/capabilities/filesystem/permission` | BuildApprovalRequest 已填 scope、operation、backend、workspace_metadata_write 等 metadata | 文件 matcher 可以直接消费现有 metadata，减少工具改造 |
| `internal/capabilities/filesystem/permission/runtime` | 文件授权能记 session/project，但 approval service 只记 session | 第一批保留文件 runtime grant，通用 GrantStore 的 project 持久化后续做 |
| `products/cli/bootstrap/container.go` | `buildBaseApprovalService` 固定 static policy；文件和命令共用文件权限感知 approval | 第一批改为从 config 构建 policy engine，并保持 quiet requester 行为 |

## 3. 第一批落地范围

第一批目标：让项目拥有统一 policy 配置结构、配置驱动的审批策略、文件路径 allow/deny 的最小实现，并让 CLI 使用它。

### 3.1 配置结构

修改文件：

- `internal/platform/config/config.go`
- `internal/platform/config/config_test.go`
- `configs/config.yaml`

新增结构建议：

```go
type Config struct {
    LLM        LLMConfig        `mapstructure:"llm"`
    HTTPClient HTTPClientConfig `mapstructure:"http_client"`
    Secrets    SecretsConfig    `mapstructure:"secrets"`
    Policy     PolicyConfig     `mapstructure:"policy"`
    Agent      AgentConfig      `mapstructure:"agent"`
    Log        LogConfig        `mapstructure:"log"`
    Server     ServerConfig     `mapstructure:"server"`
}
```

第一版 `PolicyConfig` 只实现会被用到的字段，避免完整 schema 一次性膨胀：

```go
type PolicyConfig struct {
    Defaults PolicyDefaultsConfig `mapstructure:"defaults"`
    Files    PolicyFilesConfig    `mapstructure:"files"`
    Commands PolicyCommandsConfig `mapstructure:"commands"`
    Web      PolicyWebConfig      `mapstructure:"web"`
    Sandbox  PolicySandboxConfig  `mapstructure:"sandbox"`
}
```

第一批真正参与审批执行：`defaults` 与 `files`。`commands.default`、`web.search.default`、`web.fetch.default`、`sandbox.default_mode` 先进入配置结构和默认值，用于保持 schema 方向一致；在命令、网络和沙箱 matcher 接入前，不宣称它们已经产生运行时治理效果。

默认值：

- `defaults.unknown=ask`
- `defaults.dangerous=ask`
- `defaults.critical=deny`
- `defaults.allowed_grant_scopes=[once, turn, session, project]`
- `files.default=ask`
- `files.workspace.read/list/walk/write/edit=allow`
- `files.workspace.delete=ask`
- `files.external.read/list/walk/write/edit=ask`
- `files.external.delete=deny`
- `files.protected.default=deny`
- `files.workspace_metadata.write=deny`
- `commands.default=ask`（第一批仅解析，不接入命令工具）
- `web.search.default=ask`（第一批仅解析，不接入网络工具）
- `web.fetch.default=ask`（第一批仅解析，不接入网络工具）
- `sandbox.default_mode=disabled`（第一批仅解析，不覆盖 CLI `--sandbox` 行为）

同时修复：`validate` 中调用 `applySecretsDefaults(&cfg.Secrets)`。

### 3.2 新增 policy 能力域

新增目录：

```text
internal/capabilities/policy/
  contract/
    evaluator.go
  model/
    model.go
  service/
    evaluator.go
  adapter/
    approval/
      engine.go
    config/
      builder.go
  matcher/
    filesystem/
      matcher.go
```

职责：

- `model`：第一批只定义 `Input`、`Subject`、`Environment`、`Decision`、`MatchResult` 和文件规则所需模型；Action/Resource 复用 `approvalmodel.Request` 里的字段，避免重复。
- `contract`：第一批只定义 `Evaluator`、`Matcher`，不提前定义 `RuleSource`、`GrantStore`，避免无实现端口漂在架构里。
- `service`：执行 matcher 调用、deny 优先、默认值兜底、普通授权 scope 过滤。
- `adapter/config`：把 `platform/config.PolicyConfig` 转成 policy service 可消费的规则对象。
- `adapter/approval`：把新 policy evaluator 适配成 `approvalcontract.PolicyEngine`，输入仍是 `approvalmodel.Request`，输出仍是 `approvalmodel.PolicyResult`。
- `matcher/filesystem`：根据 request metadata 中的 `scope`、`operation`、`backend`、`workspace_rel`、`workspace_metadata_write` 执行文件策略。

第一批不做：

- DB PolicyStore。
- project grant 文件持久化。
- Enterprise tenant/global 策略。
- command/web/credential/mcp 专用 matcher 完整实现。

### 3.3 文件系统 matcher 规则

输入来源：现有 `filesystem/permission.BuildApprovalRequest` 已经生成 metadata。

执行顺序：

1. 如果 `critical/protected/workspace_metadata_write` 命中，按 `files.protected.default` 或 `files.workspace_metadata.write` 返回 deny。
2. 如果命中 `files.deny_paths`，返回 deny。
3. 如果命中 `files.allow_paths` 且 operation 匹配，返回 allow 或 ask，取决于 path rule 的 decision/scope 设计。第一批可以让 allow_paths 只代表 allow，不存 scope。
4. 根据 `scope=workspace/external/protected` 和 operation 查表。
5. 如果 `destructive=true` 或 operation=delete，提升 risk。
6. 未分类走 `defaults.unknown`。

路径匹配第一批要求：

- `filepath.Clean`。
- Windows 下大小写归一。
- 路径边界使用分隔符判断，禁止单纯字符串前缀。
- symlink/junction 最终目标校验先留 TODO，不在第一批实现，因为当前 `ResolvedPath` 已是 backend path，但尚未统一暴露最终真实路径。

### 3.4 Approval service 接入

修改文件：

- `products/cli/bootstrap/container.go`
- 必要时调整 shared builder 的 BuildOptions，但优先不改 shared builder。

计划：

1. `buildProductRuntime` 或 `Container.Service` 已经拿到 `configDir`，但 `buildBaseApprovalService` 目前只接收 quiet。
2. 将 `buildBaseApprovalService(quiet bool)` 改为 `buildBaseApprovalService(quiet bool, cfg config.Config)` 或传入 `cfg.Policy`。
3. 第一批在 CLI bootstrap 中调用 `platform/config.Load(configDir)` 读取 policy 配置；因为产品工具构建发生在 `shared.BuildAgentService` 之前，暂时无法复用 shared builder 内部加载的 cfg。
4. 用 `policy/adapter/config` 构建 evaluator，再用 `policy/adapter/approval` 转成 `approvalcontract.PolicyEngine`。
5. 如果 policy 构建失败，返回启动错误，不静默回退到 static policy。
6. quiet requester 保持 `approvaldeny.NewRequester()`，确保 ask 自动 deny。

注意：当前 `shared.BuildAgentService` 在内部加载 config，而 CLI 的产品工具和 approval service 在调用 shared builder 之前构建。第一批允许 CLI bootstrap 重读一次 config，但只读取本地文件和环境变量，不产生外部副作用；后续应把已解析 `Config` 提升为产品级配置上下文，避免双读。

### 3.5 Grant scope 处理

第一批不改变 approval service 的通用缓存语义：

- 通用 approval service 继续只缓存 session。
- 文件系统外层 `RuntimeFilePermissions` 继续支持 session/project。
- policy engine 只负责返回 `SuggestedScopes`，并限制 tenant/global 不进入普通 requester。

后续 Phase 2 再新增通用 GrantStore：

- memory store 支持 turn/session。
- file store 支持 project。
- Enterprise DB store 支持 tenant/global。

### 3.6 配置样例

`configs/config.yaml` 新增安全默认值，避免明文密钥：

```yaml
policy:
  defaults:
    unknown: ask
    dangerous: ask
    critical: deny
    allowed_grant_scopes: [once, turn, session, project]
  files:
    default: ask
    workspace:
      read: allow
      list: allow
      walk: allow
      write: allow
      edit: allow
      delete: ask
    external:
      read: ask
      list: ask
      walk: ask
      write: ask
      edit: ask
      delete: deny
    protected:
      default: deny
    workspace_metadata:
      write: deny
      paths: [.git, .agents, .codex]
  commands:
    default: ask
  web:
    search:
      default: ask
    fetch:
      default: ask
  sandbox:
    default_mode: disabled
```

## 6. 风险与取舍

| 风险 | 取舍 |
|---|---|
| 新 policy 模型和 approval 模型重复 | 第一批 adapter 兼容现有 approval contract，policy model 只放通用主体/环境/规则，不复制 Request 全量字段 |
| CLI bootstrap 重复加载 config | 第一批接受双读，因为工具构建早于 shared builder；双读必须无外部副作用，Phase 2 收敛为产品级配置上下文 |
| project grant 语义不完整 | 第一批保留文件 runtime grant，不做通用 project grant 持久化 |
| symlink/junction 最终目标未完全覆盖 | 第一批保持当前 PathResolver 行为，matcher 中留清晰 TODO，Phase 2 与 PathResolver 统一补真实路径字段 |
| web/command/sandbox 配置先有字段但不执行 | 在结构注释、配置样例和测试名中标记为预留；真正执行放 Phase 2，避免伪安全 |
| config.local 可含密钥与“普通配置不含明文”冲突 | 文档和注释统一表述为本地开发可用但不推荐，生产不依赖；CredentialStore 是长期方向 |

## 7. 后续 Phase 2

1. 通用 GrantStore 支持 turn/session/project，CLI project grant 写 `.genesis/grants.yaml`。
2. command matcher 接入 `command.exec`，支持 program、args_prefix、cwd_scope、dangerous deny。
3. web matcher 接入 `web_search` / `web_fetch`，实现 domain/IP/redirect/SSRF 规则。
4. sandbox policy 与 `shared/local/sandbox.Manager` 打通，`SandboxRequire` 不可降级。
5. connection/credential resolver 接入 `connection.use`、`credential.resolve` 策略。
6. Desktop 权限中心和 Enterprise DB/RBAC/admin policy。
