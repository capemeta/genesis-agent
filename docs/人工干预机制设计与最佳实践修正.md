# 人工干预（Human-in-the-Loop）机制设计与最佳实践修正

> 本文档针对当前代码中"人工干预 / 审批"实现做一次基于代码事实的最佳实践评估，指出与既有设计目标（`docs/agent loop设计-SSE与重试策略设计.md`、`docs/统一配置权限与审批治理设计.md`）的差距，并给出可落地的修正架构与迁移路线。
>
> 结论先行：**当前实现是"工具执行链路上的同步阻塞式审批"，对 CLI 本地交互是合理的局部最优；但它不是面向多产品、长任务、企业异步审批场景的最佳实践，且与项目自身的 SSE / 干预设计脱节。** 需要引入统一的 `Intervention`（干预）抽象、Run 级暂停/恢复状态机、可持久化的干预请求，以及"同步阻塞"与"挂起-恢复"两种可替换的传输策略。

---

## 一、现状：当前实现是怎么做的

### 1.1 核心链路

人工干预当前**不发生在 Run/Strategy 层，而是发生在 Tool 内部**：工具在真正执行副作用前，构造 `approval.Request` 调用 `approval.Service.Authorize`；`Authorize` 走 `policy.Evaluate` 得到 `allow/ask/deny`；命中 `ask` 时同步调用产品侧 `Requester.RequestApproval` 阻塞等待用户决策。

```text
Tool.Execute
  └─ BuildApprovalRequest → approval.Service.Authorize(ctx, req)
        ├─ policy.Evaluate → allow | ask | deny
        ├─ allow  → 直接放行
        ├─ deny   → 直接拒绝（作为工具错误回给 LLM，Loop 继续）
        └─ ask    → 查 session grant 缓存
                     └─ 未命中 → Requester.RequestApproval(ctx, req, result) 【同步阻塞】
                                   ├─ CLI TUI：p.Send(ApprovalRequiredMsg) + select{<-decisionCh}
                                   ├─ CLI 终端：stdin ReadString 循环
                                   └─ Enterprise：headlessAskApprover 自动批准(session)
```

关键接口（同步返回 `Decision`）：

```20:23:internal/capabilities/approval/contract/interface.go
// Requester 负责把 ask 决策交给产品侧确认。
type Requester interface {
	RequestApproval(ctx context.Context, req model.Request, result model.PolicyResult) (model.Decision, error)
}
```

编排服务对 `ask` 的处理（同步阻塞 + session grant 缓存）：

```83:109:internal/capabilities/approval/service/service.go

		if decision, ok, err := s.cached(ctx, req); err != nil {
			// ...

		}

```

CLI TUI 通过 channel 解耦 UI 与阻塞 goroutine（这一点做得好）：

```34:63:products/cli/internal/approval/tui_requester.go

```

### 1.2 当前实现做得好的部分

这些设计是合理的，修正时应保留：

- **统一 `approval.Request` + `allow/ask/deny` 策略模型**：能力域各自 `BuildApprovalRequest`，产品差异隔离在 `Requester`。
- **`policy` 与 `approval` 分层**：`policy` 判断规则，`approval` 编排人机交互，职责清晰。
- **deny 优先、审计 `approval.decision`、session grant 缓存**。
- **CLI TUI 用 channel 解耦 UI 线程与业务 goroutine**：这是同步 HITL 的正确写法。
- **审批通过后 `NotifyApprovalGranted` 联动 Repeat Guard**：避免"批准后又被重复保护拦截"。

## 二、最佳实践评估：差距与反模式

判断一个 HITL 机制是否为最佳实践，主要看四条：**(A) 干预是否是可持久化、可恢复的一等状态；(B) 是否支持客户端断开后重连；(C) 传输策略是否可替换（同步/异步）而不改业务代码；(D) 抽象是否覆盖"审批"以外的干预类型。** 当前实现在这四条上都有明显差距。

| 维度 | 当前实现 | 最佳实践目标 | 差距 |
|---|---|---|---|
| 干预状态 | Run 始终 `running`，无 `paused` | Run 进入 `paused/waiting_intervention` | ❌ 缺状态机 |
| 传输方式 | 同步阻塞 `RequestApproval` 占用执行 goroutine | 可选同步/挂起-恢复，业务无感知 | ❌ 强耦合同步 |
| 断连恢复 | HTTP 断开 = `context cancel` = denied | 断连后干预仍 pending，可重连 respond | ❌ 无 |
| 持久化 | 仅进程内内存缓存，重启丢失 | 干预请求持久化 + `resume_token` | ❌ 无 |
| 抽象范围 | 仅"审批"(tool_approval) | tool_approval / manual_override / param_input 等 | ❌ 窄 |
| Abort 语义 | 只影响当前工具调用 | 上升为 Run 取消 | ❌ 未打通 |
| Enterprise | `ask` 一律自动批准 | 走真实审批流（人/RBAC/策略） | ❌ 反模式 |

### 2.1 反模式一：Enterprise 静默自动批准 `ask`

```186:203:products/enterprise/bootstrap/container.go
// headlessAskApprover 用于无交互 HTTP 服务：PolicyAsk 自动批准（session 范围）
```

这等于在企业环境里**关闭了人工干预**。策略判为"需要人来看一眼"的高危动作被无人值守地放行，且审计里记录为"已批准"，具有误导性。这是本次修正**优先级最高**要消除的问题。

### 2.2 反模式二：同步阻塞占用执行资源、无法跨断连恢复

`Requester.RequestApproval` 在返回 `Decision` 之前一直阻塞。对 CLI（进程即会话、终端常驻）没问题；但对 HTTP/SSE：

- 审批期间执行 goroutine 被占用（长任务多次审批 = 大量 goroutine 挂起）。
- 客户端刷新页面 / 网络抖动 → `ctx.Done()` → 直接判 `denied`，用户"还没点就被拒了"。
- 进程重启后正在等待的干预全部丢失，Run 无法恢复。

### 2.3 反模式三：领域已预留但未接线（死代码）

以下类型/常量已定义却**全仓库无引用**，说明设计意图存在但实现缺失：

```6:14:internal/domain/errors.go
var (
	ErrRunNotFound          = errors.New("run not found")
	ErrRunAlreadyResuming   = errors.New("run already resuming")
	ErrRunPaused            = errors.New("run is paused")
	// ...
	ErrInterventionRequired = errors.New("human intervention required")
)
```

- `RunStatus` 仅 `created/running/completed/failed/cancelled`，无 `paused`。
- `TaskStatusWaitingApproval`、`EventTypeApprovalGranted/Rejected` 定义了但没人用。
- `DecisionTimedOut` 定义了但没有超时流程产生它。

### 2.4 反模式四：抽象名只表达"审批"，无法承载更宽的干预

`Requester` / `ApprovalRequiredMsg` / `RequestApproval` 命名与语义都锁死在"审批"。但 `docs/agent loop设计-SSE与重试策略设计.md` 已经定义了更宽的干预族：`tool_approval`、`manual_override`（多次失败后人工决策/接管）、`param_input`（要求补充参数）等。当前抽象无法表达"暂停 Run，让人选一条路径/补一个参数再继续"。

### 2.5 反模式五：`abort` 语义未打通

用户在审批卡片选"中止任务"返回 `DecisionAbort`，但它被 `isApproved()` 当作"未批准"，等价于 `denied`：只让当前工具失败，Loop 继续，**并不会终止整个 Run**。这与用户直觉不符。

## 三、修正目标架构

核心思想：**把"干预"提升为 Run 的一等概念，把"决策如何从人送达等待方"与"执行流是否需要挂起"两个关注点解耦。** 业务代码（工具、策略）只声明"这里需要一次人工决策"，具体是同步阻塞还是挂起-恢复，由可替换的 `InterventionBroker` 实现决定。

### 3.1 分层与依赖方向

```text
┌────────────────────────────────────────────────────────────┐
│ Tool / 能力域                                                 │
│   BuildApprovalRequest → approval.Service.Authorize           │  ← 不变
├────────────────────────────────────────────────────────────┤
│ approval.Service                                              │
│   policy.Evaluate → allow/deny 直接返回                        │
│   ask → InterventionBroker.Await(ctx, InterventionRequest)    │  ← 由 Requester 改为 Broker
├────────────────────────────────────────────────────────────┤
│ InterventionBroker（port，可替换传输策略）                     │
│   ├─ InProcessBroker  ：同步阻塞 + 按 intervention_id 路由      │  CLI / 单进程 SSE
│   └─ DurableBroker    ：持久化挂起 + 恢复（返回 ErrInterventionRequired）│ Enterprise / 长任务
├────────────────────────────────────────────────────────────┤
│ InterventionStore（持久化 pending 干预 + resume_token）        │
│ Run 状态机（running ↔ paused/waiting_intervention）            │
│ 传输：CLI TUI channel / SSE intervention.* + REST respond      │
└────────────────────────────────────────────────────────────┘
```

依赖方向不变：能力域只依赖 `approval.Service`；`approval.Service` 依赖 `InterventionBroker` port；具体 Broker、Store、Run 状态机、SSE/REST 由产品 bootstrap 注入。

### 3.2 统一干预模型（新增，与 approval 模型对齐）

在 `internal/capabilities/intervention/model`（或复用 approval 域）新增：

```go
// InterventionType 覆盖审批以外的干预场景，与 SSE 事件 intervention_type 对齐。
type InterventionType string

const (
	InterventionToolApproval   InterventionType = "tool_approval"   // 高危工具执行前授权
	InterventionManualOverride InterventionType = "manual_override" // 多次失败/不确定，人工决策或接管
	InterventionParamInput     InterventionType = "param_input"     // 要求补充/修正参数
	InterventionAuthorization  InterventionType = "authorization"   // 调用前缺少授权，引导完成授权
)

// InterventionRequest 是一次可持久化的干预请求。
type InterventionRequest struct {
	ID             string           // intervention_id（持久化主键）
	RunID          string
	SessionID      string
	Type           InterventionType
	Approval       *approvalmodel.Request  // Type=tool_approval 时携带原审批请求
	Title          string
	Description     string
	AllowedActions []Action         // 可选动作（allow/deny/abort/自定义路径/参数表单）
	ResumeToken    string           // 防伪造，绑定该干预
	CreatedAt      time.Time
	ExpiresAt      *time.Time       // 超时策略
	Status         Status           // pending | resolved | expired | cancelled
}

// Resolution 是人对干预的响应。
type Resolution struct {
	InterventionID string
	ActionID       string            // 必须来自 AllowedActions
	Decision       approvalmodel.Decision // tool_approval 时映射为 approved/denied/abort + scope
	Params         map[string]any    // param_input 时的补充参数
	ResolvedBy     string
	ResolvedAt     time.Time
}
```

> 说明：`tool_approval` 复用现有 `approval.Request/Decision`，避免推翻已工作的策略与审计；`InterventionRequest` 是外层信封，让"审批"成为"干预"的一个子类型。

### 3.3 InterventionBroker port

把现有 `Requester` 泛化为 Broker，并明确"挂起"信号：

```go
type InterventionBroker interface {
	// Await 声明一次人工干预。
	//  - 同步实现：阻塞直到 Resolution 返回或 ctx 取消。
	//  - 挂起实现：持久化请求后返回 (Resolution{}, ErrInterventionRequired)，
	//    由上层将 Run 置为 paused；恢复时通过 Resolve 唤醒并重放。
	Await(ctx context.Context, req InterventionRequest) (Resolution, error)
}

// Resolver 供 REST/CLI 侧回写决策（挂起实现需要）。
type Resolver interface {
	Resolve(ctx context.Context, res Resolution) error
}
```

`approval.Service.Authorize` 在 `ask` 分支改为：构造 `InterventionRequest{Type: tool_approval, Approval: &req}` → `broker.Await(...)`；对 `ErrInterventionRequired` **向上透传**（不再当场判 denied）。

### 3.4 两种 Broker 实现

| 实现 | 适用 | 机制 |
|---|---|---|
| `InProcessBroker` | CLI；单进程 + 长连接 SSE | 生成 `intervention_id`，登记到内存 `map[id]chan Resolution`；`Await` 阻塞 `select{ctx.Done / <-ch}`；`Resolve` 按 id 写 channel。CLI TUI/终端与 SSE-live 都用它，UI 差异下沉到"如何把请求展示给人 + 如何调用 Resolve"。 |
| `DurableBroker` | Enterprise；长任务；需跨断连/重启 | `Await` 把 `InterventionRequest` 写入 `InterventionStore` 并生成 `resume_token`，随即返回 `ErrInterventionRequired`；Run 引擎捕获后落盘 Run 状态=`paused` 并结束当前执行栈；`Resolve`（REST 端点触发）写入 `Resolution` 并唤醒/重新调度 Run，从断点重放该工具调用。 |

> 关键：`InProcessBroker` 保留当前 CLI 体验（几乎零改造），`DurableBroker` 解决企业/长任务问题。二者对 `approval.Service` 完全透明。

### 3.5 Run 状态机与恢复

- `RunStatus` 增加 `paused`（或细分 `waiting_intervention`）。
- Run 引擎在工具执行返回 `ErrInterventionRequired` 时：持久化 Run 快照（含 ReAct 上下文、Repeat Guard、待重放的 tool call）、发 `run.paused`、结束当前 goroutine。
- `Resolve` 成功后：校验 `resume_token`/权限 → 写 `Resolution` → 将 Run 重新入队 → 从断点重放被拦截的工具调用（此时 approval 命中 grant 或直接使用 Resolution 的 decision）→ 发 `run.resumed`。
- 启用已存在的 `ErrRunPaused / ErrInterventionRequired / ErrRunAlreadyResuming`（消除死代码）。

### 3.6 传输与 API（对齐 SSE 设计文档）

严格复用 `docs/agent loop设计-SSE与重试策略设计.md` 已定义的语义，不再另起炉灶：

- SSE 事件：`intervention.required` → `run.paused`；`Resolve` 后 `block.stop(resolution)` → `run.resumed`；超时 `intervention.expired`；取消 `intervention.cancelled`。
- REST 恢复端点：`POST /v1/runs/{run_id}/interventions/{intervention_id}/respond`，携带 `action_id` + `resume_token`（+ `params`）。
- 幂等：重复 respond 返回 409；`resume_token` 失效返回 410/401。

### 3.7 Abort 与超时语义

- `abort` 动作 → 上层将 Run 置为 `cancelled`（而非仅工具失败）。
- `DurableBroker` 到 `ExpiresAt` 未响应 → 产生 `DecisionTimedOut`，按可配置默认策略（deny / 继续 / 失败）处理，发 `intervention.expired`。

## 四、修正后的关键流程对比

**同步（CLI）——体验不变，只换抽象名：**

```text
Tool → Authorize → ask → broker.Await → InProcessBroker 阻塞 → 用户按键 Resolve → Decision 返回 → 工具继续
```

**异步（Enterprise / 长任务）——新增能力：**

```text
Tool → Authorize → ask → broker.Await → DurableBroker 持久化 + 返回 ErrInterventionRequired
      → Run 引擎落盘 paused + emit intervention.required / run.paused（连接可断开）
── 用户稍后（可换设备/重连）POST /v1/runs/{id}/interventions/{id}/respond ──
      → Resolve 写决策 + 唤醒 Run → 重放工具调用 → emit run.resumed → 继续
```

## 五、迁移路线（分阶段，不破坏 CLI）

### Phase 0：抽象归一（低风险，先行）
- 新增 `InterventionRequest/Resolution/InterventionBroker` 模型与 port。
- `approval.Service` 的 `ask` 分支由 `Requester.RequestApproval` 改为 `broker.Await`（`tool_approval` 子类型）。
- 提供 `RequesterBrokerAdapter`：把现有 CLI `Requester` 包装成 `InProcessBroker`，保证 CLI 零行为变化。
- 验收：CLI 审批交互与现状完全一致；单测覆盖 allow/deny/ask/grant 缓存。

### Phase 1：消除 Enterprise 反模式（高优先级）
- 移除 `headlessAskApprover` 的"自动批准"，替换为：要么接 `DurableBroker`（推荐），要么在未配置审批通道时对 `ask` **默认 deny**（安全兜底）并记审计 warning。
- 验收：Enterprise 下 `ask` 不再被静默放行；审计如实记录 deny/pending。

### Phase 2：Run 暂停/恢复状态机 + 持久化
- `RunStatus` 增加 `paused`；实现 `InterventionStore` + Run 快照持久化 + `resume_token`。
- 实现 `DurableBroker`（Await 挂起、Resolve 唤醒重放）。
- 启用 `ErrRunPaused/ErrInterventionRequired/ErrRunAlreadyResuming`、`DecisionTimedOut`、`TaskStatusWaitingApproval`。
- 验收：进程重启后 pending 干预可恢复；断连重连后可 respond。

### Phase 3：SSE + REST 干预通道
- 按 SSE 设计文档实现 `intervention.*`、`run.paused/resumed` 事件与 `/interventions/{id}/respond` 端点。
- 超时 `intervention.expired`、取消 `intervention.cancelled`、幂等与权限校验。
- 验收：Web 端可视化审批闭环；`resume_token`/`action_id` 校验有测试。

### Phase 4：扩展干预类型与 abort
- 支持 `manual_override`（对齐重试策略 `on_final_failure=intervention`）、`param_input`。
- `abort` 打通为 Run `cancelled`。
- 验收：多次失败后可人工接管/改参；中止任务真正终止 Run。

## 七、验证要点

实现修正后至少覆盖：

- Phase 0 后 CLI 审批行为与改造前逐项一致（allow/deny/ask/session grant/abort）。
- Enterprise `ask` 不再被自动批准；未配置通道时 deny 且审计 warning。
- `DurableBroker`：Await 返回 `ErrInterventionRequired` → Run 落盘 `paused` → 重启进程后仍 pending → Resolve 后重放工具调用并 `run.resumed`。
- 客户端断开重连后仍能对 pending 干预 respond；重复 respond 幂等（409）。
- `resume_token` 伪造/过期被拒；无权限用户不能 respond。
- `abort` → Run `cancelled`；超时 → `DecisionTimedOut` + `intervention.expired` + 默认策略生效。
- 审计 `approval.decision` / 干预记录如实反映 pending / resolved / expired，不把 pending 记为 approved。
