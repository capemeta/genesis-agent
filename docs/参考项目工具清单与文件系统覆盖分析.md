## 四、Codex 可确认的工具

### 4.1 与 Kode 的关键差异

Codex **没有** Kode 式的 `Read` / `Write` / `LS` / `Glob` / `Grep` 等专用文件系统工具。文件相关能力主要由以下组合承担：

- `apply_patch`：结构化 patch 编辑
- `exec_command` / `shell_command`：通过 shell 间接读写
- `view_image`：图片查看

### 4.3 多 Agent 工具（v1 / v2 互斥）

两套实现由 `multi_agent_version` 配置切换，**不是同一批工具同时可用**。v1 工具使用 namespace `multi_agent_v1.*`；v2 默认 plain name，也可按配置再加 namespace。

Genesis 不采用 Codex 的 `spawn_agent` / `send_message` 等拆分工具名，而是用固定三件套 **`Task` / `TaskOutput` / `TaskStop`** 覆盖委派、取结果、取消。

## 六、结论

1. **Kode-CLI 与 Codex 对照完整性**
   Kode / Codex 工具枚举仍完整保留。Genesis 侧已按自身契约落地：显式 FS 工具集、统一 `run_command`/`write_stdin`、plan 三件套（`todo_*`）、Skill 固定网关、MCP 动态工具、SubAgent 三件套（`Task`/`TaskOutput`/`TaskStop`）。

2. **相对上一版文档的主要校正**
   - `Task` / `TaskOutput` / `TaskStop` 已实现，且归属 **subagent**，不是 `execution/session_manager` 的 LLM 工具别名。
   - `WebSearch` / `WebFetch`、MCP list/read/search 与 `mcp__*` 已从「待实现」转为「已实现」（受 Profile / `MCP.Enabled` 约束）。
   - `EnterPlanMode`/`ExitPlanMode` 仅为设计决策（Runtime 限工具），**专用模式闸门尚未落地** → ◐。
   - `Think` 无独立工具 → ❌。
   - `write_file` 新鲜度为 **sha256**，不是 MD5。
   - 记忆为运行时存储接线，**无 Codex 式 `memories.*` 工具** → ◐。

3. **仍明确缺口**
   LSP、NotebookEdit、view_image、AskUserQuestion / request_user_input、clock.sleep、插件市场安装、goal、code-mode、Remove/Copy/Rename 专用工具、MCP resource templates、多 Agent 细粒度消息（send_input/send_message/list_agents）等仍未做或仅部分覆盖。

4. **安全与隔离边界**
   仍遵循 AGENTS.md：文件系统工具经 `FileSystemBackend` / `PathResolver` / Approval；命令执行经 `CommandRunner` / `SandboxRunner`，不在 tool 内直接 `os`/`exec.Command`；genesis-sandbox client 与产品默认策略分离装配。
