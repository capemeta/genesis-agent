## 五、Genesis 文件系统相关覆盖分析

| 能力 | 参考项目对应 | Genesis 当前代码 | 文件系统方案覆盖 | 是否缺口 | 建议 |
|---|---|---|---|---:|---|
| 读文件 | Kode `Read` | 未实现 | `read_file` | 否 | 第一轮落地核心工具 |
| 写文件 | Kode `Write` | 未实现 | `write_file` | 否 | 必须带 `ExpectedHash`、原子写和覆盖策略 |
| 编辑文件 | Kode `Edit`、Codex `apply_patch` | 未实现 | `edit_file` | 部分缺口 | `edit_file` 覆盖基础编辑；建议后续单独设计 `apply_patch` 或 patch mode |
| 列目录 | Kode `LS` | 未实现 | `list_dir` | 否 | 需要 bounded output 和权限检查 |
| 遍历目录 | Codex bounded walk | 未实现 | `walk_dir` | 否 | 必须限制 depth/entries/bytes/symlink |
| Glob 文件匹配 | Kode `Glob` | 未实现 | 未作为第一轮核心工具 | 是 | 建议后续放 `internal/capabilities/search/tool/glob` 或 filesystem search 子能力，基于 backend/walk 实现 |
| Grep 内容搜索 | Kode `Grep` | 未实现 | 未作为第一轮核心工具 | 是 | 建议后续放 `search/grep`，不要直接在 tool 内 shell out 到 rg；本地/远程由 backend 或 runner 适配 |
| Bash/命令执行 | Kode `Bash`、Codex `exec_command`/`shell_command` | 未实现 | `execution/tool/bash` | 否 | 不属于 filesystem；需要独立 execution port、sandbox、权限和并发设计 |
| 持续命令 stdin | Codex `write_stdin` | 未实现 | 未覆盖 | 是 | 后续 execution 阶段补 `write_stdin`/PTY/session output |
| 后台任务输出/停止 | Kode `TaskOutput`/`TaskStop` | 未实现 | 未覆盖 | 是 | 后续 execution task runtime，不属于文件系统核心 |
| Notebook 编辑 | Kode `NotebookEdit` | 未实现 | 未覆盖 | 是 | 后续作为专用文档/Notebook 能力，不进入第一轮通用 FS |
| 图片查看 | Codex `view_image` | 未实现 | 未覆盖 | 是 | 后续 media inspection 工具；依赖 FileSystemBackend 读 bytes，但不属于基础 FS 写读工具 |
| Patch 应用 | Codex `apply_patch` | 未实现 | 部分由 `edit_file` 承接 | 是 | 代码 Agent 场景建议优先补；语义应包含 diff 校验、上下文匹配、原子写和审计 |

| 文件新鲜度 | Kode freshness | 未实现 | `FreshnessTracker` | 否 | 防止覆盖外部修改，是写工具前置能力 |
| 运行时授权 | Kode runtime dirs、Codex approvals | 未实现 | `RuntimeFilePermissions` + PermissionEngine | 否 | session/run scoped，不使用全局变量 |

| 沙箱文件后端 | Codex sandbox FS、genesis-sandbox 目标 | 未实现 | sandbox FileSystemBackend adapter | 否 | CLI/Desktop/Enterprise 都可选择注入，不是 Enterprise 专属 |
| Docker/genesis-sandbox | Codex remote/sandbox 思路 | 未实现 | 已设计为 CLI/Desktop 可选、Enterprise 默认外部 sandbox | 否 | 通过 sandbox client adapter 注入，不泄漏宿主机路径 |
| Remove/Copy/Rename | 常见文件工具 | 未实现 | 明确第一轮不做 | 否 | 高风险操作，等权限/审计/人审稳定后单独设计 |

## 六、结论

| 问题 | 结论 |
|---|---|
| 是否缺 bash 命令工具 | 不算文件系统方案缺口；它应作为 `execution/tool/bash` 独立落地，与文件系统共享 PathResolver、权限、锁和 sandbox policy |
| 文件系统第一轮是否覆盖核心能力 | 覆盖了 read/write/edit/list/walk、路径解析、权限、新鲜度、并发、sandbox backend 边界 |
| 最值得补充的文件相关能力 | `glob`、`grep`、`apply_patch`、`view_image`、Notebook 编辑、持续命令 stdin/输出订阅 |
| 最优先的后续补充 | 对代码 Agent 来说，优先级建议是 `apply_patch` > `grep` > `glob` > `view_image` > Notebook；`write_stdin` 属于 execution 阶段 |
| 需要守住的边界 | 文件工具只依赖 `FileSystemBackend`、`PathResolver`、`FreshnessTracker`、权限/审计 port；命令工具只依赖 `CommandRunner` / `SandboxRunner`，产品实现由 bootstrap 注入 |
